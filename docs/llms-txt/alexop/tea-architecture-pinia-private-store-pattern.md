Enter The Elm Architecture (TEA) combined with the \&quot;private store\&quot; technique from [Mastering Pinia](https://masteringpinia.com/blog/how-to-create-private-state-in-stores) by Eduardo San Martin Morote (creator of Pinia).\n\nThis hybrid approach gives you:\n\n- **Pure, testable business logic** that&#39;s framework-agnostic\n- **Controlled state mutations** through a single dispatch function\n- **Seamless Vue integration** with Pinia&#39;s reactivity\n- **Full devtools support** for debugging\n\nYou&#39;ll use a private internal store for mutable state, expose only selectors and a dispatch function publicly, and keep your update logic pure and framework-agnostic.\n\n&lt;Aside type=\&quot;tip\&quot; title=\&quot;When Should You Use This Pattern?\&quot;&gt;\n  This pattern shines when you have complex business logic, need framework\n  portability, or want rock-solid testing. For simple CRUD operations with\n  minimal logic, traditional Pinia stores are perfectly fine. Ask yourself:\n  \&quot;Would I benefit from testing this logic in complete isolation?\&quot; If yes, this\n  pattern is worth it.\n&lt;/Aside&gt;\n\n&lt;Aside type=\&quot;info\&quot; title=\&quot;Historical Context\&quot;&gt;\n  The Elm Architecture emerged from the [Elm programming\n  language](https://guide.elm-lang.org/architecture/), which pioneered a purely\n  functional approach to building web applications. This pattern later inspired\n  Redux&#39;s architecture in the JavaScript ecosystem, demonstrating the value of\n  unidirectional data flow and immutable updates. This makes them trivial to test without any framework dependencies.\n\n## How It Works: Combining TEA with Private State\n\nThe pattern uses three key pieces: a private internal store for mutable state, pure update functions for business logic, and a public store that exposes only selectors and dispatch.\n\n### The Private Internal Store\n\nFirst, create a private store that holds the mutable model. This stays in the same file as your public store but is not exported:\n\n```ts\n// Inside stores/todos.ts - NOT exported!\nconst useTodosPrivate = defineStore(\&quot;todos-private\&quot;, () =&gt; {\n  const model = ref&lt;TodosModel&gt;({\n    todos: [],\n  });\n\n  return { model };\n});\n```\n\nThe key here: no `export` keyword means components can&#39;t access this directly.\n\n### Pure Update Function\n\nNext, define your business logic as pure functions:\n\n&lt;Aside type=\&quot;info\&quot; title=\&quot;What Are Pure Functions?\&quot;&gt;\n  A pure function always returns the same output for the same inputs and has no\n  side effects. No API calls, no mutations of external state, no `console.log`.\n  Just input → transformation → output. This makes them trivially easy to test\n  and reason about: `update(state, action)` always produces the same new state.\n&lt;/Aside&gt;\n\n```ts\n// stores/todos-update.ts\nimport type { TodosModel, TodosMessage } from \&quot;./todos-model\&quot;;\n\nexport function update(model: TodosModel, message: TodosMessage): TodosModel {\n  switch (message.type) {\n    case \&quot;ADD_TODO\&quot;:\n      return {\n        ...model,\n        todos: [\n          ...model.todos,\n          { id: Date.now(), text: message.text, done: false },\n        ],\n      };\n\n    case \&quot;TOGGLE_TODO\&quot;:\n      return {\n        ...model,\n        todos: model.todos.map(todo =&gt;\n          todo.id === message.id ? { ...todo, done: !todo.done } : todo\n        ),\n      };\n\n    default:\n      return model;\n  }\n}\n```\n\nThis update function is completely framework-agnostic. You can test it with simple assertions:\n\n```ts\nimport { describe, it, expect } from \&quot;vitest\&quot;;\nimport { update } from \&quot;./todos-update\&quot;;\n\ndescribe(\&quot;update\&quot;, () =&gt; {\n  it(\&quot;adds a todo\&quot;, () =&gt; {\n    const initial = { todos: [] };\n    const result = update(initial, { type: \&quot;ADD_TODO\&quot;, text: \&quot;Test\&quot; });\n\n    expect(result.todos).toHaveLength(1);\n    expect(result.todos[0].text).toBe(\&quot;Test\&quot;);\n  });\n});\n```\n\n&lt;Aside type=\&quot;info\&quot; title=\&quot;Familiar with Redux?\&quot;&gt;\n  If you&#39;ve used Redux, this pattern will feel familiar—the `update` function is\n  like a reducer, and `TodosMessage` is like an action. The key difference?\n  We&#39;re using Pinia&#39;s reactivity instead of Redux&#39;s subscription model, and\n  we&#39;re keeping the private store pattern to prevent direct state access. This\n  gives you Redux&#39;s testability with Pinia&#39;s developer experience.\n&lt;/Aside&gt;\n\n### Public Store with Selectors + Dispatch\n\nFinally, combine everything in a single file. The private store is defined but not exported:\n\n```ts\n// stores/todos.ts (this is what components import)\nimport { defineStore } from \&quot;pinia\&quot;;\nimport { ref, computed } from \&quot;vue\&quot;;\nimport { update } from \&quot;./todos-update\&quot;;\nimport type { TodosModel, TodosMessage } from \&quot;./todos-model\&quot;;\n\n// Private store - not exported!\nconst useTodosPrivate = defineStore(\&quot;todos-private\&quot;, () =&gt; {\n  const model = ref&lt;TodosModel&gt;({\n    todos: [],\n  });\n\n  return { model };\n});\n\n// Public store - this is what gets exported\nexport const useTodosStore = defineStore(\&quot;todos\&quot;, () =&gt; {\n  const privateStore = useTodosPrivate();\n\n  // Selectors\n  const todos = computed(() =&gt; privateStore.model.todos);\n\n  // Dispatch\n  function dispatch(message: TodosMessage) {\n    privateStore.model = update(privateStore.model, message);\n  }\n\n  return { todos, dispatch };\n});\n```\n\n```mermaid\ngraph LR\n    Component[Component]\n    Component --&gt;|dispatch message| Public[Public Store]\n    Public --&gt;|call| Update[Update Function&lt;br/&gt;Pure Logic]\n    Update --&gt;|new state| Private[Private Store]\n    Private --&gt;|selectors| Public\n    Public --&gt;|reactive data| Component\n```\n\n&lt;Aside type=\&quot;caution\&quot; title=\&quot;️Private Store Limitation\&quot;&gt;\nReddit user [maertensen](https://www.reddit.com/user/maertensen/) helpfully pointed out that the private store pattern only prevents direct mutation of primitives, **not arrays and objects**. Components can still mutate through selectors:\n\n```vue\n&lt;script setup lang=\&quot;ts\&quot;&gt;\nimport { useTodosStore } from \&quot;./todos.ts\&quot;;\n\nconst publicTodosStore = useTodosStore();\n\nfunction mutateTodosSelector() {\n  // This still works and bypasses dispatch! ✗\n  publicTodosStore.todos.push({\n    id: Date.now(),\n    text: \&quot;Mutated by selector\&quot;,\n    done: false,\n  });\n}\n&lt;/script&gt;\n```\n\nThis is why I recommend using **`readonly` only** (shown below) instead of the private store pattern.\n\n&lt;/Aside&gt;\n\n### Usage in Components\n\nComponents interact with the public store:\n\n```vue\n&lt;script setup lang=\&quot;ts\&quot;&gt;\nimport { useTodosStore } from \&quot;@/stores/todos\&quot;;\n\nconst store = useTodosStore();\n&lt;/script&gt;\n\n&lt;template&gt;\n  &lt;div&gt;\n    &lt;input\n      @keyup.enter=\&quot;\n        store.dispatch({ type: &#39;ADD_TODO&#39;, text: $event.target.value })\n      \&quot;\n    /&gt;\n\n    &lt;div v-for=\&quot;todo in store.todos\&quot; :key=\&quot;todo.id\&quot;&gt;\n      &lt;input\n        type=\&quot;checkbox\&quot;\n        :checked=\&quot;todo.done\&quot;\n        @change=\&quot;store.dispatch({ type: &#39;TOGGLE_TODO&#39;, id: todo.id })\&quot;\n      /&gt;\n      {{ todo.text }}\n    &lt;/div&gt;\n  &lt;/div&gt;\n&lt;/template&gt;\n```\n\n## Simpler Alternative: Using Vue&#39;s readonly\n\nIf you want to prevent direct state mutations without creating a private store, Vue&#39;s `readonly` utility provides a simpler approach:\n\n```ts\n// stores/todos.ts\nimport { defineStore } from \&quot;pinia\&quot;;\nimport { ref, readonly } from \&quot;vue\&quot;;\nimport { update } from \&quot;./todos-update\&quot;;\nimport type { TodosModel, TodosMessage } from \&quot;./todos-model\&quot;;\n\nexport const useTodosStore = defineStore(\&quot;todos\&quot;, () =&gt; {\n  const model = ref&lt;TodosModel&gt;({\n    todos: [],\n  });\n\n  // Dispatch\n  function dispatch(message: TodosMessage) {\n    model.value = update(model.value, message);\n  }\n\n  // Only expose readonly state\n  return {\n    todos: readonly(model),\n    dispatch,\n  };\n});\n```\n\nWith `readonly`, any attempt to mutate the state from a component will fail:\n\n```vue\n&lt;script setup lang=\&quot;ts\&quot;&gt;\nconst store = useTodosStore();\n\n// ✓ Works - using dispatch\nstore.dispatch({ type: \&quot;ADD_TODO\&quot;, text: \&quot;Learn Vue\&quot; });\n\n// ✓ Works - accessing readonly state\nconst todos = store.model.todos;\n\n// ✗ TypeScript error - readonly prevents mutation\nstore.model.todos.push({ id: 1, text: \&quot;Hack\&quot;, done: false });\n&lt;/script&gt;\n```\n\n&lt;Aside type=\&quot;tip\&quot; title=\&quot;Use readonly, Not Private Store\&quot;&gt;\n  **Prefer `readonly` over the private store pattern.** The private store\n  pattern has a critical flaw: it doesn&#39;t prevent mutation of arrays and objects\n  (see warning above). Using `readonly` is simpler, more effective, and truly\n  prevents all direct state mutations.\n&lt;/Aside&gt;\n\n## Benefits of This Approach\n\n1. **Pure business logic**: The `update` function has zero dependencies on Vue or Pinia\n2. **Easy testing**: Test your update function with simple unit tests\n3. **Framework flexibility**: Could swap Vue for React without changing update logic\n4. **Type safety**: TypeScript ensures message types are correct\n5. **Devtools support**: Still works with Pinia devtools since we&#39;re using real stores\n6. **Encapsulation**: Private store is an implementation detail\n\n```mermaid\ngraph TB\n    subgraph T[\&quot;Traditional Pinia\&quot;]\n        TC[Component]\n        TC --&gt;|direct| TS[State]\n        TC --&gt;|actions| TA[Actions]\n        TA --&gt; <h2 id="the-solution-tea--private-store-pattern">The Solution: TEA + Private Store Pattern<a class="heading-link" aria-label="Link to section" href="#the-solution-tea--private-store-pattern"><span class="heading-link-icon">#</span></a></h2>
<p>What if we could keep Pinia’s excellent developer experience while adding the predictability and testability of functional patterns? Enter The Elm Architecture (TEA) combined with the “private store” technique from <a href="https://masteringpinia.com/blog/how-to-create-private-state-in-stores" rel="noopener noreferrer" target="_blank">Mastering Pinia</a> by Eduardo San Martin Morote (creator of Pinia).</p>
<p>This hybrid approach gives you:</p>
<ul>
<li><strong>Pure, testable business logic</strong> that’s framework-agnostic</li>
<li><strong>Controlled state mutations</strong> through a single dispatch function</li>
<li><strong>Seamless Vue integration</strong> with Pinia’s reactivity</li>
<li><strong>Full devtools support</strong> for debugging</li>
</ul>
<p>You’ll use a private internal store for mutable state, expose only selectors and a dispatch function publicly, and keep your update logic pure and framework-agnostic.</p>
<aside aria-label="When Should You Use This Pattern?" class="aside aside-tip astro-37uy2q7c"> <div class="aside-content astro-37uy2q7c"> <p class="aside-title astro-37uy2q7c"> <span class="aside-emoji astro-37uy2q7c" role="img" aria-hidden="true">💡</span> When Should You Use This Pattern? </p> <section class="aside-body astro-37uy2q7c"> <p>This pattern shines when you have complex business logic, need framework
portability, or want rock-solid testing. For simple CRUD operations with
minimal logic, traditional Pinia stores are perfectly fine. Ask yourself:
“Would I benefit from testing this logic in complete isolation?” If yes, this
pattern is worth it.</p> </section> </div> </aside> 
<aside aria-label="Historical Context" class="aside aside-info astro-37uy2q7c"> <div class="aside-content astro-37uy2q7c"> <p class="aside-title astro-37uy2q7c"> <span class="aside-emoji astro-37uy2q7c" role="img" aria-hidden="true">ℹ️</span> Historical Context </p> <section class="aside-body astro-37uy2q7c"> <p>The Elm Architecture emerged from the <a target="_blank" rel="noopener noreferrer" href="https://guide.elm-lang.org/architecture/" rel="noopener noreferrer" target="_blank">Elm programming
language</a>, which pioneered a purely
functional approach to building web applications. This pattern later inspired
Redux’s architecture in the JavaScript ecosystem, demonstrating the value of
unidirectional data flow and immutable updates. While Elm enforces these
patterns through its type system, we can achieve similar benefits in Vue with
disciplined patterns.</p> </section> </div> </aside>
