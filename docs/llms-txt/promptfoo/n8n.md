markdown"><div style="position:relative"><header><h1>Using Promptfoo in n8n Workflows</h1></header>
<p>This guide shows how to run Promptfoo evaluations from an <strong>n8n</strong> workflow so you can:</p>
<ul>
<li class="">schedule nightly or ad‑hoc LLM tests,</li>
<li class="">gate downstream steps (Slack/Teams alerts, merge approvals, etc.) on pass‑rates, and</li>
<li class="">publish rich results links generated by Promptfoo.</li>
</ul>
<h2 class="anchor anchorTargetStickyNavbar_tleR" id="prerequisites">Prerequisites<a href="#prerequisites" class="hash-link" aria-label="Direct link to Prerequisites" title="Direct link to Prerequisites" translate="no">​</a></h2>
<table><thead><tr><th>What</th><th>Why</th></tr></thead><tbody><tr><td><strong>Self‑hosted n8n ≥ v1</strong> (Docker or bare‑metal)</td><td>Gives access to the “Execute Command” node.</td></tr><tr><td><strong>Promptfoo CLI</strong> available in the container/host</td><td>Needed to run <code>promptfoo eval</code>.</td></tr><tr><td>(Optional) <strong>LLM provider API keys</strong> set as environment variables or n8n credentials</td><td>Example: <code>OPENAI_API_KEY</code>, <code>ANTHROPIC_API_KEY</code>, …</td></tr><tr><td>(Optional) <strong>Slack / email / GitHub nodes</strong> in the same workflow</td><td>For notifications or comments once the eval finishes.</td></tr></tbody></table>
<h3 class="anchor anchorTargetStickyNavbar_tleR" id="shipping-a-custom-docker-image-recommended">Shipping a custom Docker image (recommended)<a href="#shipping-a-custom-docker-image-recommended" class="hash-link" aria-label="Direct link to Shipping a custom Docker image (recommended)" title="Direct link to Shipping a custom Docker image (recommended)" translate="no">​</a></h3>

<h2 class="anchor anchorTargetStickyNavbar_tleR" id="basic-run--alert-workflow">Basic “Run &amp; Alert” workflow<a href="#basic-run--alert-workflow" class="hash-link" aria-label="Direct link to Basic “Run &amp; Alert” workflow" title="Direct link to Basic “Run &amp; Alert” workflow" translate="no">​</a></h2>
<p>Below is the minimal pattern most teams start with:</p>
<table><thead><tr><th>#</th><th>Node</th><th>Purpose</th></tr></thead><tbody><tr><td>1</td><td><strong>Trigger</strong> (Cron or Webhook)</td><td>Decide <em>when</em> to evaluate (nightly, on Git push webhook …).</td></tr><tr><td>2</td><td><strong>Execute Command</strong></td><td>Runs Promptfoo and emits raw stdout / stderr.</td></tr><tr><td>3</td><td><strong>Code / Set</strong> node</td><td>Parses the resulting JSON, extracts pass/fail counts &amp; share‑URL.</td></tr><tr><td>4</td><td><strong>IF</strong> node</td><td>Branches on “failures &gt; 0”.</td></tr><tr><td>5</td><td><strong>Slack / Email / GitHub</strong></td><td>Sends alert or PR comment when the gate fails.</td></tr></tbody></table>

<p>An <strong>IF</strong> node can then route execution:</p>
<ul>
<li class=""><strong>failures = 0</strong> → take <em>green</em> path (maybe just archive the results).</li>
<li class=""><strong>failures &gt; 0</strong> → post to Slack or comment on the pull request.</li>
</ul>
<h2 class="anchor anchorTargetStickyNavbar_tleR" id="advanced-patterns">Advanced patterns<a href="#advanced-patterns" class="hash-link" aria-label="Direct link to Advanced patterns" title="Direct link to Advanced patterns" translate="no">​</a></h2>
<h3 class="anchor anchorTargetStickyNavbar_tleR" id="run-different-configs-in-parallel">Run different configs in parallel<a href="#run-different-configs-in-parallel" class="hash-link" aria-label="Direct link to Run different configs in parallel" title="Direct link to Run different configs in parallel" translate="no">​</a></h3>
<p>Make the first <strong>Execute Command</strong> node loop over an array of model IDs or config
files and push each run as a separate item.
Downstream nodes will automatically fan‑out and handle each result independently.</p>
<h3 class="anchor anchorTargetStickyNavbar_tleR" id="versioncontrolled-prompts">Version‑controlled prompts<a href="#versioncontrolled-prompts" class="hash-link" aria-label="Direct link to Version‑controlled prompts" title="Direct link to Version‑controlled prompts" translate="no">​</a></h3>
<p>Mount your prompts directory and config file into the container at
<code>/data</code>. When you commit new prompts to Git, your CI/CD system can call the
<strong>n8n REST API</strong> or a <strong>Webhook trigger</strong> to re‑evaluate immediately.</p>
<h3 class="anchor anchorTargetStickyNavbar_tleR" id="autofail-the-whole-workflow">Auto‑fail the whole workflow<a href="#autofail-the-whole-workflow" class="hash-link" aria-label="Direct link to Auto‑fail the whole workflow" title="Direct link to Auto‑fail the whole workflow" translate="no">​</a></h3>
<p>If you run n8n <strong>headless</strong> via <code>n8n start --tunnel</code>, you can call this workflow
from CI pipelines (GitHub Actions, GitLab, …) with the <a href="https://docs.n8n.io/hosting/cli-commands/" target="_blank" rel="noopener noreferrer" class="">CLI <code>n8n execute</code>
command</a> and then check the HTTP
response code; returning <code>exit 1</code> from the Execute Command node will propagate
the failure.</p>
<h2 class="anchor anchorTargetStickyNavbar_tleR" id="security--best-practices">Security &amp; best practices<a href="#security--best-practices" class="hash-link" aria-label="Direct link to Security &amp; best practices" title="Direct link to Security &amp; best practices" translate="no">​</a></h2>
<ul>
<li class=""><strong>Keep API keys secret</strong> – store them in the n8n credential store or inject as
environment variables from Docker secrets, not hard‑coded in workflows.</li>
<li class=""><strong>Resource usage</strong> – Promptfoo supports caching via
<code>PROMPTFOO_CACHE_PATH</code>; mount that directory to persist across runs.</li>
<li class=""><strong>Timeouts</strong> – wrap <code>promptfoo eval</code> with <code>timeout --signal=SIGKILL 15m …</code>
(Linux) if you need hard execution limits.</li>
<li class=""><strong>Logging</strong> – route the <code>stderr</code> field of Execute Command to a dedicated log
channel so you don’t miss stack traces.</li>
</ul>
<h2 class="anchor anchorTargetStickyNavbar_tleR" id="troubleshooting">Troubleshooting<a href="#troubleshooting" class="hash-link" aria-label="Direct link to Troubleshooting" title="Direct link to Troubleshooting" translate="no">​</a></h2>
<table><thead><tr><th>Symptom</th><th>Likely cause / fix</th></tr></thead><tbody><tr><td><strong><code>Execute Command node not available</code></strong></td><td>You’re on n8n Cloud; switch to self‑hosted.</td></tr><tr><td><strong><code>promptfoo: command not found</code></strong></td><td>Promptfoo not installed inside the container. Rebuild your Docker image or add an install step.</td></tr><tr><td><strong>Run fails with <code>ENOENT</code> on config paths</strong></td><td>Make sure the prompts/config volume is mounted at the same path you reference in the command.</td></tr><tr><td><strong>Large evals time‑out</strong></td><td>Increase the node’s “Timeout (s)” setting <em>or</em> chunk your test cases and iterate inside the workflow.</td></tr></tbody></table>
