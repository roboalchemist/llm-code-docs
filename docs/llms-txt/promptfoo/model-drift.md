style="position:relative"><header><h1>Detecting Model Drift with Red Teaming</h1></header>
<p>Model drift occurs when an LLM&#x27;s behavior changes over time. This can happen due to provider model updates, fine-tuning changes, prompt modifications, or guardrail adjustments. From a security perspective, drift can mean your model becomes more vulnerable to attacks that previously failed—or that previously working attacks no longer succeed.</p>
<p>Red teaming provides a systematic way to detect these changes by running consistent adversarial tests over time and comparing results.</p>

<h2 class="anchor anchorTargetStickyNavbar_tleR" id="why-red-team-for-drift-detection">Why Red Team for Drift Detection<a href="#why-red-team-for-drift-detection" class="hash-link" aria-label="Direct link to Why Red Team for Drift Detection" title="Direct link to Why Red Team for Drift Detection" translate="no">​</a></h2>
<p>Traditional monitoring captures production incidents after they occur. Red teaming with drift detection catches security regressions before they reach users:</p>
<ul>
<li class=""><strong>Quantifiable metrics</strong>: Attack Success Rate (ASR) provides a concrete measure of security posture</li>
<li class=""><strong>Consistent test coverage</strong>: The same attacks run against the same target reveal behavioral changes</li>
<li class=""><strong>Early warning</strong>: Detect weakened defenses before attackers exploit them</li>
<li class=""><strong>Compliance evidence</strong>: Demonstrate ongoing security testing for audits and regulatory requirements</li>
</ul>

<h2 class="anchor anchorTargetStickyNavbar_tleR" id="drift-detection-with-custom-tests">Drift Detection with Custom Tests<a href="#drift-detection-with-custom-tests" class="hash-link" aria-label="Direct link to Drift Detection with Custom Tests" title="Direct link to Drift Detection with Custom Tests" translate="no">​</a></h2>
<p>While generated red team tests cover a broad attack surface, custom test cases let you track specific behaviors that matter to your application. Use the standard eval configuration with <code>tests</code> to monitor:</p>
<ul>
<li class=""><strong>Refusal consistency</strong>: Does the model still refuse specific requests?</li>
<li class=""><strong>Output format compliance</strong>: Does the model follow your formatting requirements?</li>
<li class=""><strong>Factual accuracy</strong>: Does the model still provide correct information for known queries?</li>

<p><strong>Risk score trends</strong>: The <a class="" href="/docs/red-team/risk-scoring/">risk scoring</a> system provides severity-weighted metrics. A rising system risk score is a clear signal of drift.</p>

<table><thead><tr><th>Drift Type</th><th>Indicator</th><th>Likely Cause</th></tr></thead><tbody><tr><td>Security regression</td><td>ASR increases</td><td>Model update weakened safety training, guardrail disabled, prompt change</td></tr><tr><td>Security improvement</td><td>ASR decreases</td><td>Better guardrails, improved prompt, model update with stronger safety</td></tr><tr><td>Category-specific drift</td><td>Single category ASR changes</td><td>Targeted guardrail change, model fine-tuning on specific content</td></tr><tr><td>Volatility</td><td>ASR fluctuates between runs</td><td>Non-deterministic model behavior, rate limiting, infrastructure issues</td></tr></tbody></table>

<h2 class="anchor anchorTargetStickyNavbar_tleR" id="best-practices">Best Practices<a href="#best-practices" class="hash-link" aria-label="Direct link to Best Practices" title="Direct link to Best Practices" translate="no">​</a></h2>
<ol>
<li class=""><strong>Start with a baseline</strong>: Run a comprehensive scan before deploying, then track changes from that point</li>
<li class=""><strong>Use consistent test cases</strong>: Re-run existing tests for accurate drift detection; regenerate periodically for coverage</li>
<li class=""><strong>Automate with CI/CD</strong>: Manual drift detection doesn&#x27;t scale; schedule regular scans</li>
<li class=""><strong>Set actionable thresholds</strong>: Define clear pass/fail criteria tied to your risk tolerance</li>
<li class=""><strong>Version your configuration</strong>: Track red team config changes alongside code changes</li>
<li class=""><strong>Investigate anomalies</strong>: A sudden ASR change warrants investigation, whether up or down</li>
<li class=""><strong>Document your baseline</strong>: Record the initial ASR and risk score as your security baseline</li>
</ol>
