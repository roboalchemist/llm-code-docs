page</button></div><div class="theme-doc-markdown markdown"><div style="position:relative"><header><h1>Best-of-N (BoN) Jailbreaking Strategy</h1></header>
<p>Best-of-N (BoN) is a simple but effective black-box jailbreaking algorithm that works by repeatedly sampling variations of a prompt with modality-specific augmentations until a harmful response is elicited.</p>
<p>Introduced by <a href="https://arxiv.org/abs/2412.03556" target="_blank" rel="noopener noreferrer" class="">Hughes et al. (2024)</a>, it achieves high attack success rates across text, vision, and audio modalities.</p>

<p>BoN Jailbreaking works through a simple three-step process:</p>
<ol>
<li class="">
<p><strong>Generate Variations</strong>: Creates multiple versions of the input prompt using modality-specific augmentations:</p>
<ul>
<li class="">Text: Random capitalization, character scrambling, character noising</li>
<li class="">Vision: Font variations, background colors, text positioning</li>
<li class="">Audio: Speed, pitch, volume, background noise modifications</li>
</ul>
</li>
<li class="">
<p><strong>Concurrent Testing</strong>: Tests multiple variations simultaneously against the target model</p>
</li>
<li class="">
<p><strong>Success Detection</strong>: Monitors responses until a harmful output is detected or the maximum attempts are reached</p>
</li>
</ol>
<p>The strategy&#x27;s effectiveness comes from exploiting the stochastic nature of LLM outputs and their sensitivity to small input variations.</p>
<h2 class="anchor anchorTargetStickyNavbar_tleR" id="configuration-parameters">Configuration Parameters<a href="#configuration-parameters" class="hash-link" aria-label="Direct link to Configuration Parameters" title="Direct link to Configuration Parameters" translate="no">​</a></h2>
<h3 class="anchor anchorTargetStickyNavbar_tleR" id="usebasicrefusal">useBasicRefusal<a href="#usebasicrefusal" class="hash-link" aria-label="Direct link to useBasicRefusal" title="Direct link to useBasicRefusal" translate="no">​</a></h3>
<p><strong>Type:</strong> <code>boolean</code><br>
<strong>Default:</strong> <code>false</code></p>
<p>When enabled, uses a simple refusal check instead of LLM-as-a-judge assertions. This is much faster and cheaper than using an LLM judge, making it ideal for testing when the typical response of an LLM to a prompt is a refusal.</p>
<p>We recommend using this setting whenever possible if the default response to your original prompts is a &quot;Sorry, I can&#x27;t do that&quot;-style refusal.</p>
<h3 class="anchor anchorTargetStickyNavbar_tleR" id="maxconcurrency">maxConcurrency<a href="#maxconcurrency" class="hash-link" aria-label="Direct link to maxConcurrency" title="Direct link to maxConcurrency" translate="no">​</a></h3>
<p><strong>Type:</strong> <code>number</code><br>
<strong>Default:</strong> <code>3</code></p>
<p>Maximum number of prompt variations to test simultaneously. Higher values increase throughput. We recommend setting this as high as your rate limits allow.</p>
<h3 class="anchor anchorTargetStickyNavbar_tleR" id="nsteps">nSteps<a href="#nsteps" class="hash-link" aria-label="Direct link to nSteps" title="Direct link to nSteps" translate="no">​</a></h3>
<p><strong>Type:</strong> <code>number</code><br>
<strong>Default:</strong> <code>undefined</code></p>
<p>Maximum number of total attempts before giving up. Each step generates <code>maxCandidatesPerStep</code> variations. Higher values increase success rate but also cost. The original paper achieved best results with 10,000 steps.</p>
<h3 class="anchor anchorTargetStickyNavbar_tleR" id="maxcandidatesperstep">maxCandidatesPerStep<a href="#maxcandidatesperstep" class="hash-link" aria-label="Direct link to maxCandidatesPerStep" title="Direct link to maxCandidatesPerStep" translate="no">​</a></h3>
<p><strong>Type:</strong> <code>number</code><br>
<strong>Default:</strong> <code>1</code></p>
<p>Number of prompt variations to generate in each batch. Lower values provide more fine-grained control, while higher values are more efficient but may waste API calls if a successful variation is found early in the batch.</p>
<p>Usually best to set this to <code>1</code> and increase <code>nSteps</code> until you get a successful jailbreak.</p>

<h2 class="anchor anchorTargetStickyNavbar_tleR" id="performance">Performance<a href="#performance" class="hash-link" aria-label="Direct link to Performance" title="Direct link to Performance" translate="no">​</a></h2>
<p>BoN achieves impressive attack success rates across different models and modalities:</p>
<ul>
<li class="">Text: 89% on GPT-4, 78% on Claude 3.5 Sonnet (10,000 samples)</li>
<li class="">Vision: 56% on GPT-4 Vision</li>
<li class="">Audio: 72% on GPT-4 Audio</li>
</ul>
<p>The attack success rate follows a power-law scaling with the number of samples, meaning it reliably improves as more variations are tested. This illustrates why <a class="" href="/blog/asr-not-portable-metric/">ASR comparisons must account for attempt budget</a>: a 1% per-attempt method becomes 98% with 392 tries.</p>
<h2 class="anchor anchorTargetStickyNavbar_tleR" id="key-features">Key Features<a href="#key-features" class="hash-link" aria-label="Direct link to Key Features" title="Direct link to Key Features" translate="no">​</a></h2>
<ul>
<li class=""><strong>Simple Implementation</strong>: No need for gradients or model internals</li>
<li class=""><strong>Multi-modal Support</strong>: Works across text, vision, and audio inputs</li>
<li class=""><strong>Highly Parallelizable</strong>: Can test multiple variations concurrently</li>
<li class=""><strong>Predictable Scaling</strong>: Success rate follows power-law behavior</li>
