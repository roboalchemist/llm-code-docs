class="theme-doc-markdown markdown"><div style="position:relative"><header><h1>How to red team RAG applications</h1></header>
<p>Retrieval-Augmented Generation (RAG) is an increasingly popular LLM-based architecture for knowledge-based AI products. This guide focuses on application-layer attacks that developers deploying RAGs should consider.</p>
<p>For each attack type, we explain how to detect the vulnerability by testing your application with Promptfoo, an open-source LLM red teaming tool.</p>
<p>The end result is a report that looks like this:</p>
<p><img decoding="async" loading="lazy" alt="llm red team report for RAG" src="/assets/images/riskreport-1@2x-4c0fbea80c8816901144bc951702ed91.png" width="2540" height="1946" class="img_SS3x"></p>
<h2 class="anchor anchorTargetStickyNavbar_tleR" id="prompt-injection">Prompt Injection<a href="#prompt-injection" class="hash-link" aria-label="Direct link to Prompt Injection" title="Direct link to Prompt Injection" translate="no">​</a></h2>
<p>Prompt injection attacks involve manipulating the input to an LLM-based system to execute unauthorized actions or bypass security measures. In RAG systems, this can be particularly dangerous as it may lead to the retrieval and exposure of sensitive information.</p>
<p>Widely known since ChatGPT&#x27;s debut in <a href="https://simonwillison.net/2022/Sep/12/prompt-injection/" target="_blank" rel="noopener noreferrer" class="">2022</a>, prompt injections have become more difficult but new ones are still frequently discovered.</p>

<p>Context injection attacks are similar to prompt injection, but the malicious content is inserted into the retrieved context rather than the user input.</p>
<p>This can be particularly effective because many organization treat their knowledge base as trusted. Instructions may be in the system prompt and therefore privileged.</p>

<h2 class="anchor anchorTargetStickyNavbar_tleR" id="data-poisoning">Data Poisoning<a href="#data-poisoning" class="hash-link" aria-label="Direct link to Data Poisoning" title="Direct link to Data Poisoning" translate="no">​</a></h2>
<p>Data poisoning attacks target the retrieval component of RAG systems by introducing malicious or misleading information into the knowledge base. This can lead to the LLM generating incorrect or biased responses.</p>
<h4 class="anchor anchorTargetStickyNavbar_tleR" id="example-2">Example<a href="#example-2" class="hash-link" aria-label="Direct link to Example" title="Direct link to Example" translate="no">​</a></h4>
<p>An attacker gains access to a company&#x27;s customer feedback database used for RAG. They insert numerous fake reviews praising a competitor&#x27;s product, causing the RAG system to recommend the competitor when customers ask for product advice.</p>
<h4 class="anchor anchorTargetStickyNavbar_tleR" id="mitigations-2">Mitigations<a href="#mitigations-2" class="hash-link" aria-label="Direct link to Mitigations" title="Direct link to Mitigations" translate="no">​</a></h4>
<ol>
<li class="">Implement access controls and monitoring for data sources</li>
<li class="">Continuously test for failure modes and company policies.</li>
</ol>

<p>Data exfiltration in RAG systems involves attempts to extract sensitive information or personally identifiable information (PII) from the knowledge base or the model&#x27;s training data.</p>
<p>This can be particularly dangerous in RAG systems because they have access to large amounts of potentially sensitive information.</p>

<li class="">Implement output filtering to catch and redact potential PII or sensitive data</li>
</ol>

<h2 class="anchor anchorTargetStickyNavbar_tleR" id="dataset-manipulation">Dataset Manipulation<a href="#dataset-manipulation" class="hash-link" aria-label="Direct link to Dataset Manipulation" title="Direct link to Dataset Manipulation" translate="no">​</a></h2>
<p>Retrieval Manipulation attacks exploit the process by which documents are fetched in RAG systems, causing the system to retrieve desired information.</p>
<p>These can be achieved by in several ways, including by exploiting vulnerabilities in the retrieval algorithm (e.g. via the scoring mechanism), or by crafting inputs that trigger specific retrieval patterns.</p>
<h4 class="anchor anchorTargetStickyNavbar_tleR" id="example-4">Example<a href="#example-4" class="hash-link" aria-label="Direct link to Example" title="Direct link to Example" translate="no">​</a></h4>
<p>An attacker discovers that a company&#x27;s RAG-based legal advice system weighs recency heavily in its retrieval process. They flood the knowledge base, a public legal forum, with misleading interpretations of a new law, which the system then retrieves and uses to generate inaccurate legal advice.</p>
<h4 class="anchor anchorTargetStickyNavbar_tleR" id="mitigations-4">Mitigations<a href="#mitigations-4" class="hash-link" aria-label="Direct link to Mitigations" title="Direct link to Mitigations" translate="no">​</a></h4>
<ol>
<li class="">Implement diverse retrieval strategies (e.g., combining semantic and keyword-based search) that include credibility scoring</li>
<li class="">Use ensemble methods for retrieval to reduce the impact of manipulated sources</li>
<li class="">Establish routine testing procedures</li>
</ol>

<h2 class="anchor anchorTargetStickyNavbar_tleR" id="context-window-overflow">Context Window Overflow<a href="#context-window-overflow" class="hash-link" aria-label="Direct link to Context Window Overflow" title="Direct link to Context Window Overflow" translate="no">​</a></h2>
<p>Context Window Overflow attacks exploit the limited context window of LLMs by overloading it with irrelevant information, pushing out important context or instructions.</p>
<p>In RAG systems, this can cause the application to ignore system instructions and behave according to the whims of the attacker.</p>
<p>LLMs have a fixed-size context window, typically measured in tokens, which limits the amount of information they can process at once. In RAG applications, this window needs to accommodate:</p>
<ol>
<li class="">System instructions</li>
<li class="">Retrieved context</li>
<li class="">User input</li>
<li class="">Generated output</li>
</ol>
<p>Attackers can exploit this limitation by crafting inputs that consume a large portion of the context window, pushing out other information.</p>
<h4 class="anchor anchorTargetStickyNavbar_tleR" id="example-5">Example<a href="#example-5" class="hash-link" aria-label="Direct link to Example" title="Direct link to Example" translate="no">​</a></h4>
<p>An attacker inputs a long, seemingly relevant text to a financial advice RAG system, followed by a short query about stock recommendations:</p>
<div class="language-text codeBlockContainer_mQmQ theme-code-block" style="--prism-color:#393A34;--prism-background-color:#f6f8fa"><div class="codeBlockContent_t_Hd"><pre tabindex="0" class="prism-code language-text codeBlock_RMoD thin-scrollbar" style="color:#393A34;background-color:#f6f8fa"><code class="codeBlockLines_AclH"><span class="token-line" style="color:#393A34"><span class="token plain">[A very long, detailed description of various financial concepts and market trends, consuming most of the context window...]</span><br></span><span class="token-line" style="color:#393A34"><span class="token plain" style="display:inline-block"></span><br></span><span class="token-line" style="color:#393A34"><span class="token plain">Given this comprehensive market analysis, what stocks do you recommend I buy right now for maximum profit?</span><br></span></code></pre></div></div>
<p>This attack aims to push out the system&#x27;s risk warnings from the context window, potentially producing unfounded stock advice.</p>
<h4 class="anchor anchorTargetStickyNavbar_tleR" id="mitigations-5">Mitigations<a href="#mitigations-5" class="hash-link" aria-label="Direct link to Mitigations" title="Direct link to Mitigations" translate="no">​</a></h4>
<ol>
<li class="">
<p>Implement strict input length limits</p>
<ul>
<li class="">Set a maximum token count for user inputs</li>
<li class="">Truncate or reject inputs exceeding the limit</li>
</ul>
</li>
<li class="">
<p>Use dynamic context window management techniques that prioritize instructions and critical context</p>
<ul>
<li class="">Prioritize recent and relevant information</li>
<li class="">Implement sliding window approaches for long conversations</li>
</ul>
</li>
<li class="">
<p>Implement multi-stage processing for long inputs</p>
<ul>
<li class="">Break down long inputs into smaller chunks</li>
<li class="">Process each chunk separately and combine the results</li>
</ul>
</li>
</ol>
