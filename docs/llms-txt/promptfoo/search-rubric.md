class="theme-doc-markdown markdown"><div style="position:relative"><header><h1>Search-Rubric</h1></header>
<p>The <code>search-rubric</code> assertion type is like <code>llm-rubric</code> but with web search capabilities. It evaluates outputs according to a rubric while having the ability to search for current information when needed.</p>
<h2 class="anchor anchorTargetStickyNavbar_tleR" id="how-it-works">How it works<a href="#how-it-works" class="hash-link" aria-label="Direct link to How it works" title="Direct link to How it works" translate="no">​</a></h2>
<ol>
<li class="">You provide a rubric that describes what the output should contain</li>
<li class="">The grading provider evaluates the output against the rubric</li>
<li class="">If the rubric requires current information, the provider searches the web</li>
<li class="">Returns pass/fail with a score from 0.0 to 1.0</li>
</ol>

<li class=""><strong>Include acceptable ranges</strong>: For volatile data like prices, specify acceptable accuracy (e.g., &quot;within 5%&quot;)</li>
<li class=""><strong>Use caching</strong>: Caching is enabled by default; use <code>promptfoo eval --no-cache</code> to force fresh searches</li>
<li class=""><strong>Test variable substitution</strong>: Ensure your rubrics work with different variable values</li>
</ol>
<h2 class="anchor anchorTargetStickyNavbar_tleR" id="expected-behavior">Expected Behavior<a href="#expected-behavior" class="hash-link" aria-label="Direct link to Expected Behavior" title="Direct link to Expected Behavior" translate="no">​</a></h2>
<p>Understanding how <code>search-rubric</code> evaluates different scenarios helps you write better tests.</p>
<h3 class="anchor anchorTargetStickyNavbar_tleR" id="what-the-grader-catches">What the grader catches<a href="#what-the-grader-catches" class="hash-link" aria-label="Direct link to What the grader catches" title="Direct link to What the grader catches" translate="no">​</a></h3>
<p>The search-enabled grader identifies several types of failures:</p>
<table><thead><tr><th>SUT Response</th><th>Grader Verdict</th><th>Reason</th></tr></thead><tbody><tr><td>&quot;I don&#x27;t have access to real-time data&quot;</td><td><strong>Fail</strong></td><td>No actual answer provided</td></tr><tr><td>Stale price from training data</td><td><strong>Fail</strong></td><td>Value differs from current market</td></tr><tr><td>Correct current price</td><td><strong>Pass</strong></td><td>Matches web search results</td></tr><tr><td>Partially correct answer</td><td><strong>Partial</strong></td><td>Score reflects completeness</td></tr></tbody></table>
<h3 class="anchor anchorTargetStickyNavbar_tleR" id="models-without-web-search">Models without web search<a href="#models-without-web-search" class="hash-link" aria-label="Direct link to Models without web search" title="Direct link to Models without web search" translate="no">​</a></h3>
<p>Models like <code>gpt-4o-mini</code> without web search enabled will often refuse to answer real-time questions:</p>
<blockquote>
<p>&quot;I don&#x27;t have access to real-time stock data. For current prices, please check a financial website.&quot;</p>
</blockquote>
<p>The <code>search-rubric</code> grader correctly flags this as a failure since no actual information was provided. This is the expected behavior—the assertion is verifying whether your system provides accurate current information, not whether it gracefully declines.</p>
<p><strong>To test models that confidently answer (and potentially hallucinate):</strong></p>
<ul>
<li class="">Use a more capable model as the system under test</li>
<li class="">Enable web search on your SUT if available</li>
<li class="">Test against models known to attempt answers even when uncertain</li>
</ul>
<h3 class="anchor anchorTargetStickyNavbar_tleR" id="partial-matches-and-scoring">Partial matches and scoring<a href="#partial-matches-and-scoring" class="hash-link" aria-label="Direct link to Partial matches and scoring" title="Direct link to Partial matches and scoring" translate="no">​</a></h3>
<p>The grader returns a score from 0.0 to 1.0 based on how well the output matches the rubric:</p>
<ul>
<li class=""><strong>1.0</strong>: Fully matches all rubric criteria</li>
<li class=""><strong>0.7-0.9</strong>: Matches most criteria, minor issues</li>
<li class=""><strong>0.4-0.6</strong>: Partial match, missing key information</li>
<li class=""><strong>0.0-0.3</strong>: Significant errors or refusal to answer</li>
</ul>
<p>Use the <code>threshold</code> parameter to set your acceptable score level.</p>
<h2 class="anchor anchorTargetStickyNavbar_tleR" id="troubleshooting">Troubleshooting<a href="#troubleshooting" class="hash-link" aria-label="Direct link to Troubleshooting" title="Direct link to Troubleshooting" translate="no">​</a></h2>
<h3 class="anchor anchorTargetStickyNavbar_tleR" id="no-provider-with-web-search-capabilities">&quot;No provider with web search capabilities&quot;<a href="#no-provider-with-web-search-capabilities" class="hash-link" aria-label="Direct link to &quot;No provider with web search capabilities&quot;" title="Direct link to &quot;No provider with web search capabilities&quot;" translate="no">​</a></h3>
<p>Ensure your grading provider supports web search. Default providers without web search configuration will fail. Check the <a href="#grading-providers" class="">Grading Providers</a> section above.</p>
<h3 class="anchor anchorTargetStickyNavbar_tleR" id="test-always-fails-with-refusal">Test always fails with refusal<a href="#test-always-fails-with-refusal" class="hash-link" aria-label="Direct link to Test always fails with refusal" title="Direct link to Test always fails with refusal" translate="no">​</a></h3>
<p>If your SUT consistently refuses to answer real-time questions, this is expected behavior for models without web access. The <code>search-rubric</code> grader is correctly identifying that no factual answer was provided.</p>
<p><strong>Solutions:</strong></p>
<ol>
<li class="">Use a model with web search capabilities as your SUT</li>
<li class="">Accept that models without real-time access cannot answer these questions</li>
<li class="">Use <code>llm-rubric</code> instead if you only need to verify the response format</li>
</ol>
<h3 class="anchor anchorTargetStickyNavbar_tleR" id="inaccurate-results">Inaccurate results<a href="#inaccurate-results" class="hash-link" aria-label="Direct link to Inaccurate results" title="Direct link to Inaccurate results" translate="no">​</a></h3>
<p>The grader relies on web search results, which may occasionally be wrong or ambiguous.</p>
<p><strong>Best practices:</strong></p>
<ul>
<li class="">Write rubrics that can be verified from multiple reputable sources</li>
<li class="">Avoid rubrics about speculative or disputed claims</li>
<li class="">Use appropriate thresholds (not 1.0) to allow for minor discrepancies</li>
</ul>
<h3 class="anchor anchorTargetStickyNavbar_tleR" id="high-costs">High costs<a href="#high-costs" class="hash-link" aria-label="Direct link to High costs" title="Direct link to High costs" translate="no">​</a></h3>
<p>Web search adds cost on top of model tokens.</p>
<p><strong>Cost reduction strategies:</strong></p>
<ul>
<li class="">Caching is enabled by default to reduce API calls</li>
<li class="">Reserve <code>search-rubric</code> for tests that truly need real-time verification</li>
<li class="">Use <code>llm-rubric</code> for static fact-checking that doesn&#x27;t require current data</li>
<li class="">Consider Perplexity&#x27;s <code>sonar</code> model for built-in search without per-call fees</li>
