Integration</h1></header>
<p>Promptfoo Enterprise provides webhooks to notify external systems when security vulnerabilities (issues) are created or updated.</p>
<h2 class="anchor anchorTargetStickyNavbar_tleR" id="what-is-an-issue">What is an Issue?<a href="#what-is-an-issue" class="hash-link" aria-label="Direct link to What is an Issue?" title="Direct link to What is an Issue?" translate="no">​</a></h2>
<p>An &quot;issue&quot; in Promptfoo Enterprise refers to a <strong>security vulnerability</strong> or weakness detected during AI security testing. Issues are created when red team plugins identify potential security risks such as prompt injections, data leaks, harmful content generation, or other AI-specific vulnerabilities.</p>
<h2 class="anchor anchorTargetStickyNavbar_tleR" id="event-types">Event Types<a href="#event-types" class="hash-link" aria-label="Direct link to Event Types" title="Direct link to Event Types" translate="no">​</a></h2>
<p>The following webhook event types are available:</p>
<ul>
<li class=""><code>issue.created</code>: Triggered when a new security vulnerability is detected and created</li>
<li class=""><code>issue.updated</code>: Triggered when a vulnerability is updated (such as when multiple attributes change at once)</li>
<li class=""><code>issue.status_changed</code>: Triggered when a vulnerability&#x27;s status changes (e.g., from open to fixed)</li>
<li class=""><code>issue.severity_changed</code>: Triggered when a vulnerability&#x27;s severity level changes</li>
<li class=""><code>issue.comment_added</code>: Triggered when a comment is added to a vulnerability</li>
</ul>
<blockquote>
<p>Note: When multiple properties of a vulnerability are updated simultaneously (for example, both status and severity), a single issue.updated event will be sent rather than separate issue.status_changed and issue.severity_changed events. This helps prevent webhook consumers from receiving multiple notifications for what is logically a single update operation.</p>
</blockquote>
<h2 class="anchor anchorTargetStickyNavbar_tleR" id="managing-webhooks">Managing Webhooks<a href="#managing-webhooks" class="hash-link" aria-label="Direct link to Managing Webhooks" title="Direct link to Managing Webhooks" translate="no">​</a></h2>
<p>Webhooks can be managed via the API. Each webhook is associated with an organization and can be configured to listen for specific event types.</p>
<h3 class="anchor anchorTargetStickyNavbar_tleR" id="creating-a-webhook">Creating a Webhook<a href="#creating-a-webhook" class="hash-link" aria-label="Direct link to Creating a Webhook" title="Direct link to Creating a Webhook" translate="no">​</a></h3>

<h3 class="anchor anchorTargetStickyNavbar_tleR" id="siem-integration">SIEM Integration<a href="#siem-integration" class="hash-link" aria-label="Direct link to SIEM Integration" title="Direct link to SIEM Integration" translate="no">​</a></h3>
<p>When integrating with a SIEM system, you might want to listen for <code>issue.created</code> and <code>issue.updated</code> events. This allows your security team to be notified of new security vulnerabilities detected by Promptfoo Enterprise and track their resolution. The complete vulnerability state provided with each webhook makes it easy to keep your SIEM system synchronized.</p>
<h3 class="anchor anchorTargetStickyNavbar_tleR" id="task-tracking-integration">Task Tracking Integration<a href="#task-tracking-integration" class="hash-link" aria-label="Direct link to Task Tracking Integration" title="Direct link to Task Tracking Integration" translate="no">​</a></h3>
<p>For task tracking systems like JIRA, you can:</p>
<ul>
<li class="">Listen for <code>issue.created</code> to create new tickets for vulnerabilities</li>
<li class="">Listen for <code>issue.updated</code> to update tickets when any vulnerability properties change</li>
<li class="">Listen for <code>issue.status_changed</code> if you only care about vulnerability status transitions</li>
<li class="">Listen for <code>issue.comment_added</code> to sync comments between systems</li>
</ul>
<p>The <code>changes</code> array included with <code>issue.updated</code> events makes it easy to add appropriate comments to your task tracking system (e.g., &quot;Vulnerability status changed from open to fixed&quot;).</p>
<h3 class="anchor anchorTargetStickyNavbar_tleR" id="custom-notification-system">Custom Notification System<a href="#custom-notification-system" class="hash-link" aria-label="Direct link to Custom Notification System" title="Direct link to Custom Notification System" translate="no">​</a></h3>
<p>You could build a custom notification system that:</p>
<ol>
<li class="">Creates different notification channels based on event types</li>
<li class="">Routes notifications to different teams based on severity levels</li>
<li class="">Uses the <code>changes</code> information in <code>issue.updated</code> events to craft appropriately detailed messages</li>
<li class="">Filters out specific types of changes that aren&#x27;t relevant to particular teams</li>
