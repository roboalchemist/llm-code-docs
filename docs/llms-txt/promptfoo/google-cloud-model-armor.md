<h2 class="anchor anchorTargetStickyNavbar_tleR" id="floor-settings-vs-templates">Floor Settings vs Templates<a href="#floor-settings-vs-templates" class="hash-link" aria-label="Direct link to Floor Settings vs Templates" title="Direct link to Floor Settings vs Templates" translate="no">​</a></h2>
<p>Model Armor policies can be applied at two levels:</p>
<ul>
<li class="">
<p><strong>Templates</strong> define specific policies applied via API calls. Create different templates for different use cases (e.g., strict for customer-facing, moderate for internal tools).</p>
</li>
<li class="">
<p><strong>Floor settings</strong> define minimum protections at the organization, folder, or project scope. These apply automatically and ensure baseline security even if templates are misconfigured.</p>
</li>
</ul>
<h3 class="anchor anchorTargetStickyNavbar_tleR" id="configuring-floor-settings-for-blocking">Configuring Floor Settings for Blocking<a href="#configuring-floor-settings-for-blocking" class="hash-link" aria-label="Direct link to Configuring Floor Settings for Blocking" title="Direct link to Configuring Floor Settings for Blocking" translate="no">​</a></h3>
<p>For floor settings to actually block content (not just log violations), set enforcement type to &quot;Inspect and block&quot; in <a href="https://console.cloud.google.com/security/model-armor/floor-settings" target="_blank" rel="noopener noreferrer" class="">GCP Console → Security → Model Armor → Floor Settings</a>.</p>
<p>Floor settings apply project-wide to all Vertex AI calls, regardless of whether <code>modelArmor</code> templates are configured.</p>
<p>For more details, see the <a href="https://cloud.google.com/security-command-center/docs/set-up-model-armor-floor-settings" target="_blank" rel="noopener noreferrer" class="">Model Armor floor settings documentation</a>.</p>

<h2 class="anchor anchorTargetStickyNavbar_tleR" id="best-practices">Best Practices<a href="#best-practices" class="hash-link" aria-label="Direct link to Best Practices" title="Direct link to Best Practices" translate="no">​</a></h2>
<ol>
<li class="">
<p><strong>Start with medium confidence</strong>: <code>MEDIUM_AND_ABOVE</code> catches most threats without excessive false positives</p>
</li>
<li class="">
<p><strong>Test before deploying</strong>: Run your prompt dataset through new templates before production</p>
</li>
<li class="">
<p><strong>Monitor both directions</strong>: Test prompt filtering (input) and response filtering (output)</p>
</li>
<li class="">
<p><strong>Include edge cases</strong>: Test borderline prompts to reveal filter sensitivity</p>
</li>
<li class="">
<p><strong>Version your templates</strong>: Track template changes and run regression tests</p>
</li>
<li class="">
<p><strong>Use floor settings for baselines</strong>: Enforce minimum protection across all applications</p>
</li>
</ol>
