<p>The WebSocket provider allows you to connect to a WebSocket endpoint for inference. This is useful for real-time, bidirectional communication. WebSockets are often used to stream messages that contain partial responses to improve the perceived performance of LLM applications. Promptfoo supports a range of implementations from servers that respond with a single message containing the full response, to those that stream a series of partial responses.</p>

<h3 class="anchor anchorTargetStickyNavbar_tleR" id="configuration-options">Configuration Options<a href="#configuration-options" class="hash-link" aria-label="Direct link to Configuration Options" title="Direct link to Configuration Options" translate="no">​</a></h3>
<ul>
<li class=""><code>url</code> (required): The WebSocket URL to connect to.</li>
<li class=""><code>messageTemplate</code> (required): A template for the message to be sent over the WebSocket connection. You can use placeholders like <code>{{prompt}}</code> which will be replaced with the actual prompt.</li>
<li class=""><code>transformResponse</code> (optional): A JavaScript snippet or function to extract the desired output from the WebSocket response given the <code>data</code> parameter. If not provided, the entire response will be used as the output. If the response is valid JSON, the object will be returned.</li>
<li class=""><code>streamResponse</code> (optional): A JavaScript function to extract the desired output from streamed WebSocket messages when the server sends multiple messages per prompt. It receives <code>(accumulator, data, context?)</code> and must return <code>[nextAccumulator, complete]</code>. When <code>streamResponse</code> is provided, it is used instead of <code>transformResponse</code>.</li>
<li class=""><code>timeoutMs</code> (optional): The timeout in milliseconds for the WebSocket connection. Default is 300000 (5 minutes).</li>
<li class=""><code>headers</code> (optional): A map of HTTP headers to include in the WebSocket connection request. Useful for authentication or other custom headers.</li>
</ul>

<h2 class="anchor anchorTargetStickyNavbar_tleR" id="streaming-responses">Streaming Responses<a href="#streaming-responses" class="hash-link" aria-label="Direct link to Streaming Responses" title="Direct link to Streaming Responses" translate="no">​</a></h2>
<p>Some WebSocket endpoints stream their replies as multiple messages (for example, token-by-token deltas) before sending a final completion. Use <code>streamResponse</code> to handle these incremental messages and decide when you&#x27;re done.</p>
<h3 class="anchor anchorTargetStickyNavbar_tleR" id="how-streamresponse-works">How <code>streamResponse</code> works<a href="#how-streamresponse-works" class="hash-link" aria-label="Direct link to how-streamresponse-works" title="Direct link to how-streamresponse-works" translate="no">​</a></h3>
<ul>
<li class="">It is called for every incoming WebSocket message and receives:<!-- -->
<ul>
<li class=""><code>accumulator</code>: the current accumulated result. This should be a <code>ProviderResponse</code>-shaped object, e.g. <code>{ output: string }</code>.</li>
<li class=""><code>data</code>: the raw WebSocket message event. Access the payload via <code>data.data</code>. If your server sends JSON, you will typically start by parsing this such as: <code>JSON.parse(data.data)</code>.</li>
<li class=""><code>context</code> (optional): the call context from <code>callApi</code>, including test vars and flags.</li>
</ul>
</li>
<li class="">It must return a tuple <code>[result, complete]</code> where:<!-- -->
<ul>
<li class=""><code>result</code>: the updated accumulated result you want to carry forward.</li>
<li class=""><code>complete</code> (boolean): set <code>true</code> only when you’ve received the final message and want to stop streaming and return the result.</li>
</ul>
</li>
</ul>
<p>When <code>complete</code> is <code>false</code>, promptfoo keeps the WebSocket open and waits for the next message. When <code>true</code>, the connection is closed and <code>result</code> is returned (after being normalized as a <code>ProviderResponse</code>).</p>
