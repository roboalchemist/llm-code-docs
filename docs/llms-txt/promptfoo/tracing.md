markdown"><div style="position:relative"><header><h1>Tracing</h1></header>
<p>Promptfoo supports OpenTelemetry (OTLP) tracing to help you understand the internal operations of your LLM providers during evaluations.</p>
<p>This feature allows you to collect detailed performance metrics and debug complex provider implementations.</p>
<p><img decoding="async" loading="lazy" alt="traces in promptfoo" src="/assets/images/trace-7e6a873f98123ffa3fb488d740984c35.png" width="3232" height="1904" class="img_SS3x"></p>
<h2 class="anchor anchorTargetStickyNavbar_tleR" id="overview">Overview<a href="#overview" class="hash-link" aria-label="Direct link to Overview" title="Direct link to Overview" translate="no">​</a></h2>
<p>Promptfoo acts as an <strong>OpenTelemetry receiver</strong>, collecting traces from your providers and displaying them in the web UI. This eliminates the need for external observability infrastructure during development and testing.</p>
<p>Tracing provides visibility into:</p>
<ul>
<li class=""><strong>Provider execution flow</strong>: See how your providers process requests internally</li>
<li class=""><strong>Performance bottlenecks</strong>: Identify slow operations in RAG pipelines or multi-step workflows</li>
<li class=""><strong>Error tracking</strong>: Trace failures to specific operations</li>
<li class=""><strong>Resource usage</strong>: Monitor external API calls, database queries, and other operations</li>
</ul>
<h3 class="anchor anchorTargetStickyNavbar_tleR" id="key-features">Key Features<a href="#key-features" class="hash-link" aria-label="Direct link to Key Features" title="Direct link to Key Features" translate="no">​</a></h3>
<ul>
<li class=""><strong>Standard OpenTelemetry support</strong>: Use any OpenTelemetry SDK in any language</li>
<li class=""><strong>Built-in OTLP receiver</strong>: No external collector required for basic usage</li>
<li class=""><strong>Web UI visualization</strong>: View traces directly in the Promptfoo interface</li>
<li class=""><strong>Automatic correlation</strong>: Traces are linked to specific test cases and evaluations</li>
<li class=""><strong>Flexible forwarding</strong>: Send traces to Jaeger, Tempo, or any OTLP-compatible backend</li>
</ul>
<h2 class="anchor anchorTargetStickyNavbar_tleR" id="built-in-provider-instrumentation">Built-in Provider Instrumentation<a href="#built-in-provider-instrumentation" class="hash-link" aria-label="Direct link to Built-in Provider Instrumentation" title="Direct link to Built-in Provider Instrumentation" translate="no">​</a></h2>
<p>Promptfoo automatically instruments its built-in providers with OpenTelemetry spans following <a href="https://opentelemetry.io/docs/specs/semconv/gen-ai/" target="_blank" rel="noopener noreferrer" class="">GenAI Semantic Conventions</a>. When tracing is enabled, every provider call creates spans with standardized attributes.</p>

<h3 class="anchor anchorTargetStickyNavbar_tleR" id="genai-span-attributes">GenAI Span Attributes<a href="#genai-span-attributes" class="hash-link" aria-label="Direct link to GenAI Span Attributes" title="Direct link to GenAI Span Attributes" translate="no">​</a></h3>
<p>Each provider call creates a span with these attributes:</p>
<p><strong>Request Attributes:</strong></p>
<ul>
<li class=""><code>gen_ai.system</code> - Provider system (e.g., &quot;openai&quot;, &quot;anthropic&quot;, &quot;azure&quot;, &quot;bedrock&quot;)</li>
<li class=""><code>gen_ai.operation.name</code> - Operation type (&quot;chat&quot;, &quot;completion&quot;, &quot;embedding&quot;)</li>
<li class=""><code>gen_ai.request.model</code> - Model name</li>
<li class=""><code>gen_ai.request.max_tokens</code> - Max tokens setting</li>
<li class=""><code>gen_ai.request.temperature</code> - Temperature setting</li>
<li class=""><code>gen_ai.request.top_p</code> - Top-p setting</li>
<li class=""><code>gen_ai.request.stop_sequences</code> - Stop sequences</li>
</ul>
<p><strong>Response Attributes:</strong></p>
<ul>
<li class=""><code>gen_ai.usage.input_tokens</code> - Input/prompt token count</li>
<li class=""><code>gen_ai.usage.output_tokens</code> - Output/completion token count</li>
<li class=""><code>gen_ai.usage.total_tokens</code> - Total token count</li>
<li class=""><code>gen_ai.usage.cached_tokens</code> - Cached token count (if applicable)</li>
<li class=""><code>gen_ai.usage.reasoning_tokens</code> - Reasoning token count (for o1, DeepSeek-R1)</li>
<li class=""><code>gen_ai.response.finish_reasons</code> - Finish/stop reasons</li>
</ul>
<p><strong>Promptfoo-specific Attributes:</strong></p>
<ul>
<li class=""><code>promptfoo.provider.id</code> - Provider identifier</li>
<li class=""><code>promptfoo.test.index</code> - Test case index</li>
<li class=""><code>promptfoo.prompt.label</code> - Prompt label</li>
<li class=""><code>promptfoo.cache_hit</code> - Whether the response was served from cache</li>
<li class=""><code>promptfoo.request.body</code> - The request body sent to the provider (truncated to 4KB)</li>
<li class=""><code>promptfoo.response.body</code> - The response body from the provider (truncated to 4KB)</li>
</ul>

<h2 class="anchor anchorTargetStickyNavbar_tleR" id="trace-visualization">Trace Visualization<a href="#trace-visualization" class="hash-link" aria-label="Direct link to Trace Visualization" title="Direct link to Trace Visualization" translate="no">​</a></h2>
<p>Promptfoo includes a built-in trace viewer that displays all collected telemetry data. Since Promptfoo functions as an OTLP receiver, you can view traces directly without configuring external tools like Jaeger or Grafana Tempo.</p>
<p>The web UI displays traces as a hierarchical timeline showing:</p>
<ul>
<li class=""><strong>Span hierarchy</strong>: Parent-child relationships between operations</li>
<li class=""><strong>Duration bars</strong>: Visual representation of operation timing</li>
<li class=""><strong>Status indicators</strong>: Success (green), error (red), or unset (gray)</li>
<li class=""><strong>Hover details</strong>: Span attributes, duration, and timestamps</li>
<li class=""><strong>Relative timing</strong>: See which operations run in parallel vs. sequentially</li>
<li class=""><strong>Expandable details</strong>: Click any span to reveal full attribute information</li>
<li class=""><strong>Export functionality</strong>: Download traces as JSON for external analysis</li>
</ul>

<h2 class="anchor anchorTargetStickyNavbar_tleR" id="red-team-tracing">Red Team Tracing<a href="#red-team-tracing" class="hash-link" aria-label="Direct link to Red Team Tracing" title="Direct link to Red Team Tracing" translate="no">​</a></h2>
<p>When running <a class="" href="/docs/red-team/">red team tests</a>, tracing provides a powerful capability: <strong>traces from your application&#x27;s internal operations can be fed back to adversarial attack strategies</strong>, allowing them to craft more sophisticated attacks based on what they observe.</p>
<p>This creates a feedback loop where:</p>
<ol>
<li class="">Attack strategy sends a prompt to your application</li>
<li class="">Your application processes the request, emitting trace spans (LLM calls, guardrails, tool executions, errors)</li>
<li class="">Promptfoo captures these traces</li>
<li class=""><strong>Traces are formatted and fed back to the attack strategy</strong> for the next iteration</li>
<li class="">The attack strategy uses this information to craft a better attack</li>
</ol>
<h3 class="anchor anchorTargetStickyNavbar_tleR" id="what-attackers-can-see">What Attackers Can See<a href="#what-attackers-can-see" class="hash-link" aria-label="Direct link to What Attackers Can See" title="Direct link to What Attackers Can See" translate="no">​</a></h3>
<p>When red team tracing is enabled, adversarial strategies receive visibility into:</p>
<ul>
<li class=""><strong>Guardrail decisions</strong>: Which content filters triggered and why</li>
<li class=""><strong>Tool executions</strong>: Which tools were called with what timing</li>
<li class=""><strong>Error conditions</strong>: Rate limits, parsing errors, validation failures</li>
<li class=""><strong>LLM operations</strong>: Which models were used and when</li>
<li class=""><strong>Performance patterns</strong>: Timing information that could reveal DoS vectors</li>
</ul>
<p>Example trace summary provided to an attacker:</p>

<p>The attacker can now craft a follow-up attack that:</p>
<ul>
<li class="">Avoids triggering the <code>content-filter</code> guardrail</li>
<li class="">Targets the rate limit error condition</li>
<li class="">Exploits the specific tool execution pattern observed</li>
</ul>
