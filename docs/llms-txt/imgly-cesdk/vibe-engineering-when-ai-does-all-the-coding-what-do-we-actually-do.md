                    <h4 class="toc-heading">On this Page</h4>
                </div></aside>
                <p>This isn't an article about AI replacing engineers, it's about discovering what (software) engineering will become when freed from the mechanics of coding itself.</p><p>Vibe-Coding, and Vibe-Engineering are omnipresent in my social feeds these days. As with every new trend, it's hard to judge what really works and what is pure marketing hype. But the central question nagged at me: if AI really can do all the coding, what exactly are we humans supposed to do?</p><p>Therefore, I wanted to find out for myself. I didn't want to build just a <em>Hello World</em> example, so I pulled an old idea out of the closet and dusted it off.</p><h2 id="the-experiment-ai-agents-code-humans-curate">The Experiment: AI Agents Code, Humans Curate</h2><figure class="kg-card kg-image-card"><img src="https://blog.img.ly/2025/07/20250705_1708_Human-Robot-Coding-Collaboration_remix_01jzdj1by9ebmaxjtmw6z04z8q--1-.png" class="kg-image" alt="" loading="lazy" width="1536" height="1024"></figure><p>For some time, I had considered porting our <a href="https://github.com/imgly/background-removal-js?ref=img.ly">IMG.LY background removal library</a> from JavaScript to other platforms using Rust. However, I had postponed this side project due to the anticipated effort involved. This seemed like the perfect project to discover what my role would be in a world where AI does the heavy lifting. To pull it off, I self-imposed only one strict rule:</p><blockquote><strong>Hands off the code!</strong></blockquote><p>Every commit, every fix, every feature would be handled by an AI coding agent. My role? Being the curator, formulating my intent, reviewing, play-testing, and guiding the agent with feedback.</p><p>The agent should be the sole coder, from start to finish.</p><p>This was a true hands-off experiment: could an AI build an entire Rust library if I never touched the code?</p><p>Is it really something new? After years of handing off development to teammates as a CTO or teaching students, this didn't feel completely different.</p><p>So, what happens when you <strong>let an AI do all the coding</strong>, and you just review and give feedback? What does it actually mean to be a human in this new paradigm?</p><p>The result is baffling and gives a glimpse of what we as engineers can expect for the future.</p><blockquote><strong>10,000+ lines of production-ready code. Authored by AI, orchestrated by a human.</strong></blockquote><p>Seeing is believing? Then explore the results and judge for yourself at <a href="https://github.com/imgly/background-removal-rs/tree/v0.2.0?ref=img.ly">github.com/imgly/background-removal-rs</a>.<br>I open-sourced the full code and the agent rules used during this experience.</p><p>Here's what I learned: vibe-engineering an entire Rust library with a coding agent CLI as my only pair programmer.</p><p>On a quick note, I chose to use Claude Code with Opus 4 and Sonnet 4, as I was already familiar with them and had achieved the best results so far. The costs were kept in check by using a Claude Max Account, which at the time is around ~€100.</p><h3 id="the-general-workflow-that-emerged">The General Workflow That Emerged</h3><p>After some initial time to get used to working with an AI agent, I realized that the workflow follows pretty closely a typical software engineering flow, but with an important twist that answers our central question: what do humans actually do when AI codes? While most people focus on the <code>code writing</code> part, I discovered that the <code>quality assurance</code> part becomes the human's primary domain. In my experiments, I realized that to keep the code intact over time, this QA phase is essential—and this is where humans truly shine. Most interestingly, with the right rules and prompting, we can let the agent do the heavy lifting here, too.</p><p>Here's how the collaboration actually worked in practice:</p><p><strong>Getting Started</strong><br>The project began with me setting up the environment and configuring timeouts for long-running tasks—because AI agents need time to compile Rust code and run extensive test suites. Together, the AI and I created a <code>CLAUDE.md</code> file with coding standards and rules (you can see the <a href="https://github.com/imgly/background-removal-rs/tree/main/.claude/rules?ref=img.ly" rel="noreferrer">actual rules we used</a>). We also wrote project requirements documents collaboratively, with me providing the vision and the AI helping structure the technical details.</p><p><strong>The Development Dance</strong><br>Once we got rolling, a natural rhythm emerged. I'd describe what I wanted—something like "add support for different image formats with color profile preservation." The AI would then create a detailed implementation plan, breaking down the work into manageable chunks. I'd review this plan, often tweaking the approach or scope, then give the green light.</p><p>What happened next was fascinating: the AI would write the code, run correctness checks, update all the tests (unit tests, documentation tests, end-to-end tests), run the linter, analyze coverage, and even update documentation and examples. Sometimes it would run performance benchmarks without me asking—the overzealous builder at work.</p><p><strong>My New Role: The Quality Guardian</strong><br>My job became verifying that the intent was met and the API actually made sense. I'd play-test the functionality, try to break it in ways a real user might, and provide feedback. When something worked well, I'd document what we learned for future development. The AI would then write tests to lock in that new functionality, ensuring we never accidentally broke what was working.</p><p>This cycle repeated for every feature, with the AI handling the heavy lifting while I focused on direction, quality, and user experience.</p><p>The interesting part was that the more I got used to the workflow, the more I handed off to the agent. The validation and general experience had to be largely influenced or orchestrated by me, but building, testing, and even updating documentation could be delegated. In the beginning, I started just building things, but the more we built, the more the AI agent failed to keep existing functionality intact. With those small context windows, it's kind of understandable after all.</p><h3 id="quality-assurance-knowledge-capturing-matters-more-than-ever">Quality Assurance &amp; Knowledge Capturing Matters More Than Ever</h3><p>As engineers, we know that at some points we cannot grasp every influence of our code changes on the whole code base anymore. Therefore, we use tests and tooling to help us keep our sanity.</p><p>The good thing is that the Rust and Cargo toolchains are exceptionally high quality when it comes to providing the right tooling also for agents.</p><p>Due to its lack of long-term context and limited knowledge of all the library's capabilities, the tests and verification mechanisms become crucial. While these <em>are</em> part of the context, they're not stored in the agent's memory but baked into the codebase itself. With the tests and the tooling being available to the agent, it will still make mistakes, as I do, but they can effectively <code>auto-heal</code> or <code>auto-correct</code> their own mistakes with the provided feedback.</p><p>I can only advise being even stricter with QA from the very beginning.</p><p>Another thing that became apparent very quickly is that, unlike a human colleague with lots of context and a good memory, agents <em>still</em> need significant help to remember things to avoid rediscovering the same information repeatedly. This revealed another crucial human role: <code>context engineering</code> becomes critically important. As such, we engineers become the keepers of institutional knowledge, the ones that help the agent remember what worked and what didn't. I would describe it as <code>knowledge capture</code>,  for lack of a better term.</p><h2 id="tips-tricks">Tips &amp; Tricks</h2><h3 id="ai-is-eager-resourceful-and-has-memory-like-a-sieve">AI Is Eager, Resourceful, and Has Memory Like a Sieve</h3><p>Here are the key insights I gained from building the library with an AI coding agent:</p><p><strong>Code that works:</strong> This might seem obvious, but it wasn't clear to me if the library would ever be usable or publishable, but it is. The AI proved it could deliver production-ready code.</p><p><strong>Time Effectiveness:</strong> While creating the library took around 3 weeks, don't let this fool you. Most of the time, the agent worked alone, and I only had to check in once in a while to review and give feedback. Iteration cycles and rewriting still take time. So I wouldn't say development was faster than if I did it myself, but much more scalable and effective with my own time. Here's what I actually spent my time on: reviewing architectural decisions, ensuring the API made sense, and play-testing the final product.</p><p><strong>Stamina:</strong> The agent never complained, even when fixing 400+ warnings after setting the linter to pedantic—it pushed through. The AI powered through hundreds of warnings, even clustering them and suggesting which to fix first. To be fair, most humans would have either gone crazy or aborted the effort claiming "it's good enough." This revealed something important: as humans, we don't need to be the ones grinding through tedious tasks anymore.</p><p><strong>Overzealous Builder:</strong> The agent often built more than I asked for. Scoping and clear implementation plans were essential. This taught me that one of the key human roles is setting boundaries and maintaining focus.</p><p><strong>Premature "Done":</strong> Sometimes the agent claimed it was finished but left code mocked or incomplete. I learned to always ask, "Is there anything left to implement?" Quality assurance becomes fundamentally a human responsibility.</p><p><strong>Old Knowledge:</strong> The AI sometimes relied on outdated library knowledge. I had to remind it to check the latest docs and version of dependencies, but when asked, it searched the web, GitHub, crates.io, etc., and gathered all necessary information for using a library. Humans become the validators of information freshness.</p><p><strong>Forgetting and Compaction:</strong> Larger features grow outside the context window, so the agent would "compact" context, sometimes forgetting important details. Forcing it to write implementation plans in markdown files helps maintain flow and allows better resuming and bookkeeping. We humans become the keepers of long-term project memory.</p><p><strong>Helps Establishing Rules:</strong> While working with the agent, I saw repeating patterns like it forgetting how to check, build, and test the application, or best practices like using <code>git worktrees</code>. I started adding new rules to <code>Claude.md</code> by myself at first, but quickly realized that the agent is far better at formulating and adding new rules if I asked it to. The human role evolved into being the pattern recognizer and rule creator.</p><p><strong>Doesn't Always Obey the Rules:</strong> Most of the time, the rules are followed, but occasionally it doesn't follow them and forgets to automatically run all tests. This reinforced that humans must remain the final guardians of process and quality.</p><p><strong>Code Deletion:</strong> Occasionally, the AI "fixed" issues by removing important code paths. Solution: Insist on never removing functionality without discussion, and build a robust test suite. We become the protectors of existing functionality.</p><p><strong>Process Management:</strong> The AI struggled with background processes, understanding when to run things in the background, and keeping track of what it started. For example, starting a web server via <a href="https://docs.anthropic.com/en/docs/agents-and-tools/tool-use/bash-tool?ref=img.ly" rel="noreferrer"><code>Bash</code></a> (see <a href="https://docs.anthropic.com/en/docs/agents-and-tools/tool-use/bash-tool?ref=img.ly" rel="noreferrer">Claude Code Bash tool docs</a>), and then trying to use a <a href="https://github.com/microsoft/playwright?ref=img.ly"><code>playwright</code> tool</a> (see also <a href="https://github.com/microsoft/playwright-mcp?ref=img.ly">playwright-mcp</a>) to access this server was blocked by these issues. Complex orchestration remains a distinctly human skill.</p><p><strong>Overly Agreeable:</strong> Too often, it seems to just agree with whatever I said. For the future, I wish agents wouldn't be so obedient. This highlighted that humans need to be the challengers and devil's advocates in the process.</p><h3 id="further-remarks">Further Remarks</h3><p>After three weeks of this experiment, the patterns became clear. Here's what I learned about the human role in AI-assisted development:</p><p><strong>Ensure Verify</strong>,<strong> and Self-Correct:</strong> Always ask the AI to check, format, lint, test, and benchmark its work. Your job becomes quality orchestration, not quality execution.</p><p><strong>Always Plan First:</strong> Insist on a clear implementation plan before coding begins. Humans excel at high-level architectural thinking and breaking down complex problems.</p><p><strong>Write It Down:</strong> Let an agent keep notes, todos, and open issues in markdown files. Ideally in the repository itself. Don't rely on the AI's memory. We become the institutional memory keepers.</p><p><strong>Scope Features:</strong> Break tasks into small, manageable pieces. Feature scoping and boundary setting become a core human skill.</p><p><strong>Test Suite is King:</strong> A robust test suite prevents accidental "fixes" that remove functionality. Humans become the guardians of existing behavior.</p><p><strong>Stay Involved:</strong> Fast iteration and discussion with the AI is crucial. Don't go fully hands-off. Active curation and guidance remain essential.</p><h2 id="conclusion">Conclusion</h2><p>So, what do humans actually do when AI does all the coding?</p><p>To answer that, let's look at what AI agents are today:</p><blockquote>Agents are super-talented, overly obedient coding assistants with lots of stamina and endless potential.</blockquote><p>They're fast, never complain, and iterate like champs. But they need your guidance, structure, and a healthy dose of skepticism.</p><blockquote>In the end, the best results come from a true partnership: you plan and guide—the AI builds, fixes, and learns.</blockquote><p>If you follow these concepts, then</p><blockquote>"Vibe engineering is a highly engaging experience."</blockquote><p>And if you're wondering: yes, I'd do it again. But next time, I'll make sure the AI writes everything down too, and I'll put even more effort into validating and securing new features as soon as they're play-tested and verified. The human role isn't disappearing—it's evolving into something more strategic and impactful.</p><p>Through this experiment, I've identified three fundamental shifts that define what engineering becomes in an AI-assisted world:</p><p><strong>From Execution to Orchestration</strong><br>We're no longer the ones typing code—we're the conductors. Our new skills revolve around prompting (framing tasks so AI understands our intent), directing workflows (knowing when to use AI versus human judgment), and tool fluency (combining different AI tools to create something greater than their parts).</p><p><strong>From Knowledge to Reasoning</strong><br>AI can recall facts and syntax better than any human ever could. But what matters now is our ability to interpret, contextualize, and make sense of uncertainty—something current AI still struggles with. We become the ones who ask "why" and "what if" rather than just "how."</p><p><strong>From Repetition to Adaptation</strong><br>Skills rooted in routine are increasingly automated. The enduring value lies in adaptive thinking, problem-framing, and reinventing approaches when the usual patterns don't apply. We're the ones who recognize when something feels off, even if we can't immediately articulate why.</p><h2 id="afterthought">Afterthought</h2><p>During this experiment, I saw that the agent repeated similar tasks over and over again. One thing that struck me was looking up documentation. Whenever something with a third-party library didn't pan out, it started web-searching for info and guides about the library. It rarely started out with reading the Rust docs first, so I had to tell it to use the Rust docs. Therefore, I assume that providing quick access via tools to guides, docs, API references, and examples will spare me some roundtrips, time, and tokens and accelerate the process. I know that there are tools like <a href="https://context7.com/?ref=img.ly">context7</a>, but it seems to be focused on JavaScript. At least for the Rust ecosystem, the docs are centralized and standardized and can even be used locally. A tool to access those quickly and find things in them would be highly beneficial.</p><p>Additionally, for coding projects, there are typical steps to follow to verify if the code is sound. It's probably also best to provide specific tools or workflows to follow these steps exactly. Rules already help with this, though.</p><p>Last but not least, a natural next step would be for an agent to analyze its history periodically and provide new memory entries (rules) based on that to reduce repetition and common mistakes. For more details, see the <a href="https://img.ly/blog/vibe-engineering-when-ai-does-all-the-coding-what-do-we-actually-do/#appendix-most-used-prompts" rel="noreferrer"><em>Appendix: Most used prompts</em></a> section, where I analyzed the top 10 most common prompts to help agents proactively develop useful rules.</p><p>If you're curious about improvements, I've put together a <a href="https://img.ly/blog/vibe-engineering-when-ai-does-all-the-coding-what-do-we-actually-do/#appendix" rel="noreferrer"><em>wishlist for Claude Code</em></a> that would make the agent even more useful.</p><hr><h2 id="try-it-yourself">Try It Yourself</h2><p>If you want to experience the results of vibe-engineering firsthand:</p><pre><code class="language-bash"># Clone and explore the codebase
git clone https://github.com/imgly/background-removal-rs.git
git checkout --tag v0.2.0

# Or install the CLI directly
cargo install --git https://github.com/imgly/background-removal-rs.git --tag v0.2.0

Judge for yourself whether this new paradigm produces production-ready results.</p><hr><h2 id="appendix">Appendix</h2><h3 id="appendix-claude-code-wishlist">Appendix: Claude Code Wishlist</h3><p>During the experiment, I encountered some issues whose resolution would improve the agent.</p><ul><li>Project-scoped history (not global).</li><li>Project-scoped todos (not global).</li><li>Project-scoped implementation plans (not global).</li><li>Customizable <code>/compact</code> prompt or choose other compact strategies.</li><li>Improved handling of background processes.</li><li>History analytics with memory proposal.</li><li>Code-specific predefined <code>Code and Validate</code> workflows.</li><li>Allow forking of multiple agents from a single point in the session history to create multiple trials with the same intent and context.</li></ul><h3 id="appendix-most-used-prompts">Appendix: Most Used Prompts</h3><p>Claude code stores the history under <code>~/.claude/history</code> as json formats.
