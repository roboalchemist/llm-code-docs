font-bold" id="dna-wire-protocol">DNA wire protocol</h1>
<a href="#cursor-message" class="
                block font-heading mt-12 border-b pb-2 text-2xl font-semibold tracking-tight first:mt-0
          "><h2 id="cursor-message" class="hover:after:content-[&#x27;#&#x27;] after:relative after:pl-2"><code class="text-sm">Cursor</code> message</h2></a>
<p class="leading-7 [&amp;:not(:first-child)]:mt-6">Before explaining the DNA protocol in more detail, we&#x27;re going to discuss the
<code class="text-sm">Cursor</code> message type. This type is used by all methods discussed later and
plays a central role in how DNA works.</p>
<p class="leading-7 [&amp;:not(:first-child)]:mt-6">DNA models a blockchain as a sequence of blocks. The distance of a block from
the first block in the chain (the genesis block) is known as chain height. The
genesis block has height <code class="text-sm">0</code>. Ideally, a blockchain should always build a block
on top of the most recent block, but that&#x27;s not always the case. For this
reason, a block&#x27;s height isn&#x27;t enough to uniquely identify a block in the
blockchain. A <em>chain reorganization</em> is when a chain produces blocks that are
not building on top of the most recent block. As we will see later, the DNA
protocol detects and handles chain reorganizations.
A block that can&#x27;t be part of a chain reorganization is <em>finalized</em>.</p>
<p class="leading-7 [&amp;:not(:first-child)]:mt-6">DNA uses a <em>cursor</em> to uniquely identify blocks on the chain. A cursor contains
two fields:</p>
<ul class="my-6 ml-6 list-disc">
<li class="mt-2"><code class="text-sm">order_key</code>: the block&#x27;s height.</li>
<li class="mt-2"><code class="text-sm">unique_key</code>: the block&#x27;s unique identifier. Depending on the chain, it&#x27;s
the block hash or state root.</li>
</ul>
<a href="#status-method" class="
                block font-heading mt-12 border-b pb-2 text-2xl font-semibold tracking-tight first:mt-0
          "><h2 id="status-method" class="hover:after:content-[&#x27;#&#x27;] after:relative after:pl-2"><code class="text-sm">Status</code> method</h2></a>
<p class="leading-7 [&amp;:not(:first-child)]:mt-6">The <code class="text-sm">Status</code> method is used to retrieve the state of the DNA server. The request
is an empty message. The response has the following fields:</p>
<ul class="my-6 ml-6 list-disc">
<li class="mt-2"><code class="text-sm">last_ingested</code>: returns the last block ingested by the server. This is the most
recent block available for streaming.</li>
<li class="mt-2"><code class="text-sm">finalized</code>: the most recent finalized block.</li>
<li class="mt-2"><code class="text-sm">starting</code>: the first available block. Usually this is the genesis block,
but DNA server operators can prune older nodes to save on storage space.</li>
</ul>
<a href="#streamdata-method" class="
                block font-heading mt-12 border-b pb-2 text-2xl font-semibold tracking-tight first:mt-0
          "><h2 id="streamdata-method" class="hover:after:content-[&#x27;#&#x27;] after:relative after:pl-2"><code class="text-sm">StreamData</code> method</h2></a>
<p class="leading-7 [&amp;:not(:first-child)]:mt-6">The <code class="text-sm">StreamData</code> method is used to start a DNA stream. It accepts a <code class="text-sm">StreamDataRequest</code>
message and returns an infinite stream of <code class="text-sm">StreamDataResponse</code> messages.</p>
<h3 class="font-heading mt-8 text-xl font-semibold tracking-tight" id="request">Request</h3>
<p class="leading-7 [&amp;:not(:first-child)]:mt-6">The request message is used to configure the stream. All fields except <code class="text-sm">filter</code>
are optional.</p>
<ul class="my-6 ml-6 list-disc">
<li class="mt-2"><code class="text-sm">starting_cursor</code>: resume the stream from the provided cursor. The first
block received in the stream will be the block following the provided
cursor. If no cursor is provided, the stream will start from the genesis
block. Notice that since <code class="text-sm">starting_cursor</code> is a cursor, the DNA server can
detect if that block has been part of a chain&#x27;s reorganization while the
indexer was offline.</li>
<li class="mt-2"><code class="text-sm">finality</code>: the stream contains data with at least the specified finality.
Possible values are <em>finalized</em> (only receive finalized data), <em>accepted</em>
(receive finalized and non-finalized blocks), and <em>pending</em> (receive
finalized, non-finalized, and pending blocks).</li>
<li class="mt-2"><code class="text-sm">filter</code>: a non-empty list of chain-specific data filters.</li>
<li class="mt-2"><code class="text-sm">heartbeat_interval</code>: the stream will send an heartbeat message if there are
no messages for the specified amount of time. This is useful to detect if
the stream hangs. Value must be between 10 and 60 seconds.</li>
</ul>
<h3 class="font-heading mt-8 text-xl font-semibold tracking-tight" id="response">Response</h3>
<p class="leading-7 [&amp;:not(:first-child)]:mt-6">Once the server validates and accepts the request, it starts streaming data.
Each stream message can be one of the following message types:</p>
<ul class="my-6 ml-6 list-disc">
<li class="mt-2"><code class="text-sm">data</code>: receive data about a block.</li>
<li class="mt-2"><code class="text-sm">invalidate</code>: the specified blocks don&#x27;t belong to the canonical chain
anymore because they were part of a chain reorganization.</li>
<li class="mt-2"><code class="text-sm">finalize</code>: the most recent finalized block moved forward.</li>
<li class="mt-2"><code class="text-sm">heartbeat</code>: an heartbeat message.</li>
<li class="mt-2"><code class="text-sm">system_message</code>: used to send messages from the server to the client.</li>
</ul>
<h4 class="font-heading mt-8 text-lg font-semibold tracking-tight" id="data-message"><code class="text-sm">Data</code> message</h4>
<p class="leading-7 [&amp;:not(:first-child)]:mt-6">Contains the requested data for a single block. All data messages cursors are
monotonically increasing, unless an <code class="text-sm">Invalidate</code> message is received.</p>
<p class="leading-7 [&amp;:not(:first-child)]:mt-6">The message contains the following fields:</p>
<ul class="my-6 ml-6 list-disc">
<li class="mt-2"><code class="text-sm">cursor</code>: the cursor of the block before this message. If the client
reconnects using this cursor, the first message will be the same as this
message.</li>
<li class="mt-2"><code class="text-sm">end_cursor</code>: this block&#x27;s cursor. Reconnecting to the stream using this
cursor will resume the stream.</li>
<li class="mt-2"><code class="text-sm">finality</code>: finality status of this block.</li>
<li class="mt-2"><code class="text-sm">production</code>: how the block was produced. Either <code class="text-sm">backfill</code> or <code class="text-sm">live</code>.</li>
<li class="mt-2"><code class="text-sm">data</code>: a list of encoded block data.</li>
</ul>
<p class="leading-7 [&amp;:not(:first-child)]:mt-6">Notice how the <code class="text-sm">data</code> field is a <em>list of block data</em>. This sounds
counter-intuitive since the <code class="text-sm">Data</code> message contains data about a <em>single
block</em>. The reason is that, as we&#x27;ve seen in the <em>&quot;Request&quot;</em> section,
the client can specify a list of filters. The <code class="text-sm">data</code> field has the same
length as the request&#x27;s <code class="text-sm">filters</code> field.
In most cases, the client specifies a single filter and receives a single block
of data. For advanced use cases (like tracking contracts deployed by a
factory), the client uses multiple filters to have parallel streams of data
synced on the block number.</p>
<h4 class="font-heading mt-8 text-lg font-semibold tracking-tight" id="invalidate-message"><code class="text-sm">Invalidate</code> message</h4>
<p class="leading-7 [&amp;:not(:first-child)]:mt-6">This message warns the client about a chain reorganization. It contains the
following fields:</p>
<ul class="my-6 ml-6 list-disc">
<li class="mt-2"><code class="text-sm">cursor</code>: the new chain&#x27;s head. All previously received messages where the
<code class="text-sm">end_cursor.order_key</code> was greater than (<code class="text-sm">&gt;</code>) this message
<code class="text-sm">cursor.order_key</code> should be considered invalid/recalled.</li>
<li class="mt-2"><code class="text-sm">removed</code>: a list of cursors that used to belong to the canonical chain.</li>
</ul>
<h4 class="font-heading mt-8 text-lg font-semibold tracking-tight" id="finalize-message"><code class="text-sm">Finalize</code> message</h4>
<p class="leading-7 [&amp;:not(:first-child)]:mt-6">This message contains a single <code class="text-sm">cursor</code> field with the cursor of the most
recent finalized block. All data at or before this block can&#x27;t be part of a
chain reorganization.</p>
<p class="leading-7 [&amp;:not(:first-child)]:mt-6">This message is useful to prune old data.</p>
<h4 class="font-heading mt-8 text-lg font-semibold tracking-tight" id="heartbeat-message"><code class="text-sm">Heartbeat</code> message</h4>
<p class="leading-7 [&amp;:not(:first-child)]:mt-6">This message is sent at regular intervals once the stream reaches the chain&#x27;s head.</p>
<p class="leading-7 [&amp;:not(:first-child)]:mt-6">Clients can detect if the stream hang by adding a timeout to the stream&#x27;s <em>receive</em>
method.</p>
<h4 class="font-heading mt-8 text-lg font-semibold tracking-tight" id="sytemmessage-message"><code class="text-sm">SytemMessage</code> message</h4>
<p class="leading-7 [&amp;:not(:first-child)]:mt-6">This message is used by the server to send out-of-band messages to the client.
It contains text messages such as data usage, warnings about reaching the free
quota, or information about upcoming system upgrades.</p>
