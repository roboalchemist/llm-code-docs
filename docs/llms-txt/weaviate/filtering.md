tocCollapsibleButton_TO0P">On this page</button></div><div class="theme-doc-markdown markdown"><header><h1>Filtering</h1></header><p>Weaviate provides powerful filtered vector search capabilities, allowing you to combine vector searches with structured, scalar filters. This enables you to find the closest vectors to a query vector that also match certain conditions.</p>
<p>Filtered vector search in Weaviate is based on the concept of pre-filtering. This means that the filter is constructed before the vector search is performed. Unlike some pre-filtering implementations, Weaviate&#x27;s pre-filtering does not require a brute-force vector search and is highly efficient.</p>
<p>Starting in <code>v1.34</code>, Weaviate uses the <a href="#acorn" class=""><code>ACORN</code></a> filter strategy as the default. This filtering method significantly improves performance for large datasets, especially when the filter has low correlation with the query vector.</p>
<h2 class="anchor anchorTargetStickyNavbar_Vzrq" id="post-filtering-vs-pre-filtering">Post-Filtering vs Pre-Filtering<a href="#post-filtering-vs-pre-filtering" class="hash-link" aria-label="Direct link to Post-Filtering vs Pre-Filtering" title="Direct link to Post-Filtering vs Pre-Filtering" translate="no">​</a></h2>
<p>Systems that cannot make use of pre-filtering typically have to make use of post-filtering. This is an approach where a vector search is performed first and then some results are removed which do not match the filter. This leads to two major disadvantages:</p>
<ol>
<li class="">You cannot easily predict how many elements will be contained in the search, as the filter is applied to an already reduced list of candidates.</li>
<li class="">If the filter is very restrictive, i.e. it matches only a small percentage of data points relative to the size of the data set, there is a chance that the original vector search does not contain any match at all.</li>
</ol>
<p>The limitations of post-filtering are overcome by pre-filtering. Pre-Filtering describes an approach where eligible candidates are determined before a vector search is started. The vector search then only considers candidates that are present on the &quot;allow&quot; list.</p>
<div class="theme-admonition theme-admonition-note admonition_xJq3 alert alert--secondary"><div class="admonitionHeading_Gvgb"><span class="admonitionIcon_Rf37"><svg viewBox="0 0 14 16"><path fill-rule="evenodd" d="M6.3 5.69a.942.942 0 0 1-.28-.7c0-.28.09-.52.28-.7.19-.18.42-.28.7-.28.28 0 .52.09.7.28.18.19.28.42.28.7 0 .28-.09.52-.28.7a1 1 0 0 1-.7.3c-.28 0-.52-.11-.7-.3zM8 7.99c-.02-.25-.11-.48-.31-.69-.2-.19-.42-.3-.69-.31H6c-.27.02-.48.13-.69.31-.2.2-.3.44-.31.69h1v3c.02.27.11.5.31.69.2.2.42.31.69.31h1c.27 0 .48-.11.69-.31.2-.19.3-.42.31-.69H8V7.98v.01zM7 2.3c-3.14 0-5.7 2.54-5.7 5.68 0 3.14 2.56 5.7 5.7 5.7s5.7-2.55 5.7-5.7c0-3.15-2.56-5.69-5.7-5.69v.01zM7 .98c3.86 0 7 3.14 7 7s-3.14 7-7 7-7-3.12-7-7 3.14-7 7-7z"></path></svg></span>note</div><div class="admonitionContent_BuS1"><p>Some authors make a distinction between &quot;pre-filtering&quot; and &quot;single-stage filtering&quot; where the former implies a brute-force search and the latter does not. We do not make this distinction, as Weaviate does not have to resort to brute-force searches, even when pre-filtering due to the its combined inverted index and HNSW index.</p></div></div>
<h2 class="anchor anchorTargetStickyNavbar_Vzrq" id="efficient-pre-filtered-searches-in-weaviate">Efficient Pre-Filtered Searches in Weaviate<a href="#efficient-pre-filtered-searches-in-weaviate" class="hash-link" aria-label="Direct link to Efficient Pre-Filtered Searches in Weaviate" title="Direct link to Efficient Pre-Filtered Searches in Weaviate" translate="no">​</a></h2>
<p>In the section about Storage, <a class="" href="/weaviate/concepts/storage">we have described in detail which parts make up a shard in Weaviate</a>. Most notably, each shard contains an inverted index right next to the HNSW index. This allows for efficient pre-filtering. The process is as follows:</p>
<ol>
<li class="">An inverted index (similar to a traditional search engine) is used to create an allow-list of eligible candidates. This list is essentially a list of <code>uint64</code> ids, so it can grow very large without sacrificing efficiency.</li>
<li class="">A vector search is performed where the allow-list is passed to the HNSW index. The index will move along any node&#x27;s edges normally, but will only add ids to the result set that are present on the allow list. The exit conditions for the search are the same as for an unfiltered search: The search will stop when the desired limit is reached and additional candidates no longer improve the result quality.</li>
</ol>
<h2 class="anchor anchorTargetStickyNavbar_Vzrq" id="filter-strategy">Filter strategy<a href="#filter-strategy" class="hash-link" aria-label="Direct link to Filter strategy" title="Direct link to Filter strategy" translate="no">​</a></h2>
<p>Weaviate supports two filter strategies: <code>sweeping</code> and <code>acorn</code> specifically for the HNSW index type.</p>
<h3 class="anchor anchorTargetStickyNavbar_Vzrq" id="acorn">ACORN<a href="#acorn" class="hash-link" aria-label="Direct link to ACORN" title="Direct link to ACORN" translate="no">​</a></h3>
<p>The Weaviate filtering algorithm <code>ACORN</code> is based on the paper <a href="https://arxiv.org/html/2403.04871v1" target="_blank" rel="noopener noreferrer" class="">ACORN: Performant and Predicate-Agnostic Search Over Vector Embeddings and Structured Data</a>. We refer to this as <code>ACORN</code>, but the actual implementation in Weaviate is a custom implementation that is inspired by the paper. (References to <code>ACORN</code> in this document refer to the Weaviate implementation.)</p>
<p>The <code>ACORN</code> algorithm is designed to speed up filtered searches with the <a class="" href="/weaviate/concepts/vector-index#hierarchical-navigable-small-world-hnsw-index">HNSW index</a> by the following:</p>
<ul>
<li class="">Objects that do not meet the filters are ignored in distance calculations.</li>
<li class="">The algorithm reaches the relevant part of the HNSW graph faster, by using a multi-hop approach to evaluate the neighborhood of candidates.</li>
<li class="">Additional entrypoints matching the filter are randomly seeded to speed up convergence to the filtered zone.</li>
</ul>
<p>The <code>ACORN</code> algorithm is especially useful when the filter has low correlation with the query vector. In other words, when a filter excludes many objects in the region of the graph most similar to the query vector.</p>
<p>Our internal testing indicates that for lowly correlated, restrictive filters, the <code>ACORN</code> algorithm can be significantly faster, especially for large datasets. If this has been a bottleneck for your use case, we recommend enabling the <code>ACORN</code> algorithm.</p>
<h3 class="anchor anchorTargetStickyNavbar_Vzrq" id="sweeping">Sweeping<a href="#sweeping" class="hash-link" aria-label="Direct link to Sweeping" title="Direct link to Sweeping" translate="no">​</a></h3>
<p>The <code>sweeping</code> strategy is based on the concept of &quot;sweeping&quot; through the HNSW graph.</p>
<p>The algorithm starts at the root node and traverses the graph, evaluating the distance to the query vector at each node, while keeping the &quot;allow list&quot; of the filter as context. If the filter is not met, the node is skipped and the traversal continues. This process is repeated until the desired number of results is reached.</p>
<p>The <code>sweeping</code> algorithm can be enabled by setting the <code>filterStrategy</code> field for the relevant HNSW vector index <a class="" href="/weaviate/manage-collections/vector-config#set-vector-index-parameters">in the collection configuration</a>.</p>
<h2 class="anchor anchorTargetStickyNavbar_Vzrq" id="indexFilterable"><code>indexFilterable</code><a href="#indexFilterable" class="hash-link" aria-label="Direct link to indexFilterable" title="Direct link to indexFilterable" translate="no">​</a></h2>
<p>The <code>indexFilterable</code> index speeds up match-based filtering through use of Roaring Bitmaps. Roaring Bitmaps employ various strategies to add efficiencies, whereby it divides data into chunks and applies an appropriate storage strategy to each one. This enables high data compression and set operations speeds, resulting in faster filtering speeds for Weaviate.</p>
<p>If you are dealing with a large dataset, this will likely improve your filtering performance significantly and we therefore encourage you to migrate and re-index.</p>
<p>In addition, our team maintains our underlying Roaring Bitmap library to address any issues and make improvements as needed.</p>
<h4 class="anchor anchorTargetStickyNavbar_Vzrq" id="indexfilterable-for-text-properties"><code>indexFilterable</code> for <code>text</code> properties<a href="#indexfilterable-for-text-properties" class="hash-link" aria-label="Direct link to indexfilterable-for-text-properties" title="Direct link to indexfilterable-for-text-properties" translate="no">​</a></h4>
<p>A roaring bitmap index for <code>text</code> properties is implemented using two separate (<code>filterable</code> &amp; <code>searchable</code>) indexes, which replaces the existing single index. You can configure the new <code>indexFilterable</code> and <code>indexSearchable</code> parameters to determine whether to create the roaring set index and the BM25-suitable Map index, respectively. (Both are enabled by default.)</p>
<div class="theme-admonition theme-admonition-info admonition_xJq3 alert alert--info"><div class="admonitionHeading_Gvgb"><span class="admonitionIcon_Rf37"><svg viewBox="0 0 14 16"><path fill-rule="evenodd" d="M7 2.3c3.14 0 5.7 2.56 5.7 5.7s-2.56 5.7-5.7 5.7A5.71 5.71 0 0 1 1.3 8c0-3.14 2.56-5.7 5.7-5.7zM7 1C3.14 1 0 4.14 0 8s3.14 7 7 7 7-3.14 7-7-3.14-7-7-7zm1 3H6v5h2V4zm0 6H6v2h2v-2z"></path></svg></span>Read more</div><div class="admonitionContent_BuS1"><p>To learn more about Weaviate&#x27;s roaring bitmaps implementation, see the <a href="https://pkg.go.dev/github.com/weaviate/weaviate/adapters/repos/db/lsmkv/roaringset" target="_blank" rel="noopener noreferrer" class="">in-line documentation</a>.</p></div></div>
<h2 class="anchor anchorTargetStickyNavbar_Vzrq" id="indexrangefilters"><code>indexRangeFilters</code><a href="#indexrangefilters" class="hash-link" aria-label="Direct link to indexrangefilters" title="Direct link to indexrangefilters" translate="no">​</a></h2>
<p>The <code>indexRangeFilters</code> index is a range-based index for filtering by numerical ranges. This index is available for <code>int</code>, <code>number</code>, or <code>date</code> properties. The index is not available for arrays of these data types.</p>
<p>Internally, rangeable indexes are implemented as roaring bitmap slices. This data structure limits the index to values that can be stored as 64 bit integers.</p>
<p><code>indexRangeFilters</code> is only available for new properties. Existing properties cannot be converted to use the rangeable index.</p>
<h2 class="anchor anchorTargetStickyNavbar_Vzrq" id="recall-on-pre-filtered-searches">Recall on Pre-Filtered Searches<a href="#recall-on-pre-filtered-searches" class="hash-link" aria-label="Direct link to Recall on Pre-Filtered Searches" title="Direct link to Recall on Pre-Filtered Searches" translate="no">​</a></h2>
<p>Thanks to Weaviate&#x27;s custom HNSW implementation, which persists in following all links in the HNSW graph normally and only applying the filter condition when considering the result set, graph integrity is kept intact. The recall of a filtered search is typically not any worse than that of an unfiltered search.</p>
<p>The graphic below shows filters of varying levels of restrictiveness. From left (100% of dataset matched) to right (1% of dataset matched) the filters become more restrictive without negatively affecting recall on <code>k=10</code>, <code>k=15</code> and <code>k=20</code> vector searches with filters.</p>
<p><img decoding="async" loading="lazy" alt="Recall for filtered vector search" src="/assets/images/recall-of-filtered-vector-search-d077dd892373cbf0dbda9f2246fe08ca.png" title="Recall of filtered vector search in Weaviate" width="2000" height="1146" class="img_ev3q"></p>
<h2 class="anchor anchorTargetStickyNavbar_Vzrq" id="flat-search-cutoff">Flat-Search Cutoff<a href="#flat-search-cutoff" class="hash-link" aria-label="Direct link to Flat-Search Cutoff" title="Direct link to Flat-Search Cutoff" translate="no">​</a></h2>
<p>Weaviate offers an option to automatically switch to a flat (brute-force) vector search when a filter becomes too restrictive. This scenario only applies to combined vector and scalar searches. For a detailed explanation of why HNSW requires switching to a flat search on certain filters, see this article at <a href="https://medium.com/data-science/effects-of-filtered-hnsw-searches-on-recall-and-latency-434becf8041c" target="_blank" rel="noopener noreferrer" class="">medium</a>. In short, if a filter is very restrictive (i.e. a small percentage of the dataset is matched), an HNSW traversal becomes exhaustive. In other words, the more restrictive the filter becomes, the closer the performance of HNSW is to a brute-force search on the entire dataset. However, with such a restrictive filter, we have already narrowed down the dataset to a small fraction. So if the performance is close to brute-force anyway, it is much more efficient to only search on the matching subset as opposed to the entire dataset.</p>
<p>The following graphic shows filters with varying restrictiveness. From left (0%) to right (100%), the filters become more restrictive. The <strong>cut-off is configured at ~15% of the dataset</strong> size.  This means the right side of the dotted line uses a brute-force search.</p>
<p><img decoding="async" loading="lazy" alt="Prefiltering with flat search cutoff" src="/assets/images/prefiltering-response-times-with-filter-cutoff-b8c34a04311e39dfabcaa2de0b7fbae4.png" title="Prefiltering with flat search cutoff" width="2000" height="1154" class="img_ev3q"></p>
<p>As a comparison, with pure HNSW - without the cutoff - the same filters would look like the following:</p>
<p><img decoding="async" loading="lazy" alt="Prefiltering with pure HNSW" src="/assets/images/prefiltering-pure-hnsw-without-cutoff-8a742335e9d51e8047456658b817b14b.png" title="Prefiltering without cutoff, i.e. pure HNSW" width="2125" height="1216" class="img_ev3q"></p>
<p>The cutoff value can be configured as <a class="" href="/weaviate/config-refs/indexing/vector-index#hnsw-index">part of the <code>vectorIndexConfig</code> settings in the schema</a> for each collection separately.</p>
