While you design you system, keep in mind CPU and memory management. CPU and memory are the primary resources for Weaviate instances. Depending on the modules you use, GPUs may also play a role.</p>
<h2 class="anchor anchorTargetStickyNavbar_Vzrq" id="limit-available-resources">Limit available resources<a href="#limit-available-resources" class="hash-link" aria-label="Direct link to Limit available resources" title="Direct link to Limit available resources" translate="no">​</a></h2>
<p>You can set <a class="" href="/deploy/configuration/env-vars">environment variables</a> to manage Weaviate&#x27;s resource usage, as to prevent Weaviate from using all available resources. The following environment variables are available:</p>
<ul>
<li class="">
<p><a class="" href="/deploy/configuration/env-vars#LIMIT_RESOURCES"><code>LIMIT_RESOURCES</code></a>: When set to true, Weaviate automatically limits its resource usage. It sets memory usage to 80% of the total memory and uses all but one CPU core. It overrides any <code>GOMEMLIMIT</code> values but respects <code>GOMAXPROCS</code> settings.</p>
</li>
<li class="">
<p><a class="" href="/deploy/configuration/env-vars#GOMEMLIMIT"><code>GOMEMLIMIT</code></a>: This sets the memory limit for the Go runtime, which should be around 80-90% of the total memory available for Weaviate. It controls the aggressiveness of the Garbage Collector as memory usage approaches this limit.</p>
</li>
<li class="">
<p><a class="" href="/deploy/configuration/env-vars#GOMAXPROCS"><code>GOMAXPROCS</code></a>: This sets the maximum number of threads for concurrent execution. If set, it&#x27;s respected by <code>LIMIT_RESOURCES</code>, allowing users to specify the exact number of CPU cores Weaviate should use.</p>
</li>
</ul>
<p>These settings help in optimizing Weaviate&#x27;s performance by balancing resource utilization with the available system resources.</p>
<h2 class="anchor anchorTargetStickyNavbar_Vzrq" id="the-role-of-cpus">The role of CPUs<a href="#the-role-of-cpus" class="hash-link" aria-label="Direct link to The role of CPUs" title="Direct link to The role of CPUs" translate="no">​</a></h2>
<div class="theme-admonition theme-admonition-tip admonition_xJq3 alert alert--success"><div class="admonitionHeading_Gvgb"><span class="admonitionIcon_Rf37"><svg viewBox="0 0 12 16"><path fill-rule="evenodd" d="M6.5 0C3.48 0 1 2.19 1 5c0 .92.55 2.25 1 3 1.34 2.25 1.78 2.78 2 4v1h5v-1c.22-1.22.66-1.75 2-4 .45-.75 1-2.08 1-3 0-2.81-2.48-5-5.5-5zm3.64 7.48c-.25.44-.47.8-.67 1.11-.86 1.41-1.25 2.06-1.45 3.23-.02.05-.02.11-.02.17H5c0-.06 0-.13-.02-.17-.2-1.17-.59-1.83-1.45-3.23-.2-.31-.42-.67-.67-1.11C2.44 6.78 2 5.65 2 5c0-2.2 2.02-4 4.5-4 1.22 0 2.36.42 3.22 1.19C10.55 2.94 11 3.94 11 5c0 .66-.44 1.78-.86 2.48zM4 14h5c-.23 1.14-1.3 2-2.5 2s-2.27-.86-2.5-2z"></path></svg></span>Rule of thumb</div><div class="admonitionContent_BuS1"><p>The CPU has a direct effect on query and import speed, but does not affect dataset size.</p></div></div>
<p>Vector search is the most CPU intensive process in Weaviate operations. Queries are CPU-bound, but imports are also CPU-bound because imports rely on vector search for indexing. Weaviate uses the HNSW (Hierarchical Navigable Small World) algorithm to index vectors. You can <a class="" href="/weaviate/config-refs/indexing/vector-index">tune the HNSW index</a> on a per collection basis in order to maximize performance for your primary use case.</p>
<p>To use multiple CPUs efficiently, create multiple shards for your collection. For the fastest imports, create multiple shards even on a single node.</p>
<p>Each insert, or search, is single-threaded. However, if you make multiple searches or inserts at the same time, Weaviate can make use of multiple threads. <a class="" href="/weaviate/manage-objects/import">Batch inserts</a> use multiple threads to process data in parallel.</p>
<h3 class="anchor anchorTargetStickyNavbar_Vzrq" id="when-to-add-more-cpus">When to add more CPUs<a href="#when-to-add-more-cpus" class="hash-link" aria-label="Direct link to When to add more CPUs" title="Direct link to When to add more CPUs" translate="no">​</a></h3>
<p>When CPU utilization is high during importing, add CPUs to increases import speed.</p>
<p>When search throughput is limited, add CPUs to increase the number of queries per second.</p>
<h2 class="anchor anchorTargetStickyNavbar_Vzrq" id="the-role-of-memory">The role of memory<a href="#the-role-of-memory" class="hash-link" aria-label="Direct link to The role of memory" title="Direct link to The role of memory" translate="no">​</a></h2>
<div class="theme-admonition theme-admonition-tip admonition_xJq3 alert alert--success"><div class="admonitionHeading_Gvgb"><span class="admonitionIcon_Rf37"><svg viewBox="0 0 12 16"><path fill-rule="evenodd" d="M6.5 0C3.48 0 1 2.19 1 5c0 .92.55 2.25 1 3 1.34 2.25 1.78 2.78 2 4v1h5v-1c.22-1.22.66-1.75 2-4 .45-.75 1-2.08 1-3 0-2.81-2.48-5-5.5-5zm3.64 7.48c-.25.44-.47.8-.67 1.11-.86 1.41-1.25 2.06-1.45 3.23-.02.05-.02.11-.02.17H5c0-.06 0-.13-.02-.17-.2-1.17-.59-1.83-1.45-3.23-.2-.31-.42-.67-.67-1.11C2.44 6.78 2 5.65 2 5c0-2.2 2.02-4 4.5-4 1.22 0 2.36.42 3.22 1.19C10.55 2.94 11 3.94 11 5c0 .66-.44 1.78-.86 2.48zM4 14h5c-.23 1.14-1.3 2-2.5 2s-2.27-.86-2.5-2z"></path></svg></span>Rule of thumb</div><div class="admonitionContent_BuS1"><p>Memory determines the maximum supported dataset size. Memory does not directly influence query speed.</p></div></div>
<p>The HNSW index must be stored in memory. The memory required is directly related to the size of your dataset. There is no correlation between the size of your dataset and the current query load. You can use <a class="" href="/weaviate/concepts/vector-quantization#product-quantization"><code>product quantization (PQ)</code></a> to compress the vectors in your dataset in increase the number of vectors your can hold in memory.</p>
<p>Weaviate let&#x27;s you configure a limit to the number of vectors held in memory in order to prevent unexpected Out-of-Memory (&quot;OOM&quot;) situations. The default value is one trillion (<code>1e12</code>) objects.  per collection. To adjust the number of objects, update the value of <a class="" href="/weaviate/config-refs/indexing/vector-index"><code>vectorCacheMaxObjects</code></a> in your index settings.</p>
<p>Weaviate also uses <a href="https://en.wikipedia.org/wiki/Memory-mapped_file" target="_blank" rel="noopener noreferrer" class="">memory-mapped files</a> for data stored on disks. Memory-mapped files are efficient, but disk storage is much slower than in-memory storage.</p>
<h3 class="anchor anchorTargetStickyNavbar_Vzrq" id="which-factors-drive-memory-usage">Which factors drive memory usage?<a href="#which-factors-drive-memory-usage" class="hash-link" aria-label="Direct link to Which factors drive memory usage?" title="Direct link to Which factors drive memory usage?" translate="no">​</a></h3>
<p>The HNSW vector index is the primary driver of memory usage. These factors influence the amount of memory Weaviate uses:</p>
<ul>
<li class=""><strong>The total number of object vectors</strong>. The number of vectors is important, but the raw size of the original objects is not important. Only the vector is stored in memory. The size of the original text or other data is not a limiting factor.</li>
<li class=""><strong>The <code>maxConnections</code> HNSW index setting</strong>. Each object in memory has at most <a class="" href="/weaviate/config-refs/indexing/vector-index"><code>maxConnections</code></a> connections per layer. Each of the connections uses 8-10B of memory. Note that the base layer allows for <code>2 * maxConnections</code>.</li>
</ul>
<h3 class="anchor anchorTargetStickyNavbar_Vzrq" id="an-example-calculation">An example calculation<a href="#an-example-calculation" class="hash-link" aria-label="Direct link to An example calculation" title="Direct link to An example calculation" translate="no">​</a></h3>
<div class="theme-admonition theme-admonition-note admonition_xJq3 alert alert--secondary"><div class="admonitionHeading_Gvgb"><span class="admonitionIcon_Rf37"><svg viewBox="0 0 14 16"><path fill-rule="evenodd" d="M6.3 5.69a.942.942 0 0 1-.28-.7c0-.28.09-.52.28-.7.19-.18.42-.28.7-.28.28 0 .52.09.7.28.18.19.28.42.28.7 0 .28-.09.52-.28.7a1 1 0 0 1-.7.3c-.28 0-.52-.11-.7-.3zM8 7.99c-.02-.25-.11-.48-.31-.69-.2-.19-.42-.3-.69-.31H6c-.27.02-.48.13-.69.31-.2.2-.3.44-.31.69h1v3c.02.27.11.5.31.69.2.2.42.31.69.31h1c.27 0 .48-.11.69-.31.2-.19.3-.42.31-.69H8V7.98v.01zM7 2.3c-3.14 0-5.7 2.54-5.7 5.68 0 3.14 2.56 5.7 5.7 5.7s5.7-2.55 5.7-5.7c0-3.15-2.56-5.69-5.7-5.69v.01zM7 .98c3.86 0 7 3.14 7 7s-3.14 7-7 7-7-3.12-7-7 3.14-7 7-7z"></path></svg></span>note</div><div class="admonitionContent_BuS1"><p>The following calculation assumes that you want to hold all vectors in memory. For a hybrid approach that combines in memory and on-disk storage, see <a href="#vector-cache" class="">Vector Cache</a> below.</p></div></div>
<p>To estimate your memory needs, use the following rule of thumb:</p>
<p><code>Memory usage = 2 * (the memory footprint of all vectors)</code></p>
<p>For example, consider a model that has one million 384-dimensional vectors of type <code>float32</code>.</p>
<ul>
<li class="">
<p>The memory requirement for a single vector is: <code>384 * 4 B = 1536 B</code>.</p>
</li>
<li class="">
<p>The memory requirement for one million objects is: <code>1e6 * 1536 B = 1.5G B</code></p>
</li>
</ul>
<p>The rule of thumb says to double the memory requirement. The total memory requirement for one million 384-dimensional vectors of type <code>float32</code> is: <code>2 * 1e6 * 1536 B = 3 GB</code>.</p>
<p>For a more accurate calculation, include a factor for the <a class="" href="/weaviate/config-refs/indexing/vector-index"><code>maxConnections</code></a> setting instead of multiplying the base requirement by two.</p>
<p>For example, if <code>maxConnections</code> is 64 and the other values are the same, a more accurate memory estimate is <code>1e6 * (1536B + (64 * 10)) = 2.2 GB</code>.</p>
<p>The estimate that includes <code>maxConnections</code> is smaller than the rule of thumb estimate. However, the <code>maxConnections</code> estimate doesn&#x27;t account for garbage collection. Garbage collection adds overhead that is explained in the next section.</p>
<h2 class="anchor anchorTargetStickyNavbar_Vzrq" id="effects-of-garbage-collection">Effects of garbage collection<a href="#effects-of-garbage-collection" class="hash-link" aria-label="Direct link to Effects of garbage collection" title="Direct link to Effects of garbage collection" translate="no">​</a></h2>
<p>Weaviate is written in Go, which is a garbage-collected language. This means some memory is not immediately available for reuse when it is no longer needed. The application has to wait for an asynchronous process, the garbage collector, to free up the memory. This has two distinct effects on memory use:</p>
<ul>
<li class=""><a href="#memory-overhead-for-the-garbage-collector" class="">Memory overhead</a></li>
<li class=""><a href="#out-of-memory-issues-due-to-garbage-collection" class="">Out-of-memory issues</a></li>
</ul>
<h3 class="anchor anchorTargetStickyNavbar_Vzrq" id="memory-overhead-for-the-garbage-collector">Memory overhead for the garbage collector<a href="#memory-overhead-for-the-garbage-collector" class="hash-link" aria-label="Direct link to Memory overhead for the garbage collector" title="Direct link to Memory overhead for the garbage collector" translate="no">​</a></h3>
<p>The memory calculation that includes <code>maxConnections</code> describes the system state at rest. However, while Weaviate imports vectors, additional memory is allocated and eventually freed by the garbage collector. Since garbage collection is an asynchronous process, this additional memory must also be accounted for. The &#x27;rule of thumb&#x27; formula accounts for garbage collection.</p>
<h3 class="anchor anchorTargetStickyNavbar_Vzrq" id="out-of-memory-issues-due-to-garbage-collection">Out-of-Memory issues due to garbage collection<a href="#out-of-memory-issues-due-to-garbage-collection" class="hash-link" aria-label="Direct link to Out-of-Memory issues due to garbage collection" title="Direct link to Out-of-Memory issues due to garbage collection" translate="no">​</a></h3>
<p>In rare situations - typically on large machines with very high import speeds - Weaviate can allocate memory faster than the garbage collector can free it. When this happens, the system kernel can trigger an <code>out of memory kill (OOM-Kill)</code>. This is a known issue that Weaviate is actively working on.</p>
<h3 class="anchor anchorTargetStickyNavbar_Vzrq" id="data-import">Data import<a href="#data-import" class="hash-link" aria-label="Direct link to Data import" title="Direct link to Data import" translate="no">​</a></h3>
<p>To avoid out-of-memory issues during imports, set <code>LIMIT_RESOURCES</code> to <code>True</code> or configure the <code>GOMEMLIMIT</code> environment variable. For details, see <a class="" href="/deploy/configuration/env-vars">Environment variables</a>.</p>
<h2 class="anchor anchorTargetStickyNavbar_Vzrq" id="strategies-to-reduce-memory-usage">Strategies to reduce memory usage<a href="#strategies-to-reduce-memory-usage" class="hash-link" aria-label="Direct link to Strategies to reduce memory usage" title="Direct link to Strategies to reduce memory usage" translate="no">​</a></h2>
<p>The following tactics can help to reduce Weaviate&#x27;s memory usage:</p>
<ul>
<li class="">
<p><strong>Use vector compression</strong>. Product quantization (PQ) is a technique that reduces the size of vectors. Vector compression impacts recall performance, so we recommend testing PQ on your dataset before using it in production. <br><br> For more information, see <a class="" href="/weaviate/concepts/vector-quantization">Product Quantization</a>. <br> To configure PQ, see <a class="" href="/weaviate/configuration/compression/pq-compression">Compression</a>.</p>
</li>
<li class="">
<p><strong>Reduce the dimensionality of your vectors.</strong> The most effective approach to reducing memory size, is to reduce the number of dimensions per vector. If you have high dimension vectors, consider using a model that uses fewer dimensions. For example, a model that has 384 dimensions uses far less memory than a model with 1536 dimensions.</p>
</li>
<li class="">
<p><strong>Reduce the number of <a class="" href="/weaviate/config-refs/indexing/vector-index"><code>maxConnections</code></a> in your HNSW index settings</strong>. Each object in memory has up to <code>maxConnections</code> connections. Each of those connections uses 8-10B of memory. To reduce the overall memory footprint, reduce <code>maxConnections</code>.</p>
</li>
</ul>
<p>Reducing <code>maxConnections</code> adversely affects HNSW recall performance. To mitigate this effect, increase one or both of the <code>efConstruction</code> and <code>ef</code> parameters.</p>
<ul>
<li class="">
<p>Increasing <code>efConstruction</code> increases import time without affecting query times.</p>
</li>
<li class="">
<p>Increasing <code>ef</code> increases query times without affecting import times.</p>
</li>
<li class="">
<p><strong>Use a vector cache that is smaller than the total amount of your vectors (not recommended)</strong>. This strategy is described under <a href="#vector-cache" class="">Vector Cache</a> below. It has a significant performance impact, and is only recommended in specific, limited situations.</p>
</li>
</ul>
<h2 class="anchor anchorTargetStickyNavbar_Vzrq" id="vector-cache">Vector Cache<a href="#vector-cache" class="hash-link" aria-label="Direct link to Vector Cache" title="Direct link to Vector Cache" translate="no">​</a></h2>
<p>For optimal search and import performance, all previously imported vectors need to be held in memory. The size of the vector cache is specified by the <a class="" href="/weaviate/config-refs/indexing/vector-index"><code>vectorCacheMaxObjects</code></a> parameter in the collection definition. By default this limit is set to one trillion (<code>1e12</code>) objects when you create a new collection.</p>
<p>You can reduce the size of <code>vectorCacheMaxObjects</code>, but a disk lookup for a vector is orders of magnitudes slower than memory lookup. Only reduce the size of <code>vectorCacheMaxObjects</code> with care and as a last resort.</p>
<p>Generally we recommend that:</p>
<ul>
<li class="">
<p>During import set <code>vectorCacheMaxObjects</code> high enough that all vectors can be held in memory. Each import requires multiple searches. Import performance drop drastically when there isn&#x27;t enough memory to hold all of the vectors in the cache.</p>
</li>
<li class="">
<p>After import, when your workload is mostly querying, experiment with vector cache limits that are less than your total dataset size.</p>
<p>Vectors that aren&#x27;t currently in cache are added to the cache if there is still room. If the cache fills, Weaviate drops the whole cache. All future vectors have to be read from disk for the first time. Then, subsequent queries runs against the cache, until it fills again and the procedure repeats. Note that the cache can be a very valuable tool if you have a large dataset, and a large percentage of users only query a specific subset of vectors. In this case you might be able to serve the largest user group from cache while requiring disk lookups for &quot;irregular&quot; queries.</p>
</li>
</ul>
<h3 class="anchor anchorTargetStickyNavbar_Vzrq" id="when-to-add-more-memory-to-your-weaviate-machine-or-cluster">When to add more Memory to your Weaviate machine or cluster<a href="#when-to-add-more-memory-to-your-weaviate-machine-or-cluster" class="hash-link" aria-label="Direct link to When to add more Memory to your Weaviate machine or cluster" title="Direct link to When to add more Memory to your Weaviate machine or cluster" translate="no">​</a></h3>
<p>Consider adding more memory if:</p>
<ul>
<li class="">You want to import a larger dataset (more common).</li>
<li class="">Exact lookups are disk-bound and more memory will improve page-caching (less common).</li>
</ul>
<h2 class="anchor anchorTargetStickyNavbar_Vzrq" id="the-role-of-gpus-in-weaviate">The role of GPUs in Weaviate<a href="#the-role-of-gpus-in-weaviate" class="hash-link" aria-label="Direct link to The role of GPUs in Weaviate" title="Direct link to The role of GPUs in Weaviate" translate="no">​</a></h2>
<p>Weaviate Database itself does not make use of GPUs. However, some of the models that Weaviate includes as modules are meant to run with GPUs, for example <code>text2vec-transformers</code>, <code>qna-transformers</code>, and <code>ner-transformers</code>. These modules run in isolated containers, so you can run the module containers on GPU-accelerated hardware while running Weaviate Database on low-cost CPU-only hardware.</p>
<h2 class="anchor anchorTargetStickyNavbar_Vzrq" id="disks-ssd-vs-spinning-disk">Disks: SSD vs Spinning Disk<a href="#disks-ssd-vs-spinning-disk" class="hash-link" aria-label="Direct link to Disks: SSD vs Spinning Disk" title="Direct link to Disks: SSD vs Spinning Disk" translate="no">​</a></h2>
<p>Weaviate is optimized to work with Solid-State Disks (SSDs). However, spinning hard-disks can also be used with some performance penalties.</p>
<h2 class="anchor anchorTargetStickyNavbar_Vzrq" id="file-system">File system<a href="#file-system" class="hash-link" aria-label="Direct link to File system" title="Direct link to File system" translate="no">​</a></h2>
<p>For optimal performance and reliability, avoid using <code>NFS</code> or similar file systems for the Weaviate persistent volume (<a class="" href="/deploy/configuration/env-vars"><code>PERSISTENCE_DATA_PATH</code></a>).</p>
<p>Instead, use file systems like <code>Ext4</code> or <code>XFS</code> in combination with SAN storage (e.g. <code>EBS</code>) to ensure the best performance.</p>
