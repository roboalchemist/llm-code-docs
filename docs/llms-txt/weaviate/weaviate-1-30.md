type="button" class="clean-btn tocCollapsibleButton_TO0P">On this page</button></div><div class="theme-doc-markdown markdown"><header><h1>BlockMax WAND migration guide</h1></header><p>This comprehensive guide covers how to migrate existing data to use the BlockMax WAND algorithm for the inverted index in Weaviate <code>v1.30</code> and above.</p>
<p><strong>BlockMax WAND</strong> offers improved search performance for BM25 and hybrid searches. While new collections created in Weaviate <code>v1.30</code> or newer will automatically use this optimized format, existing collections created before <code>v1.30</code> require migration to take advantage of these improvements.</p>
<p>To learn more about the technical details of this change, you can refer to <a href="https://weaviate.io/blog/blockmax-wand" target="_blank" rel="noopener noreferrer" class="">the BlockMax WAND blog post</a>.</p>
<h2 class="anchor anchorTargetStickyNavbar_Vzrq" id="prerequisites">Prerequisites<a href="#prerequisites" class="hash-link" aria-label="Direct link to Prerequisites" title="Direct link to Prerequisites" translate="no">​</a></h2>
<p>Before beginning the migration, ensure that:</p>
<ul>
<li class="">You have a Weaviate version before <code>v1.30</code> that doesn&#x27;t use the BlockMax WAND algorithm</li>
<li class="">You&#x27;re on the latest version of Weaviate before migrating (currently <code>1.34.4</code>)</li>
</ul>
<h2 class="anchor anchorTargetStickyNavbar_Vzrq" id="migration-considerations">Migration considerations<a href="#migration-considerations" class="hash-link" aria-label="Direct link to Migration considerations" title="Direct link to Migration considerations" translate="no">​</a></h2>
<p>Before starting the migration process, be aware of the following:</p>
<ul>
<li class="">Migration is recommended primarily if you use BM25 or hybrid search and aren&#x27;t satisfied with your current query times</li>
<li class="">Migration will increase server resource load, especially during the re-indexing stage</li>
<li class="">Migration can take hours for shards with millions of documents, depending on the number of searchable properties</li>
<li class="">If possible, schedule migration to avoid large data imports/updates/deletes, especially during the re-indexing stage</li>
<li class="">BlockMax WAND may return slightly different scores than WAND if you search on multiple properties or have a delete-heavy workflow, as it computes term statistics differently</li>
</ul>
<h2 class="anchor anchorTargetStickyNavbar_Vzrq" id="migration-process">Migration process<a href="#migration-process" class="hash-link" aria-label="Direct link to Migration process" title="Direct link to Migration process" translate="no">​</a></h2>
<p>The migration involves three main stages:</p>
<ol>
<li class=""><strong>Re-indexing</strong>: Converting the internal representation from <code>mapcollection</code> to the inverted <code>blockmax</code> format</li>
<li class=""><strong>Swapping</strong>: Making the system use the new format while maintaining the ability to roll back</li>
<li class=""><strong>Cleanup</strong>: Cleaning up old data after successful migration</li>
</ol>

<h2 class="anchor anchorTargetStickyNavbar_Vzrq" id="multi-tenancy-specific-notes">Multi-tenancy specific notes<a href="#multi-tenancy-specific-notes" class="hash-link" aria-label="Direct link to Multi-tenancy specific notes" title="Direct link to Multi-tenancy specific notes" translate="no">​</a></h2>
<p>Due to the dynamic nature of tenants, multi-tenancy collections behave slightly differently:</p>
<ul>
<li class="">With <code>REINDEX_MAP_TO_BLOCKMAX_AT_STARTUP</code> set, tenants will be migrated as they are activated</li>
<li class="">Deactivation stops migration and should be avoided as the migration may not make enough progress if the tenant is only activated for a short period</li>
<li class="">Reactivation of a tenant (active → cold → active) is equivalent to a restart:<!-- -->
<ul>
<li class="">To swap buckets on a tenant, you need to reactivate it (with <code>REINDEX_MAP_TO_BLOCKMAX_SWAP_BUCKETS=true</code> set)</li>
<li class="">For tidying up, if <code>REINDEX_MAP_TO_BLOCKMAX_TIDY_BUCKETS=true</code> is set, a swapped tenant will tidy on reactivation</li>
<li class="">Setting server variables between steps still requires a restart</li>
</ul>
</li>
<li class="">Tenants created during migration will be created in the old format and migrated like other tenants<!-- -->
<ul>
<li class="">They&#x27;ll start using BlockMax format by default if <code>REINDEX_MAP_TO_BLOCKMAX_TIDY_BUCKETS</code> is set</li>
</ul>
</li>
<li class="">After the final tidying step, the server should keep all migration variables set to ensure all tenants are eventually migrated</li>
</ul>
<h2 class="anchor anchorTargetStickyNavbar_Vzrq" id="monitoring-migration-status">Monitoring migration status<a href="#monitoring-migration-status" class="hash-link" aria-label="Direct link to Monitoring migration status" title="Direct link to Monitoring migration status" translate="no">​</a></h2>
<p>The migration process progresses through several stages, which can be monitored via the status endpoint:</p>
<ol>
<li class="">
<p><strong>Not active</strong> (Multi-tenancy only)</p>
<ul>
<li class="">Status: <code>shard_not_loaded</code></li>
<li class="">Message: <code>shard not loaded</code></li>
<li class="">Tenant isn&#x27;t active</li>
</ul>
</li>
<li class="">
<p><strong>Not Started</strong></p>
<ul>
<li class="">Status: <code>not_started</code></li>
<li class="">Message: <code>no searchable_map_to_blockmax found</code> or <code>no started.mig found</code></li>
<li class="">No migration files exist yet or process hasn&#x27;t been initiated</li>
</ul>
</li>
<li class="">
<p><strong>Started</strong></p>
<ul>
<li class="">Status: <code>started</code></li>
<li class="">Records start time from started.mig</li>
<li class="">If properties.mig doesn&#x27;t exist, message is <code>computing properties to reindex</code></li>
</ul>
</li>
<li class="">
<p><strong>In Progress</strong></p>
<ul>
<li class="">Status: <code>in progress</code></li>
<li class="">Tracks progress through <code>progress.mig.*</code> files (snapshots)</li>
<li class="">Updates <code>latest_snapshot</code> approximately every 15 minutes</li>
<li class="">If no progress files found, message is <code>no progress.mig.* files found, no snapshots created yet</code></li>
</ul>
</li>
<li class="">
<p><strong>Reindexed</strong></p>
<ul>
<li class="">Status: <code>reindexed</code></li>
<li class="">Message: <code>reindexing done</code> or <code>reindexing done, merging buckets</code></li>
<li class="">Indicates reindexing is complete</li>
</ul>
</li>
<li class="">
<p><strong>Merged</strong></p>
<ul>
<li class="">Status: <code>merged</code></li>
<li class="">Message: <code>merged reindex and ingest buckets</code></li>
<li class="">Buckets have been merged but not yet swapped</li>
</ul>
</li>
<li class="">
<p><strong>Swapped</strong></p>
<ul>
<li class="">Status: <code>swapped</code></li>
<li class="">Message: <code>swapped buckets</code> or <code>swapped X files</code></li>
<li class="">Multiple <code>swapped.mig.*</code> files may exist</li>
</ul>
</li>
<li class="">
<p><strong>Done</strong></p>
<ul>
<li class="">Status: <code>done</code></li>
<li class="">Message: <code>reindexing done</code></li>
<li class="">Final state indicating migration is complete</li>
</ul>
</li>
<li class="">
<p><strong>Error</strong> (can occur at any stage)</p>
<ul>
<li class="">Status: <code>error</code></li>
<li class="">Various error messages depending on which file operation failed</li>
</ul>
</li>
</ol>
<h2 class="anchor anchorTargetStickyNavbar_Vzrq" id="troubleshooting">Troubleshooting<a href="#troubleshooting" class="hash-link" aria-label="Direct link to Troubleshooting" title="Direct link to Troubleshooting" translate="no">​</a></h2>
<h3 class="anchor anchorTargetStickyNavbar_Vzrq" id="weaviate-crashes-during-migration">Weaviate crashes during migration<a href="#weaviate-crashes-during-migration" class="hash-link" aria-label="Direct link to Weaviate crashes during migration" title="Direct link to Weaviate crashes during migration" translate="no">​</a></h3>
<ul>
<li class="">The conversion will resume when Weaviate is restarted if the migration variables are still set</li>
<li class="">If your environment variables are reset during restart, the migration will stop</li>
<li class="">With variables unset and new data incoming, you&#x27;ll need to restart the migration process from the beginning</li>
</ul>
<h3 class="anchor anchorTargetStickyNavbar_Vzrq" id="aborting-migration-and-rolling-back">Aborting migration and rolling back<a href="#aborting-migration-and-rolling-back" class="hash-link" aria-label="Direct link to Aborting migration and rolling back" title="Direct link to Aborting migration and rolling back" translate="no">​</a></h3>
<p>If you have issues with the reindexing process and want to stop it:</p>
<ol>
<li class="">
<p>Call the abort endpoint:</p>
<div class="language-text codeBlockContainer_Ckt0 theme-code-block" style="--prism-color:#393A34;--prism-background-color:#f6f8fa"><div class="codeBlockContent_QJqH"><pre tabindex="0" class="prism-code language-text codeBlock_bY9V thin-scrollbar" style="color:#393A34;background-color:#f6f8fa"><code class="codeBlockLines_e6Vv"><span class="token-line" style="color:#393A34"><span class="token plain">http://&lt;node_name&gt;:&lt;&lt;debug_endpoint_port&gt;/debug/index/rebuild/inverted/abort</span><br></span></code></pre></div></div>
</li>
<li class="">
<p>To fully stop and rollback data to the initial stage, restart the server with:</p>
<div class="language-text codeBlockContainer_Ckt0 theme-code-block" style="--prism-color:#393A34;--prism-background-color:#f6f8fa"><div class="codeBlockContent_QJqH"><pre tabindex="0" class="prism-code language-text codeBlock_bY9V thin-scrollbar" style="color:#393A34;background-color:#f6f8fa"><code class="codeBlockLines_e6Vv"><span class="token-line" style="color:#393A34"><span class="token plain">REINDEX_MAP_TO_BLOCKMAX_AT_STARTUP=true</span><br></span><span class="token-line" style="color:#393A34"><span class="token plain">REINDEX_MAP_TO_BLOCKMAX_ROLLBACK=true</span><br></span></code></pre></div></div>
</li>
</ol>
<div class="theme-admonition theme-admonition-caution admonition_xJq3 alert alert--warning"><div class="admonitionHeading_Gvgb"><span class="admonitionIcon_Rf37"><svg viewBox="0 0 16 16"><path fill-rule="evenodd" d="M8.893 1.5c-.183-.31-.52-.5-.887-.5s-.703.19-.886.5L.138 13.499a.98.98 0 0 0 0 1.001c.193.31.53.501.886.501h13.964c.367 0 .704-.19.877-.5a1.03 1.03 0 0 0 .01-1.002L8.893 1.5zm.133 11.497H6.987v-2.003h2.039v2.003zm0-3.004H6.987V5.987h2.039v4.006z"></path></svg></span>caution</div><div class="admonitionContent_BuS1"><p>Rolling back is only possible before cleanup!</p></div></div>
<h2 class="anchor anchorTargetStickyNavbar_Vzrq" id="environment-variable-reference">Environment variable reference<a href="#environment-variable-reference" class="hash-link" aria-label="Direct link to Environment variable reference" title="Direct link to Environment variable reference" translate="no">​</a></h2>
<ul>
<li class=""><code>REINDEX_MAP_TO_BLOCKMAX_AT_STARTUP=&lt;node_names&gt;</code>: Enables and starts the migration process that converts the <code>property_&lt;property_name&gt;_searchable</code> from mapcollection to the inverted blockmax format (double writes are done). Required for other <code>REINDEX_MAP_TO_BLOCKMAX_*</code> variables to work</li>
<li class=""><code>REINDEX_MAP_TO_BLOCKMAX_SWAP_BUCKETS=&lt;node_names&gt;</code>: Swaps <code>mapcollection</code> buckets to inverted/<code>blockmax</code>, keeping double writes. Only runs on restart and if migration is finished</li>
<li class=""><code>REINDEX_MAP_TO_BLOCKMAX_UNSWAP_BUCKETS=&lt;node_names&gt;</code>: Unswaps inverted/<code>blockmax</code> buckets back to <code>mapcollection</code>, keeping double writes. Only runs on restart and if buckets were already swapped</li>
<li class=""><code>REINDEX_MAP_TO_BLOCKMAX_TIDY_BUCKETS=&lt;node_names&gt;</code>: Deletes the <code>mapcollection</code> buckets and stops double writes</li>
<li class=""><code>REINDEX_MAP_TO_BLOCKMAX_ROLLBACK=&lt;node_names&gt;</code>: Rollback migration process, restores <code>mapcollection</code> buckets (if not yet tidied) and removes created inverted/<code>blockmax</code> buckets</li>
</ul>
<h2 class="anchor anchorTargetStickyNavbar_Vzrq" id="using-blockmax-wand-in-v129-technical-preview">Using BlockMax WAND in <code>v1.29</code> (technical preview)<a href="#using-blockmax-wand-in-v129-technical-preview" class="hash-link" aria-label="Direct link to using-blockmax-wand-in-v129-technical-preview" title="Direct link to using-blockmax-wand-in-v129-technical-preview" translate="no">​</a></h2>
<div class="theme-admonition theme-admonition-caution admonition_xJq3 alert alert--warning"><div class="admonitionHeading_Gvgb"><span class="admonitionIcon_Rf37"><svg viewBox="0 0 16 16"><path fill-rule="evenodd" d="M8.893 1.5c-.183-.31-.52-.5-.887-.5s-.703.19-.886.5L.138 13.499a.98.98 0 0 0 0 1.001c.193.31.53.501.886.501h13.964c.367 0 .704-.19.877-.5a1.03 1.03 0 0 0 .01-1.002L8.893 1.5zm.133 11.497H6.987v-2.003h2.039v2.003zm0-3.004H6.987V5.987h2.039v4.006z"></path></svg></span>BlockMax WAND technical preview</div><div class="admonitionContent_BuS1"><p>BlockMax WAND algorithm is available in <code>v1.29</code> as a <strong>technical preview</strong>. <strong>We do not recommend using this feature in production environments in this version and suggest you use <code>v1.30+</code>.</strong></p></div></div>
<p><strong>To use BlockMax WAND in Weaviate <code>v1.29</code>, it must be enabled prior to collection creation.</strong> As of this version, Weaviate will not migrate existing collections to use BlockMax WAND.</p>
<p>Enable BlockMax WAND by setting the environment variables <code>USE_BLOCKMAX_WAND</code> and <code>USE_INVERTED_SEARCHABLE</code> to <code>true</code>.</p>
<p>Now, all new data added to Weaviate will use BlockMax WAND for BM25 and hybrid searches. However, preexisting data will continue to use the default WAND algorithm.</p>
