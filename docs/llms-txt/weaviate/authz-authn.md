authorization</h1></header><div class="theme-admonition theme-admonition-info admonition_xJq3 alert alert--info"><div class="admonitionHeading_Gvgb"><span class="admonitionIcon_Rf37"><svg viewBox="0 0 14 16"><path fill-rule="evenodd" d="M7 2.3c3.14 0 5.7 2.56 5.7 5.7s-2.56 5.7-5.7 5.7A5.71 5.71 0 0 1 1.3 8c0-3.14 2.56-5.7 5.7-5.7zM7 1C3.14 1 0 4.14 0 8s3.14 7 7 7 7-3.14 7-7-3.14-7-7-7zm1 3H6v5h2V4zm0 6H6v2h2v-2z"></path></svg></span>Authentication and authorization</div><div class="admonitionContent_BuS1"><p>Authentication and authorization are closely related concepts, and sometimes abbreviated as <code>AuthN</code> and <code>AuthZ</code>. Authentication (<code>AuthN</code>) is the process of verifying the identity of a user, while authorization (<code>AuthZ</code>) is the process of determining what permissions the user has.</p></div></div>
<h2 class="anchor anchorTargetStickyNavbar_Vzrq" id="authentication">Authentication<a href="#authentication" class="hash-link" aria-label="Direct link to Authentication" title="Direct link to Authentication" translate="no">​</a></h2>
<p>Weaviate controls access through user authentication via API keys or OpenID Connect (OIDC), with an option for anonymous access. Users can then be assigned different <a class="" href="/deploy/configuration/authorization">authorization</a> levels, as shown in the diagram below.</p>
<!-- -->
<p>For example, a user logging in with the API key <code>jane-secret</code> may be granted administrator permissions, while another user logging in with the API key <code>ian-secret</code> may be granted read-only permissions.</p>
<p>In summary, Weaviate allows the following authentication methods:</p>
<ul>
<li class=""><a class="" href="/deploy/configuration/authentication#api-key-authentication">API key</a></li>
<li class=""><a class="" href="/deploy/configuration/authentication#oidc-authentication">OpenID Connect (OIDC)</a></li>
<li class=""><a class="" href="/deploy/configuration/authentication#anonymous-access">Anonymous access</a> (no authentication, strongly discouraged except for development or evaluation)</li>
</ul>
<p>Note that API key and OIDC authentication can be both enabled at the same time.</p>
<p>The way to configure authentication differs by your deployment method, depending on whether you are running Weaviate in Docker or Kubernetes. Below, we provide examples for both.</p>
<div class="theme-admonition theme-admonition-info admonition_xJq3 alert alert--info"><div class="admonitionHeading_Gvgb"><span class="admonitionIcon_Rf37"><svg viewBox="0 0 14 16"><path fill-rule="evenodd" d="M7 2.3c3.14 0 5.7 2.56 5.7 5.7s-2.56 5.7-5.7 5.7A5.71 5.71 0 0 1 1.3 8c0-3.14 2.56-5.7 5.7-5.7zM7 1C3.14 1 0 4.14 0 8s3.14 7 7 7 7-3.14 7-7-3.14-7-7-7zm1 3H6v5h2V4zm0 6H6v2h2v-2z"></path></svg></span>What about Weaviate Cloud (WCD)?</div><div class="admonitionContent_BuS1"><p>For Weaviate Cloud (WCD) instances, authentication is pre-configured with OIDC and API key access. You can <a class="" href="/weaviate/connections/connect-cloud">authenticate against Weaviate</a> with your WCD credentials using OIDC, or <a class="" href="/cloud/manage-clusters/connect">with API keys</a>.</p></div></div>
<h3 class="anchor anchorTargetStickyNavbar_Vzrq" id="api-key">API key<a href="#api-key" class="hash-link" aria-label="Direct link to API key" title="Direct link to API key" translate="no">​</a></h3>
<p>For more details on how to work with API keys in Weaviate, check out the <a class="" href="/deploy/configuration/authentication#api-key-authentication">authentication guide</a>.</p>
<p>We recommend using a client library to authenticate against Weaviate. See <a class="" href="/weaviate/connections">How-to: Connect</a> pages for more information.</p>
<h3 class="anchor anchorTargetStickyNavbar_Vzrq" id="oidc">OIDC<a href="#oidc" class="hash-link" aria-label="Direct link to OIDC" title="Direct link to OIDC" translate="no">​</a></h3>
<p>For more details on how to work with OIDC authentication in Weaviate, check out the <a class="" href="/deploy/configuration/authentication#oidc-authentication">authentication guide</a>.</p>
<p>The OIDC standard allows for many different methods <em>(flows)</em> of obtaining tokens. The appropriate method can vary depending on your situation, including configurations at the token issuer, and your requirements.</p>
<p>OIDC authentication flows are outside the scope of this documentation, but here are some options to consider:</p>
<ol>
<li class="">Use the <code>client credentials flow</code> for machine-to-machine authorization. (Note that this authorizes an app, not a user.)<!-- -->
<ul>
<li class="">Validated using Okta and Azure as identity providers; GCP does not support client credentials grant flow (as of December 2022).</li>
<li class="">Weaviate&#x27;s Python client directly supports this method.</li>
<li class="">Client credential flows usually do not come with a refresh token and the credentials are saved in the respective clients to acquire a new access token on expiration of the old one.</li>
</ul>
</li>
<li class="">Use the <code>resource owner password flow</code> for trusted applications like <a class="" href="/cloud/manage-clusters/connect">Weaviate Cloud</a>.</li>
<li class="">Use <code>hybrid flow</code> if Azure is your token issuer or if you would like to prevent exposing passwords.</li>
</ol>
<h3 class="anchor anchorTargetStickyNavbar_Vzrq" id="support-for-weaviate-clients">Support for Weaviate clients<a href="#support-for-weaviate-clients" class="hash-link" aria-label="Direct link to Support for Weaviate clients" title="Direct link to Support for Weaviate clients" translate="no">​</a></h3>
<p>If Weaviate Database is configured to use the <code>client credentials grant</code> flow or the <code>resource owner password flow</code>, a Weaviate client can instantiate a connection to Weaviate Database that incorporates the authentication flow.</p>

<h3 class="anchor anchorTargetStickyNavbar_Vzrq" id="get-and-pass-tokens-manually">Get and pass tokens manually<a href="#get-and-pass-tokens-manually" class="hash-link" aria-label="Direct link to Get and pass tokens manually" title="Direct link to Get and pass tokens manually" translate="no">​</a></h3>
<details class="details_lb9f alert alert--info details_b_Ee" data-collapsed="true"><summary><p>Manually obtaining and passing tokens</p></summary><div><div class="collapsibleContent_i85q"><p>For cases or workflows where you may wish to manually obtain a token, we outline below the steps to do so, for the resource owner password flow and hybrid flow.</p><h4 class="anchor anchorTargetStickyNavbar_Vzrq" id="resource-owner-password-flow">Resource owner password flow<a href="#resource-owner-password-flow" class="hash-link" aria-label="Direct link to Resource owner password flow" title="Direct link to Resource owner password flow" translate="no">​</a></h4><ol>
<li class="">Send a GET request to <code>WEAVIATE_INSTANCE_URL/v1/.well-known/openid-configuration</code> to fetch Weaviate&#x27;s OIDC configuration (<code>wv_oidc_config</code>). Replace WEAVIATE_INSTANCE_URL with your instance URL.</li>
<li class="">Parse the <code>clientId</code> and <code>href</code> from <code>wv_oidc_config</code>.</li>
<li class="">Send a GET request to <code>href</code> to fetch the token issuer&#x27;s OIDC configuration (<code>token_oidc_config</code>).</li>
<li class="">If <code>token_oidc_config</code> includes the optional <code>grant_types_supported</code> key, check that <code>password</code> is in the list of values.<!-- -->
<ul>
<li class="">If <code>password</code> is not in the list of values, the token issuer is likely not configured for <code>resource owner password flow</code>. You may need to reconfigure the token issuer or use another method.</li>
<li class="">If the <code>grant_types_supported</code> key is not available, you may need to contact the token issuer to see if <code>resource owner password flow</code> is supported.</li>
</ul>
</li>
<li class="">Send a POST request to the <code>token_endpoint</code> of <code>token_oidc_config</code> with the body:<!-- -->
<ul>
<li class=""><code>{&quot;grant_type&quot;: &quot;password&quot;, &quot;client_id&quot;: client_id, &quot;username&quot;: USERNAME, &quot;password&quot;: PASSWORD</code>. Replace <code>USERNAME</code> and <code>PASSWORD</code> with the actual values.</li>
</ul>
</li>
<li class="">Parse the response (<code>token_resp</code>), and look for <code>access_token</code> in <code>token_resp</code>. This is your Bearer token.</li>
</ol><h4 class="anchor anchorTargetStickyNavbar_Vzrq" id="hybrid-flow">Hybrid flow<a href="#hybrid-flow" class="hash-link" aria-label="Direct link to Hybrid flow" title="Direct link to Hybrid flow" translate="no">​</a></h4><ol>
<li class="">Send a GET request to <code>WEAVIATE_INSTANCE_URL/v1/.well-known/openid-configuration</code> to fetch Weaviate&#x27;s OIDC configuration (<code>wv_oidc_config</code>). Replace WEAVIATE_INSTANCE_URL with your instance URL.</li>
<li class="">Parse the <code>clientId</code> and <code>href</code> from <code>wv_oidc_config</code></li>
<li class="">Send a GET request to <code>href</code> to fetch the token issuer&#x27;s OIDC configuration (<code>token_oidc_config</code>)</li>
<li class="">Construct a URL (<code>auth_url</code>) with the following parameters, based on <code>authorization_endpoint</code> from <code>token_oidc_config</code>. This will look like the following:<!-- -->
<ul>
<li class=""><code>{authorization_endpoint}</code>?client_id=<code>{clientId}</code>&amp;response_type=code%20id_token&amp;response_mode=fragment&amp;redirect_url=<code>{redirect_url}</code>&amp;scope=openid&amp;nonce=abcd</li>
<li class="">the <code>redirect_url</code> must have been <a href="https://openid.net/specs/openid-connect-core-1_0.html#AuthRequest" target="_blank" rel="noopener noreferrer" class="">pre-registered</a> with your token issuer.</li>
</ul>
</li>
<li class="">Go to the <code>auth_url</code> in your browser, and log in if prompted. If successful, the token issuer will redirect the browser to the <code>redirect_url</code>, with additional parameters that include an <code>id_token</code> parameter.</li>
<li class="">Parse the <code>id_token</code> parameter value. This is your Bearer token.</li>

<h3 class="anchor anchorTargetStickyNavbar_Vzrq" id="add-a-bearer-to-a-request">Add a Bearer to a Request<a href="#add-a-bearer-to-a-request" class="hash-link" aria-label="Direct link to Add a Bearer to a Request" title="Direct link to Add a Bearer to a Request" translate="no">​</a></h3>

<p>When you use an API key to authenticate to Weaviate, add the API key in the request header.</p>
<p>The format is: <code>Authorization: Bearer WEAVIATE_API_KEY</code>. Replace <code>WEAVIATE_API_KEY</code> with the API key for your Weaviate instance.</p>
<p>For example, the cURL command looks like this:</p>

<h2 class="anchor anchorTargetStickyNavbar_Vzrq" id="authorization">Authorization<a href="#authorization" class="hash-link" aria-label="Direct link to Authorization" title="Direct link to Authorization" translate="no">​</a></h2>
<p>Weaviate provides differentiated access through authorization levels, based on the user&#x27;s <a href="#authentication" class="">authentication</a> status. A user can be granted admin permission, read-only permission, or no permission at all. From <code>v1.29.0</code>, Weaviate also supports <a class="" href="/weaviate/configuration/rbac">Role-Based Access Control (RBAC)</a> for more fine-grained control over user permissions.</p>
<p>The following diagram illustrates the flow of a user request through the authentication and authorization process:</p>
<!-- -->
<p>The following authorization schemes are available in Weaviate:</p>
<ul>
<li class=""><a class="" href="/deploy/configuration/authorization#role-based-access-control-rbac">Role-Based Access Control (RBAC)</a></li>
<li class=""><a class="" href="/deploy/configuration/authorization#admin-list">Admin list</a></li>
<li class=""><a class="" href="/deploy/configuration/authorization#undifferentiated-access">Undifferentiated access</a></li>
</ul>
<p>In the Admin list authorization scheme, <a class="" href="/deploy/configuration/authorization#anonymous-users">anonymous users</a> can be granted permissions.</p>
<p>The way to configure authorization differs by your deployment method, depending on whether you are running Weaviate in Docker or Kubernetes. Below, we provide examples for both.</p>
