<h2 class="anchor anchorTargetStickyNavbar_Vzrq" id="collection-definition-an-introduction">Collection definition: An Introduction<a href="#collection-definition-an-introduction" class="hash-link" aria-label="Direct link to Collection definition: An Introduction" title="Direct link to Collection definition: An Introduction" translate="no">​</a></h2>
<p>The collection definition specifies how data is stored, organized and retrieved in Weaviate.</p>
<p>If <a class="" href="/weaviate/config-refs/collections#auto-schema">auto-schema</a> is enabled, Weaviate can infer missing elements and add them to the collection definition. However, it is a best practice to manually define as much of the data schema as possible since manual definition gives you the most control.</p>
<p>Let&#x27;s begin with a simple example before diving into the details.</p>
<h3 class="anchor anchorTargetStickyNavbar_Vzrq" id="basic-collection-creation">Basic collection creation<a href="#basic-collection-creation" class="hash-link" aria-label="Direct link to Basic collection creation" title="Direct link to Basic collection creation" translate="no">​</a></h3>
<p>This example creates a collection called <code>Question</code>. The collection has three properties, <code>answer</code>, <code>question</code>, and <code>category</code>. The definition specifies the <code>text2vec-openai</code> vectorizer and the <code>generative-cohere</code> module for RAG.</p>

<h2 class="anchor anchorTargetStickyNavbar_Vzrq" id="collection-definitions-in-detail">Collection definitions in detail<a href="#collection-definitions-in-detail" class="hash-link" aria-label="Direct link to Collection definitions in detail" title="Direct link to Collection definitions in detail" translate="no">​</a></h2>
<p>Conceptually, it may be useful to think of each Weaviate instance as consisting of multiple collections, each of which is a set of objects that share a common structure.</p>
<p>For example, you might have a movie database with <code>Movie</code> and <code>Actor</code> collections, each with their own properties. Or you might have a news database with <code>Article</code>, <code>Author</code> and <code>Publication</code> collections.</p>
<h3 class="anchor anchorTargetStickyNavbar_Vzrq" id="available-settings">Available settings<a href="#available-settings" class="hash-link" aria-label="Direct link to Available settings" title="Direct link to Available settings" translate="no">​</a></h3>
<p>For the most part, each collection should be thought of as isolated from the others (in fact, they are!). Accordingly, they can be configured independently. Each collection has:</p>
<ul>
<li class="">A set of <code>properties</code> specifying the object data structure.</li>
<li class="">Multi-tenancy settings.</li>
<li class="">Vectorizer and generative modules.</li>
<li class="">Index settings (for vector and inverted indexes).</li>
<li class="">Replication and sharding settings.</li>
</ul>
<p>And depending on your needs, you might want to change any number of these.</p>
<h3 class="anchor anchorTargetStickyNavbar_Vzrq" id="properties">Properties<a href="#properties" class="hash-link" aria-label="Direct link to Properties" title="Direct link to Properties" translate="no">​</a></h3>
<p>Each property has a number of settings that can be configured, such as the <code>dataType</code>, <code>tokenization</code>, and <code>vectorizePropertyName</code>. You can read more about these in the <a class="" href="/weaviate/config-refs/collections#properties">collection definition reference</a>.</p>

<h4 class="anchor anchorTargetStickyNavbar_Vzrq" id="cross-references">Cross-references<a href="#cross-references" class="hash-link" aria-label="Direct link to Cross-references" title="Direct link to Cross-references" translate="no">​</a></h4>
<!-- -->
<div class="theme-admonition theme-admonition-caution admonition_xJq3 alert alert--warning"><div class="admonitionHeading_Gvgb"><span class="admonitionIcon_Rf37"><svg viewBox="0 0 16 16"><path fill-rule="evenodd" d="M8.893 1.5c-.183-.31-.52-.5-.887-.5s-.703.19-.886.5L.138 13.499a.98.98 0 0 0 0 1.001c.193.31.53.501.886.501h13.964c.367 0 .704-.19.877-.5a1.03 1.03 0 0 0 .01-1.002L8.893 1.5zm.133 11.497H6.987v-2.003h2.039v2.003zm0-3.004H6.987V5.987h2.039v4.006z"></path></svg></span>Cross-references and query performance</div><div class="admonitionContent_BuS1"><p>Queries involving cross-references can be slower than queries that do not involve cross-references, especially at scale such as for multiple objects or complex queries.</p><p>At the first instance, we strongly encourage you to consider whether you can avoid using cross-references in your data schema. As a scalable AI database, Weaviate is well-placed to perform complex queries with vector, keyword and hybrid searches involving filters. You may benefit from rethinking your data schema to avoid cross-references where possible.</p><p>For example, instead of creating separate &quot;Author&quot; and &quot;Book&quot; collections with cross-references, consider embedding author information directly in Book objects and using searches and filters to find books by author characteristics.</p></div></div>
<p>This is also where you would specify cross-references, which are a special type of property that links to another collection.</p>
<p>Cross-references can be very useful for creating relationships between objects. For example, you might have a <code>Movie</code> collection with a <code>withActor</code> cross-reference property that points to the <code>Actor</code> collection. This will allow you to retrieve relevant actors for each movie.</p>
<p>However, cross-references can be costly in terms of performance. Use them sparingly. Additionally, cross-reference properties do not affect the object&#x27;s vector. So if you want the related properties to be considered in a vector search, they should be included in the object&#x27;s vectorized properties.</p>
<p>You can find examples of how to define and use cross-references <a class="" href="/weaviate/manage-collections/cross-references">here</a>.</p>
<h3 class="anchor anchorTargetStickyNavbar_Vzrq" id="vectorizer-and-generative-modules">Vectorizer and generative modules<a href="#vectorizer-and-generative-modules" class="hash-link" aria-label="Direct link to Vectorizer and generative modules" title="Direct link to Vectorizer and generative modules" translate="no">​</a></h3>
<p>Each collection can be configured with a vectorizer and a generative module. The vectorizer is used to generate vectors for each object and also for any un-vectorized queries, and the generative module is used to perform retrieval augmented generation (RAG) queries.</p>
<p>If you are not sure where to start, modules that integrate with popular API-based model providers such as Cohere or OpenAI are good starting points. You can find a <a class="" href="/weaviate/model-providers">list of available model integrations here</a>.</p>
<h3 class="anchor anchorTargetStickyNavbar_Vzrq" id="multi-tenancy-settings">Multi-tenancy settings<a href="#multi-tenancy-settings" class="hash-link" aria-label="Direct link to Multi-tenancy settings" title="Direct link to Multi-tenancy settings" translate="no">​</a></h3>
<p>Starting from version <code>v1.20.0</code>, each collection can be configured as a multi-tenancy collection. This allows separation of data between tenants, typically end-users, at a much lower overhead than creating separate collections for each tenant.</p>
<p>This is useful if you want to use Weaviate as a backend for a multi-tenant (e.g. SaaS) application, or if data isolation is required for any other reason.</p>

<h3 class="anchor anchorTargetStickyNavbar_Vzrq" id="index-settings">Index settings<a href="#index-settings" class="hash-link" aria-label="Direct link to Index settings" title="Direct link to Index settings" translate="no">​</a></h3>
<p>Weaviate uses two types of indexes: <a class="" href="/weaviate/concepts/vector-index">vector indexes</a> and <a class="" href="/weaviate/concepts/indexing/inverted-index">inverted indexes</a>. Vector indexes are used to store and organize vectors for fast vector similarity-based searches. Inverted indexes are used to store data for fast filtering and keyword searches.</p>
<p>The default vector index type is <a class="" href="/weaviate/concepts/vector-index#hierarchical-navigable-small-world-hnsw-index">HNSW</a>. The other options are <a class="" href="/weaviate/concepts/vector-index#flat-index">flat</a>, which is suitable for small collections, such as those in a multi-tenancy collection, or <a class="" href="/weaviate/concepts/vector-index#dynamic-index">dynamic</a>, which starts as a flat index before switching to an HNSW index if its size grows beyond a predetermined threshold.</p>

<h3 class="anchor anchorTargetStickyNavbar_Vzrq" id="replication-and-sharding-settings">Replication and sharding settings<a href="#replication-and-sharding-settings" class="hash-link" aria-label="Direct link to Replication and sharding settings" title="Direct link to Replication and sharding settings" translate="no">​</a></h3>
<h4 class="anchor anchorTargetStickyNavbar_Vzrq" id="replication">Replication<a href="#replication" class="hash-link" aria-label="Direct link to Replication" title="Direct link to Replication" translate="no">​</a></h4>
<p>Replication settings determine how many copies of the data are stored. For example, a replication setting of 3 means that each object is stored on 3 different replicas. This is important for providing redundancy and fault tolerance in production. (The default replication factor is 1.)</p>
<p>This goes hand-in-hand with consistency settings, which determine how many replicas must respond before an operation is considered successful.</p>
<p>We recommend that you read the <a class="" href="/weaviate/concepts/replication-architecture">concepts page on replication</a> for information on how replication works in Weaviate. To specify a replication factor, follow <a class="" href="/weaviate/manage-collections/multi-node-setup#replication-settings">this how-to</a>.</p>
<h4 class="anchor anchorTargetStickyNavbar_Vzrq" id="sharding">Sharding<a href="#sharding" class="hash-link" aria-label="Direct link to Sharding" title="Direct link to Sharding" translate="no">​</a></h4>
<p>Sharding settings determine how each collection is sharded and distributed across nodes. This is not a setting that is typically changed, but you can use it to control how many shards are created in a cluster, and how many virtual shards are created per physical shard (<a class="" href="/weaviate/config-refs/collections#sharding">read more here</a>).</p>
<h2 class="anchor anchorTargetStickyNavbar_Vzrq" id="collection-aliases">Collection aliases<a href="#collection-aliases" class="hash-link" aria-label="Direct link to Collection aliases" title="Direct link to Collection aliases" translate="no">​</a></h2>
<div class="theme-admonition theme-admonition-info admonition_xJq3 alert alert--info"><div class="admonitionHeading_Gvgb"><span class="admonitionIcon_Rf37"><svg viewBox="0 0 14 16"><path fill-rule="evenodd" d="M7 2.3c3.14 0 5.7 2.56 5.7 5.7s-2.56 5.7-5.7 5.7A5.71 5.71 0 0 1 1.3 8c0-3.14 2.56-5.7 5.7-5.7zM7 1C3.14 1 0 4.14 0 8s3.14 7 7 7 7-3.14 7-7-3.14-7-7-7zm1 3H6v5h2V4zm0 6H6v2h2v-2z"></path></svg></span>Added in <code>v1.32</code></div></div>
<p>Collection aliases are alternative names (pointers) for Weaviate collections that allow you to reference a collection by multiple names. When you query using an alias, Weaviate automatically routes the request to the target collection. You can set up collection aliases <a class="" href="/weaviate/manage-collections/collection-aliases">programmatically through client libraries</a> or by using the <a href="/weaviate/api/rest#tag/schema">REST endpoints</a>.</p>
<!-- -->
<div class="theme-admonition theme-admonition-info admonition_xJq3 alert alert--info"><div class="admonitionHeading_Gvgb"><span class="admonitionIcon_Rf37"><svg viewBox="0 0 14 16"><path fill-rule="evenodd" d="M7 2.3c3.14 0 5.7 2.56 5.7 5.7s-2.56 5.7-5.7 5.7A5.71 5.71 0 0 1 1.3 8c0-3.14 2.56-5.7 5.7-5.7zM7 1C3.14 1 0 4.14 0 8s3.14 7 7 7 7-3.14 7-7-3.14-7-7-7zm1 3H6v5h2V4zm0 6H6v2h2v-2z"></path></svg></span>Collection alias usage</div><div class="admonitionContent_BuS1"><p>Weaviate automatically routes alias requests to the target collection for <strong>object-related operations</strong>. You can use aliases wherever collection names are required for:</p><ul>
<li class=""><strong><a class="" href="/weaviate/manage-objects">Managing objects</a></strong>: <a class="" href="/weaviate/manage-objects/create">Create</a>, <a class="" href="/weaviate/manage-objects/import">batch import</a>, <a class="" href="/weaviate/manage-objects/read">read</a>, <a class="" href="/weaviate/manage-objects/update">update</a> and <a class="" href="/weaviate/manage-objects/delete">delete</a> objects through collection aliases.</li>
<li class=""><strong><a class="" href="/weaviate/search">Querying objects</a></strong>: <a class="" href="/weaviate/search/basics">Fetch</a> objects and perform searches (<a class="" href="/weaviate/search/similarity">vector</a>, <a class="" href="/weaviate/search/bm25">keyword</a>, <a class="" href="/weaviate/search/hybrid">hybrid</a>, <a class="" href="/weaviate/search/image">image</a>, <a class="" href="/weaviate/search/generative">generative/RAG</a>) and <a class="" href="/weaviate/search/aggregate">aggregations</a> through aliases.</li>
</ul></div></div>
