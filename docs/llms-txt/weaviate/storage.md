markdown"><header><h1>Storage</h1></header><p>Weaviate is a persistent and fault-tolerant database. This page gives you an overview of how objects and vectors are stored within Weaviate and how an inverted index is created at import time.</p>
<p>The components mentioned on this page aid Weaviate in creating some of its unique features:</p>
<ul>
<li class="">Each write operation is immediately persisted and also tolerant to application and system crashes.</li>
<li class="">On a vector search query, Weaviate returns the entire object (in other databases sometimes called a &quot;document&quot;), not just a reference, such as an ID.</li>
<li class="">When combining structured search with vector search, filters are applied prior to performing the vector search. This means that you will always receive the specified number of elements as opposed to post-filtering when the final result count is unpredictable.</li>
<li class="">Objects and their vectors can be updated or deleted at will; even while reading from the database.</li>
</ul>
<h2 class="anchor anchorTargetStickyNavbar_Vzrq" id="logical-storage-units-indexes-shards-stores">Logical Storage Units: Indexes, Shards, Stores<a href="#logical-storage-units-indexes-shards-stores" class="hash-link" aria-label="Direct link to Logical Storage Units: Indexes, Shards, Stores" title="Direct link to Logical Storage Units: Indexes, Shards, Stores" translate="no">​</a></h2>
<p>Each class in Weaviate&#x27;s user-defined schema leads to the creation of an index internally. An index is a wrapper type that is comprised of one or many shards. Shards within an index are self-contained storage units. Multiple shards can be used to distribute the load among multiple server nodes automatically.</p>
<h3 class="anchor anchorTargetStickyNavbar_Vzrq" id="components-of-a-shard">Components of a Shard<a href="#components-of-a-shard" class="hash-link" aria-label="Direct link to Components of a Shard" title="Direct link to Components of a Shard" translate="no">​</a></h3>
<p>Each shard houses three main components:</p>
<ul>
<li class="">An object store, essentially a key-value store</li>
<li class="">An <a href="https://en.wikipedia.org/wiki/Inverted_index" target="_blank" rel="noopener noreferrer" class="">inverted index</a></li>
<li class="">A vector index store (plugable, currently a <a class="" href="/weaviate/config-refs/indexing/vector-index#hnsw-index">custom implementation of HNSW</a>)</li>
</ul>
<h4 class="anchor anchorTargetStickyNavbar_Vzrq" id="object-and-inverted-index-store">Object and Inverted Index Store<a href="#object-and-inverted-index-store" class="hash-link" aria-label="Direct link to Object and Inverted Index Store" title="Direct link to Object and Inverted Index Store" translate="no">​</a></h4>
<p>Since version <code>v1.5.0</code>, the object and inverted store are implemented using an <a href="https://en.wikipedia.org/wiki/Log-structured_merge-tree" target="_blank" rel="noopener noreferrer" class="">LSM-Tree approach</a>. This means that data can be ingested at the speed of memory and after meeting a configured threshold, Weaviate will write the entire (sorted) memtable into a disk segment. When a read request comes in, Weaviate will first check the Memtable for the latest update for a specific object. If it is not present in the memtable, Weaviate will then check all previously written segments starting with the newest. To avoid checking segments which don&#x27;t contain the desired objects, <a href="https://en.wikipedia.org/wiki/Bloom_filter" target="_blank" rel="noopener noreferrer" class="">Bloom filters</a> are used.</p>
<p>Weaviate periodically merges smaller, older segments to make larger segments. Since the segments are already sorted, this is a relatively cheap operation. It happens constantly in the background. Fewer, larger segments make lookups more efficient. In the inverted index data is rarely replaced, but it is often appended. Merging means that, instead of checking all past segments and aggregating potential results, Weaviate can check a single segment (or a few large segments) and immediately find all the relevant object pointers. In addition, segments are used to remove earlier versions of an object that are out-dated because of a delete or a more recent update.</p>
<p>Considerations</p>
<p>Object storage and inverted index storage implement the LSM algorithm; they use segmentation. The vector index uses a different storage algorithm. The vector index does not use segmentation.</p>
<p>Weaviate versions before <code>v1.5.0</code> use a B+Tree storage mechanism. The LSM method is faster, it works in constant time, and it improves write performance.</p>
<p>To learn more about Weaviate&#x27;s LSM store, see the LSM library documentation in the <a href="https://pkg.go.dev/github.com/weaviate/weaviate/adapters/repos/db/lsmkv" target="_blank" rel="noopener noreferrer" class="">Go package repository</a></p>
<h4 class="anchor anchorTargetStickyNavbar_Vzrq" id="hnsw-vector-index-storage">HNSW Vector Index Storage<a href="#hnsw-vector-index-storage" class="hash-link" aria-label="Direct link to HNSW Vector Index Storage" title="Direct link to HNSW Vector Index Storage" translate="no">​</a></h4>
<p>Each shard contains a vector index that corresponds to the object and inverted index stores. The vector store and the other stores are independent. The vector store does not have to manage segmentation.</p>
<p>By grouping a vector index with the object storage within a shard, Weaviate can make sure that each shard is a fully self-contained unit which can independently serve requests for the data it owns. By placing the vector index next to the object store (instead of within), Weaviate can avoid the downsides of a segmented vector index.</p>
<p>Furthermore, its persistence and loading at startup are optimized through a combination of Write-Ahead-Logging and HNSW snapshots, detailed in the <a href="#persistence-and-crash-recovery" class="">Persistence and Crash Recovery</a> section.</p>
<h3 class="anchor anchorTargetStickyNavbar_Vzrq" id="shard-components-optimizations">Shard Components Optimizations<a href="#shard-components-optimizations" class="hash-link" aria-label="Direct link to Shard Components Optimizations" title="Direct link to Shard Components Optimizations" translate="no">​</a></h3>
<p>Weaviate&#x27;s storage mechanisms use segmentation for structured/object data. Segments are cheap to merge and even unmerged segments can be navigated efficiently thanks to Bloom filters. In turn, ingestion speed is high and does not degrade over time.</p>
<p>Weaviate keeps the vector index as large as possible within a shard. HNSW indexes cannot be merged efficiently. Querying a single large index is more efficient than sequentially querying many small indexes.</p>
<p>To use multiple CPUs efficiently, create multiple shards for your collection. For the fastest imports, create multiple shards even on a single node.</p>
<h3 class="anchor anchorTargetStickyNavbar_Vzrq" id="lazy-shard-loading">Lazy shard loading<a href="#lazy-shard-loading" class="hash-link" aria-label="Direct link to Lazy shard loading" title="Direct link to Lazy shard loading" translate="no">​</a></h3>
<div class="theme-admonition theme-admonition-info admonition_xJq3 alert alert--info"><div class="admonitionHeading_Gvgb"><span class="admonitionIcon_Rf37"><svg viewBox="0 0 14 16"><path fill-rule="evenodd" d="M7 2.3c3.14 0 5.7 2.56 5.7 5.7s-2.56 5.7-5.7 5.7A5.71 5.71 0 0 1 1.3 8c0-3.14 2.56-5.7 5.7-5.7zM7 1C3.14 1 0 4.14 0 8s3.14 7 7 7 7-3.14 7-7-3.14-7-7-7zm1 3H6v5h2V4zm0 6H6v2h2v-2z"></path></svg></span>Added in <code>v1.23</code></div></div>
<p>When Weaviate starts, it loads data from all of the shards in your deployment. This process can take a long time. Prior to v1.23, you have to wait until all of the shards are loaded before you can query your data. Since every tenant is a shard, multi-tenant deployments can have reduced availability after a restart.</p>
<p>Lazy shard loading allows you to start working with your data sooner. After a restart, shards load in the background. If the shard you want to query is already loaded, you can get your results sooner. If the shard is not loaded yet, Weaviate prioritizes loading that shard and returns a response when it is ready.</p>
<p>To enable lazy shard loading, set the <code>DISABLE_LAZY_LOAD_SHARDS</code> environment variable to <code>false</code> in your system configuration file.</p>
<div class="theme-admonition theme-admonition-caution admonition_xJq3 alert alert--warning"><div class="admonitionHeading_Gvgb"><span class="admonitionIcon_Rf37"><svg viewBox="0 0 16 16"><path fill-rule="evenodd" d="M8.893 1.5c-.183-.31-.52-.5-.887-.5s-.703.19-.886.5L.138 13.499a.98.98 0 0 0 0 1.001c.193.31.53.501.886.501h13.964c.367 0 .704-.19.877-.5a1.03 1.03 0 0 0 .01-1.002L8.893 1.5zm.133 11.497H6.987v-2.003h2.039v2.003zm0-3.004H6.987V5.987h2.039v4.006z"></path></svg></span>Disable lazy shard loading for single-tenant collections</div><div class="admonitionContent_BuS1"><p>For single-tenant collections, lazy loading can cause import operations to slow down or partially fail. In these scenarios, we recommend disabling lazy shard loading.</p></div></div>
<h2 class="anchor anchorTargetStickyNavbar_Vzrq" id="persistence-and-crash-recovery">Persistence and Crash Recovery<a href="#persistence-and-crash-recovery" class="hash-link" aria-label="Direct link to Persistence and Crash Recovery" title="Direct link to Persistence and Crash Recovery" translate="no">​</a></h2>
<h3 class="anchor anchorTargetStickyNavbar_Vzrq" id="write-ahead-log">Write-Ahead-Log<a href="#write-ahead-log" class="hash-link" aria-label="Direct link to Write-Ahead-Log" title="Direct link to Write-Ahead-Log" translate="no">​</a></h3>
<p>Both the LSM stores used for object and inverted storage, as well as the HNSW vector index store make use of memory at some point of the ingestion journey. To prevent data loss on a crash, each operation is additionally written into a <strong><a href="https://martinfowler.com/articles/patterns-of-distributed-systems/wal.html" target="_blank" rel="noopener noreferrer" class="">Write-Ahead-Log (WAL)</a></strong> (also known as a <em>commit log</em>). WALs are append-only files that are very efficient to write to and that are rarely a bottleneck for ingestion.</p>
<p>By the time Weaviate has responded with a successful status to your ingestion request, a WAL entry will have been created. If a WAL entry could not be created - for example because the disks are full - Weaviate will respond with an error to the insert or update request.</p>
<p>The LSM stores will try to flush a segment on an orderly shutdown. Only if the operation is successful, will the WAL be marked as &quot;complete&quot;. This means that if an unexpected crash happens and Weaviate encounters an &quot;incomplete&quot; WAL, it will recover from it. As part of the recovery process, Weaviate will flush a new segment based on the WAL and mark it as complete. As a result, future restarts will no longer have to recover from this WAL.</p>
<p>For the HNSW vector index, the Write-Ahead-Log (WAL) is a critical component for disaster recovery and persisting the most recent changes. The cost in building up an HNSW index is in figuring out where to place a new object and how to link it with its neighbors. The WAL contains only the result of those calculations.</p>
<p>The entire HNSW index state can be reconstructed by replaying these WAL entries.</p>
<p>For very large indexes of tens or hundreds of millions of objects, this can be time-consuming. If you have a large index and you want to speed up the startup time, you can use the <strong><a class="" href="/weaviate/configuration/hnsw-snapshots">HNSW snapshots</a></strong> feature.</p>
<h3 class="anchor anchorTargetStickyNavbar_Vzrq" id="hnsw-snapshots">HNSW snapshots<a href="#hnsw-snapshots" class="hash-link" aria-label="Direct link to HNSW snapshots" title="Direct link to HNSW snapshots" translate="no">​</a></h3>
<div class="theme-admonition theme-admonition-info admonition_xJq3 alert alert--info"><div class="admonitionHeading_Gvgb"><span class="admonitionIcon_Rf37"><svg viewBox="0 0 14 16"><path fill-rule="evenodd" d="M7 2.3c3.14 0 5.7 2.56 5.7 5.7s-2.56 5.7-5.7 5.7A5.71 5.71 0 0 1 1.3 8c0-3.14 2.56-5.7 5.7-5.7zM7 1C3.14 1 0 4.14 0 8s3.14 7 7 7 7-3.14 7-7-3.14-7-7-7zm1 3H6v5h2V4zm0 6H6v2h2v-2z"></path></svg></span>Added in <code>v1.31</code></div></div>
<p>For very large HNSW vector indexes, HNSW snapshots can significantly reduce the startup time.</p>
<p>A snapshot represents a point-in-time state of the HNSW index. When Weaviate starts, if a valid snapshot exists, it will be loaded into memory first. This significantly reduces startup time, as the number of WAL entries that need to be processed, as only the changes made after the snapshot was taken need to be replayed from the WAL.</p>
<p>If a snapshot cannot be loaded for any reason, it is safely removed, and Weaviate falls back to the traditional method of loading the full commit log from the beginning, ensuring resilience.</p>
<p>Snapshots can be created at startup and periodically based on time passed or changes in the commit log.</p>
<p>Weaviate will try to create a new snapshot during startup if there are changes in the commit log since the last snapshot. If there are no changes, then the existing snapshot will be loaded.</p>
<p>A snapshot will also be created if the corresponding conditions are met, meaning the specified time interval has passed and a sufficient number of new commits exist. This is handled by the same background process that manages commit log combining and condensing, ensuring stability as the commit logs used for snapshot creation are not mutable during this process.</p>
<p>Each new HNSW snapshot is based on the previous snapshot and newer (delta) commit logs.</p>
<p>It&#x27;s important to note that even with a fresh snapshot, the server typically still has to load at least one subsequent commit log file.</p>
<p>The WAL is still used to persist every change immediately, guaranteeing that any acknowledged write is durable. Over time, the append-only WAL will contain redundant information for operations occurring after the last snapshot. A background process continuously compacts these newer WAL files, removing redundant information. This, combined with snapshotting, keeps the disk footprint manageable and startup times fast.</p>
<p>See <strong><a class="" href="/weaviate/configuration/hnsw-snapshots">the HNSW snapshots configuration</a></strong> for more details on how to configure this feature.</p>
<p>As of <code>v1.31</code>, HNSW snapshots are disabled by default.</p>
<h2 class="anchor anchorTargetStickyNavbar_Vzrq" id="conclusions">Conclusions<a href="#conclusions" class="hash-link" aria-label="Direct link to Conclusions" title="Direct link to Conclusions" translate="no">​</a></h2>
<p>This page introduced you to the storage mechanisms of Weaviate. It outlined how all writes are persisted immediately and outlined the patterns used within Weaviate to make datasets scale well. For structured data, Weaviate makes use of segmentation to keep the write times constant. For the HNSW vector index, Weaviate avoids segmentation to keep query times efficient.</p>
