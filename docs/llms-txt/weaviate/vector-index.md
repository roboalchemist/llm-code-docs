class="theme-doc-markdown markdown"><header><h1>Vector Indexing</h1></header><p>What is vector indexing? It&#x27;s a key component of vector databases that helps to <a href="https://weaviate.io/blog/vector-search-explained" target="_blank" rel="noopener noreferrer" class="">significantly <strong>increase the speed</strong> of the search process of similarity search</a> with only a minimal tradeoff in search accuracy (<a href="#hierarchical-navigable-small-world-hnsw-index" class="">HNSW index</a>), or efficiently store many subsets of data in a small memory footprint (<a href="#flat-index" class="">flat index</a>). The <a href="#dynamic-index" class="">dynamic index</a> can even start off as a flat index and then dynamically switch to the HNSW index as it scales past a threshold.</p>
<p>Weaviate&#x27;s vector-first storage system takes care of all storage operations with a vector index. Storing data in a vector-first manner not only allows for semantic or context-based search, but also makes it possible to store <em>very</em> large amounts of data without decreasing performance (assuming scaled well horizontally or having sufficient shards for the indexes).</p>
<p>Weaviate supports these vector index types:</p>
<ul>
<li class=""><a href="#flat-index" class="">flat index</a>: a simple, lightweight index that is designed for small datasets.</li>
<li class=""><a href="#hierarchical-navigable-small-world-hnsw-index" class="">HNSW index</a>: a more complex index that is slower to build, but it scales well to large datasets as queries have a logarithmic time complexity.</li>
<li class=""><a href="#dynamic-index" class="">dynamic index</a>: allows you to automatically switch from a flat index to an HNSW index as object count scales</li>
</ul>
<p>This page explains what vector indexes are, and what purpose they serve in the Weaviate vector database.</p>

Indexing vector databases properly is crucial for performance, and different index types serve different purposes - from the simple flat index to more sophisticated approaches like HNSW.</p></div></div>
<h2 class="anchor anchorTargetStickyNavbar_Vzrq" id="why-do-you-need-vector-indexing">Why do you need vector indexing?<a href="#why-do-you-need-vector-indexing" class="hash-link" aria-label="Direct link to Why do you need vector indexing?" title="Direct link to Why do you need vector indexing?" translate="no">​</a></h2>
<p><a href="https://weaviate.io/blog/vector-embeddings-explained" target="_blank" rel="noopener noreferrer" class="">Vector embeddings</a> are a great way to represent meaning. Understanding how to index a vector is crucial for working with vector databases effectively. Vectors embeddings are arrays of elements that can capture meaning from different data types, such as texts, images, videos, and other content. The number of elements are called dimensions. High dimension vectors capture more information, but they are harder to work with.</p>
<p>Vector databases make it easier to work with high dimensional vectors. Consider search; Vector databases efficiently measure semantic similarity between data objects. When you run a <a class="" href="/weaviate/search/similarity">similarity search</a>, a vector database like Weaviate uses a vectorized version of the query to find objects in the database that have vectors similar to the query vector.</p>
<p>Vectors are like coordinates in a multi-dimensional space. A very simple vector might represent objects, <em>words</em> in this case, in a 2-dimensional space.</p>
<p>In the graph below, the words <code>Apple</code> and <code>Banana</code> are shown close to each other. <code>Newspaper</code> and <code>Magazine</code> are also close to each other, but they are far away from <code>Apple</code> and <code>Banana</code> in the same vector space.</p>
<p>Within each pair, the distance between words is small because the objects have similar vector embeddings. The distance between the pairs is larger because the difference between the vectors is larger. Intuitively, fruits are similar to each other, but fruits are not similar to reading material.</p>
<p>For more details of this representation, see: (<a href="https://github.com/stanfordnlp/GloVe" target="_blank" rel="noopener noreferrer" class="">GloVe</a>) and <a href="https://weaviate.io/blog/vector-embeddings-explained#what-exactly-are-vector-embeddings" target="_blank" rel="noopener noreferrer" class="">vector embeddings</a>.</p>
<p><img decoding="async" loading="lazy" alt="2D Vector embedding visualization" src="/assets/images/vectors-2d-8ffad96de876a3de52a75c6bd8e4c59e.png" title="2D Vectors visualization" width="373" height="376" class="img_ev3q"></p>
<p>Another way to think of this is how products are placed in a supermarket. You&#x27;d expect to find <code>Apples</code> close to <code>Bananas</code>, because they are both fruit. But when you are searching for a <code>Magazine</code>, you would move away from the <code>Apples</code> and <code>Bananas</code>, more towards the aisle with, for example, <code>Newspapers</code>. This is how the semantics of concepts can be stored in Weaviate as well, depending on the module you&#x27;re using to calculate the numbers in the vectors. Not only words or text can be indexed as vectors, but also images, video, DNA sequences, etc. Read more about which model to use <a class="" href="/weaviate/modules">here</a>.</p>

<p>Let&#x27;s explore how to index a vector using different approaches supported by Weaviate.</p>
<h2 class="anchor anchorTargetStickyNavbar_Vzrq" id="vector-index-types">Vector index types<a href="#vector-index-types" class="hash-link" aria-label="Direct link to Vector index types" title="Direct link to Vector index types" translate="no">​</a></h2>
<p>Many different types of vector indexes exist. A majority of them are designed to speed up searches by reducing the number of vectors that need to be compared. However, they do this in different ways, and each has its own strengths and weaknesses.</p>
<h3 class="anchor anchorTargetStickyNavbar_Vzrq" id="graph-indexes">Graph indexes<a href="#graph-indexes" class="hash-link" aria-label="Direct link to Graph indexes" title="Direct link to Graph indexes" translate="no">​</a></h3>
<p>Graph indexes form a network of vectors, such that similar vectors are connected to each other. This allows for fast &quot;traversal&quot; of the graph to find similar vectors to a query vector.</p>
<p>HNSW, or &quot;Hierarchical Navigable Small World&quot;, is the most common graph index type. It creates a set of &quot;layers&quot; of vectors, to enable fast traversal of the graph.</p>
<p>They are very scalable, allow incremental updates, and efficient for high-dimensional vectors.</p>
<p>This is the default index type in Weaviate.</p>
<h3 class="anchor anchorTargetStickyNavbar_Vzrq" id="tree-based-indexes">Tree-based indexes<a href="#tree-based-indexes" class="hash-link" aria-label="Direct link to Tree-based indexes" title="Direct link to Tree-based indexes" translate="no">​</a></h3>
<p>Tree-based indexes divide the vectors into a tree structure.</p>
<p>ANNOY, or &quot;Approximate Nearest Neighbors Oh Yeah&quot;, is a well-known tree-based index. It divides the vectors into a binary tree structure.</p>
<p>They can be memory-efficient, and are good for low-dimensional vectors. However, it may be costly to update the index over time, as the tree may need to be rebuilt.</p>
<h3 class="anchor anchorTargetStickyNavbar_Vzrq" id="cluster-based-indexes">Cluster-based indexes<a href="#cluster-based-indexes" class="hash-link" aria-label="Direct link to Cluster-based indexes" title="Direct link to Cluster-based indexes" translate="no">​</a></h3>
<p>Cluster-based indexes group vectors based on their similarity. As a result, the search space is reduced to only the cluster(s) that is most likely to contain the nearest neighbors.</p>
<p>Their search accuracy (recall and precision) may generally be lower than graph-based indexes, but they can be more memory-efficient.</p>
<h3 class="anchor anchorTargetStickyNavbar_Vzrq" id="flat-index">Flat index<a href="#flat-index" class="hash-link" aria-label="Direct link to Flat index" title="Direct link to Flat index" translate="no">​</a></h3>
<p>A flat index is the simplest type of index. It stores all vectors in a single list, and searches through all of them to find the nearest neighbors.</p>
<p>This is extremely memory-efficient, but does not scale well, as the search time grows linearly with the number of vectors.</p>
<p>The first method is the HNSW index.</p>
<h2 class="anchor anchorTargetStickyNavbar_Vzrq" id="hierarchical-navigable-small-world-hnsw-index">Hierarchical Navigable Small World (HNSW) index<a href="#hierarchical-navigable-small-world-hnsw-index" class="hash-link" aria-label="Direct link to Hierarchical Navigable Small World (HNSW) index" title="Direct link to Hierarchical Navigable Small World (HNSW) index" translate="no">​</a></h2>
<p><strong>Hierarchical Navigable Small World (HNSW)</strong> is an algorithm that works on multi-layered graphs. It is also an index type, and refers to vector indexes that are created using the HNSW algorithm. HNSW indexes enable very fast queries, but rebuilding the index when you add new vectors can be resource intensive.</p>
<p>Weaviate&#x27;s <code>hnsw</code> index is a <a class="" href="/weaviate/more-resources/faq#q-does-weaviate-use-hnswlib">custom implementation</a> of the Hierarchical Navigable Small World (<a href="https://arxiv.org/abs/1603.09320" target="_blank" rel="noopener noreferrer" class="">HNSW</a>) algorithm that offers full <a href="https://db-engines.com/en/blog_post/87" target="_blank" rel="noopener noreferrer" class="">CRUD-support</a>.</p>
<p>At build time, the HNSW algorithm creates a series of layers. At query time, the HNSW algorithm uses the layers to build a list of approximate nearest neighbors (ANN) quickly and efficiently.</p>
<p>Consider this diagram of a vector index using HNSW.</p>
<p><img decoding="async" loading="lazy" alt="HNSW explained" src="/assets/images/hnsw_1_explained-4d0a8715d1aa9da14f3111ab32b0c730.png" title="HNSW explained" width="2080" height="1200" class="img_ev3q"></p>
<p>An individual object can exist in more than one layer, but every object in the database is represented in the lowest layer (layer zero in the picture). The layer zero data objects are very well connected to each other. Each layer above the lowest layer has fewer data object, and fewer connections. The data objects in the higher layers correspond to the objects in the lower layers, but each higher layer has exponentially fewer objects than the layer below. The HNSW algorithm takes advantage of the layers to efficiently process large amounts of data.</p>
<p><img decoding="async" loading="lazy" alt="HNSW search" src="/assets/images/hnsw_2_search-4f4f3937d3723d678272c00e335baaf4.png" title="HNSW search" width="2080" height="1200" class="img_ev3q"></p>
<p>When a search query comes in, the HNSW algorithm finds the closest matching data points in the highest layer. Then, HNSW goes one layer deeper, and finds the closest data points in that layer to the ones in the higher layer. These are the nearest neighbors. The algorithm searches the lower layer to create a new list of nearest neighbors. Then, HNSW uses the new list and repeats the process on the next layer down. When it gets to the deepest layer, the HNSW algorithm returns the data objects closest to the search query.</p>
<p>Since there are relatively few data objects on the higher layers, HNSW has to search fewer objects. This means HNSW &#x27;jumps&#x27; over large amounts of data that it doesn&#x27;t need to search. When a data store has only one layer, the search algorithm can&#x27;t skip unrelated objects. It has to search significantly more data objects even though they are unlikely to match.</p>
<h3 class="anchor anchorTargetStickyNavbar_Vzrq" id="resource-requirements">Resource requirements<a href="#resource-requirements" class="hash-link" aria-label="Direct link to Resource requirements" title="Direct link to Resource requirements" translate="no">​</a></h3>
<p>HNSW is an in-memory index, where each node in the graph as well as each edge between nodes are stored in memory.</p>
<p>This means that the size of the index in memory is directly proportional to the number of vectors in the index, as well as the number of connections between vectors.</p>
<p>The size of an HNSW index is dominated by the number of vectors; take a look at the table below for an example:</p>
<table><thead><tr><th>Component</th><th>Size derivation</th><th>Typical size</th><th>Size @1M vectors</th><th>Size @100M vectors</th></tr></thead><tbody><tr><td>Node</td><td>4B (float) x N dimensions</td><td>2-12kB</td><td>2-12GB</td><td>200-1200GB</td></tr><tr><td>Edge</td><td>10B x 20 connections</td><td>200B</td><td>200MB</td><td>20GB</td></tr></tbody></table>
<p>As you can see, the memory requirements of an HNSW index can quickly become a bottleneck. This is where <a class="" href="/weaviate/concepts/vector-quantization">quantization</a> can be used to reduce the size of the index in memory.</p>
<p>HNSW is very fast, memory efficient, approach to similarity search. The memory cache only stores the highest layer instead of storing all of the data objects in the lowest layer. When the search moves from a higher layer to a lower one, HNSW only adds the data objects that are closest to the search query. This means HNSW uses a relatively small amount of memory compared to other search algorithms.</p>
<p>Have another look at the diagram; it demonstrates how the HNSW algorithm searches. The search vector in the top layer connects to a partial result in layer one. The objects in layer one lead HNSW to the result set in layer zero. This allows HNSW to skip objects that are unrelated to the search query.</p>
<p>Inserting a vector into an HNSW index works in a similar way. The HNSW algorithm finds the closest data objects in the highest layer, and then moves down to the next layer. It continues until it finds the best place to insert the new vector. The HNSW algorithm then connects the new vector to the existing vectors in that layer.</p>
<p><img decoding="async" loading="lazy" alt="HNSW insertion" src="/assets/images/hnsw_3_insertion-114be0c5912d8e63d03100906443589c.png" title="HNSW insertion" width="2080" height="1240" class="img_ev3q"></p>
<h3 class="anchor anchorTargetStickyNavbar_Vzrq" id="managing-search-quality-vs-speed-tradeoffs">Managing search quality vs speed tradeoffs<a href="#managing-search-quality-vs-speed-tradeoffs" class="hash-link" aria-label="Direct link to Managing search quality vs speed tradeoffs" title="Direct link to Managing search quality vs speed tradeoffs" translate="no">​</a></h3>
<p>HNSW parameters can be adjusted to adjust search quality against speed.</p>
<p><img decoding="async" loading="lazy" alt="HNSW parameters" src="/assets/images/hnsw_4_parameters-6d6594cb2ad2671341f2fb4cde072bc2.png" title="HNSW parameters" width="2080" height="1320" class="img_ev3q"></p>
<p>The <code>ef</code> parameter is a critical setting for balancing the trade-off between search speed and quality.</p>
<p>The <code>ef</code> parameter dictates the size of the dynamic list used by the HNSW algorithm during the search process. A higher <code>ef</code> value results in a more extensive search, enhancing accuracy but potentially slowing down the query.</p>
<p>In contrast, a lower <code>ef</code> makes the search faster but might compromise on accuracy. This balance is crucial in scenarios where either speed or accuracy is a priority. For instance, in applications where rapid responses are critical, a lower <code>ef</code> might be preferable, even at the expense of some accuracy. Conversely, in analytical or research contexts where precision is paramount, a higher <code>ef</code> would be more suitable, despite the increased query time.</p>
<p><code>ef</code> can be configured explicitly or dynamically. This feature is particularly beneficial in environments with varying query patterns. When <code>ef</code> is configured dynamically, Weaviate optimizes the balance between speed and recall based on real-time query requirements.</p>
<p>To enable dynamic <code>ef</code>, set <code>ef</code>: -1. Weaviate adjusts the size of the ANN list based on the query response limit. The calculation also takes into account the values of <code>dynamicEfMin</code>, <code>dynamicEfMax</code>, and <code>dynamicEfFactor</code>.</p>
<h3 class="anchor anchorTargetStickyNavbar_Vzrq" id="dynamic-ef">Dynamic ef<a href="#dynamic-ef" class="hash-link" aria-label="Direct link to Dynamic ef" title="Direct link to Dynamic ef" translate="no">​</a></h3>
<p>The <code>ef</code> parameter controls the size of the ANN list at query time. You can configure a specific list size or else let Weaviate configure the list dynamically. If you choose dynamic <code>ef</code>, Weaviate provides several options to control the size of the ANN list.</p>
<p>The length of the list is determined by the query response limit that you set in your query. Weaviate uses the query limit as an anchor and modifies the size of ANN list according to the values you set for the <code>dynamicEf</code> parameters.</p>
<ul>
<li class=""><code>dynamicEfMin</code> sets a lower bound on the list length.</li>
<li class=""><code>dynamicEfMax</code> sets an upper bound on the list length.</li>
<li class=""><code>dynamicEfFactor</code> sets a range for the list.</li>
</ul>
<p>The dynamic list size will be set as the query limit multiplied by <code>dynamicEfFactor</code>, modified by a minimum of <code>dynamicEfMin</code> and a maximum of <code>dynamicEfMax</code>.</p>

Weaviate doesn&#x27;t exceed <code>dynamicEfMax</code> even if the query limit is large enough to suggest a higher value. If the query limit is higher than <code>dynamicEfMax</code>, <code>dynamicEfMax</code> does not have any effect. In this case, dynamic <code>ef</code> value is equal to the query limit.</p>
<p>To determine the length of the ANN list, Weaviate multiples the query limit by <code>dynamicEfFactor</code>. The list range is modified by <code>dynamicEfMin</code> and <code>dynamicEfMax</code>.</p>

<p>The resulting search list has these characteristics.</p>
<ul>
<li class="">A potential length of 40 objects ( (&quot;dynamicEfFactor&quot;: 10) * (limit: 4) ).</li>
<li class="">A minimum length of 5 objects (&quot;dynamicEfMin&quot;: 5).</li>
<li class="">A maximum length of 25 objects (&quot;dynamicEfMax&quot;: 25).</li>
<li class="">An actual size of 5 to 25 objects.</li>
</ul>
<p>If you use the <a class="" href="/deploy/installation-guides/docker-installation"><code>docker-compose.yml</code> file from Weaviate</a> to run your local instance, the <code>QUERY_DEFAULTS_LIMIT</code> environment variable sets a reasonable default query limit. To prevent out of memory errors,<code>QUERY_DEFAULTS_LIMIT</code> is significantly lower than <code>QUERY_MAXIMUM_RESULTS</code>.</p>
<p>To change the default limit, edit the value for <code>QUERY_DEFAULTS_LIMIT</code> when you configure your Weaviate instance.</p>
<h3 class="anchor anchorTargetStickyNavbar_Vzrq" id="deletions">Deletions<a href="#deletions" class="hash-link" aria-label="Direct link to Deletions" title="Direct link to Deletions" translate="no">​</a></h3>
<p>Cleanup is an async process runs that rebuilds the HNSW graph after deletes and updates. Prior to cleanup, objects are marked as deleted, but they are still connected to the HNSW graph. During cleanup, the edges are reassigned and the objects are deleted for good.</p>
<h3 class="anchor anchorTargetStickyNavbar_Vzrq" id="asynchronous-indexing">Asynchronous indexing<a href="#asynchronous-indexing" class="hash-link" aria-label="Direct link to Asynchronous indexing" title="Direct link to Asynchronous indexing" translate="no">​</a></h3>
<p>This feature relates to the vector index, specifically only to the HNSW index.</p>
<p>Asynchronous indexing can be enabled by opting in as follows:</p>
<ul>
<li class="">Open-source users can do this by setting the <code>ASYNC_INDEXING</code> environment variable to <code>true</code>.</li>
<li class="">Weaviate Cloud users can do this by toggling the <code>Enable async indexing</code> switch in the Weaviate Cloud Console.</li>
</ul>
<p>With synchronous indexing, the vector index is updated in lockstep with the object store. Updating an HNSW index can be an expensive operation, especially as the size of the index grows. As a result, the indexing operation can be the bottleneck in the system, slowing down the time for user requests to be completed.</p>
<p>When asynchronous indexing is enabled, all vector indexing operations go through a queue. This applies to not only batch imports, but also to single object imports, deletions, and updates.</p>
<p>This means that the object store can be updated quickly to finish performing user requests while the vector index updates in the background. Asynchronous indexing is especially useful for importing large amounts of data.</p>
<p>This means that there will be a short delay between object creation and the object being available for vector search using the HNSW index. The number of objects in the queue can be monitored per node <a class="" href="/deploy/configuration/nodes">as shown here</a>.</p>
<div class="theme-admonition theme-admonition-info admonition_xJq3 alert alert--info"><div class="admonitionHeading_Gvgb"><span class="admonitionIcon_Rf37"><svg viewBox="0 0 14 16"><path fill-rule="evenodd" d="M7 2.3c3.14 0 5.7 2.56 5.7 5.7s-2.56 5.7-5.7 5.7A5.71 5.71 0 0 1 1.3 8c0-3.14 2.56-5.7 5.7-5.7zM7 1C3.14 1 0 4.14 0 8s3.14 7 7 7 7-3.14 7-7-3.14-7-7-7zm1 3H6v5h2V4zm0 6H6v2h2v-2z"></path></svg></span>Changes in <code>v1.28</code></div><div class="admonitionContent_BuS1"><p>In Weaviate <code>v1.22</code> to <code>v1.27</code>, the async indexing feature only affected batch import operations, using an in-memory queue.</p><br><p>Starting in <code>v1.28</code>, the async indexing feature has been expanded to include single object imports, deletions, and updates. Additionally, the in-memory queue has been replaced with a persistent, on-disk queue. This change allows for more robust handling of indexing operations, and improves performance though reduction of lock contention and memory usage.</p><br><p>The use of an on-disk queue may result in a slight increase in disk usage, however this is expected to be a small percentage of the total disk usage.</p></div></div>
<h2 class="anchor anchorTargetStickyNavbar_Vzrq" id="flat-index-1">Flat index<a href="#flat-index-1" class="hash-link" aria-label="Direct link to Flat index" title="Direct link to Flat index" translate="no">​</a></h2>
<div class="theme-admonition theme-admonition-info admonition_xJq3 alert alert--info"><div class="admonitionHeading_Gvgb"><span class="admonitionIcon_Rf37"><svg viewBox="0 0 14 16"><path fill-rule="evenodd" d="M7 2.3c3.14 0 5.7 2.56 5.7 5.7s-2.56 5.7-5.7 5.7A5.71 5.71 0 0 1 1.3 8c0-3.14 2.56-5.7 5.7-5.7zM7 1C3.14 1 0 4.14 0 8s3.14 7 7 7 7-3.14 7-7-3.14-7-7-7zm1 3H6v5h2V4zm0 6H6v2h2v-2z"></path></svg></span>Added in <code>v1.23</code></div></div>
<p>The <strong>flat index</strong> is one of the fundamental ways to implement vector indexing in databases. As the name suggests, it&#x27;s a simple, lightweight index that is fast to build and has a very small memory footprint. This index type is a good choice for use cases where each end user (i.e. tenant) has their own, isolated, dataset, such as in a SaaS product for example, or a database of isolated record sets.</p>
<p>As the name suggests, the flat index is a single layer of disk-backed data objects and thus a very small memory footprint. The flat index is a good choice for small collections, such as for multi-tenancy use cases.</p>
<p>A drawback of the flat index is that it does not scale well to large collections as it has a linear time complexity as a function of the number of data objects, unlike the <code>hnsw</code> index which has a logarithmic time complexity.</p>
<h2 class="anchor anchorTargetStickyNavbar_Vzrq" id="dynamic-index">Dynamic index<a href="#dynamic-index" class="hash-link" aria-label="Direct link to Dynamic index" title="Direct link to Dynamic index" translate="no">​</a></h2>
<div class="theme-admonition theme-admonition-caution admonition_xJq3 alert alert--warning"><div class="admonitionHeading_Gvgb"><span class="admonitionIcon_Rf37"><svg viewBox="0 0 16 16"><path fill-rule="evenodd" d="M8.893 1.5c-.183-.31-.52-.5-.887-.5s-.703.19-.886.5L.138 13.499a.98.98 0 0 0 0 1.001c.193.31.53.501.886.501h13.964c.367 0 .704-.19.877-.5a1.03 1.03 0 0 0 .01-1.002L8.893 1.5zm.133 11.497H6.987v-2.003h2.039v2.003zm0-3.004H6.987V5.987h2.039v4.006z"></path></svg></span>Experimental feature</div><div class="admonitionContent_BuS1"><p>Available starting in <code>v1.25</code>. This is an experimental feature. Use with caution.</p></div></div>
<!-- -->
<div class="theme-admonition theme-admonition-info admonition_xJq3 alert alert--info"><div class="admonitionHeading_Gvgb"><span class="admonitionIcon_Rf37"><svg viewBox="0 0 14 16"><path fill-rule="evenodd" d="M7 2.3c3.14 0 5.7 2.56 5.7 5.7s-2.56 5.7-5.7 5.7A5.71 5.71 0 0 1 1.3 8c0-3.14 2.56-5.7 5.7-5.7zM7 1C3.14 1 0 4.14 0 8s3.14 7 7 7 7-3.14 7-7-3.14-7-7-7zm1 3H6v5h2V4zm0 6H6v2h2v-2z"></path></svg></span>Dynamic index requires <code>ASYNC_INDEXING</code></div><div class="admonitionContent_BuS1"><p>Dynamic indexes require asynchronous indexing. To enable asynchronous indexing in a self-hosted Weaviate instance, set the <code>ASYNC_INDEXING</code> <a class="" href="/deploy/configuration/env-vars#general">environment variable</a> to <code>true</code>. If your instance is hosted in Weaviate Cloud, use the  Weaviate Cloud console to enable asynchronous indexing.</p></div></div>
<p>The flat index is ideal for use cases with a small object count and provides lower memory overhead and good latency. As the object count increases the HNSW index provides a more viable solution as HNSW speeds up search. The goal of the dynamic index is to shorten latencies during querying time at the cost of a larger memory footprint as you scale.</p>
<p>By configuring a dynamic index, you can automatically switch from flat to HNSW indexes. This switch occurs when the object count exceeds a prespecified threshold (by default 10,000). This functionality only works with async indexing enabled. When the threshold is hit while importing, all the data piles up in the async queue, the HNSW index is constructed in the background and when ready the swap from flat to HNSW is completed.</p>
<p>Currently, this is only a one-way upgrade from a flat to an HNSW index, it does not support changing back to a flat index even if the object count goes below the threshold due to deletion.</p>
<p>This is particularly useful in a multi-tenant setup where building an HNSW index per tenant would introduce extra overhead. With a dynamic index, as individual tenants grow their index will switch from flat to HNSW, while smaller tenants&#x27; indexes remain flat.</p>
<h2 class="anchor anchorTargetStickyNavbar_Vzrq" id="vector-cache-considerations">Vector cache considerations<a href="#vector-cache-considerations" class="hash-link" aria-label="Direct link to Vector cache considerations" title="Direct link to Vector cache considerations" translate="no">​</a></h2>
<p>For optimal search and import performance, previously imported vectors need to be in memory. A disk lookup for a vector is orders of magnitudes slower than memory lookup, so the disk cache should be used sparingly. However, Weaviate can limit the number of vectors in memory. By default, this limit is set to one trillion (<code>1e12</code>) objects when a new collection is created.</p>
<p>During import set <code>vectorCacheMaxObjects</code> high enough that all vectors can be held in memory. Each import requires multiple searches. Import performance drops drastically when there isn&#x27;t enough memory to hold all of the vectors in the cache.</p>
<p>After import, when your workload is mostly querying, experiment with vector cache limits that are less than your total dataset size.</p>
<p>Vectors that aren&#x27;t currently in cache are added to the cache if there is still room. If the cache fills, Weaviate drops the whole cache. All future vectors have to be read from disk for the first time. Then, subsequent queries run against the cache until it fills again and the procedure repeats. Note that the cache can be a very valuable tool if you have a large dataset, and a large percentage of users only query a specific subset of vectors. In this case you might be able to serve the largest user group from cache while requiring disk lookups for &quot;irregular&quot; queries.</p>
<h2 class="anchor anchorTargetStickyNavbar_Vzrq" id="vector-indexing-faq">Vector indexing FAQ<a href="#vector-indexing-faq" class="hash-link" aria-label="Direct link to Vector indexing FAQ" title="Direct link to Vector indexing FAQ" translate="no">​</a></h2>
<h3 class="anchor anchorTargetStickyNavbar_Vzrq" id="can-i-use-vector-indexing-with-vector-quantization">Can I use vector indexing with vector quantization?<a href="#can-i-use-vector-indexing-with-vector-quantization" class="hash-link" aria-label="Direct link to Can I use vector indexing with vector quantization?" title="Direct link to Can I use vector indexing with vector quantization?" translate="no">​</a></h3>
<p>Yes, you can read more about it in <a class="" href="/weaviate/concepts/vector-quantization">vector quantization (compression)</a>.</p>
<h3 class="anchor anchorTargetStickyNavbar_Vzrq" id="which-vector-index-is-right-for-me">Which vector index is right for me?<a href="#which-vector-index-is-right-for-me" class="hash-link" aria-label="Direct link to Which vector index is right for me?" title="Direct link to Which vector index is right for me?" translate="no">​</a></h3>
<p>A simple heuristic is that for use cases such as SaaS products where each end user (i.e. tenant) has their own, isolated, dataset, the <code>flat</code> index is a good choice. For use cases with large collections, the <code>hnsw</code> index may be a better choice.</p>
<p>When choosing an index type, use the following as a guide:</p>
<ul>
<li class=""><strong>Flat index</strong>: Use for small collections with a known size.</li>
<li class=""><strong>HNSW index</strong>: Use for large collections with a known size.</li>
<li class=""><strong>Dynamic index</strong>: Use for collections with an unknown size or collections that may grow over time.</li>
</ul>
<p>Note that the vector index type parameter only specifies how the vectors of data objects are <em>indexed</em>. The index is used for data retrieval and similarity search.</p>
<p>The <code>vectorizer</code> parameter determines how the data vectors are created (which numbers the vectors contain). <code>vectorizer</code> specifies a <a class="" href="/weaviate/modules">module</a>, such as <code>text2vec-contextionary</code>, that Weaviate uses to create the vectors. (You can also set to <code>vectorizer</code> to <code>none</code> if you want to import your own vectors).</p>
<p>To learn more about configuring the collection, see <a class="" href="/weaviate/manage-collections/vector-config">this how-to page</a>.</p>
<h3 class="anchor anchorTargetStickyNavbar_Vzrq" id="which-distance-metrics-can-i-use-with-vector-indexing">Which distance metrics can I use with vector indexing?<a href="#which-distance-metrics-can-i-use-with-vector-indexing" class="hash-link" aria-label="Direct link to Which distance metrics can I use with vector indexing?" title="Direct link to Which distance metrics can I use with vector indexing?" translate="no">​</a></h3>
<p>All of <a class="" href="/weaviate/config-refs/distances">the distance metrics</a>, such as cosine similarity, can be used with any vector index type.</p>
<h3 class="anchor anchorTargetStickyNavbar_Vzrq" id="how-to-configure-the-vector-index-type-in-weaviate">How to configure the vector index type in Weaviate?<a href="#how-to-configure-the-vector-index-type-in-weaviate" class="hash-link" aria-label="Direct link to How to configure the vector index type in Weaviate?" title="Direct link to How to configure the vector index type in Weaviate?" translate="no">​</a></h3>
<p>The index type can be specified per data collection via the <a class="" href="/weaviate/manage-collections/vector-config#set-vector-index-type">collection definition</a> settings, according to available <a class="" href="/weaviate/config-refs/indexing/vector-index">vector index settings</a>.</p>
<h3 class="anchor anchorTargetStickyNavbar_Vzrq" id="when-to-skip-indexing">When to skip indexing<a href="#when-to-skip-indexing" class="hash-link" aria-label="Direct link to When to skip indexing" title="Direct link to When to skip indexing" translate="no">​</a></h3>
<p>There are situations where it doesn&#x27;t make sense to vectorize a collection. For example, if the collection consists solely of references between two other collections, or if the collection contains mostly duplicate elements.</p>
<p>Importing duplicate vectors into HNSW is very expensive. The import algorithm checks early on if a candidate vector&#x27;s distance is greater than the worst candidate&#x27;s distance. When there are lots of duplicate vectors, this early exit condition is never met so each import or query results in an exhaustive search.</p>
<p>To avoid indexing a collection, set <code>&quot;skip&quot;</code> to <code>&quot;true&quot;</code>. By default, collections are indexed.</p>
