<h4 class="anchor anchorTargetStickyNavbar_Vzrq" id="q-why-would-i-use-weaviate-as-my-vector-database">Q: Why would I use Weaviate as my vector database?<a href="#q-why-would-i-use-weaviate-as-my-vector-database" class="hash-link" aria-label="Direct link to Q: Why would I use Weaviate as my vector database?" title="Direct link to Q: Why would I use Weaviate as my vector database?" translate="no">​</a></h4>
<details class="details_lb9f alert alert--info details_b_Ee" data-collapsed="true"><summary>Answer</summary><div><div class="collapsibleContent_i85q"><blockquote>
<p>Our goal is three-folded. Firstly, we want to make it as easy as possible for others to create their own semantic systems or vector search engines (hence, our APIs are GraphQL based). Secondly, we have a strong focus on the semantic element (the &quot;knowledge&quot; in &quot;vector databases,&quot; if you will). Our ultimate goal is to have Weaviate help you manage, index, and &quot;understand&quot; your data so that you can build newer, better, and faster applications. And thirdly, we want you to be able to run it everywhere. This is the reason why Weaviate comes containerized.</p>
</blockquote></div></div></details>
<h4 class="anchor anchorTargetStickyNavbar_Vzrq" id="q-what-is-the-difference-between-weaviate-and-for-example-elasticsearch">Q: What is the difference between Weaviate and for example Elasticsearch?<a href="#q-what-is-the-difference-between-weaviate-and-for-example-elasticsearch" class="hash-link" aria-label="Direct link to Q: What is the difference between Weaviate and for example Elasticsearch?" title="Direct link to Q: What is the difference between Weaviate and for example Elasticsearch?" translate="no">​</a></h4>
<details class="details_lb9f alert alert--info details_b_Ee" data-collapsed="true"><summary>Answer</summary><div><div class="collapsibleContent_i85q"><blockquote>
<p>Other database systems like Elasticsearch rely on inverted indexes, which makes search super fast. Weaviate also uses inverted indexes to store data and values. But additionally, Weaviate is also a vector-native search database, which means that data is stored as vectors, which enables semantic search. This combination of data storage is unique, and enables fast, filtered and semantic search from end-to-end.</p>
</blockquote></div></div></details>

<h2 class="anchor anchorTargetStickyNavbar_Vzrq" id="schema-and-data-structure">Schema and data structure<a href="#schema-and-data-structure" class="hash-link" aria-label="Direct link to Schema and data structure" title="Direct link to Schema and data structure" translate="no">​</a></h2>
<h4 class="anchor anchorTargetStickyNavbar_Vzrq" id="q-are-there-any-best-practices-or-guidelines-to-consider-when-designing-a-schema">Q: Are there any &#x27;best practices&#x27; or guidelines to consider when designing a schema?<a href="#q-are-there-any-best-practices-or-guidelines-to-consider-when-designing-a-schema" class="hash-link" aria-label="Direct link to Q: Are there any &#x27;best practices&#x27; or guidelines to consider when designing a schema?" title="Direct link to Q: Are there any &#x27;best practices&#x27; or guidelines to consider when designing a schema?" translate="no">​</a></h4>
<p><em>(E.g. if I was looking to perform a semantic search over a the content of a Book would I look to have Chapter and Paragraph represented in the schema etc, would this be preferred over including the entire content of the novel in a single property?)</em></p>
<details class="details_lb9f alert alert--info details_b_Ee" data-collapsed="true"><summary>Answer</summary><div><div class="collapsibleContent_i85q"><blockquote>
<p>As a rule of thumb, the smaller the units, the more accurate the search will be. Two objects of e.g. a sentence would most likely contain more information in their vector embedding than a common vector (which is essentially just the mean of sentences). At the same time more objects leads to a higher import time and (since each vector also makes up some data) more space. (E.g. when using transformers, a single vector is 768xfloat32 = 3KB. This can easily make a difference if you have millions, etc.) of vectors. As a rule of thumb, the more vectors you have the more memory you&#x27;re going to need.</p>
<p>So, basically, it&#x27;s a set of tradeoffs. Personally we&#x27;ve had great success with using paragraphs as individual units, as there&#x27;s little benefit in going even more granular, but it&#x27;s still much more precise than whole chapters, etc.</p>
<p>You can use cross-references to link e.g. chapters to paragraphs. Note that resolving a cross-references takes a performance penalty. Essentially resolving A1-&gt;B1 is the same cost as looking up both A1 and B1 indvidually. But at scale, this can add up.</p>
<p>So, consider denormalizing your data, i.e. storing the data in a way that you can resolve the cross-references without actually looking them up. This is a common pattern in databases, and it&#x27;s also a common pattern in Weaviate.</p>
</blockquote></div></div></details>
<h4 class="anchor anchorTargetStickyNavbar_Vzrq" id="q-should-i-use-references-in-my-schema">Q: Should I use references in my schema?<a href="#q-should-i-use-references-in-my-schema" class="hash-link" aria-label="Direct link to Q: Should I use references in my schema?" title="Direct link to Q: Should I use references in my schema?" translate="no">​</a></h4>
<details class="details_lb9f alert alert--info details_b_Ee" data-collapsed="true"><summary>Answer</summary><div><div class="collapsibleContent_i85q"><blockquote>
<p>In short: for convenience you can add relations to your data schema, because you need less code and queries to get data. But resolving references in queries takes some of the performance.</p>
<ol>
<li class="">If your ultimate goal is performance, references probably don&#x27;t add any value, as resolving them adds a cost.</li>
<li class="">If your goal is represent complex relationships between your data items, they can help a lot. You can resolve references in a single query, so if you have collections with multiple links, it could definitely be helpful to resolve some of those connections in a single query. On the other hand, if you have a single (bi-directional) reference in your data, you could also just denormalize the links (e.g. with an ID field) and resolve them during search.</li>
</ol>
</blockquote></div></div></details>

<h4 class="anchor anchorTargetStickyNavbar_Vzrq" id="q-the-quality-of-my-search-results-change-depending-on-the-specified-limit-why-how-can-i-fix-this">Q: The quality of my search results change depending on the specified limit. Why? How can I fix this?<a href="#q-the-quality-of-my-search-results-change-depending-on-the-specified-limit-why-how-can-i-fix-this" class="hash-link" aria-label="Direct link to Q: The quality of my search results change depending on the specified limit. Why? How can I fix this?" title="Direct link to Q: The quality of my search results change depending on the specified limit. Why? How can I fix this?" translate="no">​</a></h4>
<details class="details_lb9f alert alert--info details_b_Ee" data-collapsed="true"><summary>Answer</summary><div><div class="collapsibleContent_i85q"><p>Weaviate makes use of ANN indexes to serve vector searches. An ANN index is an approximate nearest neighbor index. The &quot;approximate&quot; part refers to an explicit recall-query-speed tradeoff. This trade-off is presented in detail in the <a class="" href="/weaviate/benchmarks/ann#benchmark-results">ANN benchmarks section</a>. For example, a 98% recall for a given set of HNSW parameters means that 2% of results will not match the true nearest neighbors. What build parameters lead to what recall depends on the dataset used. The benchmark pages shows 4 different example datasets. Based on the characteristic of each dataset you can pick the one closest to your production load and draw conclusions about the expected recall for the respective build and query-time parameters.</p><p>Generally if you need a higher recall than the default parameters provide you with, you can use stronger parameters. This can either be done at build time (<code>efConstruction</code>, <code>maxConnections</code>) or at query time (<code>ef</code>). Roughly speaking, a higher <code>ef</code> value at query time means a more thorough search. It will have a slightly higher latency, but also lead to a slightly better recall.</p><p>By changing the specified limit, you are implicitly changing the <code>ef</code> parameter. This is because the default <code>ef</code> value is set to <code>-1</code>, indicating that Weaviate should pick the parameter based on the limit. The dynamic <code>ef</code> value is controlled using the configuration fields <code>dynamicEfMin</code> which acts as a lower boundary, <code>dynamicEfMax</code> which acts as an upper boundary and <code>dynamicEfFactor</code> which is the factor to derive the target <code>ef</code> based on the limit within the lower and upper boundary.</p><p>Example: Using the default parameters <code>ef=-1</code>, <code>dynamicEfMin=100</code>, <code>dynamicEfMax=500</code>, <code>dynamicEfFactor=8</code>, you will end up with the following <code>ef</code> values based on the limit:</p><ul>
<li class=""><code>limit=1</code>, dynamically calculated: <code>ef=1*8=8</code>. This value is below the lower boundary, so <code>ef</code> is set to <code>100</code>.</li>
<li class=""><code>limit=20</code>, dynamically calculated: <code>ef=20*8=160</code>. This value is within the boundaries, so <code>ef</code> is <code>160</code>.</li>
<li class=""><code>limit=100</code>, dynamically calculated: <code>ef=100*8=800</code>. This value is above the upper boundary, so <code>ef</code> is set to <code>500</code>.</li>
</ul><p>If you need a higher search quality for a given limit you can consider the following options:</p><ol>
<li class="">Instead of using a dynamic <code>ef</code> value, use a fixed one that provides the desired recall.</li>
<li class="">If your search quality varies a lot depending on the query-time <code>ef</code> values, you should also consider choosing stronger build parameters. The <a class="" href="/weaviate/benchmarks/ann#benchmark-results">ANN benchmarks section</a> present a combination of many different parameter combination for various datasets.</li>
</ol></div></div></details>

Instead, they require you to have all of your vectors present, then you build the algorithm once. After a build, you can only query them, but cannot add more elements or change existing elements. Thus, they aren&#x27;t capable of the CRUD operations we want to support in Weaviate.</p>
</blockquote></div></div></details>

<h4 class="anchor anchorTargetStickyNavbar_Vzrq" id="q-how-does-weaviates-vector-and-scalar-filtering-work">Q: How does Weaviate&#x27;s vector and scalar filtering work?<a href="#q-how-does-weaviates-vector-and-scalar-filtering-work" class="hash-link" aria-label="Direct link to Q: How does Weaviate&#x27;s vector and scalar filtering work?" title="Direct link to Q: How does Weaviate&#x27;s vector and scalar filtering work?" translate="no">​</a></h4>
<details class="details_lb9f alert alert--info details_b_Ee" data-collapsed="true"><summary>Answer</summary><div><div class="collapsibleContent_i85q"><blockquote>
<p>It&#x27;s a 2-step process:</p>
<ol>
<li class="">The inverted index (which is <a href="#q-does-weaviate-use-hnswlib" class="">built at import time</a>) queries to produce an allowed list of the specified document ids. Then the ANN index is queried with this allow list (the list being one of the reasons for our custom implementation).</li>
<li class="">If we encounter a document id which would be a close match, but isn&#x27;t on the allow list the id is treated as a candidate (i.e. we add it to our list of links to evaluate), but is never added to the result set. Since we only add allowed IDs to the set, we don&#x27;t exit early, i.e. before the top <code>k</code> elements are reached.</li>
</ol>
<p>For more information on the technical implementations, see <a href="https://www.youtube.com/watch?v=6hdEJdHWXRE" target="_blank" rel="noopener noreferrer" class="">this video</a>.</p>
</blockquote></div></div></details>

<h4 class="anchor anchorTargetStickyNavbar_Vzrq" id="q-what-would-you-say-is-more-important-for-query-speed-in-weaviate-more-cpu-power-or-more-ram">Q: What would you say is more important for query speed in Weaviate: More CPU power, or more RAM?<a href="#q-what-would-you-say-is-more-important-for-query-speed-in-weaviate-more-cpu-power-or-more-ram" class="hash-link" aria-label="Direct link to Q: What would you say is more important for query speed in Weaviate: More CPU power, or more RAM?" title="Direct link to Q: What would you say is more important for query speed in Weaviate: More CPU power, or more RAM?" translate="no">​</a></h4>
<p>More concretely: If you had to pick between a machine that has 16 GB of RAM and 2 CPUs, or a machine that has 8 GB of RAM and 4 CPUs, which would you pick?</p>
<details class="details_lb9f alert alert--info details_b_Ee" data-collapsed="true"><summary>Answer</summary><div><div class="collapsibleContent_i85q"><blockquote>
<p>This is a very difficult to answer 100% correctly, because there are several factors in play:</p>
<ul>
<li class=""><strong>The vector search itself</strong>. This part is CPU-bound, however only with regards to throughput: A single search is single-threaded. Multiple parallel searches can use multiple threads. So if you measure the time of a single request (otherwise idle), it will be the same whether the machine has 1 core or 100. However, if your QPS approach the throughput of a CPU, you&#x27;ll see massive benefits by adding more Cores</li>
<li class=""><strong>The retrieval of the objects</strong>. Once the vector search part is done, we are essentially left with a list of n IDs which need to be resolved to actual objects. This is IO-bound in general. However, all disk files are memory-mapped. So generally, more mem will allow you to hold more of the disk state in memory. In real life however, it&#x27;s not that simple. Searches are rarely evenly distributed. So let&#x27;s pretend that 90% of searches will return just 10% of objects (because these are more popular search results). Then if those 10% of the disk objects are already cached in mem, there&#x27;s no benefit in adding more memory.</li>
</ul>
<p>Taking the above in mind: we can carefully say: If throughput is the problem, increase CPU, if response time is the problem increase mem. However, note that the latter only adds value if there are more things that can be cached. If you have enough mem to cache your entire disk state (or at least the parts that are relevant for most queries), additional memory won&#x27;t add any additional benefit.
If we are talking about imports on the other hand, they are almost always CPU-bound because of the cost of creating the HNSW index. So, if you can resize between import and query, my recommendation would be roughly prefer CPUs while importing and then gradually replace CPU with memory at query time - until you see no more benefits. (This assumes that there is a separation between importing and querying which might not always be the case in real life).</p>
</blockquote></div></div></details>

<h4 class="anchor anchorTargetStickyNavbar_Vzrq" id="q-when-scalar-and-vector-search-are-combined-will-the-scalar-filter-happen-before-or-after-the-nearest-neighbor-vector-search">Q: When scalar and vector search are combined, will the scalar filter happen before or after the nearest neighbor (vector) search?<a href="#q-when-scalar-and-vector-search-are-combined-will-the-scalar-filter-happen-before-or-after-the-nearest-neighbor-vector-search" class="hash-link" aria-label="Direct link to Q: When scalar and vector search are combined, will the scalar filter happen before or after the nearest neighbor (vector) search?" title="Direct link to Q: When scalar and vector search are combined, will the scalar filter happen before or after the nearest neighbor (vector) search?" translate="no">​</a></h4>
<details class="details_lb9f alert alert--info details_b_Ee" data-collapsed="true"><summary>Answer</summary><div><div class="collapsibleContent_i85q"><blockquote>
<p>The mixed structured vector searches in Weaviate are pre-filter. There is an inverted index which is queried first to basically form an allow-list, in the HNSW search the allow list is then used to treat non-allowed doc ids only as nodes to follow connections, but not to add to the result set.</p>
</blockquote></div></div></details>
<h4 class="anchor anchorTargetStickyNavbar_Vzrq" id="q-regarding-filtered-vector-search-since-this-is-a-two-phase-pipeline-how-big-can-that-list-of-ids-get-do-you-know-how-that-size-might-affect-query-performance">Q: Regarding &quot;filtered vector search&quot;: Since this is a two-phase pipeline, how big can that list of IDs get? Do you know how that size might affect query performance?<a href="#q-regarding-filtered-vector-search-since-this-is-a-two-phase-pipeline-how-big-can-that-list-of-ids-get-do-you-know-how-that-size-might-affect-query-performance" class="hash-link" aria-label="Direct link to Q: Regarding &quot;filtered vector search&quot;: Since this is a two-phase pipeline, how big can that list of IDs get? Do you know how that size might affect query performance?" title="Direct link to Q: Regarding &quot;filtered vector search&quot;: Since this is a two-phase pipeline, how big can that list of IDs get? Do you know how that size might affect query performance?" translate="no">​</a></h4>
<details class="details_lb9f alert alert--info details_b_Ee" data-collapsed="true"><summary>Answer</summary><div><div class="collapsibleContent_i85q"><blockquote>
<p>Essentially the list ids uses the internal doc id which is a <code>uint64</code> or 8 bytes per ID. The list can grow as long as you have memory available. So for example with 2GB of free memory, it could hold 250M ids, with 20GB it could hold 2.5B ids, etc.</p>
<p>Performance wise there are two things to consider:</p>
<ol>
<li class="">Building the lookup list</li>
<li class="">Filtering the results when vector searching</li>
</ol>
<p>Building the list is a typical inverted index look up, so depending on the operator this is just a single read on == (or a set of range reads, e.g. for &gt;7, we&#x27;d read the value rows from 7 to infinity). This process is pretty efficient, similar to how the same thing would happen in a traditional search engine, such as elasticsearch</p>
<p>Performing the filtering during the vector search depends on whether the filter is very restrictive or very loose. In the case you mentioned where a lot of IDs are included, it will be very efficient. Because the equivalent of an unfiltered search would be the one where your ID list contains all possible IDs. So the HNSW index would behave normally. There is however, a small penalty whenever a list is present: We need to check if the current ID is contained an the allow-list. This is essentially a hashmap lookup, so it should be O(1) per object. Nevertheless, there is a slight performance penalty.</p>
<p>Now the other extreme, a very restrictive list, i.e few IDs on the list, actually takes considerably more time. Because the HNSW index will find neighboring IDs, but since they&#x27;re not contained, they cannot be added as result candidates, meaning that all we can do with them is evaluating their connections, but not the points themselves. In the extreme case of a list that is very, very restrictive, say just 10 objects out of 1B in the worst case the search would become exhaustive if you the filtered ids are very far from the query. In this extreme case, it would actually be much more efficient to just skip the index and do a brute-force indexless vector search on the 10 ids. So, there is a cut-off when a brute-force search becomes more efficient than a heavily-restricted vector search with HNSW. We do not yet have any optimization to discovery such a cut-off point and skip the index, but this should be fairly simple to implement if this ever becomes an actual problem.</p>
</blockquote></div></div></details>

<h4 class="anchor anchorTargetStickyNavbar_Vzrq" id="q-what-is-weaviates-consistency-model-in-a-distributed-setup">Q: What is Weaviate&#x27;s consistency model in a distributed setup?<a href="#q-what-is-weaviates-consistency-model-in-a-distributed-setup" class="hash-link" aria-label="Direct link to Q: What is Weaviate&#x27;s consistency model in a distributed setup?" title="Direct link to Q: What is Weaviate&#x27;s consistency model in a distributed setup?" translate="no">​</a></h4>
<details class="details_lb9f alert alert--info details_b_Ee" data-collapsed="true"><summary>Answer</summary><div><div class="collapsibleContent_i85q"><blockquote>
<p>Weaviate is generally modeled to prefer Availability over Consistency (AP over CP). It is designed to deliver low search latencies under high throughput in situations where availability is more business-critical than consistency. If strict serializability is required on your data, we generally recommend storing your data in a different primary data store, use Weaviate as an auxiliary data store, and set up replication between the two. If you do not need serializability and eventual consistency is enough for your use case, Weaviate can be used as a primary datastore.</p>
<p>Weaviate has no notion of transactions, operations always affect exactly a single key, therefore Serializability is not applicable. In a distributed setup (under development) Weaviate&#x27;s consistency model is eventual consistency. When a cluster is healthy, all changes are replicated to all affected nodes by the time the write is acknowledged by the user. Objects will immediately be present in search results on all nodes after the import request completes. If a search query occurs concurrently with an import operation nodes may not be in sync yet. This means some nodes might already include the newly added or updated objects, while others don&#x27;t yet. In a healthy cluster, all nodes will have converged by the time the import request has been completed successfully. If a node is temporarily unavailable and rejoins a cluster it may temporarily be out of sync. It will then sync the missed changes from other replica nodes and eventually serve the same data again.</p>
</blockquote></div></div></details>
<h4 class="anchor anchorTargetStickyNavbar_Vzrq" id="q-with-your-aggregations-i-could-not-see-how-to-do-time-buckets-is-this-possible">Q: With your aggregations I could not see how to do time buckets, is this possible?<a href="#q-with-your-aggregations-i-could-not-see-how-to-do-time-buckets-is-this-possible" class="hash-link" aria-label="Direct link to Q: With your aggregations I could not see how to do time buckets, is this possible?" title="Direct link to Q: With your aggregations I could not see how to do time buckets, is this possible?" translate="no">​</a></h4>
<details class="details_lb9f alert alert--info details_b_Ee" data-collapsed="true"><summary>Answer</summary><div><div class="collapsibleContent_i85q"><blockquote>
<p>At the moment, we cannot aggregate over timeseries into time buckets yet, but architecturally there&#x27;s nothing in the way. If there is demand, this seems like a nice feature request, you can submit an <a href="https://github.com/weaviate/weaviate/issues" target="_blank" rel="noopener noreferrer" class="">issue here</a>. (We&#x27;re a very small company though and the priority is on Horizontal Scaling at the moment.)</p>
</blockquote></div></div></details>
