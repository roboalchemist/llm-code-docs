<h2 class="anchor anchorTargetStickyNavbar_Vzrq" id="introduction">Introduction<a href="#introduction" class="hash-link" aria-label="Direct link to Introduction" title="Direct link to Introduction" translate="no">​</a></h2>
<p>In a production environment, you might need to change your embedding model for several reasons. You may want to adopt a newer model for <strong>performance improvements</strong> like better search accuracy, or switch models due to the <strong>deprecation</strong> of your current model.</p>
<p>There are three basic steps when it comes to switching embedding models:</p>
<ol>
<li class="">
<p><strong>Baseline performance analysis</strong> Select a representative subset of the data. Using the existing embedding model, calculate a baseline for query/search accuracy. This metric will serve as the benchmark for comparison.</p>
</li>
<li class="">
<p><strong>New model evaluation</strong> Generate new vector embeddings for the identical data sample using the updated model. Re-calculate the query/search accuracy using these new embeddings.</p>
</li>
<li class="">
<p><strong>Decision &amp; deployment</strong> Compare the accuracy results from the new model against the established baseline. If the new embeddings demonstrate a clear improvement in performance, proceed with deploying the updated model system-wide.</p>
</li>
</ol>
<p>This tutorial demonstrates two approaches for switching vectorizers in your application:</p>
<ul>
<li class=""><a href="#method-a-collection-aliases-migration" class=""><strong>Method A: Collection aliases</strong></a>
This approach uses aliases to instantly switch between separate collections. It&#x27;s perfect for a complete model replacement, minimizing risk and providing an immediate rollback option.</li>
<li class=""><a href="#method-b-add-new-vector" class=""><strong>Method B: Adding new vectors</strong></a>
This method allows multiple vectors per data object within a single collection. It&#x27;s ideal for testing new models alongside existing ones.</li>
</ul>

<h2 class="anchor anchorTargetStickyNavbar_Vzrq" id="method-a-collection-aliases-migration">Method A: Collection aliases migration<a href="#method-a-collection-aliases-migration" class="hash-link" aria-label="Direct link to Method A: Collection aliases migration" title="Direct link to Method A: Collection aliases migration" translate="no">​</a></h2>
<p>Collection aliases allow instant switching between collections with different vectorizers. Instead of using a collection name in your application, you use the collection alias instead. One you create a new collection with a different embedding mode, you can update the alias to point to the new collection. Now, all of the queries using the alias will be performed with the new vector embeddings.</p>

<h3 class="anchor anchorTargetStickyNavbar_Vzrq" id="step-4-cleanup-and-important-considerations">Step 4: Cleanup and important considerations<a href="#step-4-cleanup-and-important-considerations" class="hash-link" aria-label="Direct link to Step 4: Cleanup and important considerations" title="Direct link to Step 4: Cleanup and important considerations" translate="no">​</a></h3>
<p>A critical point to remember is that you <strong>cannot delete a named vector</strong> from a collection once it has been added. Both the original and new vectors will coexist for each object.</p>
<p>This means your <strong>storage usage for this collection will permanently increase</strong>. Due to this limitation, adding new vectors best suited for temporary testing or when a permanent increase in storage is acceptable, rather than for production migrations where a clean state is needed.</p>
<hr>
<h2 class="anchor anchorTargetStickyNavbar_Vzrq" id="choosing-the-right-method">Choosing the right method<a href="#choosing-the-right-method" class="hash-link" aria-label="Direct link to Choosing the right method" title="Direct link to Choosing the right method" translate="no">​</a></h2>
<div class="theme-admonition theme-admonition-tip admonition_xJq3 alert alert--success"><div class="admonitionHeading_Gvgb"><span class="admonitionIcon_Rf37"><svg viewBox="0 0 12 16"><path fill-rule="evenodd" d="M6.5 0C3.48 0 1 2.19 1 5c0 .92.55 2.25 1 3 1.34 2.25 1.78 2.78 2 4v1h5v-1c.22-1.22.66-1.75 2-4 .45-.75 1-2.08 1-3 0-2.81-2.48-5-5.5-5zm3.64 7.48c-.25.44-.47.8-.67 1.11-.86 1.41-1.25 2.06-1.45 3.23-.02.05-.02.11-.02.17H5c0-.06 0-.13-.02-.17-.2-1.17-.59-1.83-1.45-3.23-.2-.31-.42-.67-.67-1.11C2.44 6.78 2 5.65 2 5c0-2.2 2.02-4 4.5-4 1.22 0 2.36.42 3.22 1.19C10.55 2.94 11 3.94 11 5c0 .66-.44 1.78-.86 2.48zM4 14h5c-.23 1.14-1.3 2-2.5 2s-2.27-.86-2.5-2z"></path></svg></span>Recommendation: Use collection aliases for production</div><div class="admonitionContent_BuS1"><p>While both methods achieve the goal of switching vectorizers, <strong>we strongly recommend using collection aliases for production migrations</strong>. This approach provides a reversible and efficient way to upgrade your embedding models without long-term side effects like increased storage.</p></div></div>
<h3 class="anchor anchorTargetStickyNavbar_Vzrq" id="method-a-collection-aliases-recommended">Method A: Collection aliases (Recommended)<a href="#method-a-collection-aliases-recommended" class="hash-link" aria-label="Direct link to Method A: Collection aliases (Recommended)" title="Direct link to Method A: Collection aliases (Recommended)" translate="no">​</a></h3>
<ul>
<li class=""><strong>Zero-downtime switch</strong>: Instantly redirect application traffic to the new collection.</li>
<li class=""><strong>Instant rollback</strong>: If issues arise, you can immediately switch the alias back to the old collection.</li>
<li class=""><strong>Clean architecture</strong>: Old and new models are completely isolated in separate collections.</li>
<li class=""><strong>No storage overhead</strong>: After migration, the old collection can be deleted, reclaiming all its storage space.</li>
</ul>
<h3 class="anchor anchorTargetStickyNavbar_Vzrq" id="method-b-adding-a-new-vector">Method B: Adding a new vector<a href="#method-b-adding-a-new-vector" class="hash-link" aria-label="Direct link to Method B: Adding a new vector" title="Direct link to Method B: Adding a new vector" translate="no">​</a></h3>
<ul>
<li class=""><strong>Good for experimentation</strong>: Useful for quickly comparing model performance within a single dataset.</li>
<li class=""><strong>Permanent storage increase</strong>: The old vector <strong>cannot be deleted</strong>, leading to a larger collection size and higher costs.</li>
<li class=""><strong>No simple rollback</strong>: Reverting requires application-level changes to specify the old vector name in every query.</li>
</ul>
<h2 class="anchor anchorTargetStickyNavbar_Vzrq" id="best-practices">Best practices<a href="#best-practices" class="hash-link" aria-label="Direct link to Best practices" title="Direct link to Best practices" translate="no">​</a></h2>
<ol>
<li class=""><strong>Test thoroughly</strong>: Always test the new vectorizer with a subset of data first.</li>
<li class=""><strong>Monitor quality</strong>: Compare search results between old and new models.</li>
<li class=""><strong>Plan for rollback</strong>: Keep the old setup available until you&#x27;re confident. The alias method makes it easy to rollback.</li>
<li class=""><strong>Consider costs</strong>: Different vectorizers have different pricing models. Remember that adding new vectors will increase your storage costs.</li>
<li class=""><strong>Document changes</strong>: Track which model versions and configurations you&#x27;re using.</li>
</ol>
<h2 class="anchor anchorTargetStickyNavbar_Vzrq" id="summary">Summary<a href="#summary" class="hash-link" aria-label="Direct link to Summary" title="Direct link to Summary" translate="no">​</a></h2>
<p>This tutorial demonstrated two powerful approaches for migrating vectorizers in Weaviate:</p>
<ul>
<li class=""><strong>Collection aliases</strong>: Best for instant, zero-downtime switching between collections with different vectorizers.</li>
<li class=""><strong>Adding new vectors</strong>: Creating new vector embeddings in an existing collection, best for experimentation.</li>
</ul>
<p>For production environments, the <strong>collection alias method is the preferred approach</strong> due to its safety, flexibility, and clean rollback capabilities.</p>
