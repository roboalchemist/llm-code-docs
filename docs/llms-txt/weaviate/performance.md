markdown"><header><h1>Index types and performance</h1></header><p>Weaviate uses different types of indexes to achieve performance goals.  This page focuses on <a href="https://arxiv.org/abs/1603.09320" target="_blank" rel="noopener noreferrer" class="">HNSW</a> and <a href="https://en.wikipedia.org/wiki/Inverted_index" target="_blank" rel="noopener noreferrer" class="">inverted indexes</a>.</p>
<p>These indexes are available:</p>
<p>Vector Indexes:</p>
<ul>
<li class=""><a class="" href="/weaviate/config-refs/indexing/vector-index#hnsw-index">HNSW</a></li>
<li class=""><a class="" href="/weaviate/config-refs/indexing/vector-index#flat-index">Flat</a></li>
</ul>
<p>Traditional Indexes:</p>
<ul>
<li class=""><a href="#inverted-index" class="">Inverted index</a></li>
</ul>
<h2 class="anchor anchorTargetStickyNavbar_Vzrq" id="inverted-index">Inverted index<a href="#inverted-index" class="hash-link" aria-label="Direct link to Inverted index" title="Direct link to Inverted index" translate="no">​</a></h2>
<p>The inverted index is essentially what powers all the <a class="" href="/weaviate/api/graphql/filters">GraphQL <code>where</code> filters</a>, where vectors or semantics are needed to find results. With inverted indexes, contents or data object properties such as words and numbers are mapped to its location in the database. This is the opposite of the more traditional forward index, which maps from documents to its content.</p>
<p>Inverted indexes are used often in document retrieval systems and search engines, because it allows fast full-text search and fast key-based search instead of brute-force. This fast data retrieval comes with the only cost of slight increase of processing time when a new data object is added, since the data object will indexed and stored in an inverted way, rather than only storing the index of the data object. In the database (Weaviate), there is a big lookup table which contains all the inverted indexes. If you want to retrieve objects with a specific property or content, then the database starts looking for only one row with this property which points to the relevant data objects (the row contains pointers to the data object IDs). This makes data object retrieval with these kind of queries very fast. Even if there are more than a billion entries, if you only care about the entries that contain the specific words or properties you&#x27;re looking for, only one row will be read with the document pointers.</p>
<p>The inverted index currently does not do any weighing (e.g. tf-idf) for sorting, since the vector index is used for these features like sorting. The inverted index is thus, at the moment, rather a binary operation: including or excluding data objects from the query result list, which results in an &#x27;allow list&#x27;.</p>
<h2 class="anchor anchorTargetStickyNavbar_Vzrq" id="vector-index">Vector index<a href="#vector-index" class="hash-link" aria-label="Direct link to Vector index" title="Direct link to Vector index" translate="no">​</a></h2>
<p>Everything that has a vector, thus every data object in Weaviate, is also indexed in the vector index. Weaviate currently supports <a href="https://arxiv.org/abs/1603.09320" target="_blank" rel="noopener noreferrer" class="">HNSW</a> and flat vector indexes.</p>
<p>See <a class="" href="/weaviate/concepts/vector-index">Concepts: vector index</a> for more information about the vector index.</p>
<h2 class="anchor anchorTargetStickyNavbar_Vzrq" id="costs-of-queries-and-operations">Costs of queries and operations<a href="#costs-of-queries-and-operations" class="hash-link" aria-label="Direct link to Costs of queries and operations" title="Direct link to Costs of queries and operations" translate="no">​</a></h2>
<p>This section discusses the cost of some common operations.</p>
<h3 class="anchor anchorTargetStickyNavbar_Vzrq" id="cost-of-data-import">Cost of data import<a href="#cost-of-data-import" class="hash-link" aria-label="Direct link to Cost of data import" title="Direct link to Cost of data import" translate="no">​</a></h3>
<p>At the moment, data import is relatively slow compared to the query times, because of the HNSW indexing. The cheapest data import operation is a simple &#x27;write&#x27; operation of a data object that was not seen before. It will get a completely new index. If you update a data object, the update itself is also really cheap, because in the backend an entirely new object will be created and indexed as if it was new. The &#x27;old&#x27; object will be cleaned up, which happens asynchronous and will thus add up to the operation time.</p>
<h3 class="anchor anchorTargetStickyNavbar_Vzrq" id="cost-of-queries">Cost of queries<a href="#cost-of-queries" class="hash-link" aria-label="Direct link to Cost of queries" title="Direct link to Cost of queries" translate="no">​</a></h3>
<p>Simple <code>Get</code> queries that only have a <code>where</code> filter are very cheap, because the <a href="#inverted-index" class="">inverted index</a> is used. A simple <code>Get</code> query that uses only the <code>explore</code> filter (vector search) is also very cheap, since the very efficient vector index HNSW is used. Sub-50ms 20NN-vector queries on datasets of over 1-100M objects are possible. Weaviate relies on a number of caches, but does not require keeping all vectors in memory. Thus it is also possible to run Weaviate on machines where the available memory is smaller than the size of all vectors.</p>
<p>Combining the <code>explore</code> (vector) and <code>where</code> filters in one search query (which is what makes Weaviate unique), is slightly more expensive. The inverted index is called first which returns all data items that match the <code>where</code> filter. This list is then passed on to the vector index search with HNSW. The cost of this combined operation depends on the dataset size and the amount of data returned by the inverted index search. The less items that are returned from the <code>where</code> filter search, the more items the vector search needs to skip, thus the longer it will take. These differences are however very small, perhaps not even noticeable.</p>
<h3 class="anchor anchorTargetStickyNavbar_Vzrq" id="cost-of-resolving-referencing">Cost of resolving referencing<a href="#cost-of-resolving-referencing" class="hash-link" aria-label="Direct link to Cost of resolving referencing" title="Direct link to Cost of resolving referencing" translate="no">​</a></h3>
<p>Weaviate is a database with a graph-like data model, not a pure graph database. Graph databases are built in a fashion where following links and references are very cheap, where querying links is cheaper than querying multiple items. Weaviate, on the other hand, is a <a href="https://weaviate.io/blog/what-is-a-vector-database" target="_blank" rel="noopener noreferrer" class="">vector database</a>. This means that one of the cheapest operations you can do with Weaviate is listing data. In a traditional graph database that is quite expensive. Weaviate does however have graph functionalities on top of the vector-search focus. So although its primary focus is on searching through data objects with the inverted index and/or vector index, we offer graph references between data objects. Searching, following and retrieving graph references between data objects is therefore less optimized than pure search. This means that using the graph-like features like resolving object references need more query time than pure data object search as described above.</p>
<p>Important to know is that the more connections between data objects and the deeper you try to query in a single query, the more costly the query operation gets. The best you can deal with these kind of queries is to not do <em>wide</em> and <em>deep</em> searches at the same time. If you have to resolve a lot of nested references, try to set a low limit (a low number of data objects to be returned). A second tip is to not try to resolve all the references. This could perhaps be split into separate queries, depending on your query. In practice, this could mean that you do a first search to retrieve the top 100 data objects of your query, and only get the deeper references of the top 5 results you&#x27;re actually interested in.</p>
<h3 class="anchor anchorTargetStickyNavbar_Vzrq" id="cost-of-filtering-by-reference">Cost of filtering by reference<a href="#cost-of-filtering-by-reference" class="hash-link" aria-label="Direct link to Cost of filtering by reference" title="Direct link to Cost of filtering by reference" translate="no">​</a></h3>
<p>If you have a nested reference filter, Weaviate starts by resolving the deepest reference and from there go upwards to the inner layers resolving other references. It thus finds the beacons of the deepest references first. This allows to use inverted index lookups for the other layers, which makes matching of results relatively cheap. However, queries that have nested references in the filter are still relatively costly because multiple search queries (for each nested layer) are performed and the results need to be combined into one result. The cost increases when a lot of results are returned on an inner layer, which needs to be searched through by the one layer deeper, and so on. Thus, this cost could in theory go up exponentially.</p>
<p>A tip is to avoid deeply nested filters in the queries. Additionally, try to make your queries as restrictive as possible, because a ten-level deep query would for example not be so expensive if all levels return only a single ID. In that case only ten one ID searches need to be performed, which is a lot of searches in one query, but each search is very cheap.</p>
