<h2 class="anchor anchorTargetStickyNavbar_Vzrq" id="migration-for-version-1190">Migration for version 1.19.0<a href="#migration-for-version-1190" class="hash-link" aria-label="Direct link to Migration for version 1.19.0" title="Direct link to Migration for version 1.19.0" translate="no">​</a></h2>
<p>This version introduces <code>indexFilterable</code> and <code>indexSearchable</code> variables for the new text indexes, whose values will be set based on the value of <code>indexInverted</code>.</p>
<p>Since filterable &amp; searchable are separate indexes, filterable does not exist in Weaviate instances upgraded from pre-<code>v1.19</code> to <code>v1.19</code>. The missing <code>filterable</code> index can be created though on startup for all <code>text/text[]</code> properties if env variable <code>INDEX_MISSING_TEXT_FILTERABLE_AT_STARTUP</code> is set.</p>

<h3 class="anchor anchorTargetStickyNavbar_Vzrq" id="first-multi-modal-module-clip-module-1756-1766">First Multi-modal module: CLIP Module (#1756, #1766)<a href="#first-multi-modal-module-clip-module-1756-1766" class="hash-link" aria-label="Direct link to First Multi-modal module: CLIP Module (#1756, #1766)" title="Direct link to First Multi-modal module: CLIP Module (#1756, #1766)" translate="no">​</a></h3>
<p>This release <a class="" href="/weaviate/model-providers/transformers/embeddings-multimodal">introduces the <code>multi2vec-clip</code> integration</a>, a module that allows for multi-modal vectorization within a single vector space. A class can have <code>image</code> or <code>text</code> fields or both. Similarly, the module provides both a <code>nearText</code> and a <code>nearImage</code> search and allows for various search combinations, such as text-search on image-only content and various other combinations.</p>

<h2 class="anchor anchorTargetStickyNavbar_Vzrq" id="migration-to-version-v180">Migration to version v1.8.0<a href="#migration-to-version-v180" class="hash-link" aria-label="Direct link to Migration to version v1.8.0" title="Direct link to Migration to version v1.8.0" translate="no">​</a></h2>
<h3 class="anchor anchorTargetStickyNavbar_Vzrq" id="migration-notice">Migration Notice<a href="#migration-notice" class="hash-link" aria-label="Direct link to Migration Notice" title="Direct link to Migration Notice" translate="no">​</a></h3>
<p>Version <code>v1.8.0</code> introduces multi-shard indexes and horizontal scaling. As a
result the dataset needs to be migrated. This migration is performed automatically -
without user interaction - when first starting up with Weaviate version
<code>v1.8.0</code>. However, it cannot be reversed. We, therefore, recommend carefully
reading the following migration notes and making a case-by-case decision about the
best upgrade path for your needs.</p>
<h4 class="anchor anchorTargetStickyNavbar_Vzrq" id="why-is-a-data-migration-necessary">Why is a data migration necessary?<a href="#why-is-a-data-migration-necessary" class="hash-link" aria-label="Direct link to Why is a data migration necessary?" title="Direct link to Why is a data migration necessary?" translate="no">​</a></h4>
<p>Prior to <code>v1.8.0</code> Weaviate did not support multi-shard indexes. The feature was
already planned, therefore data was already contained in a single shard with a
fixed name. A migration is necessary to move the data from a single fixed shard
into a multi-shard setup. The amount of shards is not changed. When you run
<code>v1.8.0</code> on a dataset the following steps happen automatically:</p>
<ul>
<li class="">Weaviate discovers the missing sharding configuration for your classes and fills it with the default values</li>
<li class="">When shards start-up and they do not exist on disk, but a shard with a fixed
name from <code>v1.7.x</code> exists, Weaviate automatically recognizes that a migration
is necessary and moves the data on disk</li>
<li class="">When Weaviate is up and running the data has been migrated.</li>
</ul>
<p><strong>Important Notice:</strong> As part of the migration Weaviate will assign the shard
to the (only) node available in the cluster. You need to make sure that this
node has a stable hostname. If you run on Kubernetes, hostnames are stable
(e.g. <code>weaviate-0</code> for the first node). However with <code>Docker Compose</code> hostnames
default to the id of the container. If you remove your containers (e.g.
<code>docker compose down</code>) and start them up again, the hostname will have changed.
This will lead to errors where Weaviate mentions that it cannot find the node
that the shard belongs to. The node sending the error message is the node that
owns the shard itself, but it cannot recognize it, since its own name has
changed.</p>
<p>To remedy this, you can set a stable hostname <strong>before starting up with
v1.8.0</strong> by setting the env var <code>CLUSTER_HOSTNAME=node1</code>. The actual name does
not matter, as long as it&#x27;s stable.</p>
<p>If you forgot to set a stable hostname and are now running into the error
mentioned above, you can still explicitly set the hostname that was used before
which you can derive from the error message.</p>
<p>Example:</p>
<p>If you see the error message <code>&quot;shard Knuw6a360eCY: resolve node name \&quot;5b6030dbf9ea\&quot; to host&quot;</code>, you can make Weaviate usable again, by setting
<code>5b6030dbf9ea</code> as the host name: <code>CLUSTER_HOSTNAME=5b6030dbf9ea</code>.</p>
<h4 class="anchor anchorTargetStickyNavbar_Vzrq" id="should-you-upgrade-or-reimport">Should you upgrade or reimport?<a href="#should-you-upgrade-or-reimport" class="hash-link" aria-label="Direct link to Should you upgrade or reimport?" title="Direct link to Should you upgrade or reimport?" translate="no">​</a></h4>
<p>In addition to new features, <code>v1.8.0</code> also contains a large collection of
bugfixes. Some of the bugs affect how Weaviate writes the HNSW index to disk.
A pre-1.18.0 index on disk may not be as good a freshly built <code>v1.8.0</code> index.
If you can import using a script, we generally recommend starting with a fresh
<code>v1.8.0</code> instance and reimporting instead of migrating.</p>
<h4 class="anchor anchorTargetStickyNavbar_Vzrq" id="is-downgrading-possible-after-upgrading">Is downgrading possible after upgrading?<a href="#is-downgrading-possible-after-upgrading" class="hash-link" aria-label="Direct link to Is downgrading possible after upgrading?" title="Direct link to Is downgrading possible after upgrading?" translate="no">​</a></h4>
<p>Note that the data migration which happens at the first startup of v1.8.0 is
not automatically reversible. If you plan on downgrading to <code>v1.7.x</code> again
after upgrading, you must explicitly create a backup of the state prior to
upgrading.</p>

<ul>
<li class=""><em>LSM-Tree based Storage</em>. Previous releases of Weaviate used a B+Tree based storage mechanism. This was not fast enough to keep up with the high write speed requirements of a large-scale import. This release completely rewrites the storage layer of Weaviate to use a custom LSM-tree approach. Weaviate will guess the property type based on the first time it sees a property. The defaults can be configured using the environment variables outlined in #1539. <h4 class="anchor anchorTargetStickyNavbar_Vzrq" id="consistent-casing-in-graphql-properties">Consistent casing in GraphQL properties<a href="#consistent-casing-in-graphql-properties" class="hash-link" aria-label="Direct link to Consistent casing in GraphQL properties" title="Direct link to Consistent casing in GraphQL properties" translate="no">​</a></h4>
<p>Previously, reference properties in the schema definitions are always lowercase, yet in graphQL they needed to be uppercased. E.g.: <code>Article { OfAuthor { … on Author { name } } } }</code>, even though the property is defined as ofAuthor. New is that the casing in GraphQL reflects exactly the casing in the schema definition, thus the above example would become: <code>Article { ofAuthor { … on Author { name } } } }</code></p>

<h4 class="anchor anchorTargetStickyNavbar_Vzrq" id="default-vectorizer-module">Default vectorizer module<a href="#default-vectorizer-module" class="hash-link" aria-label="Direct link to Default vectorizer module" title="Direct link to Default vectorizer module" translate="no">​</a></h4>
<p>The default vectorizer module can be specified in a new environment variable so that this doesn&#x27;t have to be specified on every data class in the schema. The environment variable is <code>DEFAULT_VECTORIZER_MODULE</code>, which can be set to for example <code>DEFAULT_VECTORIZER_MODULE=&quot;text2vec-contextionary&quot;</code>.</p>
