<h2 class="anchor anchorTargetStickyNavbar_Vzrq" id="configuration">Configuration<a href="#configuration" class="hash-link" aria-label="Direct link to Configuration" title="Direct link to Configuration" translate="no">​</a></h2>
<p>Weaviate supports four backup storage options:</p>
<table><thead><tr><th>Provider</th><th>Module Name</th><th>Best For</th><th>Multi-Node Support</th></tr></thead><tbody><tr><td>AWS S3</td><td><code>backup-s3</code></td><td>Production deployments, AWS environments</td><td>Yes</td></tr><tr><td>Google Cloud Storage</td><td><code>backup-gcs</code></td><td>Production deployments, GCP environments</td><td>Yes</td></tr><tr><td>Azure Storage</td><td><code>backup-azure</code></td><td>Production deployments, Azure environments</td><td>Yes</td></tr><tr><td>Local Filesystem</td><td><code>backup-filesystem</code></td><td>Development, testing, single-node setups</td><td>No</td></tr></tbody></table>
<p>To use any provider:</p>
<ol>
<li class="">Enable the module<!-- -->
<ul>
<li class="">Add the module name to the <code>ENABLE_MODULES</code> environment variable</li>
<li class="">On Weaviate Cloud instances, a relevant default module is enabled</li>
</ul>
</li>
<li class="">Configure the required modules<!-- -->
<ul>
<li class="">Option 1: Set the necessary environment variables</li>
<li class="">Option 2 (Kubernetes): Configure the <a href="#kubernetes-configuration" class="">Helm chart values</a></li>
</ul>
</li>
</ol>
<p>Note multiple providers can be enabled simultaneously</p>

<h4 class="anchor anchorTargetStickyNavbar_Vzrq" id="google-application-default-credentials">Google Application Default Credentials<a href="#google-application-default-credentials" class="hash-link" aria-label="Direct link to Google Application Default Credentials" title="Direct link to Google Application Default Credentials" translate="no">​</a></h4>
<p>The <code>backup-gcs</code> module follows the Google <a href="https://cloud.google.com/docs/authentication/application-default-credentials" target="_blank" rel="noopener noreferrer" class="">Application Default Credentials</a> best-practices. This means that credentials can be discovered through the environment, through a local Google Cloud CLI setup, or through an attached service account.</p>
<p>This makes it easy to use the same module in different setups. For example, you can use the environment-based approach in production, and the CLI-based approach on your local machine. This way you can easily pull a backup that was created in a remote environment to your local system. This can be helpful in debugging an issue, for example.</p>

<h4 class="anchor anchorTargetStickyNavbar_Vzrq" id="azure-credentials">Azure Credentials<a href="#azure-credentials" class="hash-link" aria-label="Direct link to Azure Credentials" title="Direct link to Azure Credentials" translate="no">​</a></h4>
<p>There are two different ways to authenticate against Azure with <code>backup-azure</code>. You can use either:</p>
<ol>
<li class="">An Azure Storage connection string, or</li>
<li class="">An Azure Storage account name and key.</li>
</ol>
<p>Both options can be implemented using environment variables as follows:</p>
<table><thead><tr><th>Environment variable</th><th>Required</th><th>Description</th></tr></thead><tbody><tr><td><code>AZURE_STORAGE_CONNECTION_STRING</code></td><td>yes (*see note)</td><td>A string that includes the authorization information required (<a href="https://learn.microsoft.com/en-us/azure/storage/common/storage-configure-connection-string" target="_blank" rel="noopener noreferrer" class="">Azure documentation</a>). <br><br> This variable is checked and used first before <code>AZURE_STORAGE_ACCOUNT</code>.</td></tr><tr><td><code>AZURE_STORAGE_ACCOUNT</code></td><td>yes (*see note)</td><td>The name of your Azure Storage account.</td></tr><tr><td><code>AZURE_STORAGE_KEY</code></td><td>no</td><td>An access key for your Azure Storage account. <br><br>For anonymous access, specify <code>&quot;&quot;</code>.</td></tr></tbody></table>

<h3 class="anchor anchorTargetStickyNavbar_Vzrq" id="create-backup">Create Backup<a href="#create-backup" class="hash-link" aria-label="Direct link to Create Backup" title="Direct link to Create Backup" translate="no">​</a></h3>

<h4 class="anchor anchorTargetStickyNavbar_Vzrq" id="asynchronous-status-checking">Asynchronous Status Checking<a href="#asynchronous-status-checking" class="hash-link" aria-label="Direct link to Asynchronous Status Checking" title="Direct link to Asynchronous Status Checking" translate="no">​</a></h4>
<p>All client implementations have a &quot;wait for completion&quot; option which will poll the backup status in the background and only return once the backup has completed (successfully or unsuccessfully).</p>
<p>If you set the &quot;wait for completion&quot; option to false, you can also check the status yourself using the Backup Creation Status API.</p>

<h3 class="anchor anchorTargetStickyNavbar_Vzrq" id="restore-backup">Restore Backup<a href="#restore-backup" class="hash-link" aria-label="Direct link to Restore Backup" title="Direct link to Restore Backup" translate="no">​</a></h3>
<p>You can restore any backup to any machine as long as the name and number of nodes between source and target are identical. The backup does not need to be created on the same instance. Once a backup backend is configured, you can restore a backup with a single request.</p>
<p>As with backup creation, the <code>include</code> and <code>exclude</code> options are mutually exclusive. You can set none or exactly one of those. In a restore operation, <code>include</code> and <code>exclude</code> are relative to the collections contained in the backup. The restore process is not aware of collections that existed on the source machine if they were not part of the backup.</p>
<p>Note that a restore fails if any of the collections already exist on this instance.</p>

<h4 class="anchor anchorTargetStickyNavbar_Vzrq" id="asynchronous-status-checking-1">Asynchronous Status Checking<a href="#asynchronous-status-checking-1" class="hash-link" aria-label="Direct link to Asynchronous Status Checking" title="Direct link to Asynchronous Status Checking" translate="no">​</a></h4>
<p>All client implementations have a &quot;wait for completion&quot; option which will poll the restore status in the background and only return once the restore has completed (successfully or unsuccessfully).</p>

<h2 class="anchor anchorTargetStickyNavbar_Vzrq" id="technical-considerations">Technical Considerations<a href="#technical-considerations" class="hash-link" aria-label="Direct link to Technical Considerations" title="Direct link to Technical Considerations" translate="no">​</a></h2>
<h3 class="anchor anchorTargetStickyNavbar_Vzrq" id="read--write-requests-while-a-backup-is-running">Read &amp; Write requests while a backup is running<a href="#read--write-requests-while-a-backup-is-running" class="hash-link" aria-label="Direct link to Read &amp; Write requests while a backup is running" title="Direct link to Read &amp; Write requests while a backup is running" translate="no">​</a></h3>
<p>The backup process is designed to be minimally invasive to a running setup. Even on very large setups, where terabytes of data need to be copied, Weaviate stays available during backup. It even accepts write requests while a backup process is running. This sections explains how backups work under the hood and why Weaviate can safely accept writes while a backup is copied.</p>
<p>Weaviate uses a custom <a class="" href="/weaviate/concepts/storage#object-and-inverted-index-store">LSM Store</a> for its object store and inverted index. LSM stores are a hybrid of immutable disk segments and an in-memory structure called a memtable that accepts all writes (including updates and deletes). Most of the time, files on disk are immutable, there are only three situations where files are changed:</p>
<ol>
<li class="">Anytime a memtable is flushed. This creates a new segment. Existing segments are not changed.</li>
<li class="">Any write into the memtable is also written into a Write-Ahead-Log (WAL). The WAL is only needed for disaster-recovery. Once a segment has been orderly flushed, the WAL can be discarded.</li>
<li class="">There is an async background process called Compaction that optimizes existing segments. It can merge two small segments into a single larger segment and remove redundant data as part of the process.</li>
</ol>
<p>Weaviate&#x27;s Backup implementation makes use of the above properties in the following ways:</p>
<ol>
<li class="">Weaviate first flushes all active memtables to disk. This process takes in the 10s or 100s of milliseconds. Any pending write requests simply waits for a new memtable to be created without any failing requests or substantial delays.</li>
<li class="">Now that the memtables are flushed, there is a guarantee: All data that should be part of the backup is present in the existing disk segments. Any data that will be imported after the backup request ends up in new disk segments. The backup references a list of immutable files.</li>
<li class="">To prevent a compaction process from changing the files on disk while they are being copied, compactions are temporarily paused until all files have been copied. They are automatically resumed right after.</li>
</ol>
<p>This way the backup process can guarantee that the files that are transferred to the remote backend are immutable (and thus safe to copy) even with new writes coming in. Even if it takes minutes or hours to backup a very large setup, Weaviate stays available without any user impact while the backup process is running.</p>
<p>It is not just safe - but even recommended - to create backups on live production instances while they are serving user requests.</p>
<h3 class="anchor anchorTargetStickyNavbar_Vzrq" id="async-nature-of-the-backup-api">Async nature of the Backup API<a href="#async-nature-of-the-backup-api" class="hash-link" aria-label="Direct link to Async nature of the Backup API" title="Direct link to Async nature of the Backup API" translate="no">​</a></h3>
<p>The backup API is built in a way that no long-running network requests are required. The request to create a new backup returns immediately. It does some basic validation, then returns to the user. The backup is now in status <code>STARTED</code>. To get the status of a running backup you can poll the <a href="#asynchronous-status-checking" class="">status endpoint</a>. This makes the backup itself resilient to network or client failures.</p>
<p>If you would like your application to wait for the background backup process to complete, you can use the &quot;wait for completion&quot; feature that is present in all language clients. The clients will poll the status endpoint in the background and block until the status is either <code>SUCCESS</code> or <code>FAILED</code>. This makes it easy to write simple synchronous backup scripts, even with the async nature of the API.</p>
<h2 class="anchor anchorTargetStickyNavbar_Vzrq" id="other-use-cases">Other Use cases<a href="#other-use-cases" class="hash-link" aria-label="Direct link to Other Use cases" title="Direct link to Other Use cases" translate="no">​</a></h2>
<h3 class="anchor anchorTargetStickyNavbar_Vzrq" id="migrating-to-another-environment">Migrating to another environment<a href="#migrating-to-another-environment" class="hash-link" aria-label="Direct link to Migrating to another environment" title="Direct link to Migrating to another environment" translate="no">​</a></h3>
<p>The flexibility around backup providers opens up new use cases. Besides using the backup &amp; restore feature for disaster recovery, you can also use it for duplicating environments or migrating between clusters.</p>
<p>For example, consider the following situation: You would like to do a load test on production data. If you would do the load test in production it might affect users. An easy way to get meaningful results without affecting uses it to duplicate your entire environment. Once the new production-like &quot;loadtest&quot; environment is up, create a backup from your production environment and restore it into your &quot;loadtest&quot; environment. This even works if the production environment is running on a completely different cloud provider than the new environment.</p>
<h2 class="anchor anchorTargetStickyNavbar_Vzrq" id="troubleshooting-and-notes">Troubleshooting and notes<a href="#troubleshooting-and-notes" class="hash-link" aria-label="Direct link to Troubleshooting and notes" title="Direct link to Troubleshooting and notes" translate="no">​</a></h2>
<ul>
<li class="">Single node backup is available starting in Weaviate <code>v1.15</code>. Multi-node backups is available starting in <code>v1.16</code>.</li>
<li class="">In some cases, backups can take a long time, or get &quot;stuck&quot;, causing Weaviate to be unresponsive. If this happens, you can <a href="#cancel-backup" class="">cancel the backup</a> and try again.</li>
<li class="">If a backup module is misconfigured, such as having an invalid backup path, it can cause Weaviate to not start. Review the system logs for any errors.</li>
<li class="">RBAC roles and users are not restored by default. You need to enable them manually through the configuration properties when <a href="#restore-backup" class="">restoring a backup</a>.</li>
</ul>
