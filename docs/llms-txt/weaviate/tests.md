<h3 class="anchor anchorTargetStickyNavbar_Vzrq" id="test-pyramid">Test Pyramid<a href="#test-pyramid" class="hash-link" aria-label="Direct link to Test Pyramid" title="Direct link to Test Pyramid" translate="no">​</a></h3>
<p>Weaviate Database follows a typical <a href="https://martinfowler.com/articles/practical-test-pyramid.html" target="_blank" rel="noopener noreferrer" class="">Test Pyramid</a> approach. As Weaviate itself contains no graphical user interface (GUI), the highest level tests test the user journeys at an API level.</p>
<p>The tests are grouped into the following three levels:</p>
<h4 class="anchor anchorTargetStickyNavbar_Vzrq" id="unit-tests">Unit tests<a href="#unit-tests" class="hash-link" aria-label="Direct link to Unit tests" title="Direct link to Unit tests" translate="no">​</a></h4>
<p>Unit tests test the smallest possible unit, mostly one &quot;class&quot; (usually a <code>struct</code> in golang) with its methods. Unit tests are designed to validate the business logic and not the internals.</p>
<p>Unit tests are stateless and do not depend on any external programs or runtime other than the Golang-built tools. (Note: We do make use of the <a href="https://github.com/stretchr/testify" target="_blank" rel="noopener noreferrer" class="">stretchr/testify</a> packages. However, they are installed with any other code-level dependency and don&#x27;t require running dedicated software).</p>
<p>This makes tests fast to execute, easy to adapt and easy to run with third-party tools like code watchers.</p>
<h4 class="anchor anchorTargetStickyNavbar_Vzrq" id="integration-tests">Integration tests<a href="#integration-tests" class="hash-link" aria-label="Direct link to Integration tests" title="Direct link to Integration tests" translate="no">​</a></h4>
<p>Integration tests test anything that crosses a boundary. A boundary could be the dependence on an external party (e.g. a third-party database) or an independent custom tool, such as the contextionary.</p>
<p>With the standalone feature we also make use of integration tests to test disk access. As outlined above, unit tests are meant to be stateless. We consider accessing the filesystem from the code as a boundary in the sense that they should be an integration test.</p>
<p>Integration tests may require third-party dependencies which can be spun up using docker. Convenience scripts are provided, see below.</p>
<p>Slow integration tests (run-time of over 10s) receive a different build tag, so those slow tests can be skipped during local development if they are not required. See the section on how to run tests for details.</p>
<h4 class="anchor anchorTargetStickyNavbar_Vzrq" id="journey-tests">Journey tests<a href="#journey-tests" class="hash-link" aria-label="Direct link to Journey tests" title="Direct link to Journey tests" translate="no">​</a></h4>
<p>The highest level tests are journey tests. As the top of the pyramid, those tests come with the highest execution cost. There should thus be a few. At the same time they provide a lot of value as they make sure all the components play together. Journey tests don&#x27;t usually care about edge cases, but rather about validating that a user journey is possible.</p>
<p>Journey-tests are black-box tests. This means the test code is completely unaware of any of the internals of Weaviate. Compare this to the unit or integration tests which tests snippets of (Golang) code. The journey tests test an application. The only way for the journey test to interact with the application is through means that are also available to users, such as the public APIs. Our Journey tests are written in Golang to keep the context-switching to a minimum for developers, but the fact that they only test APIs and not code means that they could technically be written in any language.</p>
<p>This makes sure that the UX for our users is great and the most important features are always working as intended. As a downside they come with the highest execution cost because journey tests need to compile and run the application before being able to run the tests themselves. In addition any runtime <a href="https://12factor.net/backing-services" target="_blank" rel="noopener noreferrer" class="">backing service</a> must also be present in a test scenario. To make this easy for developers, we provide convenience scripts which build both the application and all backing services and runs them in <code>Docker Compose</code>.</p>
<p>Backing services are always ephemeral and will be created solely for the tests. Weaviate will never require a test runtime that it does not create itself. This makes clean up easy and our tests very portable. New contributors should be able to run the entire test pipeline locally within seconds after first cloning the repository.</p>
<h3 class="anchor anchorTargetStickyNavbar_Vzrq" id="benchmark-tests">Benchmark tests<a href="#benchmark-tests" class="hash-link" aria-label="Direct link to Benchmark tests" title="Direct link to Benchmark tests" translate="no">​</a></h3>
<p>These tests have two functions:</p>
<ol>
<li class="">Identify regressions automatically before they are merged.</li>
<li class="">Enable performance tracking over time.</li>
</ol>
<p>They are currently very limited but will be extended over time.</p>
<h3 class="anchor anchorTargetStickyNavbar_Vzrq" id="test-coverage">Test coverage<a href="#test-coverage" class="hash-link" aria-label="Direct link to Test coverage" title="Direct link to Test coverage" translate="no">​</a></h3>
<p>We aim to have the highest useful test coverage possible. In some cases this might mean 100% test coverages, in other scenarios this might be considerably less. Golang is very explicit about it&#x27;s error handling. Especially as errors are wrapped (or &quot;annotated&quot;) you will find a lot of <code>if err != nil { ... }</code> statements. Each of those if statements is a code branch that - if left untested - will reduce the overall coverage. Whether each error case should have an explicit test case is something that you should decide based upon how much value such a test adds. Not necessarily on coverage numbers alone.</p>
<p>Nevertheless, you should aim to always make sure that your contribution does not lower the overall test coverage. We have <code>codecov</code> installed in our CI pipeline to prevent you from accidentally contributing something that would lower the coverage.</p>
<h3 class="anchor anchorTargetStickyNavbar_Vzrq" id="cross-repository-dependencies">Cross-repository dependencies<a href="#cross-repository-dependencies" class="hash-link" aria-label="Direct link to Cross-repository dependencies" title="Direct link to Cross-repository dependencies" translate="no">​</a></h3>
<p>There are various ways to interact with the Weaviate API. You can send HTTP requests directly or you could use a client, such as the <a class="" href="/weaviate/client-libraries/python">python client</a> to interact with Weaviate. In the Weaviate Database repository we have chosen not to use any of our own clients. This has the goal to minimize dependencies and allow independent development by different teams.</p>
<p>As a result all journey tests in Weaviate Database either use the go client (which is auto-generated from swagger) or plain HTTP.</p>
<h2 class="anchor anchorTargetStickyNavbar_Vzrq" id="how-to-run-tests">How to run tests<a href="#how-to-run-tests" class="hash-link" aria-label="Direct link to How to run tests" title="Direct link to How to run tests" translate="no">​</a></h2>
<h3 class="anchor anchorTargetStickyNavbar_Vzrq" id="run-the-whole-pipeline">Run the whole pipeline<a href="#run-the-whole-pipeline" class="hash-link" aria-label="Direct link to Run the whole pipeline" title="Direct link to Run the whole pipeline" translate="no">​</a></h3>
<p>There is a convenience script available which runs the entire test pipeline in the same fashion that it is run on CI. It only requires a correctly set up Golang environment, as well as <code>Docker Compose</code> to be set up on your machine.</p>
<p>You can run the entire pipeline, except the benchmark tests, using:</p>
<div class="language-sh codeBlockContainer_Ckt0 theme-code-block" style="--prism-color:#393A34;--prism-background-color:#f6f8fa"><div class="codeBlockContent_QJqH"><pre tabindex="0" class="prism-code language-sh codeBlock_bY9V thin-scrollbar" style="color:#393A34;background-color:#f6f8fa"><code class="codeBlockLines_e6Vv"><span class="token-line" style="color:#393A34"><span class="token plain">test/run.sh</span><br></span></code></pre></div></div>
<p>This script will run tests exactly the same way as on CI, i.e. all levels, including &quot;slow&quot; tests. If this script passes locally - and there are no flaky tests - the test section on CI will pass as well.</p>
<h3 class="anchor anchorTargetStickyNavbar_Vzrq" id="unit-tests-1">Unit tests<a href="#unit-tests-1" class="hash-link" aria-label="Direct link to Unit tests" title="Direct link to Unit tests" translate="no">​</a></h3>
<p>As outlined in the Philosophy, unit tests have no dependencies other than the vendor code dependencies. You can thus run them with pure <code>go test</code> commands. For example to run all unit tests, simply run:</p>
<div class="language-sh codeBlockContainer_Ckt0 theme-code-block" style="--prism-color:#393A34;--prism-background-color:#f6f8fa"><div class="codeBlockContent_QJqH"><pre tabindex="0" class="prism-code language-sh codeBlock_bY9V thin-scrollbar" style="color:#393A34;background-color:#f6f8fa"><code class="codeBlockLines_e6Vv"><span class="token-line" style="color:#393A34"><span class="token plain">go test ./...</span><br></span></code></pre></div></div>
<h4 class="anchor anchorTargetStickyNavbar_Vzrq" id="adding-new-unit-tests">Adding new unit tests<a href="#adding-new-unit-tests" class="hash-link" aria-label="Direct link to Adding new unit tests" title="Direct link to Adding new unit tests" translate="no">​</a></h4>
<ul>
<li class="">Add unit tests in the same folder as the code they are testing</li>
<li class="">Aim to write &quot;black box&quot; unit tests that test the public (&quot;exported&quot;) methods of the class under test without knowing too much about the internals.</li>
<li class="">Do not use any build tags.</li>
</ul>
<h3 class="anchor anchorTargetStickyNavbar_Vzrq" id="integration-tests-1">Integration tests<a href="#integration-tests-1" class="hash-link" aria-label="Direct link to Integration tests" title="Direct link to Integration tests" translate="no">​</a></h3>
<p>As outline in the Philosophy, integration tests require backing services to be run. We have a convenience script available which starts all required services in <code>Docker Compose</code> and runs the tests against it:</p>
<div class="language-sh codeBlockContainer_Ckt0 theme-code-block" style="--prism-color:#393A34;--prism-background-color:#f6f8fa"><div class="codeBlockContent_QJqH"><pre tabindex="0" class="prism-code language-sh codeBlock_bY9V thin-scrollbar" style="color:#393A34;background-color:#f6f8fa"><code class="codeBlockLines_e6Vv"><span class="token-line" style="color:#393A34"><span class="token plain">test/integration/run.sh</span><br></span></code></pre></div></div>
<h4 class="anchor anchorTargetStickyNavbar_Vzrq" id="adding-new-integration-tests">Adding new integration tests<a href="#adding-new-integration-tests" class="hash-link" aria-label="Direct link to Adding new integration tests" title="Direct link to Adding new integration tests" translate="no">​</a></h4>
<ul>
<li class="">Use the <code>integrationTest</code> build tag on your test, so it is ignored during unit test runs.</li>
<li class="">Make sure the test prepares for and cleans up after itself, so tests can be run in succession.</li>
<li class="">If your test requires a lot of time to execute, consider marking it as a slow test and making it optional. (see below).</li>
</ul>
<h4 class="anchor anchorTargetStickyNavbar_Vzrq" id="slow-integration-tests">Slow integration tests<a href="#slow-integration-tests" class="hash-link" aria-label="Direct link to Slow integration tests" title="Direct link to Slow integration tests" translate="no">​</a></h4>
<p>With the introduction of Standalone mode there is some behavior that needs to be tested at scale. For example, the HNSW index might work fine on a fictional test set where all entries are on layer 0, but then break once several layers need to be traversed.</p>
<p>Similarly tests which test recall (in an HNSW index) require a dataset size, where the number of nodes is considerably larger than the <code>ef</code> parameter at search. Otherwise the search is a full-dataset scan which will always have 100% recall.</p>
<p>These tests are considered &quot;slow tests&quot;. They are an important part in our test pipeline. The slow tests are important for release testing, however, individual tests might not be important for feature testing. Therefore these tests. Therefore these tests are opt-in with the <code>--include-slow</code> flag on the test runner:</p>
<div class="language-sh codeBlockContainer_Ckt0 theme-code-block" style="--prism-color:#393A34;--prism-background-color:#f6f8fa"><div class="codeBlockContent_QJqH"><pre tabindex="0" class="prism-code language-sh codeBlock_bY9V thin-scrollbar" style="color:#393A34;background-color:#f6f8fa"><code class="codeBlockLines_e6Vv"><span class="token-line" style="color:#393A34"><span class="token plain">test/integration/run.sh --include-slow</span><br></span></code></pre></div></div>
<p>Note that while slow tests are optional on the integration test runner script, the overall test script (<code>test/run.sh</code>) does set this option. Therefore any optional test becomes a required test on CI - or when running the CI-like script locally.</p>
<p>To mark an integration test as &quot;slow&quot; simply use the <code>integrationTestSlow</code> build tag, instead of the <code>integrationTest</code> tag.</p>
<h3 class="anchor anchorTargetStickyNavbar_Vzrq" id="journey-tests-1">Journey tests<a href="#journey-tests-1" class="hash-link" aria-label="Direct link to Journey tests" title="Direct link to Journey tests" translate="no">​</a></h3>
<p>As outline in the Philosophy, journey tests require the application to be compiled as well as all backing services to be running. We have a convenience script available which starts all required services in <code>Docker Compose</code> and runs the tests against it.</p>
<p>The script is part of the overall pipeline script, but you can configure it to run only the journey tests like so:</p>
<div class="language-text codeBlockContainer_Ckt0 theme-code-block" style="--prism-color:#393A34;--prism-background-color:#f6f8fa"><div class="codeBlockContent_QJqH"><pre tabindex="0" class="prism-code language-text codeBlock_bY9V thin-scrollbar" style="color:#393A34;background-color:#f6f8fa"><code class="codeBlockLines_e6Vv"><span class="token-line" style="color:#393A34"><span class="token plain">test/run.sh --acceptance-only</span><br></span></code></pre></div></div>
<h4 class="anchor anchorTargetStickyNavbar_Vzrq" id="add-a-new-journey-test">Add a new journey test<a href="#add-a-new-journey-test" class="hash-link" aria-label="Direct link to Add a new journey test" title="Direct link to Add a new journey test" translate="no">​</a></h4>
<p>Journey tests don&#x27;t use any specific build tags, however, they are all isolated in a specific folder. This folder is ignored during integration or unit test runs.</p>
<p>To add a new test, pick the most appropriate sub-folder (or add a new one) in <code>test/acceptance</code>.</p>

<h2 class="anchor anchorTargetStickyNavbar_Vzrq" id="tools-and-frameworks">Tools and Frameworks<a href="#tools-and-frameworks" class="hash-link" aria-label="Direct link to Tools and Frameworks" title="Direct link to Tools and Frameworks" translate="no">​</a></h2>
<p>We use the default Golang testing structure, to organize tests. This means a test block is wrapped in a <code>func TestMyUnit(t *testing.T) {}</code> block. Inside this the <code>t.Run(&quot;description&quot;, func(t *testing.T) {})</code> blocks are used to add more structure to the test.</p>
<p>Prefer the use of <a href="https://github.com/stretchr/testify" target="_blank" rel="noopener noreferrer" class="">stretchr/testify</a> to make assertions. We consider the readability of testify assertions higher than those of raw if statements if no assertion library was used.</p>
<p>If there are cases which cannot be solved using <code>testify</code>, write a manual assertion.</p>
<h3 class="anchor anchorTargetStickyNavbar_Vzrq" id="catastrophic-failure-of-tests">Catastrophic Failure of tests<a href="#catastrophic-failure-of-tests" class="hash-link" aria-label="Direct link to Catastrophic Failure of tests" title="Direct link to Catastrophic Failure of tests" translate="no">​</a></h3>
<p>Use the <code>assert</code> package if a failure of this tests is not catastrophic and use the <code>require</code> package if a test should not execute beyond a failure.</p>
<p>A typical scenario for this is checking for an error when we know that the other return value would be nil otherwise. For example:</p>
<div class="language-text codeBlockContainer_Ckt0 theme-code-block" style="--prism-color:#393A34;--prism-background-color:#f6f8fa"><div class="codeBlockContent_QJqH"><pre tabindex="0" class="prism-code language-text codeBlock_bY9V thin-scrollbar" style="color:#393A34;background-color:#f6f8fa"><code class="codeBlockLines_e6Vv"><span class="token-line" style="color:#393A34"><span class="token plain">res, err := DoSomethingAwesome()</span><br></span><span class="token-line" style="color:#393A34"><span class="token plain">require.Nil(t, err)</span><br></span><span class="token-line" style="color:#393A34"><span class="token plain">assert.Equal(t, &quot;foo&quot;, res.Name)</span><br></span></code></pre></div></div>
<p>If we didn&#x27;t use <code>require</code> on the error, the test would continue executing. Therefore the last line would panic as <code>res.Name</code> would try to access a property of a nil-object.</p>
<p>By using <code>require.Nil</code> we can abort this test early, if an unexpected error was returned.</p>
