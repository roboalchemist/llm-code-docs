markdown"><header><h1>Data structure</h1></header><h2 class="anchor anchorTargetStickyNavbar_Vzrq" id="data-object-concepts">Data object concepts<a href="#data-object-concepts" class="hash-link" aria-label="Direct link to Data object concepts" title="Direct link to Data object concepts" translate="no">​</a></h2>
<p>Each data object in Weaviate belongs to a <code>collection</code> and has one or more <code>properties</code>.</p>
<p>Weaviate stores <code>data objects</code> in class-based collections. Data objects are represented as JSON-documents. Objects normally include a <code>vector</code> that is derived from a machine learning model. The vector is also called an <code>embedding</code> or a <code>vector embedding</code>.</p>
<p>Each collection contains objects of the same <code>class</code>. The objects are defined by a common <code>schema</code>.</p>

<h3 class="anchor anchorTargetStickyNavbar_Vzrq" id="uuids">UUIDs<a href="#uuids" class="hash-link" aria-label="Direct link to UUIDs" title="Direct link to UUIDs" translate="no">​</a></h3>
<p>Every object stored in Weaviate has a <a href="https://en.wikipedia.org/wiki/Universally_unique_identifier" target="_blank" rel="noopener noreferrer" class="">UUID</a>. The UUID guarantees uniqueness across all collections.</p>
<p>You can <a class="" href="/weaviate/manage-objects/import#specify-an-id-value">use a deterministic UUID</a> to ensure that the same object always has the same UUID. This is useful when you want to update an object without changing its UUID.</p>
<p>If you don&#x27;t specify an ID, Weaviate generates a random UUID for you.</p>
<p>In requests without any other ordering specified, Weaviate processes them in ascending UUID order. This means that requests to <a class="" href="/weaviate/search/basics#list-objects">list objects</a>, use of the <a class="" href="/weaviate/manage-objects/read-all-objects">cursor API</a>, or requests to <a class="" href="/weaviate/manage-objects/delete#delete-multiple-objects-by-id">delete objects</a>, without any other ordering specified, will be processed in ascending UUID order.</p>
<h3 class="anchor anchorTargetStickyNavbar_Vzrq" id="cross-references">Cross-references<a href="#cross-references" class="hash-link" aria-label="Direct link to Cross-references" title="Direct link to Cross-references" translate="no">​</a></h3>
<!-- -->
<div class="theme-admonition theme-admonition-caution admonition_xJq3 alert alert--warning"><div class="admonitionHeading_Gvgb"><span class="admonitionIcon_Rf37"><svg viewBox="0 0 16 16"><path fill-rule="evenodd" d="M8.893 1.5c-.183-.31-.52-.5-.887-.5s-.703.19-.886.5L.138 13.499a.98.98 0 0 0 0 1.001c.193.31.53.501.886.501h13.964c.367 0 .704-.19.877-.5a1.03 1.03 0 0 0 .01-1.002L8.893 1.5zm.133 11.497H6.987v-2.003h2.039v2.003zm0-3.004H6.987V5.987h2.039v4.006z"></path></svg></span>Cross-references and query performance</div><div class="admonitionContent_BuS1"><p>Queries involving cross-references can be slower than queries that do not involve cross-references, especially at scale such as for multiple objects or complex queries.</p><p>At the first instance, we strongly encourage you to consider whether you can avoid using cross-references in your data schema. As a scalable AI database, Weaviate is well-placed to perform complex queries with vector, keyword and hybrid searches involving filters. You may benefit from rethinking your data schema to avoid cross-references where possible.</p><p>For example, instead of creating separate &quot;Author&quot; and &quot;Book&quot; collections with cross-references, consider embedding author information directly in Book objects and using searches and filters to find books by author characteristics.</p></div></div>
<p>If data objects are related, you can use <a class="" href="/weaviate/manage-collections/cross-references">cross-references</a> to represent the relationships. Cross-references in Weaviate are like links that help you retrieve related information. Cross-references capture relationships, but they do not change the vectors of the underlying objects.</p>
<p>To create a reference, use a property from one collection to specify the value of a related property in the other collection.</p>

<h3 class="anchor anchorTargetStickyNavbar_Vzrq" id="multiple-vector-embeddings-named-vectors">Multiple vector embeddings (named vectors)<a href="#multiple-vector-embeddings-named-vectors" class="hash-link" aria-label="Direct link to Multiple vector embeddings (named vectors)" title="Direct link to Multiple vector embeddings (named vectors)" translate="no">​</a></h3>
<!-- -->
<div class="theme-admonition theme-admonition-info admonition_xJq3 alert alert--info"><div class="admonitionHeading_Gvgb"><span class="admonitionIcon_Rf37"><svg viewBox="0 0 14 16"><path fill-rule="evenodd" d="M7 2.3c3.14 0 5.7 2.56 5.7 5.7s-2.56 5.7-5.7 5.7A5.71 5.71 0 0 1 1.3 8c0-3.14 2.56-5.7 5.7-5.7zM7 1C3.14 1 0 4.14 0 8s3.14 7 7 7 7-3.14 7-7-3.14-7-7-7zm1 3H6v5h2V4zm0 6H6v2h2v-2z"></path></svg></span>Added in v1.24.0</div><div class="admonitionContent_BuS1"><p>Weaviate collections support multiple named vectors.</p></div></div>
<p>Collections can have multiple <a class="" href="/weaviate/config-refs/collections#named-vectors">named vectors</a>.</p>
<p>The vectors in a collection can have their own configurations. Each vector space can set its own index, its own compression algorithm, and its own vectorizer. This means you can use different vectorization models, and apply different distance metrics, to the same object.</p>
<p>To work with named vectors, adjust your queries to specify a target vector for <a class="" href="/weaviate/search/similarity#named-vectors">vector search</a> or <a class="" href="/weaviate/search/hybrid#named-vectors">hybrid search</a> queries.</p>
<h4 class="anchor anchorTargetStickyNavbar_Vzrq" id="adding-a-named-vector-after-collection-creation">Adding a named vector after collection creation<a href="#adding-a-named-vector-after-collection-creation" class="hash-link" aria-label="Direct link to Adding a named vector after collection creation" title="Direct link to Adding a named vector after collection creation" translate="no">​</a></h4>
<div class="theme-admonition theme-admonition-info admonition_xJq3 alert alert--info"><div class="admonitionHeading_Gvgb"><span class="admonitionIcon_Rf37"><svg viewBox="0 0 14 16"><path fill-rule="evenodd" d="M7 2.3c3.14 0 5.7 2.56 5.7 5.7s-2.56 5.7-5.7 5.7A5.71 5.71 0 0 1 1.3 8c0-3.14 2.56-5.7 5.7-5.7zM7 1C3.14 1 0 4.14 0 8s3.14 7 7 7 7-3.14 7-7-3.14-7-7-7zm1 3H6v5h2V4zm0 6H6v2h2v-2z"></path></svg></span>Added in <code>v1.31</code></div></div>
<p>A named vector can be added to an existing collection definition after collection creation. This allows you to add new vector representations for objects without having to delete and recreate the collection.</p>
<p>When you add a new named vector to an existing collection definition, it&#x27;s important to understand that <strong>existing objects&#x27; new named vector will remain unpopulated</strong>. Only objects created after the named vector addition will receive these new vector embeddings.</p>
<p>This prevents any unintended side effects, such as incurring large vectorization time or costs for all existing objects in a collection.</p>
<p>If you want to populate the new named vector for existing objects, delete and reinsert the objects manually. This will trigger the vectorization process for the new named vector.</p>

<h2 class="anchor anchorTargetStickyNavbar_Vzrq" id="data-schema">Data Schema<a href="#data-schema" class="hash-link" aria-label="Direct link to Data Schema" title="Direct link to Data Schema" translate="no">​</a></h2>
<p>Weaviate requires a data schema before you add data. However, you don&#x27;t have to create a data schema manually. If you don&#x27;t provide one, Weaviate generates a schema based on the incoming data.</p>
<!-- -->
<p>Weaviate&#x27;s schema defines its data structure in a formal language. In other words, it is a blueprint of how to organize and store the data.</p>
<p>The schema defines data classes (i.e. collections of objects), the properties within each class (name, type, description, settings), possible graph links between data objects (<a class="" href="/weaviate/concepts/data#cross-references">cross-references</a>), and the vectorizer module (if any) to be used for the class, as well as settings such as the vectorizer module, and index configurations.</p>
<div class="theme-admonition theme-admonition-note admonition_xJq3 alert alert--secondary"><div class="admonitionHeading_Gvgb"><span class="admonitionIcon_Rf37"><svg viewBox="0 0 14 16"><path fill-rule="evenodd" d="M6.3 5.69a.942.942 0 0 1-.28-.7c0-.28.09-.52.28-.7.19-.18.42-.28.7-.28.28 0 .52.09.7.28.18.19.28.42.28.7 0 .28-.09.52-.28.7a1 1 0 0 1-.7.3c-.28 0-.52-.11-.7-.3zM8 7.99c-.02-.25-.11-.48-.31-.69-.2-.19-.42-.3-.69-.31H6c-.27.02-.48.13-.69.31-.2.2-.3.44-.31.69h1v3c.02.27.11.5.31.69.2.2.42.31.69.31h1c.27 0 .48-.11.69-.31.2-.19.3-.42.31-.69H8V7.98v.01zM7 2.3c-3.14 0-5.7 2.54-5.7 5.68 0 3.14 2.56 5.7 5.7 5.7s5.7-2.55 5.7-5.7c0-3.15-2.56-5.69-5.7-5.69v.01zM7 .98c3.86 0 7 3.14 7 7s-3.14 7-7 7-7-3.12-7-7 3.14-7 7-7z"></path></svg></span>Schema vs. Taxonomy</div><div class="admonitionContent_BuS1"><p>A Weaviate data schema is slightly different from a taxonomy. A taxonomy has a hierarchy. Read more about how taxonomies, ontologies and schemas are related in this Weaviate <a href="https://medium.com/semi-technologies/taxonomies-ontologies-and-schemas-how-do-they-relate-to-weaviate-9f76739fc695" target="_blank" rel="noopener noreferrer" class="">blog post</a>.</p></div></div>
<p>Schemas fulfill several roles:</p>
<ol>
<li class="">Schemas define collections and properties.</li>
<li class="">Schemas define cross-references that link collections, even collections that use different embeddings.</li>
<li class="">Schemas let you configure module behavior, ANN index settings, reverse indexes, and other features on a collection level.</li>
</ol>
<p>For details on configuring your schema, see the <a class="" href="/weaviate/starter-guides/managing-collections">schema tutorial</a> or <a class="" href="/weaviate/manage-collections">How-to: Manage collections</a>.</p>
<h2 class="anchor anchorTargetStickyNavbar_Vzrq" id="multi-tenancy">Multi-tenancy<a href="#multi-tenancy" class="hash-link" aria-label="Direct link to Multi-tenancy" title="Direct link to Multi-tenancy" translate="no">​</a></h2>
<div class="theme-admonition theme-admonition-info admonition_xJq3 alert alert--info"><div class="admonitionHeading_Gvgb"><span class="admonitionIcon_Rf37"><svg viewBox="0 0 14 16"><path fill-rule="evenodd" d="M7 2.3c3.14 0 5.7 2.56 5.7 5.7s-2.56 5.7-5.7 5.7A5.71 5.71 0 0 1 1.3 8c0-3.14 2.56-5.7 5.7-5.7zM7 1C3.14 1 0 4.14 0 8s3.14 7 7 7 7-3.14 7-7-3.14-7-7-7zm1 3H6v5h2V4zm0 6H6v2h2v-2z"></path></svg></span>Multi-tenancy availability</div><div class="admonitionContent_BuS1"><ul>
<li class="">Multi-tenancy added in <code>v1.20</code></li>
</ul></div></div>
<p>To separate data within a cluster, use multi-tenancy. Weaviate partitions the cluster into shards. Each shard holds data for a single tenant.</p>
<!-- -->
<p>Sharding has several benefits:</p>
<ul>
<li class="">Data isolation</li>
<li class="">Fast, efficient querying</li>
<li class="">Easy and robust setup and clean up</li>
</ul>
<p>Tenant shards are more lightweight. You can easily have 50,000, or more, active shards per node. This means that you can support 1M concurrently active tenants with just 20 or so nodes.</p>
<p>Multi-tenancy is especially useful when you want to store data for multiple customers, or when you want to store data for multiple projects.</p>
<div class="theme-admonition theme-admonition-caution admonition_xJq3 alert alert--warning"><div class="admonitionHeading_Gvgb"><span class="admonitionIcon_Rf37"><svg viewBox="0 0 16 16"><path fill-rule="evenodd" d="M8.893 1.5c-.183-.31-.52-.5-.887-.5s-.703.19-.886.5L.138 13.499a.98.98 0 0 0 0 1.001c.193.31.53.501.886.501h13.964c.367 0 .704-.19.877-.5a1.03 1.03 0 0 0 .01-1.002L8.893 1.5zm.133 11.497H6.987v-2.003h2.039v2.003zm0-3.004H6.987V5.987h2.039v4.006z"></path></svg></span>Tenant deletion == Tenant data deletion</div><div class="admonitionContent_BuS1"><p>Deleting a tenant deletes the associated shard. As a result, deleting a tenant also deletes all of its objects.</p></div></div>
<h3 class="anchor anchorTargetStickyNavbar_Vzrq" id="tenant-states">Tenant states<a href="#tenant-states" class="hash-link" aria-label="Direct link to Tenant states" title="Direct link to Tenant states" translate="no">​</a></h3>
<div class="theme-admonition theme-admonition-info admonition_xJq3 alert alert--info"><div class="admonitionHeading_Gvgb"><span class="admonitionIcon_Rf37"><svg viewBox="0 0 14 16"><path fill-rule="evenodd" d="M7 2.3c3.14 0 5.7 2.56 5.7 5.7s-2.56 5.7-5.7 5.7A5.71 5.71 0 0 1 1.3 8c0-3.14 2.56-5.7 5.7-5.7zM7 1C3.14 1 0 4.14 0 8s3.14 7 7 7 7-3.14 7-7-3.14-7-7-7zm1 3H6v5h2V4zm0 6H6v2h2v-2z"></path></svg></span>Multi-tenancy availability</div><div class="admonitionContent_BuS1"><ul>
<li class="">Tenant activity status setting added in <code>v1.21</code></li>
<li class=""><code>OFFLOADED</code> status added in <code>v1.26</code></li>
</ul></div></div>
<p>Tenants have an activity status (also called a tenant state) that reflects their availability and storage location. A tenant can be <code>ACTIVE</code>, <code>INACTIVE</code>, <code>OFFLOADED</code>, <code>OFFLOADING</code>, or <code>ONLOADING</code>.</p>
<ul>
<li class=""><code>ACTIVE</code> tenants are loaded and available for read and write operations.</li>
<li class="">In all other states, the tenant is not available for read or write access. Access attempts return an error message.<!-- -->
<ul>
<li class=""><code>INACTIVE</code> tenants are stored on local disk storage for quick activation.</li>
<li class=""><code>OFFLOADED</code> tenants are stored on cloud storage. This status is useful for long-term storage for tenants that are not frequently accessed.</li>
<li class=""><code>OFFLOADING</code> tenants are being moved to cloud storage. This is a transient status, and therefore not user-specifiable.</li>
<li class=""><code>ONLOADING</code> tenants are being loaded from cloud storage. This is a transient status, and therefore not user-specifiable. An <code>ONLOADING</code> tenant may be being warmed to a <code>ACTIVE</code> status or a <code>INACTIVE</code> status.</li>
</ul>
</li>
</ul>
<p>For more details on managing tenants, see <a class="" href="/weaviate/manage-collections/multi-tenancy">Multi-tenancy operations</a>.</p>

<div class="theme-admonition theme-admonition-info admonition_xJq3 alert alert--info"><div class="admonitionHeading_Gvgb"><span class="admonitionIcon_Rf37"><svg viewBox="0 0 14 16"><path fill-rule="evenodd" d="M7 2.3c3.14 0 5.7 2.56 5.7 5.7s-2.56 5.7-5.7 5.7A5.71 5.71 0 0 1 1.3 8c0-3.14 2.56-5.7 5.7-5.7zM7 1C3.14 1 0 4.14 0 8s3.14 7 7 7 7-3.14 7-7-3.14-7-7-7zm1 3H6v5h2V4zm0 6H6v2h2v-2z"></path></svg></span>Offloading: AWS S3 only</div><div class="admonitionContent_BuS1"><p>As of Weaviate <code>v1.26.0</code>, tenants can only be offloaded to cold storage in AWS S3. Additional storage options may be added in future releases.<br><br></p><p>To offload a tenant, use the <code>offload-s3</code> module.</p></div></div>
<p>Offloading tenants requires the relevant <code>offload-&lt;storage&gt;</code> module to be <a class="" href="/weaviate/configuration/modules">enabled</a> in the Weaviate cluster.</p>
<p>When a tenant is offloaded, the entire tenant shard is moved to cloud storage. This is useful for long-term storage of tenants that are not frequently accessed. Offloaded tenants are not available for read or write operations until they are loaded back into the cluster.</p>
<h3 class="anchor anchorTargetStickyNavbar_Vzrq" id="backups">Backups<a href="#backups" class="hash-link" aria-label="Direct link to Backups" title="Direct link to Backups" translate="no">​</a></h3>
<div class="theme-admonition theme-admonition-caution admonition_xJq3 alert alert--warning"><div class="admonitionHeading_Gvgb"><span class="admonitionIcon_Rf37"><svg viewBox="0 0 16 16"><path fill-rule="evenodd" d="M8.893 1.5c-.183-.31-.52-.5-.887-.5s-.703.19-.886.5L.138 13.499a.98.98 0 0 0 0 1.001c.193.31.53.501.886.501h13.964c.367 0 .704-.19.877-.5a1.03 1.03 0 0 0 .01-1.002L8.893 1.5zm.133 11.497H6.987v-2.003h2.039v2.003zm0-3.004H6.987V5.987h2.039v4.006z"></path></svg></span>Backups do not include inactive or offloaded tenants</div><div class="admonitionContent_BuS1"><p>Backups of multi-tenant collections will only include <code>active</code> tenants, and not <code>inactive</code> or <code>offloaded</code> tenants. <a class="" href="/weaviate/manage-collections/multi-tenancy#manage-tenant-states">Activate tenants</a> before creating a backup to ensure all data is included.</p></div></div>
<h3 class="anchor anchorTargetStickyNavbar_Vzrq" id="tenancy-and-ids">Tenancy and IDs<a href="#tenancy-and-ids" class="hash-link" aria-label="Direct link to Tenancy and IDs" title="Direct link to Tenancy and IDs" translate="no">​</a></h3>
<p>Each tenancy is like a namespace, so different tenants could, in theory, have objects with the same IDs. To avoid naming problems, object IDs in multi-tenant clusters combine the tenant ID and the object ID to create an ID that is unique across tenants.</p>
<h3 class="anchor anchorTargetStickyNavbar_Vzrq" id="tenancy-and-cross-references">Tenancy and cross-references<a href="#tenancy-and-cross-references" class="hash-link" aria-label="Direct link to Tenancy and cross-references" title="Direct link to Tenancy and cross-references" translate="no">​</a></h3>
<p>Multi-tenancy supports some cross-references.</p>
<p>Cross-references like these are supported:</p>
<ul>
<li class="">From a multi-tenancy object to a non-multi-tenancy object.</li>
<li class="">From a multi-tenancy object to another multi-tenancy object, as long as they belong to the same tenant.</li>
</ul>
<p>Cross-references like these are not supported:</p>
<ul>
<li class="">From a non-multi-tenancy object to a multi-tenancy object.</li>
<li class="">From a multi-tenancy object to another multi-tenancy object if they belong to different tenants.</li>
</ul>
<h3 class="anchor anchorTargetStickyNavbar_Vzrq" id="key-features">Key features<a href="#key-features" class="hash-link" aria-label="Direct link to Key features" title="Direct link to Key features" translate="no">​</a></h3>
<ul>
<li class="">Each tenant has a dedicated, high-performance vector index. Dedicated indexes mean faster query speeds. Instead of searching a shared index space, each tenant responds as if it was the only user on the cluster.</li>
<li class="">Each tenant&#x27;s data is isolated on a dedicated shard. This means that deletes are fast and do not affect other tenants.</li>
<li class="">To scale out, add a new node to your cluster. Weaviate does not redistribute existing tenants, however Weaviate adds new tenants to the node with the least resource usage.</li>
</ul>

<h3 class="anchor anchorTargetStickyNavbar_Vzrq" id="monitoring-metrics">Monitoring metrics<a href="#monitoring-metrics" class="hash-link" aria-label="Direct link to Monitoring metrics" title="Direct link to Monitoring metrics" translate="no">​</a></h3>
<p>To group tenants together for monitoring, set <a class="" href="/deploy/configuration/env-vars"><code>PROMETHEUS_MONITORING_GROUP = true</code></a> in your system configuration file.</p>
<h3 class="anchor anchorTargetStickyNavbar_Vzrq" id="number-of-tenants-per-node">Number of tenants per node<a href="#number-of-tenants-per-node" class="hash-link" aria-label="Direct link to Number of tenants per node" title="Direct link to Number of tenants per node" translate="no">​</a></h3>
<p>The number of tenants per node is limited by operating system constraints. The number of tenants cannot exceed the Linux open file limit per process.</p>
<p>For example, a 9-node test cluster built on <code>n1-standard-8</code> machines holds around 170k active tenants. There are 18,000 to 19,000 tenants per node.</p>
<p>Note that these numbers relate to active tenants only. If you <a class="" href="/weaviate/manage-collections/multi-tenancy#manage-tenant-states">set unused tenants as <code>inactive</code></a>, the open file per process limit does not apply.</p>

<h2 class="anchor anchorTargetStickyNavbar_Vzrq" id="summary">Summary<a href="#summary" class="hash-link" aria-label="Direct link to Summary" title="Direct link to Summary" translate="no">​</a></h2>
<ul>
<li class="">The schema defines collections and properties.</li>
<li class="">Collections contain data objects that are describe in JSON documents.</li>
<li class="">Data objects can contain a vector and properties.</li>
<li class="">Vectors come from machine learning models.</li>
<li class="">Different collections represent different vector spaces.</li>
<li class="">Cross-references link objects between schemas.</li>
<li class="">Multi-tenancy isolates data for each tenant.</li>
</ul>
