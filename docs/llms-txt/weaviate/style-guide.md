<h2 class="anchor anchorTargetStickyNavbar_Vzrq" id="content-guidelines">Content guidelines<a href="#content-guidelines" class="hash-link" aria-label="Direct link to Content guidelines" title="Direct link to Content guidelines" translate="no">​</a></h2>
<h3 class="anchor anchorTargetStickyNavbar_Vzrq" id="audience">Audience<a href="#audience" class="hash-link" aria-label="Direct link to Audience" title="Direct link to Audience" translate="no">​</a></h3>
<p>Our documentation serves <strong>AI developers</strong> across various experience levels. When creating quickstart guides, we primarily focus on developers, data scientists, and students, while also considering system evaluators who may be assessing Weaviate for their organizations. We assume readers have <strong>minimal knowledge of vector databases</strong> and provide appropriate context and explanations to help them succeed.</p>
<h3 class="anchor anchorTargetStickyNavbar_Vzrq" id="voice--tone">Voice &amp; tone<a href="#voice--tone" class="hash-link" aria-label="Direct link to Voice &amp; tone" title="Direct link to Voice &amp; tone" translate="no">​</a></h3>
<p>Address readers directly using <strong>active voice</strong> and the <strong>second person (&quot;you&quot;)</strong> to create a personal, engaging experience. Maintain a <strong>conversational, non-academic, and friendly tone</strong> throughout all documentation. Avoid overly formal language that creates distance, but also steer clear of frivolous content that undermines credibility. <strong>Adjust the assumed level of user knowledge</strong> based on the specific page content, providing more foundational explanations in introductory materials and allowing for greater technical depth in advanced topics.</p>
<h3 class="anchor anchorTargetStickyNavbar_Vzrq" id="inclusion--accessibility">Inclusion &amp; accessibility<a href="#inclusion--accessibility" class="hash-link" aria-label="Direct link to Inclusion &amp; accessibility" title="Direct link to Inclusion &amp; accessibility" translate="no">​</a></h3>
<p>Follow <strong><a href="https://developers.google.com/style/accessibility" target="_blank" rel="noopener noreferrer" class="">Google&#x27;s accessibility guidelines</a></strong> to ensure our documentation remains bias-free and respectful of all users, regardless of their background or identity. Prioritize creating <strong>clear and useful documentation</strong> that serves everyone, including users with disabilities.</p>
<p><strong>Key accessibility requirements:</strong></p>
<ul>
<li class="">Provide <strong>meaningful alt text</strong> for all images that summarizes their intent (use empty alt text only for purely decorative images)</li>
<li class="">Use <strong>descriptive headings and titles</strong> that clearly indicate content structure and purpose</li>
</ul>
<hr>
<h2 class="anchor anchorTargetStickyNavbar_Vzrq" id="types-of-documentation-pages">Types of documentation pages<a href="#types-of-documentation-pages" class="hash-link" aria-label="Direct link to Types of documentation pages" title="Direct link to Types of documentation pages" translate="no">​</a></h2>
<p>Categorize content using these types:</p>
<ul>
<li class=""><strong>Concept:</strong> Explains fundamental ideas and principles.</li>
<li class=""><strong>Guides (how-to):</strong> Provides step-by-step instructions to achieve a specific goal.</li>
<li class=""><strong>Reference:</strong> Offers detailed information about APIs, configurations, and technical specifications.</li>
<li class=""><strong>Tutorial:</strong> Guides users through a complete process, often combining concepts and guides.</li>
</ul>
<hr>
<h2 class="anchor anchorTargetStickyNavbar_Vzrq" id="documentation-directory-structure">Documentation directory structure<a href="#documentation-directory-structure" class="hash-link" aria-label="Direct link to Documentation directory structure" title="Direct link to Documentation directory structure" translate="no">​</a></h2>
<p>The documentation is organized into key sections. When contributing, understand where your content fits:</p>
<ul>
<li class="">
<p><strong>Weaviate Database - <code>/docs/weaviate</code>:</strong> Contains getting started tutorials, how-to guides for collection management and search operations, integrations with AI model providers, API reference documentation, technical concepts explanations, code examples for common tasks, release notes, performance benchmarks, module documentation, FAQ, glossary, and sample datasets.</p>
</li>
<li class="">
<p><strong>Deployment docs - <code>/docs/deploy</code>:</strong> Documentation for deploying and operating Weaviate instances. Covers installation methods including Docker and Kubernetes, configuration options for authentication and performance, production deployment patterns, scaling and high availability setups, cloud provider-specific guides, troubleshooting information, and version migration procedures.</p>
</li>
<li class="">
<p><strong>Weaviate Agents - <code>/docs/agents</code>:</strong> Documentation for Weaviate&#x27;s agent framework. Includes setup instructions, usage examples, and implementation patterns for building AI agents that can interact with Weaviate databases.</p>
</li>
<li class="">
<p><strong>Weaviate Cloud - <code>/docs/cloud</code>:</strong> Documentation for Weaviate&#x27;s managed cloud service. Contains account setup procedures, Weaviate embeddings service documentation, billing and subscription management, service limitations, and cloud-specific configuration options.</p>
</li>
<li class="">
<p><strong>Integrations - <code>/docs/integrations</code>:</strong> Documentation for third-party tools and frameworks that work with Weaviate. Covers client libraries, data import tools, visualization platforms, and other ecosystem integrations.</p>
</li>
<li class="">
<p><strong>Contributor guide - <code>/docs/contributor-guide</code>:</strong> Documentation for contributing to Weaviate&#x27;s open source projects. Includes development setup instructions, coding standards, testing procedures, and contribution workflows for the database, modules, client libraries, and contextionary components.</p>
</li>
</ul>
<hr>
<h2 class="anchor anchorTargetStickyNavbar_Vzrq" id="page-layout">Page layout<a href="#page-layout" class="hash-link" aria-label="Direct link to Page layout" title="Direct link to Page layout" translate="no">​</a></h2>
<ul>
<li class=""><strong>Frontmatter:</strong> The <code>title</code> and <a class="" href="/contributor-guide/weaviate-docs/llms#best-practices-for-page-descriptions"><code>description</code></a> fields are required.</li>
<li class=""><strong>Top-level headings:</strong> Do not include redundant top-level headings like <code>## Introduction</code> or <code>## Overview</code> within the body content; the page title serves this purpose.</li>
<li class=""><strong>Related pages:</strong> Place links to related pages at the <strong>bottom of the page</strong> within a <code>## Further resources</code> section.</li>
<li class=""><strong>QA section:</strong> Most pages should and with a <code>## Questions and feedback</code> section.</li>
</ul>
<hr>
<h2 class="anchor anchorTargetStickyNavbar_Vzrq" id="formatting--styling-rules">Formatting &amp; styling rules<a href="#formatting--styling-rules" class="hash-link" aria-label="Direct link to Formatting &amp; styling rules" title="Direct link to Formatting &amp; styling rules" translate="no">​</a></h2>
<ul>
<li class=""><strong>Capitalization:</strong> Use <strong>sentence case for all headings</strong>.<!-- -->
<ul>
<li class=""><em>Example:</em> <code>## This is a heading about Weaviate</code></li>
</ul>
</li>
<li class=""><strong>Lists:</strong>
<ul>
<li class="">Transform &quot;cascading sentences disguised as lists&quot; into proper lists.</li>
<li class="">Use <strong>numbered lists for sequences</strong>.</li>
<li class="">Use <strong>bulleted lists for most other lists</strong>.</li>
</ul>
</li>
<li class=""><strong>Headings hierarchy:</strong>
<ul>
<li class="">Skip the title heading <code>#</code>, the title will be derived from the frontmatter field <code>title:</code>.</li>
<li class="">Avoid starting pages with heading such as <code>## Introduction</code> or <code>## Overview</code> (an exception for this are tutorials).</li>
<li class="">Use Markdown for headings: <code>## Section title</code>, <code>### Subsection title</code>, <code>#### Sub-subsection title</code>.</li>
<li class="">Do not skip heading levels (e.g., go directly from <code>##</code> to <code>####</code>).</li>
<li class="">Be aware that H4 and lower headings will not appear in the right-hand-side table of contents.</li>
</ul>
</li>
<li class=""><strong>Code elements:</strong>
<ul>
<li class="">Use <strong>code font</strong> for all code-related text like method names, environment variables, etc.</li>
</ul>
</li>
<li class=""><strong>Punctuation:</strong> Use <a href="https://developers.google.com/style/commas" target="_blank" rel="noopener noreferrer" class="">serial commas</a>.</li>
<li class=""><strong>American English:</strong> Assume US dialect for English spelling and usage. Write for a global audience.</li>
</ul>
<hr>
<h2 class="anchor anchorTargetStickyNavbar_Vzrq" id="terminology">Terminology<a href="#terminology" class="hash-link" aria-label="Direct link to Terminology" title="Direct link to Terminology" translate="no">​</a></h2>
<p>Use clear, direct language. Define acronyms and abbreviations on first use. At least on the <strong>first mention</strong> of a Weaviate method or term on a page, <strong>link to its corresponding reference, concept page or how-to page</strong>.</p>
<hr>
<h2 class="anchor anchorTargetStickyNavbar_Vzrq" id="visual-assets">Visual assets<a href="#visual-assets" class="hash-link" aria-label="Direct link to Visual assets" title="Direct link to Visual assets" translate="no">​</a></h2>
<ul>
<li class=""><strong>Images</strong>
<ul>
<li class=""><strong>Dark/light mode images:</strong> Use the <a class="" href="/contributor-guide/weaviate-docs/development#themedimage-component"><code>ThemedImage</code> component</a> for specifying both light and dark mode variants of the same image.</li>
<li class=""><strong>Social/preview images:</strong> Use the <code>image:</code> frontmatter for social media previews and images available in <code>static/og/</code>.</li>
<li class=""><strong>Screenshots:</strong> Avoid using too many screenshots due to maintenance challenges. An exception to this rule are the <a class="" href="/cloud">Weaviate Cloud</a> docs where the WCD console needs to be documented visually as well.</li>
</ul>
</li>
<li class=""><strong>Diagrams:</strong> Diagrams are acceptable where they add clarity. We use <a href="https://app.excalidraw.com/" target="_blank" rel="noopener noreferrer" class="">Excalidraw</a> and <a href="https://mermaid.js.org/" target="_blank" rel="noopener noreferrer" class="">Mermaid</a> (legacy).</li>
<li class=""><strong>Videos:</strong> Videos can supplement written documentation but should not replace it. Ensure videos are kept up-to-date as the UI evolves.</li>
</ul>
