<p>We categorize storage resources using three tiers: <a href="#-hot" class=""><em>hot</em></a>, <a href="#-warm" class=""><em>warm</em></a>, and <a href="#-cold" class=""><em>cold</em></a>. Each tier has different performance characteristics and costs.</p>

<h3 class="anchor anchorTargetStickyNavbar_Vzrq" id="-hot">ðŸŸ¥ Hot<a href="#-hot" class="hash-link" aria-label="Direct link to ðŸŸ¥ Hot" title="Direct link to ðŸŸ¥ Hot" translate="no">â€‹</a></h3>
<p><img decoding="async" loading="lazy" alt="Storage Tiers - Hot" src="/assets/images/storage-tiers-hot-3e08f3be3955382d81a8f8641b7b542d.jpg" width="512" height="128" class="img_ev3q"></p>
<ul>
<li class="">Describes memory usage</li>
<li class="">Fastest and most expensive</li>
<li class="">Primarily driven by <a class="" href="/weaviate/starter-guides/managing-resources/indexing#hnsw-indexes"><code>HNSW</code></a> vector indexes</li>
<li class="">Always available (active) for use</li>
<li class="">Costs increase rapidly with scale</li>
</ul>

<h2 class="anchor anchorTargetStickyNavbar_Vzrq" id="resource-management---key-factors">Resource Management - Key Factors<a href="#resource-management---key-factors" class="hash-link" aria-label="Direct link to Resource Management - Key Factors" title="Direct link to Resource Management - Key Factors" translate="no">â€‹</a></h2>
<p>Effective resource management in Weaviate involves balancing performance, cost, and data accessibility. The key levers to manage resources are:</p>
<ul>
<li class=""><strong><a href="#vector-index-types" class="">Vector index types</a></strong>: Choose the right index type based on the number of objects and desired performance.</li>
<li class=""><strong><a href="#vector-compression" class="">Vector compression</a></strong>: Use compression techniques to reduce memory usage and improve query performance at the cost of some accuracy.</li>
<li class=""><strong><a href="#tenant-states" class="">Tenant states</a></strong>: Manage tenant states to balance cost and performance.</li>
</ul>
<h3 class="anchor anchorTargetStickyNavbar_Vzrq" id="vector-index-types">Vector index types<a href="#vector-index-types" class="hash-link" aria-label="Direct link to Vector index types" title="Direct link to Vector index types" translate="no">â€‹</a></h3>

<h3 class="anchor anchorTargetStickyNavbar_Vzrq" id="vector-compression">Vector compression<a href="#vector-compression" class="hash-link" aria-label="Direct link to Vector compression" title="Direct link to Vector compression" translate="no">â€‹</a></h3>
<p>Vector compression techniques reduce the size of vectors by quantizing them into a smaller representation.</p>
<p>This can have the impact of reducing memory usage, or improving performance by reducing the amount of data that needs to be read from disk. The trade-off is that the resulting search quality may be lower.</p>
<p>Weaviate supports the following vector compression methods:</p>
<table><thead><tr><th>Compression Method</th><th>Index Type</th><th>Requires Training</th><th>Description</th></tr></thead><tbody><tr><td>Product Quantization (PQ)</td><td>HNSW</td><td>Yes</td><td>Each vector becomes an array of integer-based centroids (<a class="" href="/weaviate/concepts/vector-quantization#product-quantization">read more</a>)</td></tr><tr><td>Binary Quantization (BQ)</td><td>HNSW, Flat</td><td>No</td><td>Each vector dimension becomes a bit (<a class="" href="/weaviate/concepts/vector-quantization#binary-quantization">read more</a>)</td></tr><tr><td>Scalar Quantization (SQ)</td><td>HNSW</td><td>Yes</td><td>Each vector dimension becomes an integer (<a class="" href="/weaviate/concepts/vector-quantization#scalar-quantization">read more</a>)</td></tr><tr><td>Rotational Quantization (RQ)</td><td>HNSW</td><td>No</td><td>Each vector is rotated then quantized to an integer (<a class="" href="/weaviate/concepts/vector-quantization#rotational-quantization">read more</a>)</td></tr></tbody></table>

<p>If you are unsure which index type to use, scalar quantization is a good starting point, provided that you have a representative sample of your likely final dataset.</p>

<h3 class="anchor anchorTargetStickyNavbar_Vzrq" id="tenant-states">Tenant states<a href="#tenant-states" class="hash-link" aria-label="Direct link to Tenant states" title="Direct link to Tenant states" translate="no">â€‹</a></h3>
<p>Multi-tenant collections enable you to efficiently manage isolated subsets of data. Each tenant share the same schema and configuration.</p>
<p>Weaviate supports the following tenant states:</p>
<table><thead><tr><th>Tenant state</th><th>CRUD &amp; Queries</th><th>Vector Index</th><th>Inverted Index</th><th>Object Data</th><th>Time to Activate</th><th>Description</th></tr></thead><tbody><tr><td>Active (default)</td><td><strong>Yes</strong></td><td>Hot/Warm</td><td>Warm</td><td>Warm</td><td>None</td><td>Tenant is available for use</td></tr><tr><td>Inactive</td><td><strong>No</strong></td><td>Warm</td><td>Warm</td><td>Warm</td><td>Fast</td><td>Tenant is locally stored but not available for use</td></tr><tr><td>Offloaded</td><td><strong>No</strong></td><td>Cold</td><td>Cold</td><td>Cold</td><td>Slow</td><td>Tenant is stored in cloud storage and not available for use</td></tr></tbody></table>
<p><em>Hot</em> tenants can be deactivated to <em>warm</em> storage to reduce memory usage, and any tenant can be offloaded to <em>cold</em> storage to reduce memory and disk usage. Conversely, any tenant can be reactivated when needed.</p>
<!-- -->
<p>Consider a strategy of deactivating tenants that are not frequently accessed, and offloading tenants that are rarely accessed.</p>

<h2 class="anchor anchorTargetStickyNavbar_Vzrq" id="tips">Tips<a href="#tips" class="hash-link" aria-label="Direct link to Tips" title="Direct link to Tips" translate="no">â€‹</a></h2>
<h3 class="anchor anchorTargetStickyNavbar_Vzrq" id="best-practices">Best Practices<a href="#best-practices" class="hash-link" aria-label="Direct link to Best Practices" title="Direct link to Best Practices" translate="no">â€‹</a></h3>
<ul>
<li class="">Start with the dynamic <a href="#vector-index-types" class="">index type</a> for new collections. This is particularly useful for multi-tenant collections, as it allows each tenant to use the most appropriate index type.</li>
<li class="">Use <a href="#vector-compression" class="">vector compression</a> techniques to optimize storage and query performance, especially for large collections or tenants.</li>
<li class="">Conduct thorough testing when changing index types or compression methods to ensure performance meets your requirements.</li>
</ul>
<h3 class="anchor anchorTargetStickyNavbar_Vzrq" id="common-pitfalls">Common Pitfalls<a href="#common-pitfalls" class="hash-link" aria-label="Direct link to Common Pitfalls" title="Direct link to Common Pitfalls" translate="no">â€‹</a></h3>
<ul>
<li class="">Overprovisioning hot storage: Keeping all data in hot storage can lead to unnecessary costs. Regularly assess what data truly needs the fastest access.</li>
<li class="">Neglecting to plan for growth: Not anticipating data growth can lead to performance issues. Always design your resource management strategy with scalability in mind.</li>
<li class="">Improper tenant management: In multi-tenant scenarios, forgetting to <a href="#tenant-states" class="">offload inactive tenants</a> can lead to resource waste. Implement automated processes to manage tenant states based on usage patterns.</li>
<li class="">Mismatch between quantization techniques, model and data: When using compression technique, ensure that the quantization technique is compatible with the model (e.g. BQ) and that the data is sufficient and representative for training (e.g. PQ, SQ).</li>
</ul>
