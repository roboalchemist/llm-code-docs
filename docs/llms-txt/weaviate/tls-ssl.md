class="theme-doc-markdown markdown"><header><h1>Securing Weaviate with SSL/TLS</h1></header><h2 class="anchor anchorTargetStickyNavbar_Vzrq" id="why-security-matters-understanding-the-risks">Why security matters: Understanding the risks<a href="#why-security-matters-understanding-the-risks" class="hash-link" aria-label="Direct link to Why security matters: Understanding the risks" title="Direct link to Why security matters: Understanding the risks" translate="no">​</a></h2>
<p>When your Weaviate database is exposed to the internet, unencrypted connections create serious vulnerabilities. Network packets can be intercepted at various points along their journey, exposing:</p>
<ul>
<li class=""><strong>Sensitive queries and search terms</strong> revealing what data you&#x27;re working with</li>
<li class=""><strong>Vector embeddings and data payloads</strong> containing your proprietary information</li>
<li class=""><strong>Authentication credentials</strong> if transmitted without encryption</li>
<li class=""><strong>API keys and tokens</strong> that could grant unauthorized access to your systems</li>
</ul>
<p>That&#x27;s exactly why it&#x27;s important to secure your database using <strong>SSL/TLS (Secure Sockets Layer/Transport Layer Security)</strong> — encryption protocols that create a secure, encrypted connection between your clients and your database. SSL/TLS ensures that even if network traffic is intercepted, the data remains unreadable to unauthorized parties.</p>
<div class="theme-admonition theme-admonition-info admonition_xJq3 alert alert--info"><div class="admonitionHeading_Gvgb"><span class="admonitionIcon_Rf37"><svg viewBox="0 0 14 16"><path fill-rule="evenodd" d="M7 2.3c3.14 0 5.7 2.56 5.7 5.7s-2.56 5.7-5.7 5.7A5.71 5.71 0 0 1 1.3 8c0-3.14 2.56-5.7 5.7-5.7zM7 1C3.14 1 0 4.14 0 8s3.14 7 7 7 7-3.14 7-7-3.14-7-7-7zm1 3H6v5h2V4zm0 6H6v2h2v-2z"></path></svg></span>Managed Weaviate deployments</div><div class="admonitionContent_BuS1"><p>If you&#x27;re using <strong>Weaviate Cloud</strong> or other managed Weaviate services, SSL/TLS is already configured and managed for you. This guide is primarily for self-hosted deployments on virtual private servers (VPS), hyperscaler infrastructure (AWS, GCP, Azure), or on-premises environments where you need to expose Weaviate to external networks.</p></div></div>
<p>This guide will take you through three different options for securing your self-hosted database. Whether you&#x27;re a small startup or an enterprise giant - there&#x27;s a solution for you!</p>
<h3 class="anchor anchorTargetStickyNavbar_Vzrq" id="when-is-ssltls-required">When is SSL/TLS required?<a href="#when-is-ssltls-required" class="hash-link" aria-label="Direct link to When is SSL/TLS required?" title="Direct link to When is SSL/TLS required?" translate="no">​</a></h3>
<p>Although testing and local deployments might not necessitate network encryption, <strong>enabling SSL/TLS is considered a best practice for any production deployment</strong>, internal or external.</p>
<h2 class="anchor anchorTargetStickyNavbar_Vzrq" id="understanding-weaviates-dual-port-architecture">Understanding Weaviate&#x27;s dual-port architecture<a href="#understanding-weaviates-dual-port-architecture" class="hash-link" aria-label="Direct link to Understanding Weaviate&#x27;s dual-port architecture" title="Direct link to Understanding Weaviate&#x27;s dual-port architecture" translate="no">​</a></h2>
<p>Before we dive into implementation, it&#x27;s important to understand that Weaviate requires <strong>two separate endpoints</strong>, both of which need SSL/TLS protection:</p>
<table><thead><tr><th><strong>Default port</strong></th><th><strong>Purpose</strong></th><th><strong>Protocol</strong></th></tr></thead><tbody><tr><td>8080</td><td>REST API and query interface (queries also available via gRPC)</td><td>HTTP</td></tr><tr><td>50051</td><td>High-performance data operations and queries</td><td>gRPC</td></tr></tbody></table>
<div class="theme-admonition theme-admonition-info admonition_xJq3 alert alert--info"><div class="admonitionHeading_Gvgb"><span class="admonitionIcon_Rf37"><svg viewBox="0 0 14 16"><path fill-rule="evenodd" d="M7 2.3c3.14 0 5.7 2.56 5.7 5.7s-2.56 5.7-5.7 5.7A5.71 5.71 0 0 1 1.3 8c0-3.14 2.56-5.7 5.7-5.7zM7 1C3.14 1 0 4.14 0 8s3.14 7 7 7 7-3.14 7-7-3.14-7-7-7zm1 3H6v5h2V4zm0 6H6v2h2v-2z"></path></svg></span>info</div><div class="admonitionContent_BuS1"><p><strong>Port 6060</strong> is used for Go profiling. In production deployments, this port should typically remain internal and not be exposed to the internet. If you need external access, apply the same SSL/TLS principles but ensure proper authentication is also in place.</p></div></div>
<p>This dual-port requirement adds complexity compared to traditional single-port applications, as each endpoint needs its own domain/subdomain and SSL certificate.</p>
