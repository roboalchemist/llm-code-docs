<h2 class="anchor anchorTargetStickyNavbar_Vzrq" id="text"><code>text</code><a href="#text" class="hash-link" aria-label="Direct link to text" title="Direct link to text" translate="no">‚Äã</a></h2>
<p>Use this type for any text data.</p>
<ul>
<li class="">Properties with the <code>text</code> type is used for vectorization and keyword search unless specified otherwise <a class="" href="/weaviate/manage-collections/vector-config#property-level-settings">in the property settings</a>.</li>
<li class="">If using <a class="" href="/weaviate/concepts/data#multiple-vector-embeddings-named-vectors">named vectors</a>, the property vectorization is defined in the <a class="" href="/weaviate/manage-collections/vector-config#define-named-vectors">named vector definition</a>.</li>
<li class="">Text properties are tokenized prior to being indexed for keyword/BM25 searches. See <a class="" href="/weaviate/config-refs/collections#tokenization">collection definition: tokenization</a> for more information.</li>
</ul>
<details class="details_lb9f alert alert--info details_b_Ee" data-collapsed="true"><summary><code>string</code> is deprecated</summary><div><div class="collapsibleContent_i85q"><p>Prior to <code>v1.19</code>, Weaviate supported an additional datatype <code>string</code>, which was differentiated by tokenization behavior to <code>text</code>. As of <code>v1.19</code>, this type is deprecated and will be removed in a future release.</p><p>Use <code>text</code> instead of <code>string</code>. <code>text</code> supports the tokenization options that are available through <code>string</code>.</p></div></div></details>

<h3 class="anchor anchorTargetStickyNavbar_Vzrq" id="note-graphql-and-int64">Note: GraphQL and <code>int64</code><a href="#note-graphql-and-int64" class="hash-link" aria-label="Direct link to note-graphql-and-int64" title="Direct link to note-graphql-and-int64" translate="no">‚Äã</a></h3>
<p>Although Weaviate supports <code>int64</code>, GraphQL currently only supports <code>int32</code>, and does not support <code>int64</code>. This means that currently <em>integer</em> data fields in Weaviate with integer values larger than <code>int32</code>, will not be returned using GraphQL queries. We are working on solving this <a href="https://github.com/weaviate/weaviate/issues/1563" target="_blank" rel="noopener noreferrer" class="">issue</a>. As current workaround is to use a <code>string</code> instead.</p>
<h2 class="anchor anchorTargetStickyNavbar_Vzrq" id="date"><code>date</code><a href="#date" class="hash-link" aria-label="Direct link to date" title="Direct link to date" translate="no">‚Äã</a></h2>
<p>A <code>date</code> in Weaviate is represented by an <a href="https://datatracker.ietf.org/doc/rfc3339/" target="_blank" rel="noopener noreferrer" class="">RFC 3339</a> timestamp in the <code>date-time</code> format. The timestamp includes the time and an offset.</p>
<p>For example:</p>
<ul>
<li class=""><code>&quot;1985-04-12T23:20:50.52Z&quot;</code></li>
<li class=""><code>&quot;1996-12-19T16:39:57-08:00&quot;</code></li>
<li class=""><code>&quot;1937-01-01T12:00:27.87+00:20&quot;</code></li>
</ul>
<p>To add a list of dates as a single entity, use an array of <code>date-time</code> formatted strings. For example: <code>[&quot;1985-04-12T23:20:50.52Z&quot;, &quot;1937-01-01T12:00:27.87+00:20&quot;]</code></p>

<h2 class="anchor anchorTargetStickyNavbar_Vzrq" id="geocoordinates"><code>geoCoordinates</code><a href="#geocoordinates" class="hash-link" aria-label="Direct link to geocoordinates" title="Direct link to geocoordinates" translate="no">‚Äã</a></h2>
<p>Geo coordinates can be used to find objects in a radius around a query location. A geo coordinate value stored as a float, and is processed as <a href="https://en.wikipedia.org/wiki/Decimal_degrees" target="_blank" rel="noopener noreferrer" class="">decimal degree</a> according to the <a href="https://www.iso.org/standard/39242.html#:~:text=For%20computer%20data%20interchange%20of,minutes%2C%20seconds%20and%20decimal%20seconds" target="_blank" rel="noopener noreferrer" class="">ISO standard</a>.</p>
<p>To supply a <code>geoCoordinates</code> property, specify the <code>latitude</code> and <code>longitude</code> as floating point decimal degrees.</p>

<p>There are two fields that accept input. <code>input</code> must always be set, while <code>defaultCountry</code> must only be set in specific situations. There are two scenarios possible:</p>
<ul>
<li class="">When you enter an international number (e.g. <code>&quot;+31 20 1234567&quot;</code>) to the <code>input</code> field, no <code>defaultCountry</code> needs to be entered. The underlying parser will automatically recognize the number&#x27;s country.</li>
<li class="">When you enter a national number (e.g. <code>&quot;020 1234567&quot;</code>), you need to specify the country in <code>defaultCountry</code> (in this case, <code>&quot;nl&quot;</code>), so that the parse can correctly convert the number into all formats. The string in <code>defaultCountry</code> should be an <a href="https://en.wikipedia.org/wiki/ISO_3166-1_alpha-2" target="_blank" rel="noopener noreferrer" class="">ISO 3166-1 alpha-2</a> country code.</li>
</ul>
<p>Weaviate will also add further read-only fields such as <code>internationalFormatted</code>, <code>countryCode</code>, <code>national</code>, <code>nationalFormatted</code> and <code>valid</code> when reading back a field of type <code>phoneNumber</code>.</p>

<h2 class="anchor anchorTargetStickyNavbar_Vzrq" id="blob"><code>blob</code><a href="#blob" class="hash-link" aria-label="Direct link to blob" title="Direct link to blob" translate="no">‚Äã</a></h2>
<p>The datatype blob accepts any binary data. The data should be <code>base64</code> encoded, and passed as a <code>string</code>. Characteristics:</p>
<ul>
<li class="">Weaviate doesn&#x27;t make assumptions about the type of data that is encoded. A module (e.g. <code>img2vec</code>) can investigate file headers as it wishes, but Weaviate itself does not do this.</li>
<li class="">When storing, the data is <code>base64</code> decoded (so Weaviate stores it more efficiently).</li>
<li class="">When serving, the data is <code>base64</code> encoded (so it is safe to serve as <code>json</code>).</li>
<li class="">There is no max file size limit.</li>
<li class="">This <code>blob</code> field is always skipped in the inverted index, regardless of setting. This mean you can not search by this <code>blob</code> field in a Weaviate GraphQL <code>where</code> filter, and there is no <code>valueBlob</code> field accordingly. Depending on the module, this field can be used in module-specific filters (e.g. <code>nearImage</code> in the <code>img2vec-neural</code> filter).</li>
</ul>
<p>To obtain the base64-encoded value of an image, you can run the following command - or use the helper methods in the Weaviate clients - to do so:</p>
<div class="language-bash codeBlockContainer_Ckt0 theme-code-block" style="--prism-color:#393A34;--prism-background-color:#f6f8fa"><div class="codeBlockContent_QJqH"><pre tabindex="0" class="prism-code language-bash codeBlock_bY9V thin-scrollbar" style="color:#393A34;background-color:#f6f8fa"><code class="codeBlockLines_e6Vv"><span class="token-line" style="color:#393A34"><span class="token plain">cat my_image.png | base64</span><br></span></code></pre></div></div>

As a scalable AI database, Weaviate is well-placed to perform complex queries with vector, keyword and hybrid searches involving filters. You may benefit from rethinking your data schema to avoid cross-references where possible.</p><p>For example, instead of creating separate &quot;Author&quot; and &quot;Book&quot; collections with cross-references, consider embedding author information directly in Book objects and using searches and filters to find books by author characteristics.</p></div></div>
<p>The <code>cross-reference</code> type allows a link to be created from one object to another. This is useful for creating relationships between collections, such as linking a <code>Person</code> collection to a <code>Company</code> collection.</p>
<p>The <code>cross-reference</code> type objects are <code>arrays</code> by default. This allows you to link to any number of instances of a given collection (including zero).</p>
<p>For more information on cross-references, see the <a class="" href="/weaviate/concepts/data#cross-references">cross-references</a>. To see how to work with cross-references, see <a class="" href="/weaviate/manage-collections/cross-references">how to manage data: cross-references</a>.</p>
