<code>v1.29</code></div><div class="admonitionContent_BuS1"><p>Role-based access control (RBAC) is generally available in Weaviate from version <code>v1.29</code>.</p></div></div>
<p>Weaviate provides differentiated access through <a class="" href="/deploy/configuration/authorization">authorization</a> levels based on the <a class="" href="/deploy/configuration/authentication">authenticated</a> user identity.</p>
<p>If <strong>role-based access control (RBAC)</strong> is enabled, access can be further restricted based on the roles of users.
The diagram below illustrates the RBAC model in Weaviate, where access is governed by defining roles and assigning them specific permissions. These permissions determine which actions users can perform on designated resource types.</p>
<p>The main components are:</p>
<ul>
<li class="">
<p><strong>Users</strong>:
Individual users (e.g., <code>user-a</code> and <code>user-b</code>) are assigned to specific roles.</p>
</li>
<li class="">
<p><strong>Roles</strong>:
Each role encapsulates a set of permissions. This abstraction lets you manage what actions a group of users can perform.</p>
</li>
<li class="">
<p><strong>Permissions</strong>:
Permissions are comprised of three parts:</p>
<ul>
<li class=""><strong>Actions</strong>: Operations like create, read, update, delete, and manage.</li>
<li class=""><strong>Resources</strong>: Specific targets for these actions, such as collections and backups.</li>
<li class=""><strong>Optional constraints</strong>: Resource-specific constraints such as filtering by collection names.</li>
</ul>
</li>
</ul>
<!-- -->
<p>This RBAC system ensures that users only have the access necessary for their roles, enhancing both security and manageability within Weaviate. Roles and permissions can be managed through the Weaviate <b><a href="/weaviate/api/rest#tag/authz">REST API</a></b> directly or through a <strong><a class="" href="/weaviate/configuration/rbac/manage-roles">client library</a></strong> programmatically.</p>
<h2 class="anchor anchorTargetStickyNavbar_Vzrq" id="roles">Roles<a href="#roles" class="hash-link" aria-label="Direct link to Roles" title="Direct link to Roles" translate="no">​</a></h2>
<h3 class="anchor anchorTargetStickyNavbar_Vzrq" id="predefined-roles">Predefined roles<a href="#predefined-roles" class="hash-link" aria-label="Direct link to Predefined roles" title="Direct link to Predefined roles" translate="no">​</a></h3>
<p>Weaviate comes with a set of predefined roles. These roles are:</p>
<ul>
<li class=""><code>root</code>: The root role has <strong>full access</strong> to all resources in Weaviate.</li>
<li class=""><code>viewer</code>: The viewer role has <strong>read-only access</strong> to all resources in Weaviate.</li>
</ul>
<p>The <code>root</code> role can be assigned to a user through the Weaviate configuration file using the <a class="" href="/deploy/configuration/env-vars#rbac-authorization"><code>AUTHORIZATION_RBAC_ROOT_USERS</code></a> environment variable. A predefined role cannot be modified. The user can, however, be assigned additional roles through the Weaviate API.</p>
<p>All roles can also be assigned through the Weaviate API, including the predefined role. The predefined roles cannot be modified, but they can be assigned to or revoked from users.</p>
<p>Refer to the <a class="" href="/deploy/configuration/configuring-rbac">RBAC: Configuration</a> page for more information on how to assign predefined roles to users.</p>
<h3 class="anchor anchorTargetStickyNavbar_Vzrq" id="custom-roles">Custom roles<a href="#custom-roles" class="hash-link" aria-label="Direct link to Custom roles" title="Direct link to Custom roles" translate="no">​</a></h3>
<p>Any authenticated user that is not assigned a predefined role has no roles or permissions by default.</p>
<p>These users&#x27; permissions can be modified through Weaviate by those with the appropriate permissions for <strong>managing roles</strong>. This allows for the creation of custom roles, which can be assigned to users as needed.</p>
<p>Role management can be performed with a <a class="" href="/deploy/configuration/configuring-rbac">predefined <code>root</code> role</a> or a custom role with <a class="" href="/weaviate/configuration/rbac/manage-roles#role-management-permissions"><code>manage_roles</code> permissions</a>.</p>
<div class="theme-admonition theme-admonition-caution admonition_xJq3 alert alert--warning"><div class="admonitionHeading_Gvgb"><span class="admonitionIcon_Rf37"><svg viewBox="0 0 16 16"><path fill-rule="evenodd" d="M8.893 1.5c-.183-.31-.52-.5-.887-.5s-.703.19-.886.5L.138 13.499a.98.98 0 0 0 0 1.001c.193.31.53.501.886.501h13.964c.367 0 .704-.19.877-.5a1.03 1.03 0 0 0 .01-1.002L8.893 1.5zm.133 11.497H6.987v-2.003h2.039v2.003zm0-3.004H6.987V5.987h2.039v4.006z"></path></svg></span>Role Management Permissions</div><div class="admonitionContent_BuS1"><p>Be careful when assigning permissions to roles that manage roles. These permissions can be used to escalate privileges by assigning additional roles to users. Only assign these permissions to trusted users.</p></div></div>
<h2 class="anchor anchorTargetStickyNavbar_Vzrq" id="permissions">Permissions<a href="#permissions" class="hash-link" aria-label="Direct link to Permissions" title="Direct link to Permissions" translate="no">​</a></h2>
<p>Permissions in Weaviate define what actions users can perform on specific resources. Each permission consists of:</p>
<ul>
<li class="">A resource type (e.g., collections, objects)</li>
<li class="">Access levels (read, write, update, delete, manage)</li>
<li class="">Optional resource-specific constraints</li>
</ul>

<h3 class="anchor anchorTargetStickyNavbar_Vzrq" id="permission-behavior">Permission behavior<a href="#permission-behavior" class="hash-link" aria-label="Direct link to Permission behavior" title="Direct link to Permission behavior" translate="no">​</a></h3>
<p>When defining permissions, setting a permission to <code>False</code> indicates that the permission is <em>not set</em>, rather than explicitly denying access. This means that if a user has multiple roles, and one role grants a permission while another sets it to <code>False</code>, the user will still have that permission through the role that grants it.</p>
<p>For example, if a user has two roles:</p>
<ul>
<li class="">Role A sets <code>read</code> to <code>False</code> for Collection X</li>
<li class="">Role B sets <code>read</code> to <code>True</code> for Collection X</li>
</ul>
<p>The user will have read access to Collection X because Role B grants the permission, while Role A&#x27;s <code>False</code> value simply indicates no permission is set rather than blocking access.</p>
<h3 class="anchor anchorTargetStickyNavbar_Vzrq" id="name-filters-in-permissions">Name filters in permissions<a href="#name-filters-in-permissions" class="hash-link" aria-label="Direct link to Name filters in permissions" title="Direct link to Name filters in permissions" translate="no">​</a></h3>
<p>Some permissions require a collection name filter to specify which collections the permission applies to.</p>
<p>In this case, <code>&quot;*&quot;</code> acts as a multi-character wildcard. As an example, setting a permission with <code>&quot;Test*&quot;</code> as the collection name filter would apply that permission to all collections that start with <code>Test</code>. Or, setting a permission with <code>&quot;*&quot;</code> as the collection filter would apply that permission to all available collections.</p>
<h3 class="anchor anchorTargetStickyNavbar_Vzrq" id="collection-and-tenant-permissions">Collection and tenant permissions<a href="#collection-and-tenant-permissions" class="hash-link" aria-label="Direct link to Collection and tenant permissions" title="Direct link to Collection and tenant permissions" translate="no">​</a></h3>
<p>A collection permission is independent of tenant permissions.</p>
<p>To have permissions to operate on a tenant that belongs to a collection, the user must have the appropriate tenant-level permissions for that collection. Collection-level permissions, such as that to create collections, do not grant the equivalent tenant-level permissions, such as that to create tenants for that collection.</p>
<p>For example, to create a tenant in a collection called <code>TestCollection</code>, the user must have permission to &quot;create&quot; tenants in that collection. This is separate from the permission to create a collection called <code>TestCollection</code>.</p>
