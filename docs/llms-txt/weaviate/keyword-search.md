class="theme-doc-markdown markdown"><header><h1>Keyword Search (BM25)</h1></header><p>Keyword search is an exact matching-based search using &quot;tokens&quot;, or strings of characters.</p>
<p>It uses the BM25 algorithm, which ranks matching documents according to their relevance to a given search query. At a high level, the BM25 algorithm uses the count of query terms in the document (term frequency) against the overall frequency of the term in the dataset (inverse document frequency) to calculate a relevance score.</p>
<p>More specifically, Weaviate uses the BM25F algorithm, which extends BM25 to support using multiple fields in the search index.</p>
<p>A keyword search determines the best matches based on the matches of exact tokens contained in the query against those of the stored objects. As a result, a keyword search is a good choice when exact matches (e.g. exact domain-specific language, precise categories or tags) are important. For example:</p>
<ul>
<li class="">Searching for documents containing specific technical terms</li>
<li class="">Identifying articles by precise keywords or tags</li>
</ul>
<p>This differs from vector search, which finds semantically similar content even when the exact words don&#x27;t match. You might use keyword search when precision is more important than finding related concepts.</p>
<h2 class="anchor anchorTargetStickyNavbar_Vzrq" id="keyword-search-in-weaviate">Keyword search in Weaviate<a href="#keyword-search-in-weaviate" class="hash-link" aria-label="Direct link to Keyword search in Weaviate" title="Direct link to Keyword search in Weaviate" translate="no">​</a></h2>
<p>In Weaviate, a keyword search will return the objects best matching the query, as measured by <a href="https://en.wikipedia.org/wiki/Okapi_BM25" target="_blank" rel="noopener noreferrer" class="">BM25F</a> &quot;score&quot;.</p>
<div class="theme-admonition theme-admonition-info admonition_xJq3 alert alert--info"><div class="admonitionHeading_Gvgb"><span class="admonitionIcon_Rf37"><svg viewBox="0 0 14 16"><path fill-rule="evenodd" d="M7 2.3c3.14 0 5.7 2.56 5.7 5.7s-2.56 5.7-5.7 5.7A5.71 5.71 0 0 1 1.3 8c0-3.14 2.56-5.7 5.7-5.7zM7 1C3.14 1 0 4.14 0 8s3.14 7 7 7 7-3.14 7-7-3.14-7-7-7zm1 3H6v5h2V4zm0 6H6v2h2v-2z"></path></svg></span>BM25F vs BM25</div><div class="admonitionContent_BuS1"><p>The &quot;F&quot; in BM25F stands for &quot;field&quot;, indicating that it is a field-specific version of BM25. This allows for different weights for different fields, or properties, of the objects.</p><br><p>In Weaviate, they are used interchangeably, as the BM25F algorithm is used to calculate the scores for keyword searches. Here we will refer to it generally as BM25.</p></div></div>
<p>A BM25 score is calculated based on the frequency of the query tokens in the object properties, as well as the length of the object properties and the query.</p>
<p>When an input string such as <code>&quot;A red Nike shoe&quot;</code> is provided as the query, Weaviate will:</p>
<ol>
<li class=""><a href="#tokenization" class="">Tokenize</a> the input (e.g. to <code>[&quot;a&quot;, &quot;red&quot;, &quot;nike&quot;, &quot;shoe&quot;]</code>)</li>
<li class="">Remove any <a href="#stopwords" class="">stopwords</a> (e.g. remove <code>a</code>, to produce <code>[&quot;red&quot;, &quot;nike&quot;, &quot;shoe&quot;]</code>)</li>
<li class="">Determine the BM25 scores against <a href="#selected-properties" class="">selected properties</a> of the database objects, based on the <a href="#bm25-parameters" class="">BM25 parameters</a> and any <a href="#property-boosting" class="">property boosting</a>.</li>
<li class="">Return the objects with the highest BM25 scores as the search results</li>
</ol>
<h3 class="anchor anchorTargetStickyNavbar_Vzrq" id="tokenization">Tokenization<a href="#tokenization" class="hash-link" aria-label="Direct link to Tokenization" title="Direct link to Tokenization" translate="no">​</a></h3>
<p>Tokenization for keyword searches refers to how each source text is split up into individual &quot;tokens&quot; to be compared and matched.</p>
<p>The default tokenization method is <code>word</code>.</p>
<p>Other tokenization methods such as <code>whitespace</code>, <code>lowercase</code>, and <code>field</code> are available, as well as specialized ones such as <code>GSE</code> or <code>kagome_kr</code> for other languages (<a class="" href="/weaviate/config-refs/collections#tokenization">more details</a>).</p>
<p>Set the tokenization option <a class="" href="/weaviate/search/bm25#set-tokenization">in the inverted index configuration</a> for a collection.</p>
<div class="theme-admonition theme-admonition-info admonition_xJq3 alert alert--info"><div class="admonitionHeading_Gvgb"><span class="admonitionIcon_Rf37"><svg viewBox="0 0 14 16"><path fill-rule="evenodd" d="M7 2.3c3.14 0 5.7 2.56 5.7 5.7s-2.56 5.7-5.7 5.7A5.71 5.71 0 0 1 1.3 8c0-3.14 2.56-5.7 5.7-5.7zM7 1C3.14 1 0 4.14 0 8s3.14 7 7 7 7-3.14 7-7-3.14-7-7-7zm1 3H6v5h2V4zm0 6H6v2h2v-2z"></path></svg></span>Tokenization in different contexts</div><div class="admonitionContent_BuS1"><p>The term &quot;tokenization&quot; is used in other contexts such as vectorization, or language generation. Note that each of these typically use different tokenizers to meet different requirements. This results in different sets of tokens, even from the same input text.</p></div></div>
<h3 class="anchor anchorTargetStickyNavbar_Vzrq" id="stopwords">Stopwords<a href="#stopwords" class="hash-link" aria-label="Direct link to Stopwords" title="Direct link to Stopwords" translate="no">​</a></h3>
<p>Stopwords are words that are filtered out before processing text.</p>
<p>Weaviate uses configurable stopwords in calculating the BM25 score. Any tokens that are contained in the stopword list will be ignored from the BM25 score calculation.</p>
<p>See the <a class="" href="/weaviate/config-refs/indexing/inverted-index#stopwords">reference page</a> for more details.</p>
<h3 class="anchor anchorTargetStickyNavbar_Vzrq" id="bm25-parameters">BM25 parameters<a href="#bm25-parameters" class="hash-link" aria-label="Direct link to BM25 parameters" title="Direct link to BM25 parameters" translate="no">​</a></h3>
<p>BM25 is a scoring function used to rank documents based on the query terms appearing in them. It has two main parameters that control its behavior:</p>
<ul>
<li class=""><code>k1</code> (default: 1.2): Controls term frequency saturation. Higher values mean that multiple occurrences of a term continue to increase the score more</li>
<li class=""><code>b</code> (default: 0.75): Controls document length normalization. Values closer to 1 mean more normalization for document length</li>
</ul>
<!-- -->
<p>Set custom <code>k1</code> and <code>b</code> values <a class="" href="/weaviate/manage-collections/inverted-index#set-inverted-index-parameters">for a collection</a>.</p>
<h3 class="anchor anchorTargetStickyNavbar_Vzrq" id="keyword-search-operators">Keyword search operators<a href="#keyword-search-operators" class="hash-link" aria-label="Direct link to Keyword search operators" title="Direct link to Keyword search operators" translate="no">​</a></h3>
<div class="theme-admonition theme-admonition-info admonition_xJq3 alert alert--info"><div class="admonitionHeading_Gvgb"><span class="admonitionIcon_Rf37"><svg viewBox="0 0 14 16"><path fill-rule="evenodd" d="M7 2.3c3.14 0 5.7 2.56 5.7 5.7s-2.56 5.7-5.7 5.7A5.71 5.71 0 0 1 1.3 8c0-3.14 2.56-5.7 5.7-5.7zM7 1C3.14 1 0 4.14 0 8s3.14 7 7 7 7-3.14 7-7-3.14-7-7-7zm1 3H6v5h2V4zm0 6H6v2h2v-2z"></path></svg></span>Added in <code>v1.31</code></div></div>
<p>Search operators define the minimum number of query <a class="" href="/weaviate/search/bm25#set-tokenization">tokens</a> that must be present in the object to be returned.</p>
<p>Conceptually, it works as though a filter is applied to the results of the BM25 score calculation. The available operators are:</p>
<ul>
<li class=""><code>and</code>: All tokens must be present in the object</li>
<li class=""><code>or</code>: At least one token must be present in the object, with the minimum number of tokens being configurable (<code>minimumOrTokensMatch</code>)</li>
</ul>
<p>As an example, a BM25 query of <code>computer networking guide</code> with the <code>and</code> operator would only return objects that contain all of the tokens <code>computer</code>, <code>networking</code>, and <code>guide</code>. In contrast, the same query with the <code>or</code> operator would return objects that contain at least one of those tokens. If the <code>or</code> operator is used with a <code>minimumOrTokensMatch</code> of <code>2</code>, then at least two of the tokens must be present in the object.</p>
<p>If not specified, the default operator is <code>or</code>, with a <code>minimumOrTokensMatch</code> of <code>1</code>. This means that at least one token must be present in the object for it to be returned.</p>
<!-- -->
<img src="/assets/images/bm25_operators_light-fe6a283b72823515e7f70dc0a6fc6d29.png" alt="BM25 operators" class="themedComponent_mlkZ themedComponent--light_NVdE"><img src="/assets/images/bm25_operators_dark-396717691fb634634cf23aafc558c704.png" alt="BM25 operators" class="themedComponent_mlkZ themedComponent--dark_xIcU">
<p>See the <a class="" href="/weaviate/search/bm25#search-operators">how-to page</a> for details on usage.</p>
<h3 class="anchor anchorTargetStickyNavbar_Vzrq" id="selected-properties">Selected properties<a href="#selected-properties" class="hash-link" aria-label="Direct link to Selected properties" title="Direct link to Selected properties" translate="no">​</a></h3>
<p>A BM25 query can optionally specify which object properties are to be included in the score calculations.</p>
<p>By default, all <code>text</code> properties are included in a BM25 calculation. There are two ways to vary this:</p>
<ul>
<li class="">In the collection configuration, <a class="" href="/weaviate/manage-collections/vector-config#property-level-settings">set <code>indexSearchable</code> for a property to <code>false</code></a>. This property will then be ignored in all BM25 searches.</li>
<li class=""><a class="" href="/weaviate/search/bm25#search-on-selected-properties-only">Specify which properties to search at query time</a>. This will only apply for that particular query.</li>
</ul>
<h3 class="anchor anchorTargetStickyNavbar_Vzrq" id="property-boosting">Property Boosting<a href="#property-boosting" class="hash-link" aria-label="Direct link to Property Boosting" title="Direct link to Property Boosting" translate="no">​</a></h3>
<p>Property boosting allows a query apply different weights to different properties when calculating the final BM25 score.</p>
<p>This is useful when certain properties are more important for search than others.</p>
<p>For example, when searching an e-commerce catalog, you could boost the title property and its categories over the product description.</p>
<p><a class="" href="/weaviate/search/bm25#use-weights-to-boost-properties">Set the property weights</a> at query time.</p>
<h2 class="anchor anchorTargetStickyNavbar_Vzrq" id="combining-with-vector-search">Combining with Vector Search<a href="#combining-with-vector-search" class="hash-link" aria-label="Direct link to Combining with Vector Search" title="Direct link to Combining with Vector Search" translate="no">​</a></h2>
<p>Keyword search can be combined with vector search in Weaviate to perform a hybrid search. This allows you to leverage both:</p>
<ul>
<li class="">Exact matching capabilities of keyword search</li>
<li class="">Semantic understanding of vector search</li>
</ul>
<p>See <a class="" href="/weaviate/concepts/search/hybrid-search">Hybrid Search</a> for more information.</p>
<h2 class="anchor anchorTargetStickyNavbar_Vzrq" id="notes-and-best-practices">Notes and Best Practices<a href="#notes-and-best-practices" class="hash-link" aria-label="Direct link to Notes and Best Practices" title="Direct link to Notes and Best Practices" translate="no">​</a></h2>
<p>Here are some key considerations when using keyword search:</p>
<ol>
<li class="">
<p><strong>Tokenization Choice</strong></p>
<ul>
<li class="">Choose based on your data and search requirements. For example, use <code>word</code> tokenization for natural language text, but consider <code>field</code> for URLs or email addresses that need exact matching as a whole.</li>
<li class="">For multilingual content, consider specialized tokenizers like <code>GSE</code> for Chinese/Japanese or <code>kagome_kr</code> for Korean</li>
<li class="">Consider special characters and case sensitivity needs</li>
<li class="">Test your tokenization choice with subsets of your data and queries to ensure it handles special characters and case sensitivity as expected. You could perform these experiments with vectorization disabled to save resources/costs, as the two processes are independent.</li>
</ul>
</li>
<li class="">
<p><strong>Performance Optimization</strong></p>
<ul>
<li class="">Index only the properties you need for search</li>
<li class="">Consider combining keyword search with vector search (i.e. perform a <a class="" href="/weaviate/concepts/search/hybrid-search">hybrid search</a>) as a starting point, especially where you cannot anticipate users&#x27; behavior</li>
</ul>
</li>
<li class="">
<p><strong>Query Optimization</strong></p>
<ul>
<li class="">Consider boosting properties that are more important for search (e.g. title, category) over others (e.g. description)</li>
<li class="">Only modify <code>k1</code> and <code>b</code> values if you have a good reason to do so, as the defaults are generally well-suited for most use cases</li>
</ul>
</li>
</ol>
