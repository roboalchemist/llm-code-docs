markdown"><header><h1>Enable and configure RBAC</h1></header><div class="theme-admonition theme-admonition-info admonition_xJq3 alert alert--info"><div class="admonitionHeading_Gvgb"><span class="admonitionIcon_Rf37"><svg viewBox="0 0 14 16"><path fill-rule="evenodd" d="M7 2.3c3.14 0 5.7 2.56 5.7 5.7s-2.56 5.7-5.7 5.7A5.71 5.71 0 0 1 1.3 8c0-3.14 2.56-5.7 5.7-5.7zM7 1C3.14 1 0 4.14 0 8s3.14 7 7 7 7-3.14 7-7-3.14-7-7-7zm1 3H6v5h2V4zm0 6H6v2h2v-2z"></path></svg></span>Added in <code>v1.29</code></div><div class="admonitionContent_BuS1"><p>Role-based access control (RBAC) is generally available in Weaviate from version <code>v1.29</code>.</p></div></div>
<p>Role-based access control (RBAC) is a method of restricting access to resources based on the roles of users. In Weaviate, RBAC allows you to <strong><a class="" href="/weaviate/configuration/rbac/manage-roles">define roles and assign permissions</a></strong> to those roles. Users can then be assigned to roles and inherit the permissions associated with those roles.</p>
<p>Weaviate comes with a set of predefined roles. These roles are:</p>
<ul>
<li class=""><code>root</code>: The root role has full access to all resources in Weaviate.</li>
<li class=""><code>viewer</code>: The viewer role has read-only access to all resources in Weaviate.</li>
</ul>
<p>The <code>root</code> role can be assigned through the Weaviate configuration file. A predefined role cannot be modified. The user can, however, be assigned additional roles through the Weaviate API.</p>
<div class="theme-admonition theme-admonition-tip admonition_xJq3 alert alert--success"><div class="admonitionHeading_Gvgb"><span class="admonitionIcon_Rf37"><svg viewBox="0 0 12 16"><path fill-rule="evenodd" d="M6.5 0C3.48 0 1 2.19 1 5c0 .92.55 2.25 1 3 1.34 2.25 1.78 2.78 2 4v1h5v-1c.22-1.22.66-1.75 2-4 .45-.75 1-2.08 1-3 0-2.81-2.48-5-5.5-5zm3.64 7.48c-.25.44-.47.8-.67 1.11-.86 1.41-1.25 2.06-1.45 3.23-.02.05-.02.11-.02.17H5c0-.06 0-.13-.02-.17-.2-1.17-.59-1.83-1.45-3.23-.2-.31-.42-.67-.67-1.11C2.44 6.78 2 5.65 2 5c0-2.2 2.02-4 4.5-4 1.22 0 2.36.42 3.22 1.19C10.55 2.94 11 3.94 11 5c0 .66-.44 1.78-.86 2.48zM4 14h5c-.23 1.14-1.3 2-2.5 2s-2.27-.86-2.5-2z"></path></svg></span>Root user requirement in older versions</div><div class="admonitionContent_BuS1"><p>In Weaviate versions <code>v1.30.6</code> and <code>v1.31.0</code> or below, if RBAC is enabled, you must configure at least one user with the built-in root role. Otherwise, Weaviate will not start. This requirement was removed in versions <code>v1.30.7</code> and <code>v1.31.1</code>.</p></div></div>

<h2 class="anchor anchorTargetStickyNavbar_Vzrq" id="rbac-and-performance">RBAC and performance<a href="#rbac-and-performance" class="hash-link" aria-label="Direct link to RBAC and performance" title="Direct link to RBAC and performance" translate="no">​</a></h2>
<p>RBAC is a powerful feature that allows you to define fine-grained access control policies. However, it can also have an impact on performance as each operation must be checked against the user&#x27;s permissions.</p>
<p>The exact performance impact will depend on your setup and use case. In our internal testing, the most significant performance impact was seen for object creation operations.</p>
<p>We did not observe additional performance penalties for using custom roles over the built-in roles.</p>
<p>Here are some tips to optimize performance when using RBAC:</p>
<ul>
<li class="">Monitor object creation performance</li>
<li class="">Use a high availability (i.e. 3+ nodes) setup to distribute the load</li>
</ul>
