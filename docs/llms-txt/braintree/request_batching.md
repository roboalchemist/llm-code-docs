class="main-article-content"><div class="ecmRichText" style="margin-bottom:0px"><h1><span>GraphQL Request Batching</span></h1><p class="markdown-p"><span>
    We are working on an "extension" of GraphQL at Braintree to allow clients to execute a series of
    GraphQL queries in a single HTTP request, and optionally share variables between them. "Extension"
    is in quotes because this feature is intended to maintain full compliance with the GraphQL spec,
    which is intentionally vague around request/response transport format. We will go straight into
    examples here, but feel free to jump to </span><a href="/braintree/graphql/guides/request_batching/#motivation"><span>the end</span></a><span> for more details on the
    motivation behind the feature.
</span></p><p class="markdown-p"><span>Here is a basic example using the Ruby </span><code><span>http</span></code><span> gem:</span></p><div class="styles_tab_container"><ol class="styles_tab_list styles_tab_list_no_label"><div><li class="styles_tab_list_item styles_tab_list_active"></li></div></ol><div class="codeblock-spacing"><button class="copy-button" aria-label="Copy"><span class="copy-status">Copy</span><svg class="snippet__copy-icon" fill="#ffffff" height="24" viewBox="0 0 24 24" width="24" xmlns="http://www.w3.org/2000/svg"><path d="M0 0h24v24H0z" fill="none"></path><path d="M16 1H4c-1.1 0-2 .9-2 2v14h2V3h12V1zm3 4H8c-1.1 0-2 .9-2 2v14c0 1.1.9 2 2 2h11c1.1 0 2-.9 2-2V7c0-1.1-.9-2-2-2zm0 16H8V7h11v14z"></path></svg></button><pre class="language-bash"><code class="language-bash">HTTP
  .basic_auth(
    user: &quot;my_public_key&quot;,
    pass: &quot;my_private_key&quot;
  )
  .headers(
    content_type: &quot;application/vnd+braintree.graphql.batch.v0+json&quot;,
    braintree_version: &quot;2020-05-06&quot;
  )
  .post(
    &quot;https://payments.sandbox.braintree-api.com/graphql&quot;,
    json: [
      {
        query: &#x27;mutation Tokenize($input: TokenizeCreditCardInput!) {
                  tokenizeCreditCard(input: $input) {
                    paymentMethod {
                      id @export(as: &quot;tokenizedId&quot;)
                      usage
                    }
                  }
                }&#x27;,
         variables: {
           input: {
             creditCard: {
               number: &quot;4111111111111111&quot;,
               expirationYear: &quot;2020&quot;,
               expirationMonth: &quot;12&quot;
             }
           }
         }
      },
      {
        query: &#x27;mutation Vault($tokenizedId: ID!) {
                  vaultPaymentMethod(input: {paymentMethodId: $tokenizedId}) {
                    paymentMethod {
                      id
                      usage
                    }
                  }
                }&#x27;
      }
    ]
  )</code></pre></div></div><p class="markdown-p"><span>
    At a high level, this example performs two GraphQL mutations in a single HTTP request to the
    Braintree API. The first mutation is </span><code><span>tokenizeCreditCard</span></code><span>, which exchanges credit card
    details for a single-use payment method, and the second is </span><code><span>vaultPaymentMethod</span></code><span>, which
    converts the single-use payment method from the previous step into a multi-use (vaulted) payment
    method. Let's break down the pieces.
</span></p><p class="markdown-p"><code><span>.basic_auth(...)</span></code><span> sets the authentication for the entire request. In this case, we use
    our public/private key pair.
</span></p><p class="markdown-p"><code><span>.headers(...)</span></code><span> is where things start to diverge from a typical GraphQL request.
    Specifically, notice the </span><code><span>content_type</span></code><span> header:
    </span><code><span>application/vnd+braintree.graphql.batch.v0+json</span></code><span>. This is a custom media type that is
    </span><em><span>required</span></em><span> in order to opt-in to this feature. Without it, we will assume you are trying to
    make a regular GraphQL request, and fail due to invalid syntax.
</span></p><p class="markdown-p"><code><span>.post(...)</span></code><span> sends the request to the Braintree Sandbox GraphQL URL, with a JSON
    request body represented as a Ruby array. Each object in this array corresponds to the GraphQL
    request payload you are used to: i.e. a </span><code><span>query</span></code><span> entry that contains the GraphQL query
    as a string, and an optional </span><code><span>variables</span></code><span> entry that contains a JSON object representing
    variables referenced by the query.
</span></p><p class="markdown-p"><span>
    One thing you may notice is that the second query, </span><code><span>vaultPaymentMethod</span></code><span> declares a
    variable </span><code><span>$tokenizedId: ID!</span></code><span> which is </span><em><span>not</span></em><span> present in the
    </span><code><span>variables</span></code><span> object for that query. In fact, the second query doesn't specify any
    variables at all. The reason this works is because of the </span><code><span>@export</span></code><span> directive in the
    first query, which propagates variables to subsequent queries behind the scenes.
</span></p><p class="markdown-p"><code><span>@export</span></code><span> is a GraphQL
    </span><a href="https://graphql.org/learn/queries/#directives"><span>query directive</span></a><span> which accepts a single
    argument </span><code><span>as</span></code><span> of type </span><code><span>String!</span></code><span>. In this case, we've passed
    </span><code><span>"tokenizedId"</span></code><span> to indicate we want the result of the field </span><code><span>id</span></code><span> to be
    available as </span><code><span>$tokenizedId</span></code><span> to subsequent queries, provided they declare the variable
    first. For instance, the </span><code><span>vaultPaymentMethod</span></code><span> mutation declares the variable on its
    first line: </span><code><span>mutation Vault($tokenizedId: ID!)</span></code><span>, and then references it with
    </span><code><span>vaultPaymentMethod(input: {paymentMethodId: $tokenizedId})</span></code><span>.
</span></p><h2 id="responses" class="test_h2 markdown-h2"><span>Responses</span><a class="icon icon--link hash-link" href="#responses" aria-label="Responses"><img src="https://www.paypalobjects.com/btdevdoc/bundles/4.3.24/_assets/anchor-PKYVW2VT.svg" alt="AnchorIcon"/></a></h2><p class="markdown-p"><span>
    Just as the request is a list of regular GraphQL request payloads, the response is a list of
    regular GraphQL response payloads, in the same order. For instance, if everything goes well, the
    above request would return something like this:

    purposes.
</span></p><p class="markdown-p"><span>

</span></p><h2 id="when-to-use-request-batching" class="test_h2 markdown-h2"><span>When to use Request Batching</span><a class="icon icon--link hash-link" href="#when-to-use-request-batching" aria-label="When to use Request Batching"><img src="https://www.paypalobjects.com/btdevdoc/bundles/4.3.24/_assets/anchor-PKYVW2VT.svg" alt="AnchorIcon"/></a></h2><p class="markdown-p"><span>
    Request batching should only be used in specific scenarios, namely when you need to mix mutations
    and queries in a single request, and/or when you need to pass variables from one to the other. If
    you simply want to perform multiple independent mutations or queries in a single request, GraphQL
    already supports that. For instance, this is a valid query:
</span></p><div><div class="styles_tab_container"><ol class="styles_tab_list"><div><li class="styles_tab_list_item styles_tab_list_active">GraphQL</li></div></ol><div class="codeblock-spacing"><button class="copy-button" aria-label="Copy"><span class="copy-status">Copy</span><svg class="snippet__copy-icon" fill="#ffffff" height="24" viewBox="0 0 24 24" width="24" xmlns="http://www.w3.org/2000/svg"><path d="M0 0h24v24H0z" fill="none"></path><path d="M16 1H4c-1.1 0-2 .9-2 2v14h2V3h12V1zm3 4H8c-1.1 0-2 .9-2 2v14c0 1.1.9 2 2 2h11c1.1 0 2-.9 2-2V7c0-1.1-.9-2-2-2zm0 16H8V7h11v14z"></path></svg></button><pre class="language-graphql"><code class="language-graphql">query MultiQuery {
  ping
  viewer {
    id
  }
}</code></pre></div></div></div><p class="markdown-p"><span>
    This will perform </span><code><span>ping</span></code><span> and </span><code><span>viewer</span></code><em><span>in parallel</span></em><span>. Similarly, the
    following is a valid mutation:
</span></p><div><div class="styles_tab_container"><ol class="styles_tab_list"><div><li class="styles_tab_list_item styles_tab_list_active">Mutation</li></div></ol><div class="codeblock-spacing"><button class="copy-button" aria-label="Copy"><span class="copy-status">Copy</span><svg class="snippet__copy-icon" fill="#ffffff" height="24" viewBox="0 0 24 24" width="24" xmlns="http://www.w3.org/2000/svg"><path d="M0 0h24v24H0z" fill="none"></path><path d="M16 1H4c-1.1 0-2 .9-2 2v14h2V3h12V1zm3 4H8c-1.1 0-2 .9-2 2v14c0 1.1.9 2 2 2h11c1.1 0 2-.9 2-2V7c0-1.1-.9-2-2-2zm0 16H8V7h11v14z"></path></svg></button><pre class="language-graphql"><code class="language-graphql">mutation MultiMutation(
  $customerInput: CreateCustomerInput!
  $tokenizeInput: TokenizeCreditCardInput!

  createCustomer(input: $customerInput) {

      id

    This will perform </span><code><span>createCustomer</span></code><span> and </span><code><span>tokenizeCreditCard</span></code><em><span>sequentially</span></em><span> (top-level mutation fields always run sequentially).
</span></p><p class="markdown-p"><span>
    In general, if your request batching can be replaced with standard GraphQL, you should prefer
    standard GraphQL.
</span></p><h2 id="prior-art" class="test_h2 markdown-h2"><span>Prior Art</span><a class="icon icon--link hash-link" href="#prior-art" aria-label="Prior Art"><img src="https://www.paypalobjects.com/btdevdoc/bundles/4.3.24/_assets/anchor-PKYVW2VT.svg" alt="AnchorIcon"/></a></h2><p class="markdown-p"><span>
    This form of request batching has already been popularized by some GraphQL server implementations,
    for instance
    </span><a href="https://chillicream.com/docs/hotchocolate/v10/execution-engine/batching"><span>Hot Chocolate</span></a><span>
    and </span><a href="https://sangria-graphql.github.io/learn/#batch-executor"><span>Sangria</span></a><span>. Our
    implementation is slightly different in that it only returns a single response instead of
    streaming each query response separately. We may add support for streaming responses in the
    future.
</span></p><h2 id="more-considerations-and-limitations" class="test_h2 markdown-h2"><span>More Considerations and Limitations</span><a class="icon icon--link hash-link" href="#more-considerations-and-limitations" aria-label="More Considerations and Limitations"><img src="https://www.paypalobjects.com/btdevdoc/bundles/4.3.24/_assets/anchor-PKYVW2VT.svg" alt="AnchorIcon"/></a></h2><p class="markdown-p"><span>This is a non-exhaustive list; it may change in future as we add functionality.</span></p><h3 id="all-queries-will-be-executed--even-if-there-are-errors" class="test_h3 markdown-h3"><span>All queries will be executed, even if there are errors.</span><a class="icon icon--link hash-link" href="#all-queries-will-be-executed--even-if-there-are-errors" aria-label="All queries will be executed, even if there are errors."><img src="https://www.paypalobjects.com/btdevdoc/bundles/4.3.24/_assets/anchor-PKYVW2VT.svg" alt="AnchorIcon"/></a></h3><p class="markdown-p"><span>
    As the previous errors example shows, all queries are executed even when previous queries have
    "failed". This is because GraphQL doesn't really have a binary representation of failure.
    Individual fields can contain errors, but that doesn't necessarily affect the rest of the query,
    and often you will be able to proceed with the data you did get. In future iterations of this
    feature, we may allow clients more flexibility in specifying how they want to proceed in the event
    of errors.
</span></p><h3 id="all-queries-are-executed-sequentially" class="test_h3 markdown-h3"><span>All queries are executed sequentially.</span><a class="icon icon--link hash-link" href="#all-queries-are-executed-sequentially" aria-label="All queries are executed sequentially."><img src="https://www.paypalobjects.com/btdevdoc/bundles/4.3.24/_assets/anchor-PKYVW2VT.svg" alt="AnchorIcon"/></a></h3><p class="markdown-p"><span>
    While we do expect clients to see slight performance improvements from this feature due to less
    overall HTTP round-trips, the underlying queries will be executed sequentially. So every query you
    add to your list will introduce that much extra time to your overall response. For that reason, we
    have limited the number of queries you can send at the time of this writing to 5. All queries
    beyond this limit will return an error.
</span></p><h3 id="only-scalar-values-can-be-exported" class="test_h3 markdown-h3"><span>Only scalar values can be exported.</span><a class="icon icon--link hash-link" href="#only-scalar-values-can-be-exported" aria-label="Only scalar values can be exported."><img src="https://www.paypalobjects.com/btdevdoc/bundles/4.3.24/_assets/anchor-PKYVW2VT.svg" alt="AnchorIcon"/></a></h3><p class="markdown-p"><span>
    This is actually a hard requirement due to the distinction between GraphQL input and output types.
    Even if we supported exporting a non-scalar field, it wouldn't be usable as input in subsequent
    queries.
</span></p><h3 id="exported-variables-are-interoperable-with-passed-in-variables" class="test_h3 markdown-h3"><span>Exported variables are interoperable with passed-in variables.</span><a class="icon icon--link hash-link" href="#exported-variables-are-interoperable-with-passed-in-variables" aria-label="Exported variables are interoperable with passed-in variables."><img src="https://www.paypalobjects.com/btdevdoc/bundles/4.3.24/_assets/anchor-PKYVW2VT.svg" alt="AnchorIcon"/></a></h3><p class="markdown-p"><span>
    There's nothing stopping you from referencing both exported variables and those in the
    </span><code><span>variables</span></code><span> object of your query. Just note that variables in the
    </span><code><span>variables</span></code><span> object will override exported ones in the event of a conflict.
</span></p><h3 id="exporting-fields-that-are-under-a-list-node-will-only-export-the-last-value-encountered" class="test_h3 markdown-h3"><span>Exporting fields that are under a list node will only export the last value encountered.</span><a class="icon icon--link hash-link" href="#exporting-fields-that-are-under-a-list-node-will-only-export-the-last-value-encountered" aria-label="Exporting fields that are under a list node will only export the last value encountered."><img src="https://www.paypalobjects.com/btdevdoc/bundles/4.3.24/_assets/anchor-PKYVW2VT.svg" alt="AnchorIcon"/></a></h3><p class="markdown-p"><span>
    We haven't decided how to support accumulating values into a list variable yet (although we're
    exploring some options). For now it will just override the exported variable for each result in
    the list, which is probably not what you want.
</span></p><h3 id="authentication-applies-to-all-queries" class="test_h3 markdown-h3"><span>Authentication applies to all queries.</span><a class="icon icon--link hash-link" href="#authentication-applies-to-all-queries" aria-label="Authentication applies to all queries."><img src="https://www.paypalobjects.com/btdevdoc/bundles/4.3.24/_assets/anchor-PKYVW2VT.svg" alt="AnchorIcon"/></a></h3><p class="markdown-p"><span>
    The examples we used here conveniently all accept public/private key basic authentication.
    However, there may be cases where different query fields will expect different authentication,
    which is not currently supported. We are currently exploring some possible solutions for this.
</span></p><h2 id="motivation" class="test_h2 markdown-h2"><span>Motivation</span><a class="icon icon--link hash-link" href="#motivation" aria-label="Motivation"><img src="https://www.paypalobjects.com/btdevdoc/bundles/4.3.24/_assets/anchor-PKYVW2VT.svg" alt="AnchorIcon"/></a></h2><p class="markdown-p"><span>
    One of our goals designing the Braintree GraphQL API has been to break down our existing API
    features into smaller, composable pieces. For instance, the </span><code><span>Transaction.sale</span></code><span> function
    in our client libraries accepts any combination of </span><code><span>payment_method_token</span></code><span>,
    </span><code><span>payment_method_nonce</span></code><span>, or raw payment method details. In addition, you can pass
    </span><code><span>store_in_vault: true</span></code><span> to indicate that you would like to persist the payment method
    after the transaction completes. This proliferation of options leads to ambiguity about things
    like parameter precedence, and makes it difficult for clients to understand the overall surface
    area of the endpoint.
</span></p><p class="markdown-p"><span>
    In order to avoid this scenario in our GraphQL API, we have designed our schema to only expose the
    basic building blocks of these operations, allowing clients to compose them in whatever way fits
    their needs. After all, one of the core tenets of GraphQL is client flexibility. However,
    previously this placed a significant burden on clients to perform multiple HTTP requests for each
    of these operations, which involves separate response handling for each step. With request
    batching, clients now only have to handle a single HTTP response.
</span></p><p class="markdown-p"><span>
    We will continue to add to this in the future, and would like to hear your feedback. Feel free to
    open an issue on our
    </span><a href="https://github.com/braintree/graphql-api/issues"><span>GraphQL API repo</span></a><span> if you have any
    questions.

We will go straight into examples here, but feel free to jump to [the end](#motivation) for more details on the motivation behind the feature.\n\nHere is a basic example using the Ruby http gem:\n\n```language-ruby\nHTTP .basic_auth( user: \"my_public_key\", pass: \"my_private_key\" ) .headers( content_type: \"application/vnd+braintree.graphql.batch.v0+json\", braintree_version: \"2020-05-06\" ) .post( \"https://payments.sandbox.braintree-api.com/graphql\", json: [ { query: 'mutation Tokenize($input: TokenizeCreditCardInput!) { tokenizeCreditCard(input: $input) { paymentMethod { id @export(as: \"tokenizedId\") usage } } }', variables: { input: { creditCard: { number: \"4111111111111111\", expirationYear: \"2020\", expirationMonth: \"12\" } } } }, { query: 'mutation Vault($tokenizedId: ID!) { vaultPaymentMethod(input: {paymentMethodId: $tokenizedId}) { paymentMethod { id usage } } }' } ] )\n```\nAt a high level, this example performs two GraphQL mutations in a single HTTP request to the Braintree API. The first mutation is tokenizeCreditCard, which exchanges credit card details for a single-use payment method, and the second is vaultPaymentMethod, which converts the single-use payment method from the previous step into a multi-use (vaulted) payment method. Let's break down the pieces.\n\n.basic_auth(...) sets the authentication for the entire request. In this case, we use our public/private key pair.\n\n.headers(...) is where things start to diverge from a typical GraphQL request. Specifically, notice the content_type header: application/vnd+braintree.graphql.batch.v0+json. This is a custom media type that is required in order to opt-in to this feature. Without it, we will assume you are trying to make a regular GraphQL request, and fail due to invalid syntax.\n\n.post(...) sends the request to the Braintree Sandbox GraphQL URL, with a JSON request body represented as a Ruby array. Each object in this array corresponds to the GraphQL request payload you are used to: i.e. a query entry that contains the GraphQL query as a string, and an optional variables entry that contains a JSON object representing variables referenced by the query.\n\nOne thing you may notice is that the second query, vaultPaymentMethod declares a variable $tokenizedId: ID! which is not present in the variables object for that query. In this case, we've passed \"tokenizedId\" to indicate we want the result of the field id to be available as $tokenizedId to subsequent queries, provided they declare the variable first. If you simply want to perform multiple independent mutations or queries in a single request, GraphQL already supports that. For instance, this is a valid query:\n\n### Query\n```graphql\nquery MultiQuery {\n  ping\n  viewer {\n    id\n  }\n}\n\n```\n\nThis will perform ping and viewer  in parallel. Similarly, the following is a valid mutation:\n\n### Mutation\n```graphql\nmutation MultiMutation(\n  $customerInput: CreateCustomerInput!\n  $tokenizeInput: TokenizeCreditCardInput!\n) {\n  createCustomer(input: $customerInput) {\n    customer {\n      id\n    }\n  }\n  tokenizeCreditCard(input: $tokenizeInput) {\n    paymentMethod {\n      id\n    }\n  }\n}\n\n```\n\nThis will perform createCustomer and tokenizeCreditCard  sequentially (top-level mutation fields always run sequentially).\n\nIn general, if your request batching can be replaced with standard GraphQL, you should prefer standard GraphQL.\n\n\n## Prior Art\n\nThis form of request batching has already been popularized by some GraphQL server implementations, for instance [Hot Chocolate](https://chillicream.com/docs/hotchocolate/v10/execution-engine/batching) and [Sangria](https://sangria-graphql.github.io/learn/#batch-executor). Our implementation is slightly different in that it only returns a single response instead of streaming each query response separately. We may add support for streaming responses in the future.\n\n\n## More Considerations and Limitations\n\nThis is a non-exhaustive list; it may change in future as we add functionality.\n\n\n### All queries will be executed, even if there are errors.\n\nAs the previous errors example shows, all queries are executed even when previous queries have \"failed\". This is because GraphQL doesn't really have a binary representation of failure. Individual fields can contain errors, but that doesn't necessarily affect the rest of the query, and often you will be able to proceed with the data you did get. In future iterations of this feature, we may allow clients more flexibility in specifying how they want to proceed in the event of errors.\n\n\n### All queries are executed sequentially.\n\nWhile we do expect clients to see slight performance improvements from this feature due to less overall HTTP round-trips, the underlying queries will be executed sequentially. So every query you add to your list will introduce that much extra time to your overall response. For that reason, we have limited the number of queries you can send at the time of this writing to 5. All queries beyond this limit will return an error.\n\n\n### Only scalar values can be exported.\n\nThis is actually a hard requirement due to the distinction between GraphQL input and output types.
