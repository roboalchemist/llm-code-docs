<h2 id="abstract" class="group mt-12 mb-4 scroll-mt-20 pt-2 text-2xl font-semibold tracking-tight text-foreground"><a class="no-underline text-inherit hover:text-foreground transition-colors" href="#abstract"><strong class="font-semibold text-foreground">Abstract</strong></a><button type="button" aria-label="Copy link to section" class="inline-flex items-center justify-center ml-2 align-middle h-5 w-5 rounded opacity-0 group-hover:opacity-60 hover:!opacity-100 transition-opacity duration-150 text-muted-foreground hover:text-foreground"><svg xmlns="http://www.w3.org/2000/svg" width="24" height="24" viewBox="0 0 24 24" fill="none" stroke="currentColor" stroke-width="2" stroke-linecap="round" stroke-linejoin="round" class="lucide lucide-link2 lucide-link-2 h-3.5 w-3.5" aria-hidden="true"><path d="M9 17H7A5 5 0 0 1 7 7h2"></path><path d="M15 7h2a5 5 0 1 1 0 10h-2"></path><line x1="8" x2="16" y1="12" y2="12"></line></svg></button></h2>
<p class="my-4 leading-7 text-muted-foreground">Agent Identity &amp; Discovery (AID) answers one question: <strong class="font-semibold text-foreground">&quot;Given a domain, where is the agent and which protocol should I speak?&quot;</strong> It does so with a single DNS TXT record at a well-known subdomain: <code>_agent.&lt;domain&gt;</code>.</p>
<p class="my-4 leading-7 text-muted-foreground">This protocol is an intentionally minimal discovery layer. After a client uses AID to find the correct endpoint or package, richer protocols such as the Model Context Protocol (MCP) or the Agent-to-Agent Protocol (A2A) take over for communication and capability negotiation.</p>
<p class="my-4 leading-7 text-muted-foreground">This document defines the record format, a simple client algorithm, and strict security rules. It does not include manifests, capability lists, or runtime orchestration. AID defines a DNS discovery entry point for a specific protocol. It does not attempt to define a universal discovery framework for all agent systems.</p>

<h2 id="2-txt-record-specification" class="group mt-12 mb-4 scroll-mt-20 pt-2 text-2xl font-semibold tracking-tight text-foreground"><a class="no-underline text-inherit hover:text-foreground transition-colors" href="#2-txt-record-specification"><strong class="font-semibold text-foreground">2. TXT Record Specification</strong></a><button type="button" aria-label="Copy link to section" class="inline-flex items-center justify-center ml-2 align-middle h-5 w-5 rounded opacity-0 group-hover:opacity-60 hover:!opacity-100 transition-opacity duration-150 text-muted-foreground hover:text-foreground"><svg xmlns="http://www.w3.org/2000/svg" width="24" height="24" viewBox="0 0 24 24" fill="none" stroke="currentColor" stroke-width="2" stroke-linecap="round" stroke-linejoin="round" class="lucide lucide-link2 lucide-link-2 h-3.5 w-3.5" aria-hidden="true"><path d="M9 17H7A5 5 0 0 1 7 7h2"></path><path d="M15 7h2a5 5 0 1 1 0 10h-2"></path><line x1="8" x2="16" y1="12" y2="12"></line></svg></button></h2>
<p class="my-4 leading-7 text-muted-foreground">A provider <strong class="font-semibold text-foreground">MUST</strong> advertise its agent service by publishing a single DNS TXT record at <code>_agent.&lt;domain&gt;</code>.</p>
<h3 id="21-format" class="group mt-8 mb-3 scroll-mt-20 text-xl font-semibold tracking-tight text-foreground"><a class="no-underline text-inherit hover:text-foreground transition-colors" href="#21-format"><strong class="font-semibold text-foreground">2.1. Format</strong></a><button type="button" aria-label="Copy link to section" class="inline-flex items-center justify-center ml-2 align-middle h-5 w-5 rounded opacity-0 group-hover:opacity-60 hover:!opacity-100 transition-opacity duration-150 text-muted-foreground hover:text-foreground"><svg xmlns="http://www.w3.org/2000/svg" width="24" height="24" viewBox="0 0 24 24" fill="none" stroke="currentColor" stroke-width="2" stroke-linecap="round" stroke-linejoin="round" class="lucide lucide-link2 lucide-link-2 h-3.5 w-3.5" aria-hidden="true"><path d="M9 17H7A5 5 0 0 1 7 7h2"></path><path d="M15 7h2a5 5 0 1 1 0 10h-2"></path><line x1="8" x2="16" y1="12" y2="12"></line></svg></button></h3>
<p class="my-4 leading-7 text-muted-foreground">For AID v1.x, agents <strong class="font-semibold text-foreground">MUST</strong> use DNS TXT records for discovery. TXT was chosen because it is universally supported by every DNS registrar and provider, enabling any developer to deploy AID discovery without requiring specialized DNS infrastructure. Alternative record types such as SVCB ([RFC9460]) and SRV ([RFC2782]) offer more structured semantics but are not yet uniformly available across registrar control panels, limiting real-world deployability. Future AID versions may adopt alternative record types as provider support matures; the <code>_agent</code> label is designed to remain stable across such transitions (see Section 5).</p>
<p class="my-4 leading-7 text-muted-foreground">The record <strong class="font-semibold text-foreground">MUST</strong> be a single string of semicolon-delimited <code>key=value</code> pairs. Clients <strong class="font-semibold text-foreground">SHOULD</strong> <code>trim()</code> leading and trailing whitespace from keys and values. Clients <strong class="font-semibold text-foreground">MUST</strong> ignore unknown keys. If a DNS server splits the record into multiple 255-octet strings, the client <strong class="font-semibold text-foreground">MUST</strong> concatenate them in order before parsing. Keep the total length under 255 bytes when possible.</p>
<p class="my-4 leading-7 text-muted-foreground">Clients <strong class="font-semibold text-foreground">MUST</strong> recognize single-letter lowercase aliases for all keys. A record <strong class="font-semibold text-foreground">MUST NOT</strong> include both a full key and its alias. Key comparisons are case-insensitive.</p>
<p class="my-4 leading-7 text-muted-foreground">Providers <strong class="font-semibold text-foreground">SHOULD</strong> emit the short-key form <code>v,u,p,a,s,d,e,k,i</code> for v1.x records. Long keys remain compatibility-only input for existing parsers and deployments.</p>

<h3 id="23-client-discovery-algorithm" class="group mt-8 mb-3 scroll-mt-20 text-xl font-semibold tracking-tight text-foreground"><a class="no-underline text-inherit hover:text-foreground transition-colors" href="#23-client-discovery-algorithm"><strong class="font-semibold text-foreground">2.3. Client Discovery Algorithm</strong></a><button type="button" aria-label="Copy link to section" class="inline-flex items-center justify-center ml-2 align-middle h-5 w-5 rounded opacity-0 group-hover:opacity-60 hover:!opacity-100 transition-opacity duration-150 text-muted-foreground hover:text-foreground"><svg xmlns="http://www.w3.org/2000/svg" width="24" height="24" viewBox="0 0 24 24" fill="none" stroke="currentColor" stroke-width="2" stroke-linecap="round" stroke-linejoin="round" class="lucide lucide-link2 lucide-link-2 h-3.5 w-3.5" aria-hidden="true"><path d="M9 17H7A5 5 0 0 1 7 7h2"></path><path d="M15 7h2a5 5 0 1 1 0 10h-2"></path><line x1="8" x2="16" y1="12" y2="12"></line></svg></button></h3>
<p class="my-4 leading-7 text-muted-foreground">An AID Client, when given a <code>&lt;domain&gt;</code>, <strong class="font-semibold text-foreground">MUST</strong> perform these steps:</p>
<ol class="my-4 ml-6 list-decimal space-y-1.5 text-muted-foreground">
<li>Normalize domain. If the domain contains non-ASCII characters, convert it to its Punycode A-label representation ([RFC5890]).</li>
<li>DNS lookup. Query the <code>TXT</code> record for <code>_agent.&lt;exact-host-user-entered&gt;</code>. Clients <strong class="font-semibold text-foreground">MUST NOT</strong> implicitly retry parent hosts such as <code>_agent.&lt;parent-domain&gt;</code>. If no record is found or the lookup fails, the client MAY attempt a <code>.well-known</code> fallback on the same exact host (Appendix E). If both fail, stop with an error.</li>
<li>Parse and validate. Parse the returned <code>TXT</code> answers as semicolon-delimited <code>key=value</code> records. Key comparisons <strong class="font-semibold text-foreground">MUST</strong> be case-insensitive. Clients <strong class="font-semibold text-foreground">MUST</strong> recognize single-letter aliases for all keys. If exactly one valid AID record is present at the queried DNS name, use it. If more than one valid AID record is present at the same queried DNS name, clients <strong class="font-semibold text-foreground">MUST</strong> fail due to ambiguity rather than choosing by answer order. Malformed answers do not matter when there is exactly one valid AID record.</li>
<li>Optional metadata. If <code>docs</code> (<code>d</code>) is present, clients MAY display it. If <code>dep</code> (<code>e</code>) is in the future, clients SHOULD warn. If <code>dep</code> is in the past, clients SHOULD fail gracefully.</li>
<li>Endpoint proof. If <code>pka</code> (<code>k</code>) is present, clients <strong class="font-semibold text-foreground">MUST</strong> perform the handshake in Appendix D. Use HTTP Message Signatures (RFC 9421) with Ed25519. Clients SHOULD warn on downgrade if a previously present <code>pka</code> is removed.</li>
<li>Return result. If validation succeeds, return the discovered details. If the client does not support the discovered protocol token, it <strong class="font-semibold text-foreground">MUST</strong> fail with the appropriate error code.</li>
</ol>

<h3 id="24-exact-host-semantics-and-explicit-delegation" class="group mt-8 mb-3 scroll-mt-20 text-xl font-semibold tracking-tight text-foreground"><a class="no-underline text-inherit hover:text-foreground transition-colors" href="#24-exact-host-semantics-and-explicit-delegation"><strong class="font-semibold text-foreground">2.4. Exact-Host Semantics and Explicit Delegation</strong></a><button type="button" aria-label="Copy link to section" class="inline-flex items-center justify-center ml-2 align-middle h-5 w-5 rounded opacity-0 group-hover:opacity-60 hover:!opacity-100 transition-opacity duration-150 text-muted-foreground hover:text-foreground"><svg xmlns="http://www.w3.org/2000/svg" width="24" height="24" viewBox="0 0 24 24" fill="none" stroke="currentColor" stroke-width="2" stroke-linecap="round" stroke-linejoin="round" class="lucide lucide-link2 lucide-link-2 h-3.5 w-3.5" aria-hidden="true"><path d="M9 17H7A5 5 0 0 1 7 7h2"></path><path d="M15 7h2a5 5 0 1 1 0 10h-2"></path><line x1="8" x2="16" y1="12" y2="12"></line></svg></button></h3>
<p class="my-4 leading-7 text-muted-foreground">Discovery is exact-host by default. If the application asks for <code>app.team.example.com</code>, the canonical base query is <code>_agent.app.team.example.com</code>.</p>
<p class="my-4 leading-7 text-muted-foreground">Clients <strong class="font-semibold text-foreground">MUST NOT</strong> walk up the DNS hierarchy looking for <code>_agent.team.example.com</code> or <code>_agent.example.com</code>.</p>
<p class="my-4 leading-7 text-muted-foreground">If an operator wants child hosts to inherit a shared record, that inheritance <strong class="font-semibold text-foreground">MUST</strong> be expressed in DNS for the exact queried name. A common pattern is a <code>CNAME</code> from the child host&#x27;s <code>_agent</code> label to a shared <code>_agent</code> record.</p>
<p class="my-4 leading-7 text-muted-foreground"><strong class="font-semibold text-foreground">Examples:</strong></p>
<pre class="my-4 overflow-x-auto rounded-lg border border-border bg-muted p-4 text-sm"><code class="language-dns">_agent.app.team.example.com. 300 IN TXT &quot;v=aid1;p=mcp;u=https://app.team.example.com/mcp&quot;

_agent.app.team.example.com. 300 IN CNAME _agent.shared.team.example.com.
_agent.shared.team.example.com. 300 IN TXT &quot;v=aid1;p=mcp;u=https://gateway.team.example.com/mcp&quot;
</code></pre>
<p class="my-4 leading-7 text-muted-foreground">Protocol-specific lookups follow the same exact-host rule. If a client explicitly requests <code>mcp</code> for <code>app.team.example.com</code>, it may query <code>_agent._mcp.app.team.example.com</code> before <code>_agent.app.team.example.com</code>, but it <strong class="font-semibold text-foreground">MUST NOT</strong> query parent hosts unless the operator has delegated the exact child name in DNS.</p>
<h3 id="25-exposing-multiple-protocols-non-normative-guidance" class="group mt-8 mb-3 scroll-mt-20 text-xl font-semibold tracking-tight text-foreground"><a class="no-underline text-inherit hover:text-foreground transition-colors" href="#25-exposing-multiple-protocols-non-normative-guidance"><strong class="font-semibold text-foreground">2.5. Exposing Multiple Protocols (Non-Normative Guidance)</strong></a><button type="button" aria-label="Copy link to section" class="inline-flex items-center justify-center ml-2 align-middle h-5 w-5 rounded opacity-0 group-hover:opacity-60 hover:!opacity-100 transition-opacity duration-150 text-muted-foreground hover:text-foreground"><svg xmlns="http://www.w3.org/2000/svg" width="24" height="24" viewBox="0 0 24 24" fill="none" stroke="currentColor" stroke-width="2" stroke-linecap="round" stroke-linejoin="round" class="lucide lucide-link2 lucide-link-2 h-3.5 w-3.5" aria-hidden="true"><path d="M9 17H7A5 5 0 0 1 7 7h2"></path><path d="M15 7h2a5 5 0 1 1 0 10h-2"></path><line x1="8" x2="16" y1="12" y2="12"></line></svg></button></h3>
<p class="my-4 leading-7 text-muted-foreground">The canonical location for discovery is the base record: <code>_agent.&lt;domain&gt;</code>. Providers <strong class="font-semibold text-foreground">MAY</strong> additionally expose distinct agent services (e.g., one for MCP and one for A2A) on protocol-specific subdomains using the underscore form <code>_agent._&lt;proto&gt;.&lt;domain&gt;</code>.</p>
<p class="my-4 leading-7 text-muted-foreground"><strong class="font-semibold text-foreground">Examples:</strong></p>
<pre class="my-4 overflow-x-auto rounded-lg border border-border bg-muted p-4 text-sm"><code class="language-dns">_agent._mcp.example.com. 300 IN TXT &quot;v=aid1;p=mcp;u=...&quot;
_agent._a2a.example.com. 300 IN TXT &quot;v=aid1;p=a2a;u=...&quot;
</code></pre>
<p class="my-4 leading-7 text-muted-foreground"><strong class="font-semibold text-foreground">Client behavior:</strong></p>
<ul class="my-4 ml-6 list-disc space-y-1.5 text-muted-foreground">
<li>By default, clients query the base <code>_agent.&lt;exact-host&gt;</code>.</li>
<li>When a specific protocol is explicitly requested (by the application), clients <strong class="font-semibold text-foreground">MAY</strong> first query the protocol-specific subdomain <code>_agent._&lt;proto&gt;.&lt;exact-host&gt;</code> and, if not found, fall back to the base record for that same exact host.</li>
<li>Providers that publish only the base record remain fully compliant.</li>
</ul>

<h2 id="3-security-rules" class="group mt-12 mb-4 scroll-mt-20 pt-2 text-2xl font-semibold tracking-tight text-foreground"><a class="no-underline text-inherit hover:text-foreground transition-colors" href="#3-security-rules"><strong class="font-semibold text-foreground">3. Security Rules</strong></a><button type="button" aria-label="Copy link to section" class="inline-flex items-center justify-center ml-2 align-middle h-5 w-5 rounded opacity-0 group-hover:opacity-60 hover:!opacity-100 transition-opacity duration-150 text-muted-foreground hover:text-foreground"><svg xmlns="http://www.w3.org/2000/svg" width="24" height="24" viewBox="0 0 24 24" fill="none" stroke="currentColor" stroke-width="2" stroke-linecap="round" stroke-linejoin="round" class="lucide lucide-link2 lucide-link-2 h-3.5 w-3.5" aria-hidden="true"><path d="M9 17H7A5 5 0 0 1 7 7h2"></path><path d="M15 7h2a5 5 0 1 1 0 10h-2"></path><line x1="8" x2="16" y1="12" y2="12"></line></svg></button></h2>
<ul class="my-4 ml-6 list-disc space-y-1.5 text-muted-foreground">
<li><strong class="font-semibold text-foreground">DNSSEC:</strong> Providers <strong class="font-semibold text-foreground">SHOULD</strong> sign their DNS records with DNSSEC. Clients <strong class="font-semibold text-foreground">SHOULD</strong> perform DNSSEC validation when DNSSEC-signed answers are available.</li>
<li><strong class="font-semibold text-foreground">HTTPS:</strong> A <code>remote</code> agent&#x27;s <code>uri</code> <strong class="font-semibold text-foreground">MUST</strong> use <code>https://</code>. Clients <strong class="font-semibold text-foreground">MUST</strong> perform standard TLS certificate and hostname validation.</li>
<li><strong class="font-semibold text-foreground">No Secrets:</strong> The TXT record is public and <strong class="font-semibold text-foreground">MUST NOT</strong> contain any secrets.</li>
<li><strong class="font-semibold text-foreground">Endpoint Proof (PKA):</strong> When the record includes <code>pka</code>/<code>k</code>, clients <strong class="font-semibold text-foreground">MUST</strong> verify server control of the private key using HTTP Message Signatures (RFC 9421) with Ed25519 (Appendix D). Providers <strong class="font-semibold text-foreground">MUST</strong> publish <code>kid</code>/<code>i</code> when <code>pka</code> is present. Clients <strong class="font-semibold text-foreground">SHOULD</strong> warn on downgrade if a domain removes <code>pka</code> after prior discovery.</li>
<li><strong class="font-semibold text-foreground">Local Execution (<code>proto=local</code>) Safeguards:</strong> Clients that support local execution <strong class="font-semibold text-foreground">MUST</strong> implement the following:<!-- -->
<ol class="my-4 ml-6 list-decimal space-y-1.5 text-muted-foreground">
<li><strong class="font-semibold text-foreground">Explicit Consent:</strong> Before the first execution, the client <strong class="font-semibold text-foreground">MUST</strong> display the full, resolved command to the user and require explicit confirmation.</li>
<li><strong class="font-semibold text-foreground">Integrity Check:</strong> The client <strong class="font-semibold text-foreground">MUST</strong> compute and cache a cryptographic fingerprint of the <code>uri</code> and <code>proto</code> values. If these values change on a subsequent lookup, the client <strong class="font-semibold text-foreground">MUST</strong> re-trigger the full consent process.</li>
<li><strong class="font-semibold text-foreground">No Shell Interpretation:</strong> Arguments derived from the <code>uri</code> <strong class="font-semibold text-foreground">MUST</strong> be passed atomically to the underlying OS execution call to prevent command injection.</li>
<li><strong class="font-semibold text-foreground">No Nested Discovery:</strong> Clients <strong class="font-semibold text-foreground">MUST</strong> reject a <code>local</code> execution <code>uri</code> that could be interpreted as a command initiating another AID discovery request.</li>
<li><strong class="font-semibold text-foreground">Sandboxing:</strong> Clients <strong class="font-semibold text-foreground">SHOULD</strong> run local agents within a sandboxed environment with minimum necessary permissions.</li>
</ol>
</li>
<li><strong class="font-semibold text-foreground">Redirect Handling:</strong> If an initial request to the discovered <code>uri</code> returns an HTTP redirect (<code>301</code>, <code>302</code>, <code>307</code>, or <code>308</code>) <strong class="font-semibold text-foreground">to a different origin</strong> (hostname or port), the client <strong class="font-semibold text-foreground">SHOULD</strong> treat this as a potential security risk. Clients <strong class="font-semibold text-foreground">MUST NOT</strong> follow such cross-origin redirects automatically. Implementations MAY either<br/>
<!-- -->a. terminate with <code>ERR_SECURITY</code>, or<br/>
<!-- -->b. require explicit user confirmation before proceeding.</li>
</ul>
<h3 id="31-threat-model" class="group mt-8 mb-3 scroll-mt-20 text-xl font-semibold tracking-tight text-foreground"><a class="no-underline text-inherit hover:text-foreground transition-colors" href="#31-threat-model"><strong class="font-semibold text-foreground">3.1 Threat Model</strong></a><button type="button" aria-label="Copy link to section" class="inline-flex items-center justify-center ml-2 align-middle h-5 w-5 rounded opacity-0 group-hover:opacity-60 hover:!opacity-100 transition-opacity duration-150 text-muted-foreground hover:text-foreground"><svg xmlns="http://www.w3.org/2000/svg" width="24" height="24" viewBox="0 0 24 24" fill="none" stroke="currentColor" stroke-width="2" stroke-linecap="round" stroke-linejoin="round" class="lucide lucide-link2 lucide-link-2 h-3.5 w-3.5" aria-hidden="true"><path d="M9 17H7A5 5 0 0 1 7 7h2"></path><path d="M15 7h2a5 5 0 1 1 0 10h-2"></path><line x1="8" x2="16" y1="12" y2="12"></line></svg></button></h3>
<p class="my-4 leading-7 text-muted-foreground">AID&#x27;s security model addresses the following threat landscape:</p>
<p class="my-4 leading-7 text-muted-foreground"><strong class="font-semibold text-foreground">Assumptions:</strong></p>
<ul class="my-4 ml-6 list-disc space-y-1.5 text-muted-foreground">
<li>DNS resolvers are trusted for transport. DNSSEC is recommended but not required, preserving deployability across all registrars.</li>
<li>HTTPS endpoints are verified via standard TLS certificate validation ([RFC6125]).</li>
<li>The TXT record is public data. No secrets are transmitted via DNS.</li>
</ul>
<p class="my-4 leading-7 text-muted-foreground"><strong class="font-semibold text-foreground">Mitigations provided:</strong></p>
<ul class="my-4 ml-6 list-disc space-y-1.5 text-muted-foreground">
<li><strong class="font-semibold text-foreground">DNS spoofing/cache poisoning:</strong> DNSSEC validation (when available) provides cryptographic proof of record authenticity. Enterprise deployments can mandate DNSSEC via policy modes (Section 3.2).</li>
<li><strong class="font-semibold text-foreground">Endpoint impersonation:</strong> PKA (Appendix D) provides endpoint proof using Ed25519 HTTP Message Signatures ([RFC9421]). AID&#x27;s Ed25519 key material is also compatible with other external key-distribution systems, though AID v1.2 does not depend on or require them.</li>
<li><strong class="font-semibold text-foreground">Downgrade attacks:</strong> Clients track previously seen PKA keys and detect removal or rotation (configurable via policy).</li>
<li><strong class="font-semibold text-foreground">Command injection (local agents):</strong> Local execution safeguards (explicit consent, integrity checks, no shell interpretation) prevent exploitation of <code>proto=local</code> records.</li>
<li><strong class="font-semibold text-foreground">Cross-origin redirect:</strong> Clients reject or flag cross-origin HTTP redirects from discovered endpoints.</li>
</ul>
<p class="my-4 leading-7 text-muted-foreground"><strong class="font-semibold text-foreground">Explicitly out of scope:</strong></p>
<ul class="my-4 ml-6 list-disc space-y-1.5 text-muted-foreground">
<li>Compromised authoritative DNS servers (DNSSEC is the mitigation; AID does not replace it)</li>
<li>Active network attackers between client and HTTPS endpoint (TLS provides this defense)</li>
<li>Revocation of compromised PKA keys beyond DNS record update (future work; see Section 5)</li>
</ul>
<h3 id="32-enterprise-policy-modes" class="group mt-8 mb-3 scroll-mt-20 text-xl font-semibold tracking-tight text-foreground"><a class="no-underline text-inherit hover:text-foreground transition-colors" href="#32-enterprise-policy-modes"><strong class="font-semibold text-foreground">3.2 Enterprise Policy Modes</strong></a><button type="button" aria-label="Copy link to section" class="inline-flex items-center justify-center ml-2 align-middle h-5 w-5 rounded opacity-0 group-hover:opacity-60 hover:!opacity-100 transition-opacity duration-150 text-muted-foreground hover:text-foreground"><svg xmlns="http://www.w3.org/2000/svg" width="24" height="24" viewBox="0 0 24 24" fill="none" stroke="currentColor" stroke-width="2" stroke-linecap="round" stroke-linejoin="round" class="lucide lucide-link2 lucide-link-2 h-3.5 w-3.5" aria-hidden="true"><path d="M9 17H7A5 5 0 0 1 7 7h2"></path><path d="M15 7h2a5 5 0 1 1 0 10h-2"></path><line x1="8" x2="16" y1="12" y2="12"></line></svg></button></h3>
<p class="my-4 leading-7 text-muted-foreground">Clients that expose enterprise controls <strong class="font-semibold text-foreground">SHOULD</strong> provide a preset mode surface and <strong class="font-semibold text-foreground">MAY</strong> also expose the underlying policy knobs directly.</p>
<p class="my-4 leading-7 text-muted-foreground">The normative preset names for v1.2.x are:</p>
<ul class="my-4 ml-6 list-disc space-y-1.5 text-muted-foreground">
<li><code>balanced</code>
<ul class="my-4 ml-6 list-disc space-y-1.5 text-muted-foreground">
<li><code>pka</code>: <code>if-present</code></li>
<li><code>dnssec</code>: <code>prefer</code></li>
<li><code>well-known</code>: <code>auto</code></li>
<li><code>downgrade</code>: <code>warn</code></li>
</ul>
</li>
<li><code>strict</code>
<ul class="my-4 ml-6 list-disc space-y-1.5 text-muted-foreground">
<li><code>pka</code>: <code>require</code></li>
<li><code>dnssec</code>: <code>require</code></li>
<li><code>well-known</code>: <code>disable</code></li>
<li><code>downgrade</code>: <code>fail</code></li>
</ul>
</li>
</ul>
<p class="my-4 leading-7 text-muted-foreground">The underlying policy knobs are:</p>
<ul class="my-4 ml-6 list-disc space-y-1.5 text-muted-foreground">
<li><strong class="font-semibold text-foreground">PKA policy:</strong> <code>if-present | require</code></li>
<li><strong class="font-semibold text-foreground">DNSSEC policy:</strong> <code>off | prefer | require</code></li>
<li><strong class="font-semibold text-foreground">Well-known policy:</strong> <code>auto | disable</code></li>
<li><strong class="font-semibold text-foreground">Downgrade policy:</strong> <code>off | warn | fail</code></li>
</ul>
<p class="my-4 leading-7 text-muted-foreground">Policy semantics:</p>
<ul class="my-4 ml-6 list-disc space-y-1.5 text-muted-foreground">
<li><strong class="font-semibold text-foreground">PKA <code>require</code>:</strong> discovery <strong class="font-semibold text-foreground">MUST</strong> fail with <code>ERR_SECURITY</code> if the selected record does not publish <code>pka</code> and <code>kid</code>.</li>
<li><strong class="font-semibold text-foreground">DNSSEC <code>prefer</code>:</strong> clients <strong class="font-semibold text-foreground">SHOULD</strong> continue when DNSSEC cannot be validated, but <strong class="font-semibold text-foreground">SHOULD</strong> surface a warning.</li>
<li><strong class="font-semibold text-foreground">DNSSEC <code>require</code>:</strong> clients <strong class="font-semibold text-foreground">MUST</strong> fail with <code>ERR_SECURITY</code> when DNSSEC validation is unavailable or unsuccessful for the selected DNS answer.</li>
<li><strong class="font-semibold text-foreground">Well-known <code>disable</code>:</strong> clients <strong class="font-semibold text-foreground">MUST NOT</strong> use <code>/.well-known/agent</code> fallback.</li>
<li><strong class="font-semibold text-foreground">Downgrade <code>warn</code>:</strong> if a previously seen <code>pka</code> disappears or <code>pka</code>/<code>kid</code> changes, clients <strong class="font-semibold text-foreground">SHOULD</strong> surface a warning.</li>
<li><strong class="font-semibold text-foreground">Downgrade <code>fail</code>:</strong> if a previously seen <code>pka</code> disappears or <code>pka</code>/<code>kid</code> changes, clients <strong class="font-semibold text-foreground">MUST</strong> fail with <code>ERR_SECURITY</code>.</li>
</ul>
<p class="my-4 leading-7 text-muted-foreground">If discovery succeeds only through <code>.well-known</code>, that result cannot satisfy <code>dnssec=require</code>.</p>
<p class="my-4 leading-7 text-muted-foreground">Operational rollout guidance for these modes lives in the <a class="font-medium text-foreground underline decoration-border underline-offset-4 hover:decoration-foreground transition-colors" href="/docs/Reference/enterprise_rollout">Enterprise Rollout Playbook</a>.</p>

<h2 id="4-dns-and-caching" class="group mt-12 mb-4 scroll-mt-20 pt-2 text-2xl font-semibold tracking-tight text-foreground"><a class="no-underline text-inherit hover:text-foreground transition-colors" href="#4-dns-and-caching"><strong class="font-semibold text-foreground">4. DNS and Caching</strong></a><button type="button" aria-label="Copy link to section" class="inline-flex items-center justify-center ml-2 align-middle h-5 w-5 rounded opacity-0 group-hover:opacity-60 hover:!opacity-100 transition-opacity duration-150 text-muted-foreground hover:text-foreground"><svg xmlns="http://www.w3.org/2000/svg" width="24" height="24" viewBox="0 0 24 24" fill="none" stroke="currentColor" stroke-width="2" stroke-linecap="round" stroke-linejoin="round" class="lucide lucide-link2 lucide-link-2 h-3.5 w-3.5" aria-hidden="true"><path d="M9 17H7A5 5 0 0 1 7 7h2"></path><path d="M15 7h2a5 5 0 1 1 0 10h-2"></path><line x1="8" x2="16" y1="12" y2="12"></line></svg></button></h2>
<ul class="my-4 ml-6 list-disc space-y-1.5 text-muted-foreground">
<li><strong class="font-semibold text-foreground">Provider TTL:</strong> It is <strong class="font-semibold text-foreground">RECOMMENDED</strong> that providers set a Time-To-Live (TTL) of <strong class="font-semibold text-foreground">300–900 seconds (5–15 minutes)</strong> on their <code>_agent.&lt;domain&gt;</code> TXT record.</li>
<li><strong class="font-semibold text-foreground">Client Caching:</strong> Clients <strong class="font-semibold text-foreground">MUST</strong> respect the TTL of the DNS record they receive and <strong class="font-semibold text-foreground">MUST NOT</strong> cache the record for longer.</li>
</ul>

<p class="my-4 leading-7 text-muted-foreground">The <code>_agent</code> DNS label is the stable discovery identifier for AID across all protocol versions. This ensures that IANA registration of <code>_agent</code> provides long-term stability for operators regardless of future record type transitions.</p>

<h2 id="appendix-d-pka-handshake-normative" class="group mt-12 mb-4 scroll-mt-20 pt-2 text-2xl font-semibold tracking-tight text-foreground"><a class="no-underline text-inherit hover:text-foreground transition-colors" href="#appendix-d-pka-handshake-normative"><strong class="font-semibold text-foreground">Appendix D: PKA Handshake (Normative)</strong></a><button type="button" aria-label="Copy link to section" class="inline-flex items-center justify-center ml-2 align-middle h-5 w-5 rounded opacity-0 group-hover:opacity-60 hover:!opacity-100 transition-opacity duration-150 text-muted-foreground hover:text-foreground"><svg xmlns="http://www.w3.org/2000/svg" width="24" height="24" viewBox="0 0 24 24" fill="none" stroke="currentColor" stroke-width="2" stroke-linecap="round" stroke-linejoin="round" class="lucide lucide-link2 lucide-link-2 h-3.5 w-3.5" aria-hidden="true"><path d="M9 17H7A5 5 0 0 1 7 7h2"></path><path d="M15 7h2a5 5 0 1 1 0 10h-2"></path><line x1="8" x2="16" y1="12" y2="12"></line></svg></button></h2>
<p class="my-4 leading-7 text-muted-foreground">When <code>pka</code>/<code>k</code> is present, clients MUST verify control of the private key. Use HTTP Message Signatures (RFC 9421) with Ed25519.</p>
<p class="my-4 leading-7 text-muted-foreground">Pseudocode (client):</p>
<pre class="my-4 overflow-x-auto rounded-lg border border-border bg-muted p-4 text-sm"><code>function performPKAHandshake(uri, pka, kid):
    nonce = generateRandomBytes(32)
    challenge = base64urlEncode(nonce)
    headers = {
        &quot;AID-Challenge&quot;: challenge,
        &quot;Date&quot;: currentUTCTime()
    }
    response = sendGET(uri, headers)

    if response.status != 200:
        failWith(ERR_SECURITY)

    sigInput = response.headers[&quot;Signature-Input&quot;]
    signature = response.headers[&quot;Signature&quot;]

    created = parseCreated(sigInput)
    if |currentTime() - created| &gt; 300 seconds:
        failWith(ERR_SECURITY)

    pubKey = multibaseDecode(pka)
    if not verifyEd25519(signature, sigInput.coveredFields, pubKey):
        failWith(ERR_SECURITY)
</code></pre>
<p class="my-4 leading-7 text-muted-foreground">Providers MUST include <code>kid</code>/<code>i</code> with <code>pka</code>/<code>k</code>. Clients SHOULD warn on downgrade if a previously present <code>pka</code> is removed.</p>
<h2 id="appendix-e-well-known-fallback-non-normative" class="group mt-12 mb-4 scroll-mt-20 pt-2 text-2xl font-semibold tracking-tight text-foreground"><a class="no-underline text-inherit hover:text-foreground transition-colors" href="#appendix-e-well-known-fallback-non-normative"><strong class="font-semibold text-foreground">Appendix E: .well-known Fallback (Non-Normative)</strong></a><button type="button" aria-label="Copy link to section" class="inline-flex items-center justify-center ml-2 align-middle h-5 w-5 rounded opacity-0 group-hover:opacity-60 hover:!opacity-100 transition-opacity duration-150 text-muted-foreground hover:text-foreground"><svg xmlns="http://www.w3.org/2000/svg" width="24" height="24" viewBox="0 0 24 24" fill="none" stroke="currentColor" stroke-width="2" stroke-linecap="round" stroke-linejoin="round" class="lucide lucide-link2 lucide-link-2 h-3.5 w-3.5" aria-hidden="true"><path d="M9 17H7A5 5 0 0 1 7 7h2"></path><path d="M15 7h2a5 5 0 1 1 0 10h-2"></path><line x1="8" x2="16" y1="12" y2="12"></line></svg></button></h2>
<p class="my-4 leading-7 text-muted-foreground">AID is a DNS-based discovery protocol. The <code>.well-known</code> fallback described here is a non-normative convenience for environments where DNS TXT record creation is restricted (e.g., certain managed hosting platforms). It does not affect the <code>_agent</code> DNS registration or its scope under RFC 8552.</p>
<ul class="my-4 ml-6 list-disc space-y-1.5 text-muted-foreground">
<li>Path: <code>GET https://&lt;domain&gt;/.well-known/agent</code></li>
<li>Format: JSON that mirrors TXT keys, including aliases.</li>
<li>Security: Relies on TLS certificate validation. PKA still applies when present.</li>
<li>Client algorithm: Use DNS first. Fallback only on <code>ERR_NO_RECORD</code> or <code>ERR_DNS_LOOKUP_FAILED</code>.</li>
<li>Errors: Use <code>ERR_FALLBACK_FAILED</code> when the fallback fails or is invalid.</li>
</ul>

bootstrap standard_\n\n**Date:** 6 February 2026\n**Editor:** Agent Community\n**Status:** Final\n\n---\n\n## **Abstract**\n\nAgent Identity \u0026 Discovery (AID) answers one question: **\"Given a domain, where is the agent and which protocol should I speak?\"** It does so with a single DNS TXT record at a well-known subdomain: `_agent.\u003cdomain\u003e`.\n\nThis protocol is an intentionally minimal discovery layer. After a client uses AID to find the correct endpoint or package, richer protocols such as the Model Context Protocol (MCP) or the Agent-to-Agent Protocol (A2A) take over for communication and capability negotiation.\n\nThis document defines the record format, a simple client algorithm, and strict security rules. It does not include manifests, capability lists, or runtime orchestration. AID defines a DNS discovery entry point for a specific protocol. It does not attempt to define a universal discovery framework for all agent systems.\n\n---\n\n## **0. Glossary**\n\n| Term                  | Meaning                                                                                    |\n| --------------------- | ------------------------------------------------------------------------------------------ |\n| **AID Client**        | Any software that performs discovery according to this specification.                      |\n| **Provider**          | The entity that controls a domain and is responsible for publishing the AID TXT record.    |\n| **\\_agent subdomain** | The well-known subdomain `_agent.\u003cdomain\u003e` where the AID TXT record **MUST** be published. |\n| **A-label**           | The Punycode representation of an Internationalized Domain Name (IDN), as per [RFC5890].   |\n\nThe key words \"MUST\", \"MUST NOT\", \"REQUIRED\", \"SHALL\", \"SHALL NOT\", \"SHOULD\", \"SHOULD NOT\", \"RECOMMENDED\", \"NOT RECOMMENDED\", \"MAY\", and \"OPTIONAL\" in this document are to be interpreted as described in [RFC2119] and [RFC8174].\n\n---\n\n## **1. Design Goals**\n\n- **Zero-Configuration:** A user types a domain, and the client automatically discovers how to connect.\n- **Decentralized and Deployable:** The standard uses DNS TXT records, which are universally supported and require no central registry.\n- **Protocol-Agnostic:** AID discovers agents speaking any protocol, including MCP, A2A, OpenAPI, or even local package-based protocols.\n- **Clear Upgrade Path:** A future AID version may adopt a more structured DNS record type, such as SRV or SVCB, while preserving the same `_agent` discovery label established in this specification.\n\n---\n\n## **2. TXT Record Specification**\n\nA provider **MUST** advertise its agent service by publishing a single DNS TXT record at `_agent.\u003cdomain\u003e`.\n\n### **2.1. Format**\n\nFor AID v1.x, agents **MUST** use DNS TXT records for discovery. TXT was chosen because it is universally supported by every DNS registrar and provider, enabling any developer to deploy AID discovery without requiring specialized DNS infrastructure. Alternative record types such as SVCB ([RFC9460]) and SRV ([RFC2782]) offer more structured semantics but are not yet uniformly available across registrar control panels, limiting real-world deployability. Future AID versions may adopt alternative record types as provider support matures; the `_agent` label is designed to remain stable across such transitions (see Section 5).\n\nThe record **MUST** be a single string of semicolon-delimited `key=value` pairs. Clients **SHOULD** `trim()` leading and trailing whitespace from keys and values. Clients **MUST** ignore unknown keys. If a DNS server splits the record into multiple 255-octet strings, the client **MUST** concatenate them in order before parsing. Keep the total length under 255 bytes when possible.\n\nClients **MUST** recognize single-letter lowercase aliases for all keys. A record **MUST NOT** include both a full key and its alias. Key comparisons are case-insensitive.\n\nProviders **SHOULD** emit the short-key form `v,u,p,a,s,d,e,k,i` for v1.x records. Long keys remain compatibility-only input for existing parsers and deployments.\n\n| Key       | Alias | Requirement  | Description                                                                                           | Example                            |\n| --------- | ----- | ------------ | ----------------------------------------------------------------------------------------------------- | ---------------------------------- |\n| `version` | `v`   | **Required** | The specification version. For v1 it **MUST** be `aid1`.                                              | `v=aid1`                           |\n| `uri`     | `u`   | **Required** | An absolute `https://` URL for a remote agent, or a package/locator for local agents. Examples**\n\n**Remote MCP Agent:**\n\n```text\n_agent.example.com. 300 IN TXT \"v=aid1;u=https://api.example.com/mcp;p=mcp;a=pat;s=Example AI Tools\"\n```\n\n**Local Agent via Docker:**\n\n```text\n_agent.grafana.com. 300 IN TXT \"v=aid1;u=docker:grafana/mcp:latest;p=local;a=pat;s=Run Grafana agent locally\"\n```\n\n**Remote MCP with PKA and metadata (v1.2):**\n\n```text\n_agent.example.com. 300 IN TXT \"v=aid1;p=mcp;u=https://api.example.com/mcp;k=z7rW8rTq8o4mM6vVf7w1k3m4uQn9p2YxCAbcDeFgHiJ;i=g1;d=https://docs.example.com/agent;e=2026-01-01T00:00:00Z;s=Secure AI Gateway\"\n```\n\n**Local Zeroconf (v1.2):**\n\n```text\n_agent.local.test. 300 IN TXT \"v=aid1;p=zeroconf;u=zeroconf:_mcp._tcp;s=Local Dev Agent\"\n```\n\n### **2.3. Client Discovery Algorithm**\n\nAn AID Client, when given a `\u003cdomain\u003e`, **MUST** perform these steps:\n\n1.  Normalize domain. If the domain contains non-ASCII characters, convert it to its Punycode A-label representation ([RFC5890]).\n2.  DNS lookup. Query the `TXT` record for `_agent.\u003cexact-host-user-entered\u003e`. Clients **MUST NOT** implicitly retry parent hosts such as `_agent.\u003cparent-domain\u003e`. If no record is found or the lookup fails, the client MAY attempt a `.well-known` fallback on the same exact host (Appendix E). If both fail, stop with an error.\n3.  Parse and validate. Parse the returned `TXT` answers as semicolon-delimited `key=value` records. Key comparisons **MUST** be case-insensitive. Clients **MUST** recognize single-letter aliases for all keys. If exactly one valid AID record is present at the queried DNS name, use it. If more than one valid AID record is present at the same queried DNS name, clients **MUST** fail due to ambiguity rather than choosing by answer order. Malformed answers do not matter when there is exactly one valid AID record.\n4.  Optional metadata. If `docs` (`d`) is present, clients MAY display it. If `dep` (`e`) is in the future, clients SHOULD warn. If `dep` is in the past, clients SHOULD fail gracefully.\n5.  Endpoint proof. If `pka` (`k`) is present, clients **MUST** perform the handshake in Appendix D. Use HTTP Message Signatures (RFC 9421) with Ed25519. Clients SHOULD warn on downgrade if a previously present `pka` is removed.\n6.  Return result. If validation succeeds, return the discovered details. If the client does not support the discovered protocol token, it **MUST** fail with the appropriate error code.\n\n#### **Table 1: Standard Client Error Codes**\n\nClient implementations **SHOULD** use these codes to report specific failure modes. Exact-Host Semantics and Explicit Delegation**\n\nDiscovery is exact-host by default. If the application asks for `app.team.example.com`, the canonical base query is `_agent.app.team.example.com`.\n\nClients **MUST NOT** walk up the DNS hierarchy looking for `_agent.team.example.com` or `_agent.example.com`.\n\nIf an operator wants child hosts to inherit a shared record, that inheritance **MUST** be expressed in DNS for the exact queried name. A common pattern is a `CNAME` from the child host's `_agent` label to a shared `_agent` record.\n\n**Examples:**\n\n```dns\n_agent.app.team.example.com. 300 IN TXT \"v=aid1;p=mcp;u=https://app.team.example.com/mcp\"\n\n_agent.app.team.example.com. 300 IN CNAME _agent.shared.team.example.com.\n_agent.shared.team.example.com. 300 IN TXT \"v=aid1;p=mcp;u=https://gateway.team.example.com/mcp\"\n```\n\nProtocol-specific lookups follow the same exact-host rule. If a client explicitly requests `mcp` for `app.team.example.com`, it may query `_agent._mcp.app.team.example.com` before `_agent.app.team.example.com`, but it **MUST NOT** query parent hosts unless the operator has delegated the exact child name in DNS.\n\n### **2.5. Exposing Multiple Protocols (Non-Normative Guidance)**\n\nThe canonical location for discovery is the base record: `_agent.\u003cdomain\u003e`. Providers **MAY** additionally expose distinct agent services (e.g., one for MCP and one for A2A) on protocol-specific subdomains using the underscore form `_agent._\u003cproto\u003e.\u003cdomain\u003e`.\n\n**Examples:**\n\n```dns\n_agent._mcp.example.com. 300 IN TXT \"v=aid1;p=mcp;u=...\"\n_agent._a2a.example.com. 300 IN TXT \"v=aid1;p=a2a;u=...\"\n```\n\n**Client behavior:**\n\n- By default, clients query the base `_agent.\u003cexact-host\u003e`.\n- When a specific protocol is explicitly requested (by the application), clients **MAY** first query the protocol-specific subdomain `_agent._\u003cproto\u003e.\u003cexact-host\u003e` and, if not found, fall back to the base record for that same exact host.\n- Providers that publish only the base record remain fully compliant.\n\n---\n\n## **3. Security Rules**\n\n- **DNSSEC:** Providers **SHOULD** sign their DNS records with DNSSEC. Clients **SHOULD** perform DNSSEC validation when DNSSEC-signed answers are available.\n- **HTTPS:** A `remote` agent's `uri` **MUST** use `https://`. Clients **MUST** perform standard TLS certificate and hostname validation.\n- **No Secrets:** The TXT record is public and **MUST NOT** contain any secrets.\n- **Endpoint Proof (PKA):** When the record includes `pka`/`k`, clients **MUST** verify server control of the private key using HTTP Message Signatures (RFC 9421) with Ed25519 (Appendix D). Providers **MUST** publish `kid`/`i` when `pka` is present. Clients **SHOULD** warn on downgrade if a domain removes `pka` after prior discovery.\n- **Local Execution (`proto=local`) Safeguards:** Clients that support local execution **MUST** implement the following:\n  1.  **Explicit Consent:** Before the first execution, the client **MUST** display the full, resolved command to the user and require explicit confirmation.\n  2.  **Integrity Check:** The client **MUST** compute and cache a cryptographic fingerprint of the `uri` and `proto` values. If these values change on a subsequent lookup, the client **MUST** re-trigger the full consent process.\n  3.  **No Shell Interpretation:** Arguments derived from the `uri` **MUST** be passed atomically to the underlying OS execution call to prevent command injection.\n  4.  **No Nested Discovery:** Clients **MUST** reject a `local` execution `uri` that could be interpreted as a command initiating another AID discovery request.\n  5.  **Sandboxing:** Clients **SHOULD** run local agents within a sandboxed environment with minimum necessary permissions.\n- **Redirect Handling:** If an initial request to the discovered `uri` returns an HTTP redirect (`301`, `302`, `307`, or `308`) **to a different origin** (hostname or port), the client **SHOULD** treat this as a potential security risk. Clients **MUST NOT** follow such cross-origin redirects automatically. Implementations MAY either  \n  a. terminate with `ERR_SECURITY`, or  \n  b. require explicit user confirmation before proceeding.\n\n### **3.1 Threat Model**\n\nAID's security model addresses the following threat landscape:\n\n**Assumptions:**\n- DNS resolvers are trusted for transport. DNSSEC is recommended but not required, preserving deployability across all registrars.\n- HTTPS endpoints are verified via standard TLS certificate validation ([RFC6125]).\n- The TXT record is public data. No secrets are transmitted via DNS.\n\n**Mitigations provided:**\n- **DNS spoofing/cache poisoning:** DNSSEC validation (when available) provides cryptographic proof of record authenticity. Enterprise deployments can mandate DNSSEC via policy modes (Section 3.2).\n- **Endpoint impersonation:** PKA (Appendix D) provides endpoint proof using Ed25519 HTTP Message Signatures ([RFC9421]). AID's Ed25519 key material is also compatible with other external key-distribution systems, though AID v1.2 does not depend on or require them.\n- **Downgrade attacks:** Clients track previously seen PKA keys and detect removal or rotation (configurable via policy).\n- **Command injection (local agents):** Local execution safeguards (explicit consent, integrity checks, no shell interpretation) prevent exploitation of `proto=local` records.\n- **Cross-origin redirect:** Clients reject or flag cross-origin HTTP redirects from discovered endpoints.\n\n**Explicitly out of scope:**\n- Compromised authoritative DNS servers (DNSSEC is the mitigation; AID does not replace it)\n- Active network attackers between client and HTTPS endpoint (TLS provides this defense)\n- Revocation of compromised PKA keys beyond DNS record update (future work; see Section 5)\n\n### **3.2 Enterprise Policy Modes**\n\nClients that expose enterprise controls **SHOULD** provide a preset mode surface and **MAY** also expose the underlying policy knobs directly.\n\nThe normative preset names for v1.2.x are:\n\n- `balanced`\n  - `pka`: `if-present`\n  - `dnssec`: `prefer`\n  - `well-known`: `auto`\n  - `downgrade`: `warn`\n- `strict`\n  - `pka`: `require`\n  - `dnssec`: `require`\n  - `well-known`: `disable`\n  - `downgrade`: `fail`\n\nThe underlying policy knobs are:\n\n- **PKA policy:** `if-present | require`\n- **DNSSEC policy:** `off | prefer | require`\n- **Well-known policy:** `auto | disable`\n- **Downgrade policy:** `off | warn | fail`\n\nPolicy semantics:\n\n- **PKA `require`:** discovery **MUST** fail with `ERR_SECURITY` if the selected record does not publish `pka` and `kid`.\n- **DNSSEC `prefer`:** clients **SHOULD** continue when DNSSEC cannot be validated, but **SHOULD** surface a warning.\n- **DNSSEC `require`:** clients **MUST** fail with `ERR_SECURITY` when DNSSEC validation is unavailable or unsuccessful for the selected DNS answer.\n- **Well-known `disable`:** clients **MUST NOT** use `/.well-known/agent` fallback.\n- **Downgrade `warn`:** if a previously seen `pka` disappears or `pka`/`kid` changes, clients **SHOULD** surface a warning.\n- **Downgrade `fail`:** if a previously seen `pka` disappears or `pka`/`kid` changes, clients **MUST** fail with `ERR_SECURITY`.\n\nIf discovery succeeds only through `.well-known`, that result cannot satisfy `dnssec=require`.\n\nOperational rollout guidance for these modes lives in the [Enterprise Rollout Playbook](Reference/enterprise_rollout.md).\n\n---\n\n## **4. DNS and Caching**\n\n- **Provider TTL:** It is **RECOMMENDED** that providers set a Time-To-Live (TTL) of **300–900 seconds (5–15 minutes)** on their `_agent.\u003cdomain\u003e` TXT record.\n- **Client Caching:** Clients **MUST** respect the TTL of the DNS record they receive and **MUST NOT** cache the record for longer.\n\n---\n\n## **5. Future Path**\n\n- **Record Type Evolution:** AID v2 may adopt a more structured DNS record type such as SRV ([RFC2782]) or SVCB ([RFC9460]) for discovery, depending on registrar support at the time of ratification. Until then, TXT lookup remains the canonical method for v1.x.\n- **IANA Registration:** Formal requests will be submitted for the `_agent` underscored DNS node name under [RFC8552] and the `agent` service name under [RFC6335].\n\n### **5.1 Label Stability**\n\nThe `_agent` DNS label is the stable discovery identifier for AID across all protocol versions. If AID evolves from v1 (TXT) to v2 (SRV, SVCB, or another record type), the `_agent` label remains the stable identifier. Clients use the version field (`v=aid1`) to determine record format. This ensures that IANA registration of `_agent` provides long-term stability for operators regardless of future record type transitions.\n\n---\n\n## **6.
