<details><summary><strong>Some notes on subpath versioning</strong></summary><p>Ultimately, the subpath versioning scheme was a necessary evil to force the ecosystem to upgrade in a non-breaking way. If I&#x27;d published <code><a href="/cdn-cgi/l/email-protection" class="__cf_email__" data-cfemail="82f8ede6c2b6acb2acb2">[email&#160;protected]</a></code> out of the gate, most libraries would have naively bumped their peer dependencies, forcing a &quot;version bump avalanche&quot; across the ecosystem.</p><p>As it stands, there is now <a href="https://x.com/colinhacks/status/1932323805705482339" rel="noreferrer noopener" target="_blank">broad support</a> for Zod 4 across the ecosystem. No migration process is totally painless, but it seems like the &quot;version avalanche&quot; I&#x27;d feared didn&#x27;t happen. By and large, libraries have been able to support Zod 3 and Zod 4 simultaneously: Hono, LangChain, React Hook Form, etc. Several ecosystem maintainers reached out to me specifically to indicate how convenient it was to incrementally add support for Zod 4 (something that would typically require a major version bump). Long story short: this approach worked great! Few other libraries are subject to the same constraints as Zod, but I strongly encourage other libraries with large associated ecosystems to consider a similar approach.</p></details>
<h2 class="flex scroll-m-28 flex-row items-center gap-2 undefined" id="versioning-in-zod-4"><a data-card="" class="peer" href="?id=versioning-in-zod-4">Versioning in Zod 4</a><svg xmlns="http://www.w3.org/2000/svg" width="24" height="24" viewBox="0 0 24 24" fill="none" stroke="currentColor" stroke-width="2" stroke-linecap="round" stroke-linejoin="round" class="lucide lucide-link size-3.5 shrink-0 text-fd-muted-foreground opacity-0 transition-opacity peer-hover:opacity-100" aria-label="Link to section"><path d="M10 13a5 5 0 0 0 7.54.54l3-3a5 5 0 0 0-7.07-7.07l-1.72 1.71"></path><path d="M14 11a5 5 0 0 0-7.54-.54l-3 3a5 5 0 0 0 7.07 7.07l1.71-1.71"></path></svg></h2>
<p>This is a writeup of Zod 4&#x27;s approach to versioning, with the goal of making it easier for users and Zod&#x27;s ecosystem of associated libraries to migrate to Zod 4.</p>
<p>The general approach:</p>
<ul>
<li>Zod 4 will not initially be published as <code><a href="/cdn-cgi/l/email-protection" class="__cf_email__" data-cfemail="403a2f2400746e706e70">[email&#160;protected]</a></code> on npm. Instead it will be exported at a subpath (<code>&quot;zod/v4&quot;</code>) alongside <code><a href="/cdn-cgi/l/email-protection" class="__cf_email__" data-cfemail="067c696246352834332836">[email&#160;protected]</a></code></li>
<li>Despite this, Zod 4 is considered stable and production-ready</li>
<li>Zod 3 will continue to be exported from the package root (<code>&quot;zod&quot;</code>) as well as a new subpath <code>&quot;zod/v3&quot;</code>. It will continue to receive bug fixes &amp; stability improvements.</li>
</ul>
<div class="my-6 flex flex-row gap-2 rounded-lg border border-s-2 bg-fd-card p-3 text-sm text-fd-card-foreground shadow-md border-s-blue-500/50"><svg xmlns="http://www.w3.org/2000/svg" width="24" height="24" viewBox="0 0 24 24" fill="none" stroke="currentColor" stroke-width="2" stroke-linecap="round" stroke-linejoin="round" class="lucide lucide-info size-5 fill-blue-500 text-fd-card"><circle cx="12" cy="12" r="10"></circle><path d="M12 16v-4"></path><path d="M12 8h.01"></path></svg><div class="min-w-0 flex-1"><div class="text-fd-muted-foreground prose-no-margin">
<p>This approach is analogous to how Golang handles major version changes: <a href="https://go.dev/doc/modules/major-version" rel="noreferrer noopener" target="_blank">https://go.dev/doc/modules/major-version</a></p>
</div></div></div>
<p>Sometime later:</p>
<ul>
<li>The package root (<code>&quot;zod&quot;</code>) will switch over from exporting Zod 3 to Zod 4</li>
<li>At this point <code><a href="/cdn-cgi/l/email-protection" class="__cf_email__" data-cfemail="a3d9ccc7e3978d938d93">[email&#160;protected]</a></code> will get published to npm</li>
<li>The <code>&quot;zod/v4&quot;</code> subpath will remain available forever</li>
</ul>
<h2 class="flex scroll-m-28 flex-row items-center gap-2 undefined" id="why"><a data-card="" class="peer" href="?id=why">Why?</a><svg xmlns="http://www.w3.org/2000/svg" width="24" height="24" viewBox="0 0 24 24" fill="none" stroke="currentColor" stroke-width="2" stroke-linecap="round" stroke-linejoin="round" class="lucide lucide-link size-3.5 shrink-0 text-fd-muted-foreground opacity-0 transition-opacity peer-hover:opacity-100" aria-label="Link to section"><path d="M10 13a5 5 0 0 0 7.54.54l3-3a5 5 0 0 0-7.07-7.07l-1.72 1.71"></path><path d="M14 11a5 5 0 0 0-7.54-.54l-3 3a5 5 0 0 0 7.07 7.07l1.71-1.71"></path></svg></h2>
<p>Zod occupies a unique place in the ecosystem. Many libraries/frameworks in the ecosystem accept user-defined Zod schemas. This means their user-facing API is strongly coupled to Zod and its various classes/interfaces/utilities. For these libraries/frameworks, a breaking change to Zod necessarily causes a breaking change for their users. A Zod 3 <code>ZodType</code> is not assignable to a Zod 4 <code>ZodType</code>.</p>
<h3 class="flex scroll-m-28 flex-row items-center gap-2 undefined" id="why-cant-libraries-just-support-v3-and-v4-simultaneously"><a data-card="" class="peer" href="?id=why-cant-libraries-just-support-v3-and-v4-simultaneously">Why can&#x27;t libraries just support v3 and v4 simultaneously?</a><svg xmlns="http://www.w3.org/2000/svg" width="24" height="24" viewBox="0 0 24 24" fill="none" stroke="currentColor" stroke-width="2" stroke-linecap="round" stroke-linejoin="round" class="lucide lucide-link size-3.5 shrink-0 text-fd-muted-foreground opacity-0 transition-opacity peer-hover:opacity-100" aria-label="Link to section"><path d="M10 13a5 5 0 0 0 7.54.54l3-3a5 5 0 0 0-7.07-7.07l-1.72 1.71"></path><path d="M14 11a5 5 0 0 0-7.54-.54l-3 3a5 5 0 0 0 7.07 7.07l1.71-1.71"></path></svg></h3>
<p>Unfortunately the limitations of peerDependencies (and inconsistencies between package managers) make it extremely difficult to elegantly support two major versions of one library simultaneously.</p>
<p>If I naively published <code><a href="/cdn-cgi/l/email-protection" class="__cf_email__" data-cfemail="3f45505b7f0b110f110f">[email&#160;protected]</a></code> to npm, the vast majority of the libraries in Zod&#x27;s ecosystem would need to publish a new major version to properly support Zod 4, include some high-profile libraries like the AI SDK. It would trigger a &quot;version bump avalanche&quot; across the ecosystem and generally create a huge amount of frustration and work.</p>
<p>With subpath versioning, we solve this problem. it provides a straightforward way for libraries to support Zod 3 and Zod 4 (including Zod Mini) simultaneously. They can continue defining a single peerDependency on <code>&quot;zod&quot;</code>; no need for more arcane solutions like npm aliases, optional peer dependencies, a <code>&quot;zod-compat&quot;</code> package, or other such hacks.</p>
<p>Libraries will need to bump the minimum version of their <code>&quot;zod&quot;</code> peer dependency to <code>zod@^3.25.0</code>. They can then reference both Zod 3 and Zod 4 in their implementation:</p>

<p>Later, once there&#x27;s broad support for v4, we&#x27;ll bump the major version on <code>npm</code> and start exporting Zod 4 from the package root, completing the transition. (This has now happened—see the note at the top of this page.)</p>
<p>As long as libraries are importing exclusively from the associated subpaths (not the root), their implementations will continue to work across the major version bump without code changes.</p>
<p>While it may seem unorthodox (at least for people who don&#x27;t use Go!), this is the only approach I&#x27;m aware of that enables a clean, incremental migration path for both Zod&#x27;s users and the libraries in the broader ecosystem.</p>
<hr/>
<p>A deeper dive into why peer dependencies don&#x27;t work in this situation.</p>
<p>Imagine you&#x27;re a library trying to build a function <code>acceptSchema</code> that accepts a Zod schema. You want to be able to accept Zod 3 or Zod 4 schemas.  In this hypothetical, I&#x27;m imagine Zod 4 was published as <code>zod@4</code> on npm, no subpaths. Here are your options:</p>
<ol>
<li>
<p>Install both zod@3 and zod@4 as <code>dependencies</code> simultaneously using npm aliases. This works but you end up including your own copies of both Zod 3 and Zod 4. You have no guarantee that your user&#x27;s Zod schemas are instances of the same z.ZodType class you&#x27;re pulling from dependencies (<code>instanceof</code> checks will probably fail).</p>
</li>
<li>
<p>Use a peer dependency that spans multiple major versions:  <code>&quot;zod@&gt;=3.0.0&quot;</code> …but when developing a library you’d still need to pick a version to develop against. Usually you&#x27;d install this as a dev dependency. The onus is on you to painstakingly ensure your code works, character-for-character,  across both versions. This is impossible in the case of Zod 3 &amp; Zod 4 because a number of very fundamental classes have simplified/different generics.</p>
</li>
<li>
<p>Optional peer dependencies. i just couldn&#x27;t find a straight answer about how to reliably determine which peer dep is installed at runtime across all platforms. Many answers online will say &quot;use dynamic imports in a try/catch to check it a package exists&quot;. Those folks are assuming you&#x27;re on the backend because no frontend bundlers have no affordance for this. They&#x27;ll fail when you try to bundle a dependency that isn&#x27;t installed. Obviuosly it doesn&#x27;t matter if you&#x27;re inside a try/catch during a build step. Also: since we&#x27;re talking about multiple versions of the same library, you&#x27;d need to use npm aliases to differentiate the two versions in your <code>package.json</code>. Versions of npm as recent as v10 cannot handle the combination of peer dependencies + npm aliases.</p>
</li>
<li>
<p><code>zod-compat</code>. This extremely hand-wavy solution you see online is &quot;define interfaces for each version that represents some basic functionality&quot;. Basically some utility types libraries can use to approximate the real deal. This is error prone, a ton of work, needs to be kept synchronized with the real implementations, and ultimately libraries are developing against a shadow version of your library that probably lacks detail. It also only works for types: if a library depends on any runtime code in Zod it falls apart.</p>
</li>
</ol>
