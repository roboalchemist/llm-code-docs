<p>With token rotation, you'll provide an extra layer of security for your access tokens.</p>
<h2 class="anchor anchorTargetStickyNavbar_Vzrq" id=overview>Overview<a href=#overview class=hash-link aria-label="Direct link to Overview" title="Direct link to Overview" translate=no>​</a></h2>
<p>An <a class="" href=/authentication/tokens>access token</a> represents an installation of your Slack app. Without token rotation, the access token never expires. With token rotation, it expires every 12 hours.</p>
<p>An OAuth flow with token rotation involves exchanging one expiring access token for a new one, using an additional token: the refresh token. The refresh token is then revoked, and a new refresh token is used to exchange the new expiring access token when it expires.</p>
<p>New access and refresh tokens need to be <em>rotated</em> in throughout the lifespan of the app for it to continue to function.</p>

<p>Before you turn on token rotation, consider how you're going to test your implementation. You don't want all your access tokens to expire before you've successfully figured out how to refresh them.</p>
<p>If your app <em>is not</em> published in the Slack Marketplace, token rotation is enabled via the <strong>OAuth & Permissions</strong> section of your <a href=https://api.slack.com/apps target=_blank rel="noopener noreferrer" class="">app settings</a>.</p>
<p>If your app <em>is</em> published in the Slack Marketplace, now is a good time to follow the advice in our <a class="" href=/slack-marketplace/slack-marketplace-app-guidelines-and-requirements>guide to testing Slack Marketplace apps</a>. Apps published in the Slack Marketplace can have token rotation enabled via the <strong>Published App Settings</strong> section of your <a href=https://api.slack.com/apps target=_blank rel="noopener noreferrer" class="">app settings</a>.</p>
<p>And if you're using a Bolt app, this is your stop! You're all good to go. If not, keep reading. The <a href=#exchange class="">next section</a> is just for you.</p>
<h2 class="anchor anchorTargetStickyNavbar_Vzrq" id=exchange>Exchange the long-lived access token<a href=#exchange class=hash-link aria-label="Direct link to Exchange the long-lived access token" title="Direct link to Exchange the long-lived access token" translate=no>​</a></h2>
<p>A Slack app implemented with OAuth will already have a long-lived access token. You need to call the <a class="" href=/reference/methods/oauth.v2.exchange><code>oauth.v2.exchange</code></a> method to exchange that long-lived access token for a refresh token and an expiring access token.</p>
<p>You'll need your app's <code>client_id</code> and <code>client_secret</code> as well, which can be found under "App Credentials" within the specific app's "Basic Information" <a href=https://api.slack.com/apps target=_blank rel="noopener noreferrer" class="">page</a>.</p>

<p>After you refresh your short-lived credentials for the first time, we'll expire your original long-lived access token. Please don't try to revoke it yourself using <a class="" href=/reference/methods/auth.revoke><code>auth.revoke</code></a>, or you'll end up making the user redo the whole OAuth installation process (<em>yikes!</em>). Instead, just call <a class="" href=/reference/methods/auth.test><code>auth.test</code></a> to make sure the original token won't work anymore. Keep in mind that this applies to <strong>both user</strong> and <strong>bot</strong> tokens, so make sure to repeat the OAuth process to get your hands on fresh credentials.</p>

<h2 class="anchor anchorTargetStickyNavbar_Vzrq" id=refresh>Refresh a token<a href=#refresh class=hash-link aria-label="Direct link to Refresh a token" title="Direct link to Refresh a token" translate=no>​</a></h2>
<p>Any time you implement OAuth with your Slack app, whether token rotation is enabled or not, the last step of the flow requires you to call the <a class="" href=/reference/methods/oauth.v2.access><code>oauth.v2.access</code></a> method. The only difference with token rotation is that you'll be calling the method throughout your app's life to get new tokens.</p>
<p>In order to refresh a token, make a call to <code>oauth.v2.access</code>, setting the new <code>grant_type</code> parameter to your refresh token:</p>

<h2 class="anchor anchorTargetStickyNavbar_Vzrq" id=uninstall>Uninstalling apps<a href=#uninstall class=hash-link aria-label="Direct link to Uninstalling apps" title="Direct link to Uninstalling apps" translate=no>​</a></h2>
<p>Slack APIs provide multiple ways to undo Slack installations. With <a class="" href=/reference/methods/apps.uninstall><code>apps.uninstall</code></a>, behavior is the same regardless of whether the app uses token rotation. The app is uninstalled and all tokens associated with the installation are revoked.</p>
<p><a class="" href=/reference/methods/auth.revoke><code>auth.revoke</code></a> is used to revoke a single token of an installation. When used on an app <strong>without</strong> token rotation, it usually revokes the authorization associated with it, meaning that the user or team that had the app installed needs to reinstall it.</p>
<p>For apps <em>with</em> token rotation, <code>auth.revoke</code> will revoke a single token (a refresh token, a bot access token that expires, or a user access token that expires) without changing the underlying installation.
