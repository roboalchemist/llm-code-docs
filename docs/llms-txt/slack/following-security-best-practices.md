<p>Keeping your apps and functions secure is an important part of developing on the Slack platform.</p>
<p>Slack’s managed hosted environment is built on the <a href=https://deno.land/ target=_blank rel="noopener noreferrer" class="">Deno</a> runtime, a secure Javascript runtime. Learn more about Deno’s <a href=https://deno.land/manual@v1.32.1/basics/permissions target=_blank rel="noopener noreferrer" class="">permissions model</a>.</p>
<p>Here are some best practices to keep in mind when developing <a class="" href=/tools/deno-slack-sdk/guides/creating-custom-functions>custom functions</a>, <a class="" href=/tools/deno-slack-sdk/guides/creating-workflows>workflows</a> and <a class="" href=/tools/deno-slack-sdk/guides/using-triggers>triggers</a> for Slack automations.</p>
<h2 class="anchor anchorTargetStickyNavbar_Vzrq" id=ACL>Set appropriate access control levels (ACLs)<a href=#ACL class=hash-link aria-label="Direct link to Set appropriate access control levels (ACLs)" title="Direct link to Set appropriate access control levels (ACLs)" translate=no>​</a></h2>
<ul>
<li class="">
<p><a class="" href=/tools/deno-slack-sdk/guides/managing-triggers#manage>Limit access</a> to your <a class="" href=/tools/deno-slack-sdk/guides/creating-custom-functions>functions</a> and <a class="" href=/tools/deno-slack-sdk/guides/using-triggers>triggers</a> to only the intended audience. Use the <code>slack function access</code> or <code>slack trigger access</code> commands to control who can use your functions or trip your triggers.</p>
</li>
<li class="">
<p>Your app collaborators can deploy your functions and manage your workflows and triggers. Only add collaborators to your app that you trust.</p>
</li>
</ul>
<h2 class="anchor anchorTargetStickyNavbar_Vzrq" id=validate-input>Validate input<a href=#validate-input class=hash-link aria-label="Direct link to Validate input" title="Direct link to Validate input" translate=no>​</a></h2>
<ul>
<li class="">
<p>It’s always important to validate inputs to your functions. If you’ve changed the distribution of your function, keep in mind it may be used in other workflows in ways you may not anticipate. Your app collaborators may also create or update triggers to start your workflows.</p>
</li>
<li class="">
<p>When using Slack <a class="" href=/tools/deno-slack-sdk/guides/using-datastores>datastores</a>, avoid <a href=https://owasp.org/Top10/A03_2021-Injection/ target=_blank rel="noopener noreferrer" class="">injection attacks</a> by properly sanitizing user input when building queries, and use the <code>expression_values</code> and <code>expression_attributes</code> fields when querying for data. Also ensure that the user has read access to data from the datastore before processing it.</p>
</li>
<li class="">
<p>Always confirm the user has access to perform whatever operation your function is being asked to perform. In complex workflows, several users may participate in the various steps, so ensure you’re checking the correct user’s permissions. For example, in a contract approval workflow, anyone may be able to request an approval, but only certain approvers may actually provide an approval.</p>
</li>
<li class="">
<p>When listening to <a class="" href=/tools/deno-slack-sdk/guides/integrating-message-metadata-events>message metadata events</a>, keep in mind that many apps may be posting messages with the same event types. If you’d like to listen to messages from only specific apps, use a filter on the <code>app_id</code> in the event trigger definition.</p>
</li>
</ul>
<h2 class="anchor anchorTargetStickyNavbar_Vzrq" id=handle-sensitive-data>Handle sensitive data<a href=#handle-sensitive-data class=hash-link aria-label="Direct link to Handle sensitive data" title="Direct link to Handle sensitive data" translate=no>​</a></h2>
<ul>
<li class="">
<p>Secrets needed by your custom functions running on Slack-managed infrastructure should be shared with Slack using the <code>slack env add</code> command and never hard-coded in your functions. Examples are API keys, OAuth client IDs and secrets, certificates, and cryptography keys. If possible, use Slack’s <a href=https://docs.slack.dev/faq#third-party target=_blank rel="noopener noreferrer" class="">third party auth support</a> to manage OAuth-based credentials.</p>
</li>
<li class="">
<p>For local apps using <code>slack run</code>, ensure that the <code>.env</code> file containing local secrets does not end up in your source control system.</p>
</li>
<li class="">
<p>When using <code>slack env add</code>, ensure the secret does not end up in your shell’s history. This can be done using shell environment variables, or by running <code>slack env add</code> without any parameters. With no parameters, <code>slack env add</code> will prompt you for the secret’s name and value in the console, using a password display.</p>
</li>
<li class="">
<p>Be careful about collecting or logging sensitive information in workflows from users, especially passwords or personally identifiable information (PII). Data may be exposed to later workflow steps, in data exports, or in activity logs.</p>
</li>
</ul>
<h2 class="anchor anchorTargetStickyNavbar_Vzrq" id=secure-credentials>Secure credentials<a href=#secure-credentials class=hash-link aria-label="Direct link to Secure credentials" title="Direct link to Secure credentials" translate=no>​</a></h2>
<ul>
<li class="">
<p>The Slack CLI stores credentials in the <code>credentials.json</code> file in the <code>.slack</code> folder in your home directory. Slack will never ask you share the tokens contained in that file.</p>
</li>
<li class="">
<p>While these credentials expire and are regularly rotated, access to this file should be limited to only you.</p>
</li>
<li class="">
<p>Never share the tokens or challenge strings generated by the <code>slack login</code> flow.</p>
</li>
<li class="">
<p>Never paste a <code>/slackauthticket</code> command given to you by another user into Slack.</p>
</li>
</ul>
<h2 class="anchor anchorTargetStickyNavbar_Vzrq" id=secure-libraries>Use secure libraries<a href=#secure-libraries class=hash-link aria-label="Direct link to Use secure libraries" title="Direct link to Use secure libraries" translate=no>​</a></h2>
<ul>
<li class="">
<p>It is your responsibility to monitor and respond to security vulnerabilities in your custom function’s code and dependencies, and to deploy new versions to Slack-managed infrastructure as needed.</p>
</li>
<li class="">
<p>Keep your Slack CLI and SDKs up to date by upgrading when prompted.</p>
</li>
<li class="">
<p>Only use a Slack CLI download by following instructions found within the <a class="" href=/tools/slack-cli/>Slack CLI docs</a>.</p>
</li>
</ul>
<h2 class="anchor anchorTargetStickyNavbar_Vzrq" id=network-egress>Handle network egress<a href=#network-egress class=hash-link aria-label="Direct link to Handle network egress" title="Direct link to Handle network egress" translate=no>​</a></h2>
<ul>
<li class="">Slack’s <code>outgoingDomains</code> configuration limits which domains your custom function code can use when making external network requests. Only list <code>outgoingDomains</code> if the domains are required by your functions.</li>
</ul>
<h2 class="anchor anchorTargetStickyNavbar_Vzrq" id=scopes-tokens>Make scopes and tokens<a href=#scopes-tokens class=hash-link aria-label="Direct link to Make scopes and tokens" title="Direct link to Make scopes and tokens" translate=no>​</a></h2>
<ul>
<li class="">
<p>Functions are given a short-lived token that can be used to make <a class="" href=/tools/deno-slack-sdk/guides/calling-slack-api-methods>Slack API</a> calls, which use the scopes requested in the app’s manifest. We recommend only sending these tokens to Slack API endpoints, and not logging them or sending them to external systems.</p>
</li>
<li class="">
<p>Only request the scopes your functions need to do their job.</p>
</li>
</ul>
