markdown"><header><h1>Authenticating with OAuth</h1></header>
<p>OAuth allows installation of your app to any workspace and is an important step in distributing your app. This is because each app installation issues unique <a href=#the-installation-object class="">access tokens with related installation information</a> that can be retrieved for incoming events and used to make scoped API requests.</p>
<p>All of the additional underlying details around authentications can be found
within <a class="" href=/authentication/installing-with-oauth>the Slack API documentation</a>!</p>
<h2 class="anchor anchorTargetStickyNavbar_Vzrq" id=configuring-the-application>Configuring the application<a href=#configuring-the-application class=hash-link aria-label="Direct link to Configuring the application" title="Direct link to Configuring the application" translate=no>​</a></h2>
<p>To set your Slack app up for distribution, you will need to enable Bolt OAuth
and store installation information securely. Bolt supports OAuth by using the
<a class="" href=/tools/node-slack-sdk/oauth><code>@slack/oauth</code></a> package to handle most of the work; this includes
setting up OAuth routes, verifying state, and passing your app an installation
object which you must store.</p>
<h3 class="anchor anchorTargetStickyNavbar_Vzrq" id=app-options>App options<a href=#app-options class=hash-link aria-label="Direct link to App options" title="Direct link to App options" translate=no>​</a></h3>
<p>The following <code>App</code> options are required for OAuth installations:</p>
<ul>
<li class=""><code>clientId</code>: <code>string</code>. An application credential found on the <strong>Basic
Information</strong> page of your <a href=https://api.slack.com/apps target=_blank rel="noopener noreferrer" class="">app settings</a>.</li>
<li class=""><code>clientSecret</code>: <code>string</code>. A secret value also found on the <strong>Basic

<h3 class="anchor anchorTargetStickyNavbar_Vzrq" id=installer-options>Installer options<a href=#installer-options class=hash-link aria-label="Direct link to Installer options" title="Direct link to Installer options" translate=no>​</a></h3>
<p>We provide several options for customizing default OAuth using the
<code>installerOptions</code> object, which can be passed in during the initialization of
<code>App</code>. You can override these common options and
<a href=https://github.com/slackapi/node-slack-sdk/blob/main/packages/oauth/src/install-provider-options.ts target=_blank rel="noopener noreferrer" class="">find others here</a>:</p>
<ul>
<li class=""><code>authVersion</code>: <code>string</code>. Settings for either new Slack apps (<code>v2</code>) or
"classic" Slack apps (<code>v1</code>). Most apps use <code>v2</code> since <code>v1</code> was available for a
Slack app model that can no longer be created. Default: <code>v2</code>.</li>
<li class=""><code>directInstall</code>: <code>boolean</code>. Skip rendering the
<a href=#add-to-slack-button class="">installation page</a> at <code>installPath</code> and redirect to the
authorization URL instead. Default: <code>false</code>.</li>
<li class=""><code>installPath</code>: <code>string</code>. Path of the URL for starting an installation.
Default: <code>/slack/install</code>.</li>
<li class=""><code>metadata</code>: <code>string</code>. Static information shared between requests as install
URL options. Optional.</li>
<li class=""><code>redirectUriPath</code>: <code>string</code>. Path of the installation callback URL. Default:
<code>/slack/oauth_redirect</code>.</li>
<li class=""><code>stateVerification</code>: <code>boolean</code>. Option to customize the state verification
logic. When set to <code>false</code>, the app does not verify the state parameter. While
not recommended for general OAuth security, some apps might want to skip this
for internal installations within an enterprise grid org. Default: <code>true</code>.</li>
<li class=""><code>userScopes</code>: <code>string[]</code>. User scopes to request during installation. Default:
<code>[]</code>.</li>
<li class=""><code>callbackOptions</code>: <a class="" href=/tools/node-slack-sdk/reference/oauth/interfaces/CallbackOptions><code>CallbackOptions</code></a>. Customized
<a class="" href=/tools/node-slack-sdk/reference/oauth/interfaces/CallbackOptions>responses to send</a> during OAuth.
<a href=https://github.com/slackapi/node-slack-sdk/blob/e5a4f3fbbd4f6aad9fdd415976f80668b01fd442/packages/oauth/src/callback-options.ts#L81-L162 target=_blank rel="noopener noreferrer" class="">Default callbacks</a>.</li>
<li class=""><code>stateStore</code>: <a class="" href=/tools/node-slack-sdk/reference/oauth/interfaces/StateStore><code>StateStore</code></a>. Customized generator and validator
for <a class="" href=/tools/node-slack-sdk/oauth#using-a-custom-state-store>OAuth state parameters</a>; the default <code>ClearStateStore</code> should work
well for most scenarios. However, if you need even better security, storing
state parameter data with a server-side database would be a good approach.
Default: <a href=https://github.com/slackapi/node-slack-sdk/blob/main/packages/oauth/src/state-stores/clear-state-store.ts target=_blank rel="noopener noreferrer" class=""><code>ClearStateStore</code></a>.</li>
</ul>

<h2 class="anchor anchorTargetStickyNavbar_Vzrq" id=completing-authentication>Completing authentication<a href=#completing-authentication class=hash-link aria-label="Direct link to Completing authentication" title="Direct link to Completing authentication" translate=no>​</a></h2>
<p>The complete authentication handshake involves requesting scopes using a
generated installation URL and processing approved installations. Bolt handles
this with a default installation and callback route, but some configurations to
the app settings are needed and changes to these routes might be desired.</p>
<div class="theme-admonition theme-admonition-info admonition_xJq3 alert alert--info"><div class=admonitionHeading_Gvgb><span class=admonitionIcon_Rf37><svg viewBox="0 0 14 16"><path fill-rule=evenodd d="M7 2.3c3.14 0 5.7 2.56 5.7 5.7s-2.56 5.7-5.7 5.7A5.71 5.71 0 0 1 1.3 8c0-3.14 2.56-5.7 5.7-5.7zM7 1C3.14 1 0 4.14 0 8s3.14 7 7 7 7-3.14 7-7-3.14-7-7-7zm1 3H6v5h2V4zm0 6H6v2h2v-2z"/></svg></span>info</div><div class=admonitionContent_BuS1><p>Bolt for JavaScript does not support OAuth for
<a class="" href=/tools/bolt-js/concepts/receiver>custom receivers</a>. If you're implementing a custom
receiver, you can instead use our <a class="" href=/tools/node-slack-sdk/oauth><code>@slack/oauth</code></a> package, which is
what Bolt for JavaScript uses under the hood.</div></div>
<h3 class="anchor anchorTargetStickyNavbar_Vzrq" id=installing-your-app>Installing your App<a href=#installing-your-app class=hash-link aria-label="Direct link to Installing your App" title="Direct link to Installing your App" translate=no>​</a></h3>
<p>Bolt for JavaScript provides an <strong>Install Path</strong> at the <code>/slack/install</code> URL
out-of-the-box. This endpoint returns a simple static page that includes an
<code>Add to Slack</code> button that links to a generated authorization URL for your app.
This has the right scopes, a valid <code>state</code>, the works.</p>
<p>For example, an app hosted at <em><a href=http://www.example.com target=_blank rel="noopener noreferrer" class="">www.example.com</a></em> will serve the install page at
<em><a href=http://www.example.com/slack/install target=_blank rel="noopener noreferrer" class="">www.example.com/slack/install</a></em> but this path can be changed with
<code>installerOptions.installPath</code>. Rendering a webpage before the authorization URL
is also optional and can be skipped using <code>installerOptions.directInstall</code>.</p>

<h3 class="anchor anchorTargetStickyNavbar_Vzrq" id=sign-in-with-slack>Sign in with Slack<a href=#sign-in-with-slack class=hash-link aria-label="Direct link to Sign in with Slack" title="Direct link to Sign in with Slack" translate=no>​</a></h3>
<p>Right now Bolt does not support <a class="" href=/authentication/sign-in-with-slack/>Sign in with Slack</a> out-of-the-box. This
still continues to remain an option using APIs from the
<a class="" href=/tools/node-slack-sdk/web-api><code>@slack/web-api</code></a> package, which aim to make implementing OpenID
Connect (OIDC) connections simple. Alternative <a class="" href=/tools/bolt-js/concepts/custom-routes>routes</a> might be
required.</p>
<p>Explore <a class="" href=/tools/node-slack-sdk/web-api#sign-in-with-slack-via-openid-connect>this relevant package documentation</a> for reference and example.</p>
<h3 class="anchor anchorTargetStickyNavbar_Vzrq" id=extra-authorizations>Extra authorizations<a href=#extra-authorizations class=hash-link aria-label="Direct link to Extra authorizations" title="Direct link to Extra authorizations" translate=no>​</a></h3>
<p>If you need additional authorizations or permissions, such as user scopes for
user tokens from users of a team where your app is already installed, or have a
reason to dynamically generate an install URL, an additional installation is
required.</p>
<p>Generating a new installation URL requires a few steps:</p>
<ol>
<li class="">Manually instantiate an <code>ExpressReceiver</code> instance.</li>
<li class="">Assign the instance to a variable named <code>receiver</code>.</li>
<li class="">Call the <code>receiver.installer.generateInstallUrl()</code> function.</li>
</ol>
<p>Read more about <code>generateInstallUrl()</code> in the
<a class="" href=/tools/node-slack-sdk/oauth/#using-handleinstallpath>"Manually generating installation page URL"</a> section of
the <code>@slack/oauth</code> docs.</p>
