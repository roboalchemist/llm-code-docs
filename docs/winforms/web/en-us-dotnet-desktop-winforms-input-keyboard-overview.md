Title: Overview of keyboard input - Windows Forms

URL Source: https://learn.microsoft.com/en-us/dotnet/desktop/winforms/input-keyboard/overview

Markdown Content:
In Windows Forms, user input is sent to applications in the form of [Windows messages](https://learn.microsoft.com/en-us/windows/win32/winmsg/about-messages-and-message-queues). A series of overridable methods process these messages at the application, form, and control level. When these methods receive keyboard messages, they raise events that can be handled to get information about the keyboard input. In many cases, Windows Forms applications are able to process all user input simply by handling these events. In other cases, an application might need to override one of the methods that process messages in order to intercept a particular message before it's received by the application, form, or control.

All Windows Forms controls inherit a set of events related to mouse and keyboard input. For example, a control can handle the [KeyPress](https://learn.microsoft.com/en-us/dotnet/api/system.windows.forms.control.keypress#system-windows-forms-control-keypress) event to determine the character code of a key that was pressed. For more information, see [Using keyboard events](https://learn.microsoft.com/en-us/dotnet/desktop/winforms/input-keyboard/events).

Forms and controls have access to the [IMessageFilter](https://learn.microsoft.com/en-us/dotnet/api/system.windows.forms.imessagefilter) interface and a set of overridable methods that process Windows messages at different points in the message queue. These methods all have a [Message](https://learn.microsoft.com/en-us/dotnet/api/system.windows.forms.message) parameter, which encapsulates the low-level details of Windows messages. You can implement or override these methods to examine the message and then either consume the message or pass it on to the next consumer in the message queue. The following table presents the methods that process all Windows messages in Windows Forms.

| Method | Notes |
| --- | --- |
| [PreFilterMessage](https://learn.microsoft.com/en-us/dotnet/api/system.windows.forms.imessagefilter.prefiltermessage) | This method intercepts queued (also known as posted) Windows messages at the application level. |
| [PreProcessMessage](https://learn.microsoft.com/en-us/dotnet/api/system.windows.forms.control.preprocessmessage) | This method intercepts Windows messages at the form and control level before they have been processed. |
| [WndProc](https://learn.microsoft.com/en-us/dotnet/api/system.windows.forms.control.wndproc) | This method processes Windows messages at the form and control level. |
| [DefWndProc](https://learn.microsoft.com/en-us/dotnet/api/system.windows.forms.control.defwndproc) | This method performs the default processing of Windows messages at the form and control level. This provides the minimal functionality of a window. |
| [OnNotifyMessage](https://learn.microsoft.com/en-us/dotnet/api/system.windows.forms.control.onnotifymessage) | This method intercepts messages at the form and control level, after they have been processed. The [EnableNotifyMessage](https://learn.microsoft.com/en-us/dotnet/api/system.windows.forms.controlstyles#system-windows-forms-controlstyles-enablenotifymessage) style bit must be set for this method to be called. |

Keyboard and mouse messages are processed by an extra set of overridable methods that are specific to those types of messages. For more information, see the [Preprocessing keys](https://learn.microsoft.com/en-us/dotnet/desktop/winforms/input-keyboard/overview#preprocessing-keys) section.

Windows Forms identifies keyboard input as virtual-key codes that are represented by the bitwise [Keys](https://learn.microsoft.com/en-us/dotnet/api/system.windows.forms.keys) enumeration. With the [Keys](https://learn.microsoft.com/en-us/dotnet/api/system.windows.forms.keys) enumeration, you can combine a series of pressed keys to result in a single value. These values correspond to the values that accompany the **WM_KEYDOWN** and **WM_SYSKEYDOWN** Windows messages. You can detect most physical key presses by handling the [KeyDown](https://learn.microsoft.com/en-us/dotnet/api/system.windows.forms.control.keydown#system-windows-forms-control-keydown) or [KeyUp](https://learn.microsoft.com/en-us/dotnet/api/system.windows.forms.control.keyup#system-windows-forms-control-keyup) events. Character keys are a subset of the [Keys](https://learn.microsoft.com/en-us/dotnet/api/system.windows.forms.keys) enumeration and correspond to the values that accompany the **WM_CHAR** and **WM_SYSCHAR** Windows messages. If the combination of pressed keys results in a character, you can detect the character by handling the [KeyPress](https://learn.microsoft.com/en-us/dotnet/api/system.windows.forms.control.keypress#system-windows-forms-control-keypress) event. Alternatively, you can use [Keyboard](https://learn.microsoft.com/en-us/dotnet/api/microsoft.visualbasic.devices.keyboard), exposed by Visual Basic programming interface, to discover which keys were pressed and send keys. For more information, see [Accessing the Keyboard (Visual Basic)](https://learn.microsoft.com/en-us/dotnet/visual-basic/developing-apps/programming/computer-resources/accessing-the-keyboard).

As listed previously, there are three keyboard related events that can occur on a control. The following sequence shows the general order of the events:

1. The user pushes the A key, the key is preprocessed, dispatched, and a [KeyDown](https://learn.microsoft.com/en-us/dotnet/api/system.windows.forms.control.keydown#system-windows-forms-control-keydown) event occurs.
2. The user holds the A key, the key is preprocessed, dispatched, and a [KeyPress](https://learn.microsoft.com/en-us/dotnet/api/system.windows.forms.control.keypress#system-windows-forms-control-keypress) event occurs. This event occurs multiple times as the user holds a key.
3. The user releases the A key, the key is preprocessed, dispatched and a [KeyUp](https://learn.microsoft.com/en-us/dotnet/api/system.windows.forms.control.keyup#system-windows-forms-control-keyup) event occurs.

Like other messages, keyboard messages are processed in the [WndProc](https://learn.microsoft.com/en-us/dotnet/api/system.windows.forms.control.wndproc) method of a form or control. However, before keyboard messages are processed, the [PreProcessMessage](https://learn.microsoft.com/en-us/dotnet/api/system.windows.forms.control.preprocessmessage) method calls one or more methods that can be overridden to handle special character keys and physical keys. You can override these methods to detect and filter certain keys before the control processes the messages. The following table shows the action that is being performed and the related method that occurs, in the order that the method occurs.

| Action | Related method | Notes |
| --- | --- | --- |
| Check for a command key such as an accelerator or menu shortcut. | [ProcessCmdKey](https://learn.microsoft.com/en-us/dotnet/api/system.windows.forms.control.processcmdkey) | This method processes a command key, which takes precedence over regular keys. If this method returns `true`, the key message isn't dispatched and a key event doesn't occur. If it returns `false`, [IsInputKey](https://learn.microsoft.com/en-us/dotnet/api/system.windows.forms.control.isinputkey) is called`.` |
| Check for a special key that requires preprocessing or a normal character key that should raise a [KeyDown](https://learn.microsoft.com/en-us/dotnet/api/system.windows.forms.control.keydown#system-windows-forms-control-keydown) event and be dispatched to a control. | [IsInputKey](https://learn.microsoft.com/en-us/dotnet/api/system.windows.forms.control.isinputkey) | If the method returns `true`, it means the control is a regular character and a [KeyDown](https://learn.microsoft.com/en-us/dotnet/api/system.windows.forms.control.keydown#system-windows-forms-control-keydown) event is raised. If `false`, [ProcessDialogKey](https://learn.microsoft.com/en-us/dotnet/api/system.windows.forms.control.processdialogkey) is called. **Note:** To ensure a control gets a key or combination of keys, you can handle the [PreviewKeyDown](https://learn.microsoft.com/en-us/dotnet/api/system.windows.forms.control.previewkeydown#system-windows-forms-control-previewkeydown) event and set [IsInputKey](https://learn.microsoft.com/en-us/dotnet/api/system.windows.forms.previewkeydowneventargs.isinputkey) of the [PreviewKeyDownEventArgs](https://learn.microsoft.com/en-us/dotnet/api/system.windows.forms.previewkeydowneventargs) to `true` for the key or keys you want. |
| Check for a navigation key (ESC, TAB, Return, or arrow keys). | [ProcessDialogKey](https://learn.microsoft.com/en-us/dotnet/api/system.windows.forms.control.processdialogkey) | This method processes a physical key that employs special functionality within the control, such as switching focus between the control and its parent. If the immediate control doesn't handle the key, the [ProcessDialogKey](https://learn.microsoft.com/en-us/dotnet/api/system.windows.forms.control.processdialogkey) is called on the parent control, and so on, to the topmost control in the hierarchy. If this method returns `true`, preprocessing is complete and a key event isn't generated. If it returns `false`, a [KeyDown](https://learn.microsoft.com/en-us/dotnet/api/system.windows.forms.control.keydown#system-windows-forms-control-keydown) event occurs. |

| Action | Related method | Notes |
| --- | --- | --- |
| Check to see the key is a normal character that should be processed by the control | [IsInputChar](https://learn.microsoft.com/en-us/dotnet/api/system.windows.forms.control.isinputchar) | If the character is a normal character, this method returns `true`, the [KeyPress](https://learn.microsoft.com/en-us/dotnet/api/system.windows.forms.control.keypress#system-windows-forms-control-keypress) event is raised and no further preprocessing occurs. Otherwise [ProcessDialogChar](https://learn.microsoft.com/en-us/dotnet/api/system.windows.forms.control.processdialogchar) is called. |
| Check to see if the character is a mnemonic (such as &OK on a button) | [ProcessDialogChar](https://learn.microsoft.com/en-us/dotnet/api/system.windows.forms.control.processdialogchar) | This method, similar to [ProcessDialogKey](https://learn.microsoft.com/en-us/dotnet/api/system.windows.forms.control.processdialogkey), is called up the control hierarchy. If the control is a container control, it checks for mnemonics by calling [ProcessMnemonic](https://learn.microsoft.com/en-us/dotnet/api/system.windows.forms.control.processmnemonic) on itself and its child controls. If [ProcessDialogChar](https://learn.microsoft.com/en-us/dotnet/api/system.windows.forms.control.processdialogchar) returns `true`, a [KeyPress](https://learn.microsoft.com/en-us/dotnet/api/system.windows.forms.control.keypress#system-windows-forms-control-keypress) event doesn't occur. |

After keyboard messages reach the [WndProc](https://learn.microsoft.com/en-us/dotnet/api/system.windows.forms.control.wndproc) method of a form or control, they're processed by a set of methods that can be overridden. Each of these methods returns a [Boolean](https://learn.microsoft.com/en-us/dotnet/api/system.boolean) value specifying whether the keyboard message has been processed and consumed by the control. If one of the methods returns `true`, then the message is considered handled, and it isn't passed to the control's base or parent for further processing. Otherwise, the message stays in the message queue and might be processed in another method in the control's base or parent. The following table presents the methods that process keyboard messages.

| Method | Notes |
| --- | --- |
| [ProcessKeyMessage](https://learn.microsoft.com/en-us/dotnet/api/system.windows.forms.control.processkeymessage) | This method processes all keyboard messages that are received by the [WndProc](https://learn.microsoft.com/en-us/dotnet/api/system.windows.forms.control.wndproc) method of the control. |
| [ProcessKeyPreview](https://learn.microsoft.com/en-us/dotnet/api/system.windows.forms.control.processkeypreview) | This method sends the keyboard message to the control's parent. If [ProcessKeyPreview](https://learn.microsoft.com/en-us/dotnet/api/system.windows.forms.control.processkeypreview) returns `true`, no key event is generated, otherwise [ProcessKeyEventArgs](https://learn.microsoft.com/en-us/dotnet/api/system.windows.forms.control.processkeyeventargs) is called. |
| [ProcessKeyEventArgs](https://learn.microsoft.com/en-us/dotnet/api/system.windows.forms.control.processkeyeventargs) | This method raises the [KeyDown](https://learn.microsoft.com/en-us/dotnet/api/system.windows.forms.control.keydown#system-windows-forms-control-keydown), [KeyPress](https://learn.microsoft.com/en-us/dotnet/api/system.windows.forms.control.keypress#system-windows-forms-control-keypress), and [KeyUp](https://learn.microsoft.com/en-us/dotnet/api/system.windows.forms.control.keyup#system-windows-forms-control-keyup) events, as appropriate. |

There are many methods available for overriding when a keyboard message is preprocessed and processed; however, some methods are better choices than others. Following table shows tasks you might want to accomplish and the best way to override the keyboard methods. For more information on overriding methods, see [Inheritance (C# Programming Guide)](https://learn.microsoft.com/en-us/dotnet/csharp/programming-guide/classes-and-structs/inheritance#abstract-and-virtual-methods) or [Inheritance (Visual Basic)](https://learn.microsoft.com/en-us/dotnet/visual-basic/programming-guide/language-features/objects-and-classes/inheritance-basics#overriding-properties-and-methods-in-derived-classes)

| Task | Method |
| --- | --- |
| Intercept a navigation key and raise a [KeyDown](https://learn.microsoft.com/en-us/dotnet/api/system.windows.forms.control.keydown#system-windows-forms-control-keydown) event. For example, you want Tab and Enter to be handled in a text box. | Override [IsInputKey](https://learn.microsoft.com/en-us/dotnet/api/system.windows.forms.control.isinputkey). Alternatively, you can handle the [PreviewKeyDown](https://learn.microsoft.com/en-us/dotnet/api/system.windows.forms.control.previewkeydown#system-windows-forms-control-previewkeydown) event and set [IsInputKey](https://learn.microsoft.com/en-us/dotnet/api/system.windows.forms.previewkeydowneventargs.isinputkey) of the [PreviewKeyDownEventArgs](https://learn.microsoft.com/en-us/dotnet/api/system.windows.forms.previewkeydowneventargs) to `true` for the key or keys you want. |
| Perform special input or navigation handling on a control. For example, you want the use of arrow keys in your list control to change the selected item. | Override [ProcessDialogKey](https://learn.microsoft.com/en-us/dotnet/api/system.windows.forms.control.processdialogkey) |
| Intercept a navigation key and raise a [KeyPress](https://learn.microsoft.com/en-us/dotnet/api/system.windows.forms.control.keypress#system-windows-forms-control-keypress) event. For example in a spin-box control you want multiple arrow key presses to accelerate progression through the items. | Override [IsInputChar](https://learn.microsoft.com/en-us/dotnet/api/system.windows.forms.control.isinputchar). |
| Perform special input or navigation handling during a [KeyPress](https://learn.microsoft.com/en-us/dotnet/api/system.windows.forms.control.keypress#system-windows-forms-control-keypress) event. For example, in a list control holding down the R key skips between items that begin with the letter **r**. | Override [ProcessDialogChar](https://learn.microsoft.com/en-us/dotnet/api/system.windows.forms.control.processdialogchar) |
| Perform custom mnemonic handling; for example, you want to handle mnemonics on owner-drawn buttons contained in a toolbar. | Override [ProcessMnemonic](https://learn.microsoft.com/en-us/dotnet/api/system.windows.forms.control.processmnemonic). |
