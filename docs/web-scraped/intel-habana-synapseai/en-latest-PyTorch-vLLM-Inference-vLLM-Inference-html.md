Inference Using vLLM[¶](https://docs.habana.ai/en/latest/PyTorch/vLLM_Inference/vLLM_Inference.html#inference-using-vllm "Permalink to this headline")
======================================================================================================================================================

The following sections provide instructions on deploying models using vLLM with Intel® Gaudi® AI accelerator. The document is based on the vLLM Inference Server [README for Gaudi](https://github.com/HabanaAI/vllm-fork/blob/v1.23.0/README_GAUDI.md). The process involves:

*   Creating a vLLM Docker image for Gaudi

*   Building and installing the vLLM Inference Server

*   Sending an Inference Request

For frequently asked questions about using Gaudi with vLLM, see [vLLM with Intel Gaudi FAQs](https://docs.habana.ai/en/latest/PyTorch/vLLM_Inference/vLLM_FAQs.html#vllm-faqs).

To enable FP8 calibration and inference on Gaudi via vLLM, see [FP8 Calibration and Inference with vLLM](https://docs.habana.ai/en/latest/PyTorch/vLLM_Inference/vLLM_FP8_Inference.html#fp8-calibration-and-inference-with-vllm).

Creating a Docker Image for Gaudi[¶](https://docs.habana.ai/en/latest/PyTorch/vLLM_Inference/vLLM_Inference.html#creating-a-docker-image-for-gaudi "Permalink to this headline")
--------------------------------------------------------------------------------------------------------------------------------------------------------------------------------

Since a vLLM server is launched within a Docker container, a Docker image tailored for Gaudi is needed. Follow the instructions in [Run Docker Image](https://github.com/HabanaAI/vllm-fork/blob/v1.23.0/README_GAUDI.md#run-docker-image).

Building and Installing vLLM for Gaudi[¶](https://docs.habana.ai/en/latest/PyTorch/vLLM_Inference/vLLM_Inference.html#building-and-installing-vllm-for-gaudi "Permalink to this headline")
------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------

Follow the instructions in [Build and Install vLLM](https://github.com/HabanaAI/vllm-fork/blob/v1.23.0/README_GAUDI.md#build-and-install-vllm).

Sending an Inference Request[¶](https://docs.habana.ai/en/latest/PyTorch/vLLM_Inference/vLLM_Inference.html#sending-an-inference-request "Permalink to this headline")
----------------------------------------------------------------------------------------------------------------------------------------------------------------------

There are two common methods to send inference requests on vLLM:

*   [Offline Batched Inference](https://docs.vllm.ai/en/stable/getting_started/quickstart.html#offline-batched-inference)

*   [OpenAI-Compatible Server](https://docs.vllm.ai/en/stable/serving/openai_compatible_server.html)

A practical jupyter notebook utilizing these methods is provided in [Gaudi Tutorials](https://github.com/HabanaAI/Gaudi-tutorials/blob/main/PyTorch/vLLM_Tutorials/Getting_Started_with_vLLM/Getting_Started_with_vLLM.ipynb).

For a list of popular models and supported configurations validated on Gaudi, see the [README for Gaudi](https://github.com/HabanaAI/vllm-fork/blob/v1.23.0/README_GAUDI.md#supported-configurations).

Basic Troubleshooting for OOM Errors[¶](https://docs.habana.ai/en/latest/PyTorch/vLLM_Inference/vLLM_Inference.html#basic-troubleshooting-for-oom-errors "Permalink to this headline")
--------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------

*   Due to various factors such as available GPU memory, model size, and input sequence length, the standard inference command may not always work for your model, potentially leading to OOM errors. The following steps help mitigate OOM errors:

> *   Increase `gpu_memory_utilization` - This addresses insufficient available memory. The vLLM pre-allocates HPU cache by using gpu_memory_utilization% of memory. By increasing this utilization, you can provide more KV cache space.
> 
>     *   Decrease `max_num_seqs` or `max_num_batched_tokens` - This may reduce the number of concurrent requests in a batch, thereby requiring less KV cache space.
> 
>     *   Increase `tensor_parallel_size` - This approach shards model weights, so each GPU has more memory available for KV cache.

*   During the development phase when evaluating a model for inference on vLLM, you may skip the warmup phase of the server. This helps to achieve faster testing turnaround times and can be set using the `VLLM_SKIP_WARMUP=true` environment variable.

Note

You may disable warmup only for development but it is highly recommended to enable it in production. 

Guidelines for Performance Tuning[¶](https://docs.habana.ai/en/latest/PyTorch/vLLM_Inference/vLLM_Inference.html#guidelines-for-performance-tuning "Permalink to this headline")
--------------------------------------------------------------------------------------------------------------------------------------------------------------------------------

Some general guidelines for tweaking performance are noted below. Your results may vary:

*   [Warmup](https://github.com/HabanaAI/vllm-fork/blob/v1.23.0/README_GAUDI.md#warmup) should be enabled during deployment with optimal number of [buckets](https://github.com/HabanaAI/vllm-fork/blob/v1.23.0/README_GAUDI.md#bucketing-mechanism). Warmup time depends on many factors, e.g. input and output sequence length, batch size, number of buckets, and data type. Warmup can even take a couple of hours, depending on the configuration. Refer to [Managing and Reducing vLLM Warmup Time](https://docs.habana.ai/en/latest/PyTorch/vLLM_Inference/Managing_vLLM_Warmup_Time.html#managing-and-reducing-vllm-warmup-time) for more information about warmup time optimization strategies.

*   Since HPU Graphs and KV Cache occupy the same memory pool (“usable memory” determined by `gpu_memory_utilization`), a balance is required between the two which can be managed using the `VLLM_GRAPH_RESERVED_MEM` environment variable (defines the ratio of memory reserved for HPU Graphs vs KV Cache):

    *   Maximizing KV Cache size helps to accommodate bigger batches resulting in increased overall throughput.

    *   Enabling [HPU Graphs](https://github.com/HabanaAI/vllm-fork/blob/v1.23.0/README_GAUDI.md#hpu-graph-capture) reduces host [overhead](https://docs.habana.ai/en/latest/PyTorch/Inference_on_PyTorch/Inference_Using_HPU_Graphs.html#reducing-host-overhead-with-hpu-graphs) times and can be useful for reducing latency.

*   For fine-grained control, the `VLLM_GRAPH_PROMPT_RATIO` environment variable determines the ratio of usable graph memory reserved for prefill and decode graphs. Allocating more memory for one stage generally means faster processing of that stage.

*   [Bucketing mechanisms](https://github.com/HabanaAI/vllm-fork/blob/v1.23.0/README_GAUDI.md#bucketing-mechanism) can be applied. vLLM server is pre-configured optimally for heavy-duty decoding expecting lots of pending requests (see `VLLM_GRAPH_DECODE_STRATEGY` default maximum batch size strategy). However, during lean periods when request rate is low, this strategy may not be best suited and can be tuned to cater to lower batch sizes. As an example, tweaking the range of buckets using `VLLM_DECODE_BS_BUCKET_{param}` might prove useful. Refer to [performance tuning knobs](https://github.com/HabanaAI/vllm-fork/blob/v1.23.0/README_GAUDI.md#environment-variables) - `VLLM_{phase}_{dim}_BUCKET_{param}` for a list of 12 environment variables configuring ranges of bucketing mechanisms.

*   Using FP8 data type for inference on large language models halves the required memory bandwidth compared to BF16. In addition, FP8 compute is twice as fast as BF16 compute, so even compute-bound workloads, such as offline inference on large batch sizes, benefit. For more details, see [FP8 Calibration and Inference with vLLM](https://docs.habana.ai/en/latest/PyTorch/vLLM_Inference/vLLM_FP8_Inference.html#fp8-calibration-and-inference-with-vllm).
