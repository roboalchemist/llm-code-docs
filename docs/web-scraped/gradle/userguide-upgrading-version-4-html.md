This chapter provides the information you need to migrate your older Gradle 4.x builds to Gradle 5.0. In most cases, you will need to apply the changes from all versions that come after the one you’re upgrading from. For example, if you’re upgrading from Gradle 4.3 to 5.0, you will also need to apply the changes since 4.4, 4.5, etc up to 5.0.

If you are using Gradle for Android, you need to move to version 3.3 or higher of both the Android Gradle Plugin and Android Studio.

If you are not already on the latest 4.10.x release, read the sections [below](https://docs.gradle.org/userguide/upgrading_version_4.html#changes_4.10) for help upgrading your project to the latest 4.10.x release. We recommend upgrading to the latest 4.10.x release to get the most useful warnings and deprecations information before moving to 5.0. Avoid upgrading Gradle and migrating to Kotlin DSL at the same time in order to ease troubleshooting in case of potential issues.

2. Try running `gradle help --scan` and view the [deprecations view](https://docs.gradle.com/develocity/get-started/#identifying_deprecated_gradle_functionality) of the generated Build Scan. If there are no warnings, the Deprecations tab will not appear.

Gradle 5.x will generate (potentially less obvious) errors if you try to upgrade directly to it. Alternatively, you can run `gradle help --warning-mode=all` to see the deprecations in the console, though it may not report as much detailed information.
3.   Update your plugins.

Some plugins will break with this new version of Gradle, for example because they use internal APIs that have been removed or changed. The previous step will help you identify potential problems by issuing deprecation warnings when a plugin does try to use a deprecated part of the API. In particular, you will need to use at least a 2.x version of the [**Shadow Plugin**](https://plugins.gradle.org/plugin/com.gradleup.shadow).
4.   Run `gradle wrapper --gradle-version 5.0` to update the project to 5.0

1. Move to Java 8 or higher if you haven’t already. Whereas Gradle 4.x requires Java 7, Gradle 5 requires Java 8 to run.

2. Read the [Upgrading from 4.10](https://docs.gradle.org/userguide/upgrading_version_4.html#changes_5.0) section and make any necessary changes.

3. Try to run the project and debug any errors using the [Troubleshooting Guide](https://docs.gradle.org/userguide/troubleshooting.html#troubleshooting).

In addition, Gradle has added several significant new and improved features that you should consider using in your builds:

* [Maven Publish and Ivy Publish Plugins](https://docs.gradle.org/userguide/upgrading_version_4.html#rel4.8:switch_to_publishing_plugins) that now support digital signatures with the [Signing Plugin](https://docs.gradle.org/userguide/signing_plugin.html#signing_plugin).

* Use native [BOM import](https://docs.gradle.org/userguide/upgrading_version_4.html#rel5.0:bom_import) in your builds.

* The [Worker API](https://docs.gradle.org/userguide/worker_api.html#worker_api) for enabling units of work to run in parallel.

* A new API for [creating and configuring tasks lazily](https://docs.gradle.org/userguide/upgrading_version_4.html#rel4.9:lazy_task_creation) that can significantly improve your build’s configuration time.

* The [default memory settings](https://docs.gradle.org/userguide/upgrading_version_4.html#rel5.0:default_memory_settings) for the command-line client, the Gradle daemon, and all workers including compilers and test executors, have been greatly reduced.

* The [default versions of several code quality plugins](https://docs.gradle.org/userguide/upgrading_version_4.html#rel5.0:default_tool_versions) have been updated.

* Several [library versions used by Gradle](https://docs.gradle.org/userguide/upgrading_version_4.html#rel5.0:library_upgrades) have been upgraded.

### [](https://docs.gradle.org/userguide/upgrading_version_4.html#other_changes)[Other changes](https://docs.gradle.org/userguide/upgrading_version_4.html#other_changes)

* The `enableFeaturePreview('IMPROVED_POM_SUPPORT')` and `enableFeaturePreview('STABLE_PUBLISHING')` flags are no longer necessary. These features are now enabled by default.

* Gradle now bundles [JAXB](https://docs.gradle.org/userguide/upgrading_version_4.html#rel5.0:jaxb_and_java9) for Java 9 and above. You can remove the `--add-modules java.xml.bind` option from `org.gradle.jvmargs`, if set.

### [](https://docs.gradle.org/userguide/upgrading_version_4.html#potential_breaking_changes)[Potential breaking changes](https://docs.gradle.org/userguide/upgrading_version_4.html#potential_breaking_changes)

The changes in this section have the potential to break your build, but the vast majority have been deprecated for quite some time and few builds will be affected by a large number of them. We strongly recommend upgrading to Gradle 4.10 first to get a report on what deprecations affect your build.

The following breaking changes are not from deprecations, but the result of changes in behavior:

* [Separation of compile and runtime dependencies when consuming POMs](https://docs.gradle.org/userguide/upgrading_version_4.html#rel5.0:pom_compile_runtime_separation)

* The evaluation of the `publishing {}` block is no longer deferred until needed but behaves like any other block. Please use `afterEvaluate {}` if you need to defer evaluation.

* The [`Javadoc`](https://docs.gradle.org/dsl/org.gradle.api.tasks.javadoc.Javadoc.html) and [`Groovydoc`](https://docs.gradle.org/dsl/org.gradle.api.tasks.javadoc.Groovydoc.html) tasks now delete the destination dir for the documentation before executing. This has been added to remove stale output files from the last task execution.

* The [Java Library Distribution Plugin](https://docs.gradle.org/userguide/java_library_distribution_plugin.html#java_library_distribution_plugin) is now based on the [Java Library Plugin](https://docs.gradle.org/userguide/java_library_plugin.html#java_library_plugin) instead of the [Java Plugin](https://docs.gradle.org/userguide/java_plugin.html#java_plugin).

While it applies the Java Plugin, it behaves slightly different (e.g. it adds the `api` configuration). Thus, make sure to check whether your build behaves as expected after upgrading.

* The `html` property on `CheckstyleReport` and `FindBugsReport` now returns a [`CustomizableHtmlReport`](https://docs.gradle.org/dsl/org.gradle.api.reporting.CustomizableHtmlReport.html) instance that is easier to configure from statically typed languages like Java and Kotlin.

* The [Configuration Avoidance API](https://docs.gradle.org/userguide/upgrading_version_4.html#rel5.0:configuration_avoidance) has been updated to prevent the creation and configuration of tasks that are never used.

* The [default memory settings](https://docs.gradle.org/userguide/upgrading_version_4.html#rel5.0:default_memory_settings) for the command-line client, the Gradle daemon, and all workers including compilers and test executors, have been greatly reduced.

* The [default versions of several code quality plugins](https://docs.gradle.org/userguide/upgrading_version_4.html#rel5.0:default_tool_versions) have been updated.

* Several [library versions used by Gradle](https://docs.gradle.org/userguide/upgrading_version_4.html#rel5.0:library_upgrades) have been upgraded.

The following breaking changes will appear as deprecation warnings with Gradle 4.10:

General

* `<<` for task definitions no longer works. In other words, you can not use the syntax `task myTask << { …​ }`.

Use the [Task.doLast()](https://docs.gradle.org/dsl/org.gradle.api.Task.html#org.gradle.api.Task:doLast(org.gradle.api.Action)) method instead, like this: ```groovy
task myTask {
    doLast {
        ...
    }
}

```  
*   You can no longer use any of the following characters in domain object names, such as project and task names: <space>`/ \ : < > " ? * |` . You should also not use `.` as a leading or trailing character.

Running Gradle & build environment

*   As mentioned before, Gradle can no longer be run on Java 7. However, you can still use [forked compilation and testing](https://docs.gradle.org/userguide/building_java_projects.html#sec:java_cross_compilation) to build and test software for Java 6 and above.

*   The `-Dtest.single` command-line option has been removed —use [test filtering](https://docs.gradle.org/userguide/java_testing.html#test_filtering) instead.

*   The `-Dtest.debug` command-line option has been removed — use the [`--debug-jvm` option](https://docs.gradle.org/userguide/java_testing.html#sec:debugging_java_tests) instead.

*   The `-u`/`--no-search-upward` command-line option has been removed —make sure all your builds have a _settings.gradle_ file.

*   The `--recompile-scripts` command-line option has been removed.

*   You can no longer have a Gradle build nested in a subdirectory of another Gradle build unless the nested build has a _settings.gradle_ file.

*   The `DirectoryBuildCache.setTargetSizeInMB(long)` method has been removed —use [DirectoryBuildCache.removeUnusedEntriesAfterDays](https://docs.gradle.org/dsl/org.gradle.caching.local.DirectoryBuildCache.html#org.gradle.caching.local.DirectoryBuildCache:removeUnusedEntriesAfterDays) instead.

*   The `org.gradle.readLoggingConfigFile` system property no longer does anything —update affected tests to work with your `java.util.logging` settings.

Working with files

*   You can no longer cast `FileCollection` objects to other types using the `as` keyword or the `asType()` method.

*   You can no longer pass `null` as the configuration action of [CopySpec.from(Object, Action)](https://docs.gradle.org/javadoc/org/gradle/api/file/CopySpec.html#from-java.lang.Object-org.gradle.api.Action-).

*   For better compatibility with the Kotlin DSL, [CopySpec.duplicatesStrategy](https://docs.gradle.org/javadoc/org/gradle/api/file/DuplicatesStrategy.html) is no longer nullable. The property setter no longer accepts `null` as a way to reset the property back to its default value. Use `DuplicatesStrategy.INHERIT` instead.

*   The `FileCollection.stopExecutionIfEmpty()` method has been removed — use the [@SkipWhenEmpty](https://docs.gradle.org/javadoc/org/gradle/api/tasks/SkipWhenEmpty.html) annotation on `FileCollection` task properties instead.

*   The `FileCollection.add()` method has been removed —use [Project.files()](https://docs.gradle.org/dsl/org.gradle.api.Project.html#org.gradle.api.Project:files(java.lang.Object[])) and [Project.fileTree()](https://docs.gradle.org/dsl/org.gradle.api.Project.html#org.gradle.api.Project:fileTree(java.lang.Object)) to create configurable file collections/file trees and add to them via [ConfigurableFileCollection.from()](https://docs.gradle.org/javadoc/org/gradle/api/file/ConfigurableFileCollection.html#from-java.lang.Object...-).

*   `SimpleFileCollection` has been removed — use [Project.files(Object…​)](https://docs.gradle.org/dsl/org.gradle.api.Project.html#org.gradle.api.Project:files(java.lang.Object[])) instead.

*   Don’t have your own classes extend `AbstractFileCollection` — use the [Project.files()](https://docs.gradle.org/dsl/org.gradle.api.Project.html#org.gradle.api.Project:files(java.lang.Object[])) method instead. This problem may exhibit as a missing `getBuildDependencies()` method.

Java builds

*   The `CompileOptions.bootClasspath` property has been removed —use [CompileOptions.bootstrapClasspath](https://docs.gradle.org/dsl/org.gradle.api.tasks.compile.CompileOptions.html#org.gradle.api.tasks.compile.CompileOptions:bootstrapClasspath) instead.

*   You can no longer use `-source-path` as a generic compiler argument —use [CompileOptions.sourcepath](https://docs.gradle.org/dsl/org.gradle.api.tasks.compile.CompileOptions.html#org.gradle.api.tasks.compile.CompileOptions:sourcepath) instead.

*   You can no longer use `-processorpath` as a generic compiler argument —use [CompileOptions.annotationProcessorPath](https://docs.gradle.org/dsl/org.gradle.api.tasks.compile.CompileOptions.html#org.gradle.api.tasks.compile.CompileOptions:annotationProcessorPath) instead.

*   Gradle will no longer automatically apply annotation processors that are on the compile classpath — use [CompileOptions.annotationProcessorPath](https://docs.gradle.org/dsl/org.gradle.api.tasks.compile.CompileOptions.html#org.gradle.api.tasks.compile.CompileOptions:annotationProcessorPath) instead.

*   The `testClassesDir` property has been removed from the [Test](https://docs.gradle.org/dsl/org.gradle.api.tasks.testing.Test.html) task —use [testClassesDirs](https://docs.gradle.org/dsl/org.gradle.api.tasks.testing.Test.html#org.gradle.api.tasks.testing.Test:testClassesDirs) instead.

*   The `classesDir` property has been removed from both the _JDepend_ task and [SourceSetOutput](https://docs.gradle.org/dsl/org.gradle.api.tasks.SourceSetOutput.html). Use the _JDepend.classesDirs_ and [SourceSetOutput.classesDirs](https://docs.gradle.org/dsl/org.gradle.api.tasks.SourceSetOutput.html#org.gradle.api.tasks.SourceSetOutput:classesDirs) properties instead.

*   The `JavaLibrary(PublishArtifact, DependencySet)` constructor has been removed —this was used by the [Shadow Plugin](https://plugins.gradle.org/plugin/com.gradleup.shadow), so make sure you upgrade to at least version 2.x of that plugin.

*   The `JavaBasePlugin.configureForSourceSet()` method has been removed.

*   You can no longer create your own instances of `JavaPluginConvention`, `ApplicationPluginConvention`, `WarPluginConvention`, `EarPluginConvention`, `BasePluginConvention`, and `ProjectReportsPluginConvention`.

*   The `Maven` Plugin used to publish the highly outdated Maven 2 metadata format. This has been changed and it will now publish Maven 3 metadata, just like the `Maven Publish` Plugin.

With the removal of Maven 2 support, the methods that configure unique snapshot behavior have also been removed. Maven 3 only supports unique snapshots, so we decided to remove them. 

Tasks & properties

*   The following legacy classes and methods related to [lazy properties](https://docs.gradle.org/userguide/lazy_configuration.html#lazy_properties) have been removed —use [ObjectFactory.property()](https://docs.gradle.org/javadoc/org/gradle/api/model/ObjectFactory.html#property-java.lang.Class-) to create `Property` instances:

    *   `PropertyState`

    *   `DirectoryVar`

    *   `RegularFileVar`

    *   `ProjectLayout.newDirectoryVar()`

    *   `ProjectLayout.newFileVar()`

    *   `Project.property(Class)`

    *   `Script.property(Class)`

    *   `ProviderFactory.property(Class)`

*   Tasks configured and registered with the [task configuration avoidance](https://docs.gradle.org/userguide/task_configuration_avoidance.html#sec:task_configuration_avoidance_migration_guidelines) APIs have more restrictions on the other methods that can be called from a configuration action.

*   The internal `@Option` and `@OptionValues` annotations — package `org.gradle.api.internal.tasks.options` — have been removed. Use the public [@Option](https://docs.gradle.org/javadoc/org/gradle/api/tasks/options/Option.html) and [@OptionValues](https://docs.gradle.org/javadoc/org/gradle/api/tasks/options/OptionValues.html) annotations instead.

*   The `Task.deleteAllActions()` method has been removed with no replacement.

*   The `Task.dependsOnTaskDidWork()` method has been removed — use [declared inputs and outputs](https://docs.gradle.org/userguide/incremental_build.html#incremental_build) instead.

*   The following properties and methods of `TaskInternal` have been removed —use task dependencies, task rules, reusable utility methods, or the [Worker API](https://docs.gradle.org/userguide/worker_api.html#worker_api) in place of executing a task directly.

    *   `execute()`

    *   `executer`

    *   `getValidators()`

    *   `addValidator()`

Gradle now provides support for importing bill of materials (BOM) files, which are effectively POM files that use `<dependencyManagement>` sections to control the versions of direct and transitive dependencies. All you need to do is declare the POM as a `platform` dependency.

The following example picks the versions of the `gson` and `dom4j` dependencies from the declared Spring Boot BOM:

```groovy
dependencies {
    // import a BOM
    implementation platform('org.springframework.boot:spring-boot-dependencies:1.5.8.RELEASE')

    implementation 'com.google.code.gson:gson'
    implementation 'dom4j:dom4j'
}
```

### [][https://docs.gradle.org/userguide/upgrading_version_4.html#rel5.0:pom_compile_runtime_separation]([5.0) Separation of compile and runtime dependencies when consuming POMs](https://docs.gradle.org/userguide/upgrading_version_4.html#rel5.0:pom_compile_runtime_separation)

Since Gradle 1.0, runtime-scoped dependencies have been included in the Java compilation classpath, which has some drawbacks:

* The compilation classpath is much larger than it needs to be, slowing down compilation.

* The compilation classpath includes runtime-scoped files that do not impact compilation, resulting in unnecessary re-compilation when those files change.

With this new behavior, the Java and Java Library plugins both honor the [separation of compile and runtime scopes](https://docs.gradle.org/userguide/java_library_plugin.html#sec:java_library_separation). This means that the compilation classpath only includes compile-scoped dependencies, while the runtime classpath adds the runtime-scoped dependencies as well. This is particularly useful if you develop and publish Java libraries with Gradle where the separation between `api` and `implementation` dependencies is reflected in the published scopes.

### [][https://docs.gradle.org/userguide/upgrading_version_4.html#rel5.0:changes_to_default_task]([5.0) Changes to property factory methods on `DefaultTask`](https://docs.gradle.org/userguide/upgrading_version_4.html#rel5.0:changes_to_default_task)

#### [](https://docs.gradle.org/userguide/upgrading_version_4.html#property_factory_methods_on_defaulttask_are_now_final)[Property factory methods on `DefaultTask` are now final](https://docs.gradle.org/userguide/upgrading_version_4.html#property_factory_methods_on_defaulttask_are_now_final)

The Property instances need to be declared as inputs or outputs in the usual ways, such as attaching annotations such as `@OutputFile` or using the runtime API to register the property.

For example, you could previously use the following syntax and have both outputFile instances registered as declared outputs:

```option

```

```option

```

open class MyTask : DefaultTask() {

    doLast { ... }
}
```

    // note: no annotation here
    final RegularFileProperty outputFile = newOutputFile()
}

task myOtherTask {
    def outputFile = newOutputFile()
    doLast { ... }

```option

    val outputFile = project.objects.fileProperty()

    doLast { ... }

build.gradle

```groovy

    final RegularFileProperty outputFile = project.objects.fileProperty()

### [](https://docs.gradle.org/userguide/upgrading_version_4.html#rel5.0:configuration_avoidance)[Configuration Avoidance API disallows common configuration errors](https://docs.gradle.org/userguide/upgrading_version_4.html#rel5.0:configuration_avoidance)

The [configuration avoidance API](https://docs.gradle.org/userguide/task_configuration_avoidance.html#task_configuration_avoidance) introduced in Gradle 4.9 allows you to avoid creating and configuring tasks that are never used.

With the existing API, this example adds two tasks (`foo` and `bar`):

```option
Kotlin
```

```option
Groovy
```

build.gradle.kts

```kotlin
tasks.create("foo") {
    tasks.create("bar")
}
```

build.gradle

```groovy
tasks.create("foo") {
    tasks.create("bar")
}
```

When converting this to use the new API, something surprising happens: `bar` doesn’t exist. The new API only executes configuration actions when necessary, so the `register()` for task `bar` only executes when `foo` is configured.

```option
Kotlin
```

```option
Groovy
```

build.gradle.kts

```kotlin
tasks.register("foo") {
    tasks.register("bar") // WRONG
}
```

build.gradle

```groovy
tasks.register("foo") {
    tasks.register("bar") // WRONG
}
```

To avoid this, Gradle now detects this and prevents modification to the underlying container (through `create()` or `register()`) when using the new API.

### [][https://docs.gradle.org/userguide/upgrading_version_4.html#rel4.10:aws_s3_permissions]([4.10) Publishing to AWS S3 requires new permissions](https://docs.gradle.org/userguide/upgrading_version_4.html#rel4.10:aws_s3_permissions)

The S3 repository transport protocol allows Gradle to publish artifacts to AWS S3 buckets. Starting with this release, every artifact uploaded to an S3 bucket will be equipped with the `bucket-owner-full-control` canned ACL. Make sure that the AWS account used to publish artifacts has the `s3:PutObjectAcl` and `s3:PutObjectVersionAcl` permissions, otherwise the upload will fail.

```json
{
    "Version":"2012-10-17",
    "Statement":[
        // ...
        {
            "Effect":"Allow",
            "Action":[
                "s3:PutObject", // necessary for uploading objects
                "s3:PutObjectAcl", // required starting with this release
                "s3:PutObjectVersionAcl" // if S3 bucket versioning is enabled
            ],
            "Resource":"arn:aws:s3:::myCompanyBucket/*"
        }
    ]
}
```

See [AWS S3 Cross Account Access](https://docs.gradle.org/userguide/supported_repository_protocols.html#sec:s3-cross-account) for more information.

Prior to Gradle 4.8, the `publishing {}` block was implicitly treated as if all the logic inside it was executed after the project was evaluated. This was confusing, because it was the only block that behaved that way. As part of the stabilization effort in Gradle 4.8, we are deprecating this behavior and asking all users to migrate their build.

The new, stable behavior can be switched on by adding the following to your settings file:

```option
Kotlin
```

```option
Groovy
```

settings.gradle.kts

```kotlin
enableFeaturePreview("STABLE_PUBLISHING")
```

settings.gradle

```groovy
enableFeaturePreview('STABLE_PUBLISHING')
```

We recommend doing a test run with a local repository to see whether all artifacts still have the expected coordinates. In most cases everything should work as before and you are done. However, your publishing block may rely on the implicit deferred configuration, particularly if it relies on values that may change during the configuration phase of the build.

For example, under the new behavior, the following logic assumes that `jar.archiveBaseName` doesn’t change after `artifactId` is set:

```option
Kotlin
```

```option
Groovy
```

build.gradle.kts

```kotlin

    publishing {
        publications {
            named<MavenPublication>("mavenJava") {
                from(components["java"])
                artifactId = tasks.jar.get().archiveBaseName.get()

        }
    }
}
```

build.gradle

```groovy
subprojects {
    publishing {
        publications {
            mavenJava {
                from components.java
                artifactId = jar.archiveBaseName
            }
        }
    }
}
```

If that assumption is incorrect or might possibly be incorrect in the future, the `artifactId` must be set within an `afterEvaluate {}` block, like so:

```option
Kotlin
```

```option
Groovy
```

build.gradle.kts

```kotlin

    publishing {
        publications {
            named<MavenPublication>("mavenJava") {
                from(components["java"])
                afterEvaluate {
                    artifactId = tasks.jar.get().archiveBbaseName.get()
                }
            }
        }
    }
}
```

build.gradle

```groovy
subprojects {
    publishing {
        publications {
            mavenJava {
                from components.java
                afterEvaluate {
                    artifactId = jar.archiveBaseName
                }
            }
        }
    }
}
```

The plain console mode now formats output consistently with the rich console, which means that the output format has changed. For example:
