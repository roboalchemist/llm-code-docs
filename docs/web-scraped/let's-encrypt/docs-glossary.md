Glossary
========

 Last updated: July 31, 2025 

[See all Documentation](https://letsencrypt.org/docs)

[ACME (Automatic Certificate Management Environment)](https://letsencrypt.org/docs/glossary/#def-ACME)The protocol implemented by [Let’s Encrypt](https://letsencrypt.org/#def-LE). Software compatible with that protocol can use it to communicate with Let’s Encrypt to ask for a [certificate](https://letsencrypt.org/#def-leaf). [ACME RFC](https://tools.ietf.org/html/rfc8555) - [Wikipedia](https://en.wikipedia.org/wiki/Automated_Certificate_Management_Environment)[ACME Client](https://letsencrypt.org/docs/glossary/#def-ACME-client)A program capable of communicating with an ACME server to ask for a [certificate](https://letsencrypt.org/#def-leaf).[ACME Server](https://letsencrypt.org/docs/glossary/#def-ACME-server)An ACME-compatible server that can generate [certificates](https://letsencrypt.org/#def-leaf). Let’s Encrypt’s software, [Boulder](https://letsencrypt.org/#def-boulder), is ACME-compatible, [with some divergences](https://github.com/letsencrypt/boulder/blob/main/docs/acme-divergences.md).[Authority Information Access (AIA)](https://letsencrypt.org/docs/glossary/#def-AIA)A certificate [extension](https://letsencrypt.org/#def-extension) used to indicate to [user agents](https://letsencrypt.org/#def-user-agent) how to obtain information about the issuer of the [certificate](https://letsencrypt.org/#def-certificate). It typically specifies the [issuer URI](https://letsencrypt.org/#def-CAI).[Baseline Requirements (BRs)](https://letsencrypt.org/docs/glossary/#def-BRs)A set of technical and policy requirements for CAs. Since all major [root programs](https://letsencrypt.org/#def-root-program) incorporate the Baseline Requirements, CAs must follow these requirements to be trusted by most browsers.[Boulder](https://letsencrypt.org/docs/glossary/#def-boulder)The software implementing ACME, developed and used by [Let’s Encrypt](https://letsencrypt.org/#def-LE). [GitHub](https://github.com/letsencrypt/boulder)[CA Issuers](https://letsencrypt.org/docs/glossary/#def-CAI)Part of the [AIA](https://letsencrypt.org/#def-AIA) field containing information about the issuer of the [certificate](https://letsencrypt.org/#def-leaf). It may be useful when the [web server](https://letsencrypt.org/#def-web-server) didn’t provide a trusted [certificate chain](https://letsencrypt.org/#def-chain).[CA/Browser Forum](https://letsencrypt.org/docs/glossary/#def-CABF)A voluntary group of certification authorities, vendors of Internet browser software, operating systems, and other PKI-enabled applications. The CA/Browser Forum publishes the [Baseline Requirements](https://letsencrypt.org/#def-BRs). [Let’s Encrypt](https://letsencrypt.org/#def-LE) is a member of the CA/Browser Forum. [Wikipedia](https://en.wikipedia.org/wiki/CA/Browser_Forum)[CAA (Certificate Authority Authorization)](https://letsencrypt.org/docs/glossary/#def-CAA)A DNS record that specifies which [CAs](https://letsencrypt.org/#def-CA) are allowed to issue certificates for the corresponding domain name. CAA records are checked by CAs, not by browsers. [Let’s Encrypt](https://letsencrypt.org/#def-LE)[honors CAA records](https://letsencrypt.org/docs/caa/) as required by the [Baseline Requirements](https://letsencrypt.org/#def-BRs). - [Wikipedia](https://en.wikipedia.org/wiki/DNS_Certification_Authority_Authorization)[Canonical Name record (CNAME)](https://letsencrypt.org/docs/glossary/#def-CNAME)A DNS entry which maps one domain name to another, referred to as the Canonical Name. [Wikipedia](https://en.wikipedia.org/wiki/CNAME_record)[Certificate](https://letsencrypt.org/docs/glossary/#def-certificate)A file in a [particular format](https://letsencrypt.org/#def-X509) that contains a public key and other data describing when to use that public key. The most common kind of certificate is a [leaf certificate](https://letsencrypt.org/#def-leaf). There are also [intermediate](https://letsencrypt.org/#def-intermediate) and [root](https://letsencrypt.org/#def-root) certificates.[Certificate Authority (CA)](https://letsencrypt.org/docs/glossary/#def-CA)An organization that issues [certificates](https://letsencrypt.org/#def-leaf). [Let’s Encrypt](https://letsencrypt.org/#def-LE), [IdenTrust](https://letsencrypt.org/#def-IdenTrust), Sectigo, and DigiCert are Certificate Authorities. [Wikipedia](https://en.wikipedia.org/wiki/Certificate_authority)[Certificate Policy (CP)](https://letsencrypt.org/docs/glossary/#def-CP)A named set of rules that indicates the applicability of a certificate to a particular community and/or class of applications with common security requirements. Specific details of issuance are outlined in a [CPS](https://letsencrypt.org/#def-CPS). CP and CPS documents may be combined into a single document. [ISRG Combined CP/CPS](https://letsencrypt.org/repository/#isrg-certificate-policy) - [RFC 3647](https://tools.ietf.org/html/rfc3647) - [Wikipedia](https://en.wikipedia.org/wiki/Certificate_policy)[Certificate Revocation List (CRL)](https://letsencrypt.org/docs/glossary/#def-CRL)A method to inform [user agents](https://letsencrypt.org/#def-user-agent) about the [revocation](https://letsencrypt.org/#def-revocation) status of a [certificate](https://letsencrypt.org/#def-leaf). This is a list of the serial numbers of all revoked certificates from a given CA, signed by that CA. [Wikipedia](https://en.wikipedia.org/wiki/Certificate_revocation_list)[Certificate Signing Request (CSR)](https://letsencrypt.org/docs/glossary/#def-CSR)A signed file containing the information required by the [CA](https://letsencrypt.org/#def-CA) to generate a certificate. Relevant information for [Let’s Encrypt](https://letsencrypt.org/#def-LE) are the [Common Name](https://letsencrypt.org/#def-CN), [Subject Alternative Names](https://letsencrypt.org/#def-SAN), and Subject Public Key Info. Usually, [client applications](https://letsencrypt.org/#def-ACME-client) automatically generate the CSR for the user, although a web hosting provider or device might also generate a CSR. [Wikipedia](https://en.wikipedia.org/wiki/Certificate_signing_request)[Certificate Store](https://letsencrypt.org/docs/glossary/#def-store)A certificate store contains a list of trusted [roots](https://letsencrypt.org/#def-root). Operating systems (such as Windows, Android or Debian) and [web browsers](https://letsencrypt.org/#def-web-browser) (such as Firefox) maintain a certificate store. Browsers without one rely on the operating systems’ certificate store. [Certificates](https://letsencrypt.org/#def-leaf) provided by [Let’s Encrypt](https://letsencrypt.org/#def-LE) are [trusted by most certificate stores](https://letsencrypt.org/certificates/).[Certificate Transparency (CT)](https://letsencrypt.org/docs/glossary/#def-CT)To improve security, certificates (or [precertificates](https://letsencrypt.org/#def-precertificate)) must be published in Certificate Transparency Logs: [https://www.certificate-transparency.org/](https://www.certificate-transparency.org/). [Let’s Encrypt](https://letsencrypt.org/#def-LE) generates and publishes [precertificates](https://letsencrypt.org/#def-precertificate), and includes in the subsequent [certificate](https://letsencrypt.org/#def-leaf) a list of [SCTs](https://letsencrypt.org/#def-SCT) for the precertificate. Some [browsers](https://letsencrypt.org/#def-web-browser), such as Google Chrome, require the presence of this verifiable promise in order to validate the certificate. [Wikipedia](https://en.wikipedia.org/wiki/Certificate_Transparency)[Certificate Transparency Log](https://letsencrypt.org/docs/glossary/#def-CT-log)A component of [Certificate Transparency](https://letsencrypt.org/#def-CT) that accepts submissions of certificates and [precertificates](https://letsencrypt.org/#def-precertificate) and incorporates them into a permanent, verifiable, publicly-accessible list.[Certificate chain](https://letsencrypt.org/docs/glossary/#def-chain)A list of [intermediate certificates](https://letsencrypt.org/#def-intermediate) that help a [user agent](https://letsencrypt.org/#def-user-agent) determine that it can trust an end-entity or [leaf certificate](https://letsencrypt.org/#def-leaf), by connecting it to a [root certificate](https://letsencrypt.org/#def-root) in its [certificate store](https://letsencrypt.org/#def-store). Note: the chain is not always unique, and when a website presents a certificate chain leading to one root, the user agent may decide to use another chain to validate the certificate. [Wikipedia](https://en.wikipedia.org/wiki/Public_key_certificate)[Certificate extension](https://letsencrypt.org/docs/glossary/#def-extension)In certificates, most fields are defined by extensions. For instance, [Subject Alternative Names](https://letsencrypt.org/#def-SAN) and [AIA](https://letsencrypt.org/#def-AIA) are extensions. The extension mechanism allows creating new fields that were not part of the original [X.509](https://letsencrypt.org/#def-X509) standard.[Certificate issuer](https://letsencrypt.org/docs/glossary/#def-issuer)The “Issuer” field of a certificate describes which certificate signed it. For instance, the Issuer field of a Let’s Encrypt end-entity certificate might be “Issuer: C = US, O = Let’s Encrypt, CN = Let’s Encrypt Authority X3”. It commonly contains fields like [Common Name](https://letsencrypt.org/#def-CN), Country, and Organization. The Issuer field always matches some certificate’s [Subject](https://letsencrypt.org/#def-subject) field. For [self-signed](https://letsencrypt.org/#def-self-signed) certificates like [roots](https://letsencrypt.org/#def-root), the Issuer is the same as the Subject. The term “issuer” may also be used to indicate a certificate that issues other certificates (an [intermediate](https://letsencrypt.org/#def-intermediate) or root), or an organization that issues certificates.[Certificate subject](https://letsencrypt.org/docs/glossary/#def-subject)The “Subject” field of a certificate indicates what a certificate is about. This commonly contains fields like [Common Name](https://letsencrypt.org/#def-CN), Country, and Organization.[Certification Practice Statement (CPS)](https://letsencrypt.org/docs/glossary/#def-CPS)A statement of the practices that a certification authority employs in issuing, managing, revoking, and renewing or re-keying certificates. A CPS must be compliant with its associated [Certificate Policy](https://letsencrypt.org/#def-CP). CP and CPS documents may be combined into a single document. [ISRG Combined CP/CPS](https://letsencrypt.org/repository/#isrg-certificate-policy) - [RFC 3647 section 3.4](https://tools.ietf.org/html/rfc3647#section-3.4)[Wikipedia](https://en.wikipedia.org/wiki/Certification_Practice_Statement)[Common Name (CN)](https://letsencrypt.org/docs/glossary/#def-CN)Part of a certificate’s [Subject](https://letsencrypt.org/#def-subject) describing what the certificate is about. For [roots](https://letsencrypt.org/#def-root) and [intermediates](https://letsencrypt.org/#def-intermediate) it’s the human-readable name of the [certificate authority](https://letsencrypt.org/#def-CA). For [leaf certificates](https://letsencrypt.org/#def-leaf) it’s one of the domain names on the certificate. Note: The common name is limited to 63 characters. It is an obsolete method of indicating a domain name to which the certificate applies, since current Internet standards expect software to check only the [Subject Alternative Names](https://letsencrypt.org/#def-SAN) in order to determine the applicability of a certificate.[Critical extension](https://letsencrypt.org/docs/glossary/#def-critical)A certificate may contain [extensions](https://letsencrypt.org/#def-extension) marked “critical.” This means that software must reject that certificate unless the software understands how to process that extension. This makes it possible to introduce new extensions that are important for security without creating risks for older software.[Cross Signing](https://letsencrypt.org/docs/glossary/#def-cross-signing)An issuing certificate may be signed by more than one [root](https://letsencrypt.org/#def-root). For example, [Let’s Encrypt](https://letsencrypt.org/#def-LE)[intermediates](https://letsencrypt.org/#def-intermediate) are cross signed by [IdenTrust](https://letsencrypt.org/#def-IdenTrust), because at launch the Let’s Encrypt root was not yet trusted by [certificate stores](https://letsencrypt.org/#def-store). Technically, it’s achieved with two issuing certificates, using the same [Subject](https://letsencrypt.org/#def-subject) and the same [Key-pair](https://letsencrypt.org/#def-key-pair), one signed by the private key of a Let’s Encrypt root and the other signed by the private key of an IdenTrust’s root: [/certificates](https://letsencrypt.org/certificates/). [Wikipedia](https://en.wikipedia.org/wiki/X.509#Certificate_chains_and_cross-certification)[DNS-based Authentication of Named Entities (DANE)](https://letsencrypt.org/docs/glossary/#def-DANE)A mechanism using DNS to indicate how to verify the authenticity of the [certificate](https://letsencrypt.org/#def-leaf) or encryption key presented. [Wikipedia](https://en.wikipedia.org/wiki/DNS-based_Authentication_of_Named_Entities)[Domain Name System Security Extensions (DNSSEC)](https://letsencrypt.org/docs/glossary/#def-DNSSEC)A mechanism to cryptographically authenticate DNS responses. DNSSEC requires deployment by TLDs, domain name owners, and recursive resolvers in order to take effect. Adoption is currently somewhat low. [Wikipedia](https://en.wikipedia.org/wiki/Domain_Name_System_Security_Extensions)[Domain-validated certificate](https://letsencrypt.org/docs/glossary/#def-DV)A [certificate](https://letsencrypt.org/#def-leaf) where the applicant has only proven its control over the domain name (and not the identity of the requesting organization). [Let’s Encrypt](https://letsencrypt.org/#def-LE) offers only DV certificates (not [OV](https://letsencrypt.org/#def-OV) or [EV](https://letsencrypt.org/#def-EV)): [FAQ](https://letsencrypt.org/docs/faq/) - [Wikipedia](https://en.wikipedia.org/wiki/Domain-validated_certificate)[ECDSA (Elliptic Curve Digital Signature Algorithm)](https://letsencrypt.org/docs/glossary/#def-ECDSA)A variant of the Digital Signature Algorithm (DSA) which uses elliptic curve cryptography. [Wikipedia](https://en.wikipedia.org/wiki/Elliptic_Curve_Digital_Signature_Algorithm). [Let’s Encrypt](https://letsencrypt.org/#def-LE) supports ECDSA for [end-entity or leaf certificates](https://letsencrypt.org/#def-leaf), but not yet for the entire [chain](https://letsencrypt.org/#def-chain): [/upcoming-features](https://letsencrypt.org/upcoming-features/)[Ed25519](https://letsencrypt.org/docs/glossary/#def-Ed25519)A specific type of [EdDSA](https://letsencrypt.org/#def-EdDSA), along with Ed448.[EdDSA (Edwards-curve Digital Signature Algorithm)](https://letsencrypt.org/docs/glossary/#def-EdDSA)A modern public-key signature system based on elliptic curves, designed to solve several common [implementation issues](https://ed25519.cr.yp.to/) with elliptic curve cryptography. Certificate Authorities like [Let’s Encrypt](https://letsencrypt.org/#def-LE) can’t provide EdDSA certificates yet. [Wikipedia](https://en.wikipedia.org/wiki/EdDSA)[Elliptic Curve Cryptography (ECC)](https://letsencrypt.org/docs/glossary/#def-ECC)A type of public-key cryptography based on elliptic curves. ECC uses smaller keys compared to non-EC cryptography while providing equivalent security. [Cloudflare](https://blog.cloudflare.com/a-relatively-easy-to-understand-primer-on-elliptic-curve-cryptography/) - [Wikipedia](https://en.wikipedia.org/wiki/Elliptic-curve_cryptography)[Extended Validation (EV)](https://letsencrypt.org/docs/glossary/#def-EV)A type of certificate validation for which the [CA](https://letsencrypt.org/#def-CA) has verified the legal entity controlling the website. They contain information about that entity. Controls from the [CA](https://letsencrypt.org/#def-CA) are more strict than for [OV](https://letsencrypt.org/#def-OV) certificates. [Let’s Encrypt](https://letsencrypt.org/#def-LE) doesn’t offer EV certificates. [Wikipedia](https://en.wikipedia.org/wiki/Extended_Validation_Certificate)[Fully qualified domain name (FQDN)](https://letsencrypt.org/docs/glossary/#def-FQDN)The complete domain name of a website. For example, `www.example.com` is an _FQDN_.[HTTP Public Key Pinning (HPKP)](https://letsencrypt.org/docs/glossary/#def-HPKP)A security mechanism that asks a browser to require that a site’s [certificate chain](https://letsencrypt.org/#def-chain) use certain public keys on future loads. Chrome introduced this mechanism to protect against CA compromises, but it caused site outages, leading Chrome to [deprecate and remove it](https://groups.google.com/a/chromium.org/forum/#!topic/blink-dev/he9tr7p3rZ8). [Wikipedia](https://en.wikipedia.org/wiki/HTTP_Public_Key_Pinning)[IdenTrust](https://letsencrypt.org/docs/glossary/#def-IdenTrust)A [Certificate Authority](https://letsencrypt.org/#def-CA). IdenTrust has [cross-signed](https://letsencrypt.org/#def-cross-signing)[Let’s Encrypt](https://letsencrypt.org/#def-LE)[intermediate certificates](https://letsencrypt.org/#def-intermediate): [/certificates](https://letsencrypt.org/certificates/). [Wikipedia](https://en.wikipedia.org/wiki/IdenTrust)[Intermediate certificate](https://letsencrypt.org/docs/glossary/#def-intermediate)A certificate signed by a [root](https://letsencrypt.org/#def-root) or another intermediate, and capable of signing other certificates. They are used to sign leaf certificates while keeping the private key of root certificate offline. Intermediates are included in [certificate chains](https://letsencrypt.org/#def-chain). [Wikipedia](https://en.wikipedia.org/wiki/Public_key_certificate#Types_of_certificate)[Internationalized Domain Name (IDN)](https://letsencrypt.org/docs/glossary/#def-IDN)Domain name with characters other than `a` to `z`, `0` to `9` and the hyphen (`-`). They can for example contain Arabic, Chinese, Cyrillic, Tamil, Hebrew or Latin alphabet-based characters with diacritics or ligatures. The encoded representation of an IDN domains starts with `xn--`. IDNs are supported by [Let’s Encrypt](https://letsencrypt.org/#def-LE): [https://letsencrypt.org/2016/10/21/introducing-idn-support.html](https://letsencrypt.org/2016/10/21/introducing-idn-support). [Wikipedia](https://en.wikipedia.org/wiki/Internationalized_domain_name) - [RFC 5890](https://tools.ietf.org/html/rfc5890) - [RFC 5891](https://tools.ietf.org/html/rfc5891)[Internationalized Domain Names for Applications (IDNA)](https://letsencrypt.org/docs/glossary/#def-IDNA)See [internationalized domain name](https://letsencrypt.org/#def-IDN).[Internet Security Research Group (ISRG)](https://letsencrypt.org/docs/glossary/#def-ISRG)The organization behind [Let’s Encrypt](https://letsencrypt.org/#def-LE): [https://www.abetterinternet.org/about/](https://www.abetterinternet.org/about/). [Wikipedia](https://en.wikipedia.org/wiki/Internet_Security_Research_Group)[Key-pair](https://letsencrypt.org/docs/glossary/#def-key-pair)A combination of a private key and public key used to sign or encrypt. The public key is commonly embedded in a certificate, while the private key is stored on its own and should be kept secret. A key pair can be used to encrypt and decrypt, to sign and verify data, or to negotiate secondary keys, depending on the application. [Wikipedia](https://en.wikipedia.org/wiki/Public-key_cryptography)[Leaf certificate (end-entity certificate)](https://letsencrypt.org/docs/glossary/#def-leaf)Most commonly, a certificate signed by an [intermediate](https://letsencrypt.org/#def-intermediate), valid for a set of domains and not able to sign other certificates. This is the type of certificate that [ACME clients](https://letsencrypt.org/#def-ACME-client) request, and that [web servers](https://letsencrypt.org/#def-web-server) use. [Wikipedia](https://en.wikipedia.org/wiki/Public_key_certificate#End-entity_or_leaf_certificate)[Let's Encrypt (LE)](https://letsencrypt.org/docs/glossary/#def-LE)The [Certificate Authority](https://letsencrypt.org/#def-CA) operated by [ISRG](https://letsencrypt.org/#def-ISRG). [Wikipedia](https://en.wikipedia.org/wiki/Let%27s_Encrypt)[Mixed content](https://letsencrypt.org/docs/glossary/#def-mixed-content)When an HTTPS webpage loads sub-resources (Javascript, CSS or images) over HTTP. [Browsers](https://letsencrypt.org/#def-web-browser) may block mixed content, or mark the page as less secure when mixed content is present: [https://developer.mozilla.org/en-US/docs/Web/Security/Mixed_content](https://developer.mozilla.org/en-US/docs/Web/Security/Mixed_content). To fix a mixed content problem, a web developer must change their pages so all resources use HTTPS URLs. [Developer tools](https://developer.mozilla.org/en-US/docs/Learn/Common_questions/What_are_browser_developer_tools) built into browsers can be used to find out which resources are causing mixed content problems.[OCSP (Online Certificate Status Protocol)](https://letsencrypt.org/docs/glossary/#def-OCSP)A method to check the [revocation](https://letsencrypt.org/#def-revocation) status of a [certificate](https://letsencrypt.org/#def-leaf). In other words, a way to check whether a [Certificate Authority](https://letsencrypt.org/#def-CA) indicates that the certificate should no longer be considered valid, even though its expiration date has not yet been reached. This request can create privacy problems because it allows the certificate authority, and Internet service providers, to directly observe who is visiting which sites. [Let’s Encrypt](https://letsencrypt.org/#def-LE) no longer provides an OCSP service. [Wikipedia](https://en.wikipedia.org/wiki/Online_Certificate_Status_Protocol)[OCSP Must-Staple](https://letsencrypt.org/docs/glossary/#def-OCSP-must-staple)A [certificate](https://letsencrypt.org/#def-leaf) extension, informing the [browser](https://letsencrypt.org/#def-web-browser) that the [web server](https://letsencrypt.org/#def-web-server) with that certificate must use [OCSP stapling](https://letsencrypt.org/#def-OCSP-stapling). It’s used to require that an up-to-date [revocation](https://letsencrypt.org/#def-revocation) status of the [certificate](https://letsencrypt.org/#def-leaf) is confirmed by the web server on every connection, making revocation more reliable. [Let’s Encrypt](https://letsencrypt.org/#def-LE) no longer supports this. [RFC 7633](https://tools.ietf.org/html/rfc7633)[OCSP stapling](https://letsencrypt.org/docs/glossary/#def-OCSP-stapling)A way for a [web server](https://letsencrypt.org/#def-web-server) to send a [browser](https://letsencrypt.org/#def-web-browser) an [OCSP](https://letsencrypt.org/#def-OCSP) response signed by the [Certificate Authority](https://letsencrypt.org/#def-CA), so the browser itself doesn’t need to make a secondary OCSP request to the CA, improving speed and privacy. Also known as TLS Certificate Status Request extension. Note that [Let’s Encrypt](https://letsencrypt.org/#def-LE) does not support OCSP. [Wikipedia](https://en.wikipedia.org/wiki/OCSP_stapling)[Cloudflare](https://blog.cloudflare.com/high-reliability-ocsp-stapling/)[Object identifier (OID)](https://letsencrypt.org/docs/glossary/#def-OID)OIDs are unique numeric identifiers standardized by the International Telecommunications Union (ITU) and ISO/IEC. OIDs are used within certificates to define extensions, fields, or policy assertions. Internet standards and [Certificate Policy](https://letsencrypt.org/#def-CP) and [Certification Practice Statement](https://letsencrypt.org/#def-CPS) documents define OID usage. [Wikipedia](https://en.wikipedia.org/wiki/Object_identifier)[Organization Validation (OV)](https://letsencrypt.org/docs/glossary/#def-OV)Certificates for which the [CA](https://letsencrypt.org/#def-CA) has verified the legal entity of the [Subscriber](https://letsencrypt.org/#def-subscriber). They contain information about that entity. [Let’s Encrypt](https://letsencrypt.org/#def-LE) doesn’t offer OV certificates. [Wikipedia](https://en.wikipedia.org/wiki/Public_key_certificate#Organization_validation)[PEM file (.pem)](https://letsencrypt.org/docs/glossary/#def-pem)A format for cryptographic information (originally specified as part of the Privacy Enhanced Mail Internet standards for secure email). A PEM document can represent information such as a private key, a public key, or a digital certificate. These files start with “-----BEGIN " and then a data type. [Wikipedia](https://en.wikipedia.org/wiki/Privacy-Enhanced_Mail)[Personal Information Exchange Files (.pfx)](https://letsencrypt.org/docs/glossary/#def-pfx)A file that may contain a [leaf certificate](https://letsencrypt.org/#def-leaf), its [chain](https://letsencrypt.org/#def-chain) up to the root and the private key of the leaf. See also [https://en.wikipedia.org/wiki/PKCS_12](https://en.wikipedia.org/wiki/PKCS_12). [Microsoft Hardware Dev Center](https://docs.microsoft.com/en-us/windows-hardware/drivers/install/personal-information-exchange---pfx--files)[Precertificate](https://letsencrypt.org/docs/glossary/#def-precertificate)Precertificates are a part of [Certificate Transparency](https://letsencrypt.org/#def-CT). A precertificate is a copy of the [certificate](https://letsencrypt.org/#def-leaf) that a CA intends to issue, with a [critical](https://letsencrypt.org/#def-critical) poison extension added to prevent the precertificate from being accepted by software in the wild. A CA submits a precertificate to [CT logs](https://letsencrypt.org/#def-CT-log) in exchange for [SCTs](https://letsencrypt.org/#def-SCT). Since a precertificate is not identical to its corresponding certificate, Certificate Transparency logs may end up containing both. [RFC 6962 Section 3.1](https://tools.ietf.org/html/rfc6962#section-3.1)[Profile](https://letsencrypt.org/docs/glossary/#def-profile)A Profile is a collection of properties which affect both the validation of and final contents of a certificate. See the [profiles documentation](https://letsencrypt.org/docs/profiles/) for descriptions of each profile, what they affect, and how to select them.[Public Suffix List (PSL)](https://letsencrypt.org/docs/glossary/#def-PSL)A list of _Public Suffixes_ maintained by Mozilla, indicating which Internet domains are available for many separate entities to register subdomains. For instance, the list indicates that both `com` and `co.uk` are Public Suffixes even though `co.uk` is not a TLD. Web browsers use the list, among other things, for preventing sites that are likely operated by different entities from sharing web cookies with one another. [Let’s Encrypt](https://letsencrypt.org/#def-LE) also uses the list for rate-limit calculations: [/docs/rate-limits](https://letsencrypt.org/docs/rate-limits/). [https://publicsuffix.org/](https://publicsuffix.org/)[RSA](https://letsencrypt.org/docs/glossary/#def-RSA)A public-key algorithm used for encryption and to digitally sign certificates. [Wikipedia](https://en.wikipedia.org/wiki/RSA_%28cryptosystem%29)[Relying Party](https://letsencrypt.org/docs/glossary/#def-relying-party)The person relying on information in a certificate. For instance, someone who visits an HTTPS web site is a Relying Party.[Revocation](https://letsencrypt.org/docs/glossary/#def-revocation)A certificate is valid until its expiration date, unless the [CA](https://letsencrypt.org/#def-CA) says it has been revoked. The certificate may be revoked for various reasons such as the compromise of the private key. Browsers may check if a certificate is revoked using a [CRL](https://letsencrypt.org/#def-CRL) or newer methods like [CRLite](https://github.com/mozilla/crlite/) and [CRLSets](https://dev.chromium.org/Home/chromium-security/crlsets). Note that in many situations, [revocation doesn’t work](https://www.imperialviolet.org/2011/03/18/revocation.html). [/docs/revoking](https://letsencrypt.org/docs/revoking/)[Root Program](https://letsencrypt.org/docs/glossary/#def-root-program)The policies an organization uses to decide which certificates to include in its [certificate store](https://letsencrypt.org/#def-store), and therefore which CAs are trusted by their software.[Root certificate](https://letsencrypt.org/docs/glossary/#def-root)A [self-signed](https://letsencrypt.org/#def-self-signed) certificate controlled by a [certificate authority](https://letsencrypt.org/#def-CA), used to sign its [intermediate](https://letsencrypt.org/#def-intermediate) certificates and included in [certificate stores](https://letsencrypt.org/#def-store). [Wikipedia](https://en.wikipedia.org/wiki/Root_certificate)[SSL (Secure Sockets Layer)](https://letsencrypt.org/docs/glossary/#def-SSL)An older name for [TLS](https://letsencrypt.org/#def-TLS), still in common use.[Self-signed certificate](https://letsencrypt.org/docs/glossary/#def-self-signed)A certificate signed by its own private key, with its [Subject](https://letsencrypt.org/#def-subject) equal to its [Issuer](https://letsencrypt.org/#def-issuer). Self-signed certificates are trusted only due to prior arrangements made in the physical world, such as inclusion on a [trusted root list](https://letsencrypt.org/#def-store). [Root certificates](https://letsencrypt.org/#def-root) are self-signed. [Wikipedia](https://en.wikipedia.org/wiki/Self-signed_certificate)[Server Name Indication (SNI)](https://letsencrypt.org/docs/glossary/#def-SNI)A field that a [user agent](https://letsencrypt.org/#def-user-agent) sends to a [server](https://letsencrypt.org/#def-web-server) during a [TLS](https://letsencrypt.org/#def-TLS) handshake, specifying the domain name to connect to. This allows the server to answer with the appropriate [certificate](https://letsencrypt.org/#def-leaf) when multiple domains are hosted behind the same IP. The web server might send a different certificate, and show different content, depending on the name that the client requested by SNI. SNI is not encrypted, but an experimental replacement, ESNI, is. [Wikipedia](https://en.wikipedia.org/wiki/Server_Name_Indication)[Signed Certificate Timestamp (SCT)](https://letsencrypt.org/docs/glossary/#def-SCT)A signed, verifiable promise to publish a certificate, from a [Certificate Transparency log](https://letsencrypt.org/#def-CT-log). Browsers that enforce [CT](https://letsencrypt.org/#def-CT) check for the presence of SCTs in a site’s certificate, or in the [TLS](https://letsencrypt.org/#def-TLS) handshake, and refuse to connect to sites that don’t meet their logging requirements. This increases the likelihood that fraudulent or inaccurate certificates will be detected. [https://www.certificate-transparency.org/how-ct-works](https://www.certificate-transparency.org/how-ct-works)[Staging](https://letsencrypt.org/docs/glossary/#def-staging)[Let’s Encrypt](https://letsencrypt.org/#def-LE) provides a staging API to test certificate requests without impacting rate limits. Certificates generated by the staging environment are _not_ publicly trusted. The staging environment should be used for testing, debugging, and ACME client development purposes. [/docs/staging-environment](https://letsencrypt.org/docs/staging-environment/)[Subject Alternative Name (SAN)](https://letsencrypt.org/docs/glossary/#def-SAN)A field of a [certificate](https://letsencrypt.org/#def-leaf) that indicates for which domain(s) the certificate is valid. It replaces the usage of the [Common Name](https://letsencrypt.org/#def-CN), which is now provided for compatibility reasons only. A single certificate may contain many SANs and be valid for many different domain names. [Wikipedia](https://en.wikipedia.org/wiki/Subject_Alternative_Name)[https://letsencrypt.org/docs/rate-limits/#names-per-certificate](https://letsencrypt.org/docs/rate-limits/#names-per-certificate)[Subscriber](https://letsencrypt.org/docs/glossary/#def-subscriber)The person or organization requesting a certificate.[TLS (Transport-Layer Security)](https://letsencrypt.org/docs/glossary/#def-TLS)The protocol used by HTTPS to encrypt and authenticate web page visits.[TLSA](https://letsencrypt.org/docs/glossary/#def-TLSA)The part of [DANE](https://letsencrypt.org/#def-DANE) specifically related to validating [TLS](https://letsencrypt.org/#def-TLS) connections.[Top-Level Domain (TLD)](https://letsencrypt.org/docs/glossary/#def-TLD)Highest level in the hierarchical Domain Name System, such as country-code top-level domains (ccTLDs) like `.de` (Germany), `.cn` (China) and generic top-level domains (gTLDs) like `.com`, `.org`. [Wikipedia](https://en.wikipedia.org/wiki/Top-level_domain)[Unified Communications Certificate (UCC)](https://letsencrypt.org/docs/glossary/#def-UCC)A description of a certificate containing multiple [Subject Alternative Names (SANs)](https://letsencrypt.org/#def-SAN).[User Agent](https://letsencrypt.org/docs/glossary/#def-user-agent)Software capable of communicating with a [web server](https://letsencrypt.org/#def-web-server). Example: a [web browser](https://letsencrypt.org/#def-web-browser) or [cURL](https://en.wikipedia.org/wiki/CURL).[Web Browser](https://letsencrypt.org/docs/glossary/#def-web-browser)A [user agent](https://letsencrypt.org/#def-user-agent) used to display web pages. Examples: _Mozilla Firefox_, _Google Chrome_ or _Safari_. [Wikipedia](https://en.wikipedia.org/wiki/Web_browser)[Web server](https://letsencrypt.org/docs/glossary/#def-web-server)Software serving web pages (or, by extension, the hardware server hosting it). [Wikipedia](https://en.wikipedia.org/wiki/Web_server)[Wildcard Certificate](https://letsencrypt.org/docs/glossary/#def-wildcard)Certificates valid for subdomains one level deep. For instance, a certificate containing a [SAN](https://letsencrypt.org/#def-SAN) for `*.example.com` is valid for `blog.example.com` and `www.example.com` but **not** for `bork.bork.example.com` or `example.com`). A wildcard is indicated by an asterisk character (*) in place of a subdomain. [Let’s Encrypt](https://letsencrypt.org/#def-LE)[provides Wildcard certificates as of March 2018](https://community.letsencrypt.org/t/acme-v2-and-wildcard-certificate-support-is-live/55579).
