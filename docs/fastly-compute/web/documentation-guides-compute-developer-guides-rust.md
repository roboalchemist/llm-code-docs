[](https://www.fastly.com/documentation/guides/compute/developer-guides/rust/#rust-language-support)Rust language support
-------------------------------------------------------------------------------------------------------------------------

The Compute platform currently supports services built using Rust versions 1.78.0 through 1.91.1. Note that version 1.91.0 is not compatible with the platform and should not be used; version 1.91.1 was released to fix the incompatibility.

As part of standard platform safeguards, Fastly engineers research each new [Rust version](https://releases.rs/) to ensure optimal compatibility with Compute. To avoid build issues with your projects, don't update Rust to a newer version until Fastly releases a software development kit (SDK) that supports it.

[](https://www.fastly.com/documentation/guides/compute/developer-guides/rust/#project-layout)Project layout
-----------------------------------------------------------------------------------------------------------

If you don't yet have a working toolchain and Compute service set up, start by [getting set up](https://www.fastly.com/documentation/guides/compute).

At the end of the initialization process, the current working directory will contain a file tree resembling the following:

`├── .cargo├── .gitignore├── Cargo.lock├── Cargo.toml├── README.md├── fastly.toml├── rust-toolchain.toml└── src    └── main.rs`

The most important file to work on is `src/main.rs`, which contains the logic you'll run on incoming requests. If you initialized your project from the default starter template, the contents of this file should match the [one in the template's repository](https://github.com/fastly/compute-starter-kit-rust-default/blob/master/src/main.rs). The other files include:

* Cargo metadata: `Cargo.toml` and `Cargo.lock` describe the dependencies of your package, managed using [Cargo](https://doc.rust-lang.org/cargo/), Rust's package manager.
* Fastly metadata: The `fastly.toml` file contains metadata required by Fastly to deploy your package to a Fastly service. It is generated by the [fastly compute init](https://www.fastly.com/documentation/reference/cli/compute/init/) command. [Learn more about `fastly.toml`](https://www.fastly.com/documentation/reference/compute/fastly-toml/).
* Rust toolchain: The `rust-toolchain.toml` file specifies that the compiler should produce a WebAssembly binary and which version of Rust to use.

[](https://www.fastly.com/documentation/guides/compute/developer-guides/rust/#using-fastly-cli-1100-or-higher)Using Fastly CLI 11.0.0 or higher
-----------------------------------------------------------------------------------------------------------------------------------------------

Fastly [CLI 11.0.0](https://github.com/fastly/cli/releases/tag/v11.0.0) was released on March 25, 2025. Beginning with that release, the CLI only supports the `wasm32-wasip1` compilation target for Rust services in Compute. As a result, it requires Rust version 1.78.0 or higher and a project that is configured for that compilation target. Use the `fastly compute init` command and select a Rust starter kit to create a new project with a compatible configuration.

To upgrade existing Compute projects previously compiled with Fastly CLI versions lower than 11.0.0, these changes will need to be made to the project:

* replace `wasm32-wasi` in the `.cargo/config.toml` file with `wasm32-wasip1`
* replace `wasm32-wasi` in the `rust-toolchain.toml` file with `wasm32-wasip1`
* replace `wasm32-wasi` in the `scripts.build` attribute of the `fastly.toml` file with `wasm32-wasip1`

-----------------------------------------------------------------------------------------------------------

The most common way to start a Compute program is to define a `main()` function with the `#[fastly::main]` attribute and the following type signature:

`}`

**HINT:** Use of the `#[fastly::main]` macro creates a simple interface in which the main function of your program receives a request and returns a response. However, there are use cases where this is not necessarily a good fit, such doing more work after starting to send the response to the client. If you wish, you may define a conventional main function and use [`fastly::Request::from_client`](https://docs.rs/fastly/latest/fastly/struct.Request.html#method.from_client) and [`fastly::Response::send_to_client`](https://docs.rs/fastly/latest/fastly/http/struct.Response.html#method.send_to_client) (or [`fastly::Response::stream_to_client`](https://docs.rs/fastly/latest/fastly/http/struct.Response.html#method.stream_to_client)) instead.

The main [fastly crate](https://docs.rs/fastly) provides the core `Request`, `Body`, `ResponseExt`, and `Error` types referenced here. The program will be invoked for each request that Fastly receives for a domain attached to your service, and it must return a response that can be served to the client.

[](https://www.fastly.com/documentation/guides/compute/developer-guides/rust/#communicating-with-backend-servers-and-the-fastly-cache)Communicating with backend servers and the Fastly cache
---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------

A `fastly::Request` can be forwarded to any [backend](https://www.fastly.com/documentation/guides/integrations/non-fastly-services/developer-guide-backends) defined on your service. If you specify a backend hostname as part of completing the [fastly compute deploy](https://www.fastly.com/documentation/reference/cli/compute/deploy/) wizard, it will be named the same as the hostname or IP address, but with `.` replaced with `_` (e.g., `151_101_129_57`). It's a good idea to define backend names as constants:

`const BACKEND_NAME: &str = "my_backend_name";`

And then reference them when you want to forward a request to a backend:

`use fastly::{Error, Request, Response};const BACKEND_NAME: &str = "my_backend_name";#[fastly::main]fn main(mut req: Request) -> Result<Response, Error> {    req.set_ttl(60);    Ok(req.send(BACKEND_NAME)?)}`

Requests forwarded to a backend will typically transit the Fastly readthrough cache interface, and the response may come from cache. See [Readthrough (HTTP) cache](https://www.fastly.com/documentation/guides/concepts/cache#readthrough-cache) for more precise or explicit control over the Fastly readthrough cache.

The Rust SDK supports [dynamic backends](https://www.fastly.com/documentation/guides/integrations/non-fastly-services/developer-guide-backends#dynamic-backends) created at runtime using the [`BackendBuilder`](https://docs.rs/fastly/latest/fastly/backend/struct.BackendBuilder.html).

Responses returned from the `send` method are compatible with the return type of `main`, so a minimal implementation of a Compute service that acts as a standard HTTP caching proxy between the client and the backend is:

`use fastly::{Error, Request, Response};const BACKEND_NAME: &str = "my_backend_name";#[fastly::main]fn main(req: Request) -> Result<Response, Error> {    Ok(req.send(BACKEND_NAME)?)}`

### [](https://www.fastly.com/documentation/guides/compute/developer-guides/rust/#customizing-cache-behavior)Customizing cache behavior

The readthrough cache interface can be further customized in the way it interacts with the backend, supporting use cases such as:

* making modifications to a request only when it is forwarded to a backend
* reading and modifying response status code and headers, and adjusting cache controls
* transforming the body of the response that is stored into the cache

See [Customizing cache interaction with the backend](https://www.fastly.com/documentation/guides/concepts/cache#customizing-cache-interaction-with-the-backend) for details.

[](https://www.fastly.com/documentation/guides/compute/developer-guides/rust/#composing-requests-and-responses)Composing requests and responses
-----------------------------------------------------------------------------------------------------------------------------------------------

In addition to the request passed into `main()` and responses returned from `send()`, requests and responses can also be constructed. This is useful if you want to make an arbitrary API call that is not derived from the client request, or if you want to make a response to the client without making any backend fetch at all.

The [`Request`](https://docs.rs/fastly/latest/fastly/http/request/struct.Request.html) struct can be used as a builder, to chain methods that customize the request:

`use fastly::{Error, Request, Response};#[fastly::main]fn main(_original_req: Request) -> Result<Response, Error> {    let req = Request::post("https://example.com/api/getFlags").with_header("some-header", "someValue");    Ok(req.send("example_backend")?)}`

Similarly, [`Response`](https://docs.rs/fastly/latest/fastly/http/struct.Response.html) has several static methods that create a new response:

`use fastly::{Error, Request, Response};#[fastly::main]fn main(_req: Request) -> Result<Response, Error> {    Ok(Response::from_body("Hi from the edge"))}`

[](https://www.fastly.com/documentation/guides/compute/developer-guides/rust/#parsing-and-transforming-responses)Parsing and transforming responses
---------------------------------------------------------------------------------------------------------------------------------------------------

Requests and responses in Compute services are streams, which allows large payloads to move through your service without buffering or running out of memory. Conversely, running methods such as [`into_string`](https://docs.rs/fastly/latest/fastly/struct.Body.html#method.into_string) on a `Body` will force the stream to be consumed entirely into memory. This can be appropriate where a response is known to be small or needs to be complete to be parsable.

This example will read a backend response into memory, replace every occurrence of "cat" with "dog" in the body, and then create a new body with the transformed string:

`use fastly::{Error, Request, Response};#[fastly::main]fn main(_req: Request) -> Result<Response, Error> {    let api_req = Request::get("https://host/api/checkAuth");    let mut beresp = api_req.send("example_backend")?;    let beresp_body = beresp.take_body();    // Take care! into_string() will consume the entire body into memory, and replace()    // will futher double the memory requirement    let new_body = beresp_body.into_string().replace("cat", "dog");    beresp.set_body(new_body);    Ok(beresp)}`

However, it is often better to avoid buffering responses in this way. Peeking at the beginning of the stream can be useful for some use cases; this example identifies when a response stream begins with the WebAssembly 'magic number':

`use fastly::{Error, Request, Response};#[fastly::main]fn main(_req: Request) -> Result<Response, Error> {    let api_req = Request::get("https://host/api/checkAuth");    let mut beresp = api_req.send("example_backend")?;    let mut beresp_body = beresp.take_body();    const MAGIC: &[u8] = b"\0asm";    let prefix = beresp_body.get_prefix_mut(MAGIC.len());    if prefix.as_slice() == MAGIC {        println!("might be Wasm!");    }    Ok(Response::from_status(204))}`

Parsing responses in Rust usually benefits from well tested dependencies such as [`serde_json`](https://docs.serde.rs/serde_json/) which works well on the Compute platform. This example tries to consume the body as a [JSON value](https://docs.rs/serde_json/latest/serde_json/value/enum.Value.html), but only up to the first 4KiB. Using `take()` here avoids writing the bytes back to the body unnecessarily:

`use fastly::{Error, Request, Response};use serde_json;#[fastly::main]fn main(_req: Request) -> Result<Response, Error> {    let api_req = Request::get("https://host/api/checkAuth");    let mut beresp = api_req.send("example_backend")?;    let mut beresp_body = beresp.take_body();    let prefix = beresp_body.get_prefix_mut(4096).take();    if let Ok(_json) = serde_json::from_slice::<serde_json::Value>(&prefix) {        println!("valid json!");    }    Ok(Response::from_status(204))}`

Other crates known to be useful for transforming or composing responses include [`lolhtml`](https://docs.rs/lol_html/) and [`horrorshow`](https://docs.rs/horrorshow/).

### [](https://www.fastly.com/documentation/guides/compute/developer-guides/rust/#compression)Compression

Fastly can compress and decompress content automatically, and it is often easier to use these features than to try to perform compression or decompression within your Rust code. Learn more about [compression with Fastly](https://www.fastly.com/documentation/guides/concepts/compression).

[](https://www.fastly.com/documentation/guides/compute/developer-guides/rust/#using-edge-data)Using edge data
-------------------------------------------------------------------------------------------------------------

Fastly allows you to configure various forms of data stores to your services, both for [dynamic configuration](https://www.fastly.com/documentation/guides/concepts/dynamic-config) and for [storing data at the edge](https://www.fastly.com/documentation/guides/compute/edge-data-storage/about-edge-data-stores). The Rust SDK exposes the [`kv_store`](https://docs.rs/fastly/latest/fastly/kv_store/index.html), [`config_store`](https://docs.rs/fastly/latest/fastly/config_store/index.html) and [`secret_store`](https://docs.rs/fastly/latest/fastly/secret_store/index.html) modules to allow access to these APIs.

All edge data resources are account-level, [service-linked resources](https://www.fastly.com/documentation/reference/api/services/resource), allowing a single store to be accessed from multiple Fastly services.

[](https://www.fastly.com/documentation/guides/compute/developer-guides/rust/#logging)Logging
---------------------------------------------------------------------------------------------

The [`log-fastly`](https://docs.rs/log-fastly) crate provides a standardized interface for sending logs to [Fastly real-time logging](https://www.fastly.com/documentation/guides/integrations/non-fastly-services/developer-guide-logging/), which can be attached to many third-party logging providers. Log endpoints are referenced in your code by name:

`log_fastly::init_simple("my_endpoint_name", log::LevelFilter::Warn);log::warn!("This will be written to my_endpoint...");log::info!("...but this won't");`

If your code panics, output will be emitted to `stderr`. You can override this behavior by specifying an endpoint to use for Rust panics with `fastly::log::set_panic_endpoint`:

`fastly::log::set_panic_endpoint("my_error_endpoint").unwrap();panic!("oh no!");// => logs "panicked at 'oh no', your/file.rs:line:col" to "my_error_endpoint"`

[](https://www.fastly.com/documentation/guides/compute/developer-guides/rust/#using-dependencies)Using dependencies
-------------------------------------------------------------------------------------------------------------------

The Compute build process compiles your code to WebAssembly and uses the WebAssembly System Interface (WASI). Because of this, it supports WASI-compatible Rust crates. To get an idea of whether a crate will work with WASI, build it using `cargo build --target=wasm32-wasip1`. If it fails, it is not currently compatible. If it succeeds, still note that some crates may use conditional compilation to exclude functionality on Wasm, or include stub implementations that fail at runtime.

### [](https://www.fastly.com/documentation/guides/compute/developer-guides/rust/#unit-testing)Unit testing

Due to Fastly's custom WASI hostcalls, some setup is required in order to run Rust unit tests in a way you might be used to. By default, the environment created by `cargo test` doesn't expose the Fastly hostcalls and instead panics when there is an attempt to read a value from a type that is provided by the `fastly` crate.

In order to provide these types, you can use a special run-mode on our local development server to run each individual test and use `cargo-nextest` to handle the surrounding orchestration. To do that, you will need to install the local testing server independently of the Fastly CLI, and modify your `cargo.toml`. Full instructions can be found in the [testing server's GitHub repository](https://github.com/fastly/Viceroy#usage-as-a-test-runner). This will enable you to use `cargo nextest run` to run unit tests that use the Fastly crate.
